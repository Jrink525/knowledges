# 🌐 Agent Communication Protocols — 深度解读

> **A Technical Taxonomy of LLM Agent Communication Protocols** | arxiv:2606.19135 | 2026-06-23
>
> **作者：** Linus Sander, Habtom Kahsay Gidey

---

## 📋 概述

随着多智能体系统试图超越单智能体的局限，通信成为了承重基础设施。但当前的协议景观是**碎片化的混乱状态**——每个框架有各自的协议，互不兼容。

这篇论文做了系统性的分类工作：沿**五维分类法**对 9 个活跃的开源协议进行编码、比较、分析，揭示重复出现的架构模式与空白区域。

---

## 🏗️ 五维分类法

论文使用公认的迭代方法论（Nickerson et al.），经过 5 轮迭代（3 轮实证→概念，2 轮概念→实证），最终确认了五个分类维度：

### 维度 1：对手方（Counterparty）

协议沟通的对象：

| 类型 | 描述 | 示例 |
|------|------|------|
| **Agent-to-Agent (A2A)** | 智能体之间通信 | A2A, IoA, SOAF |
| **Agent-to-Context (A2C)** | 智能体↔工具/数据/系统 | MCP, Function Calling |
| **Agent-to-Human (A2H)** | 智能体↔人类 | A2H（少） |

**关键发现：** 所有采样协议要么是 A2A，要么是 A2C，**目前没有协议同时覆盖两者**——这指向了短期统一压力。

### 维度 2：负载（Payload）

消息格式和内容：

| 类型 | 描述 | 协议 |
|------|------|------|
| **Text-only** | 纯文本/JSON | 少数旧协议 |
| **Structured** | 预定义 schema | MCP（tools schema）|
| **Hybrid** | 混合（文本+结构化数据） | **所有 A2A 协议** |

**关键发现：** 所有 A2A 协议都使用混合负载——这是必要但不充分的模式。

### 维度 3：交互状态（Interaction State）

是否维护对话会话：

| 类型 | 描述 | 协议 |
|------|------|------|
| **Stateless** | 每个消息独立 | HTTP 风格 |
| **Session-state** | 持久化会话 | **所有 A2A 协议** |
| **Multi-modal** | 多种持久化策略 | 少数先进协议 |

**关键发现：** 所有 A2A 协议都保持会话状态持久化——没有无状态 A2A。

### 维度 4：发现机制（Discovery Mechanism）

智能体如何互相发现：

| 类型 | 描述 | 示例 |
|------|------|------|
| **Static** | 硬编码/配置文件 | 多数早期框架 |
| **Centralized** | 注册表/目录 | DSC, IoA |
| **Decentralized** | P2P 发现 | **仍然罕见** |

**关键发现：** 去中心化发现极为罕见——这是当前协议最大的空白。

### 维度 5：Schema 灵活性（Schema Flexibility）

消息结构是固定的还是协商的：

| 类型 | 描述 | 协议 |
|------|------|------|
| **Fixed** | 预定义不可变 | 传统协议 |
| **Multiple predefined** | 支持多个 schema，开发时固定 | MCP, A2A |
| **Runtime negotiation** | 运行时协商 schema | **SOAF, IoA** |

**关键发现：** 多数协议支持多个预定义 schema；两个协议支持运行时 schema 协商——指向灵活性趋势。

## 🧩 重复出现的架构模式

**三个压倒性模式：**

1. **所有 A2A 协议 = 混合负载 + 会话持久化**
   - 这是 A2A 通信的最低要求
   - 文本+结构化数据的混合负载支持自然语言辩论 + 工具调用

2. **A2A ≠ A2C 尚未统一**
   - MCP 只做 A2C（工具/数据访问）
   - A2A 只做 A2A（智能体间通信）
   - 没有协议能同时做好两者

3. **去中心化发现极度缺乏**
   - 实践中几乎都依赖中央注册表或静态配置
   - 去中心化 P2P 发现仍是一个开放问题

## 🔮 演化预测

### 短期（1-2 年）：统一压力

论文预测协议将在短期内趋同于**统一 A2A + A2C 的协议**。原因：
- 开发者不想维护两套协议栈
- 一个框架中既有 agent-to-agent 通信又有 tool/data 访问是最自然的架构
- MCP → A2C + A2A 扩展是可能性之一

### 长期（3-5 年）：联邦化分层协议栈

论文认为**没有单一协议能同时最大化通用性、效率和可移植性**。长期趋势是：

```
┌──────────────────────────────┐
│    Layer N: Application      │ ← 特定场景（编码、科研、客服）
├──────────────────────────────┤
│    Layer 3: Federation       │ ← 互操作/网关/翻译器
├──────────────────────────────┤
│    Layer 2: Transport/Session│ ← 消息路由、会话管理
├──────────────────────────────┤
│    Layer 1: Wire Format      │ ← 序列化、传输
└──────────────────────────────┘
```

类似互联网的协议栈（TCP/IP → HTTP → ...），智能体通信也将形成分层结构。

## 🎯 核心洞见

### 1. "协议是基础设施"

论文的基调很明确：通信协议不是锦上添花，而是**多智能体系统的承重墙**。没有标准化的通信，每个 MAS 是一个孤岛。

### 2. 分类法即是地图

这篇论文最大的价值是**画了一张地图**——在混乱中建立了坐标系。任何团队在选择或设计协议时，都可以用这五个维度快速定位、权衡。

### 3. MCP 是事实上的 A2C 标准

虽然论文保持中立，但从分析看 **MCP（Model Context Protocol）** 已成为 A2C 的主导协议。A2A 领域尚未有同样统治力的标准。

### 4. 运行时 Schema 协商引人注目

SOAF 和 IoA 支持运行时协商 schema——这对应的是动态环境中的灵活适应，可能是未来协议的关键特性。

### 5. 去中心化发现是最大瓶颈

如果要实现真正的分布式 MAS（无中央控制点），去中心化发现是必须破解的难题。目前几乎没有好的解决方案。

## ⚠️ 局限

1. **只覆盖 9 个开源协议**：商业协议和更早的协议未被纳入
2. **纯分类学，不评测实现质量**：没有性能基准测试
3. **快照分析**：协议领域变化极快，分析难免过时
4. **未涉足安全/隐私深度分析**：仅在开放问题中提及
5. **定性而非定量**：缺乏系统性比较的实验数据

---

## 💡 启示

1. **选协议前先画坐标**：用五维分类法确定需求定位，再选协议
2. **短期押注 MCP**：如果只需要 A2C，MCP 是最成熟的选择
3. **A2A 领域盯紧 Google A2A 和 IoA**：最可能成为 A2A 标准的候选
4. **不要重复造协议轮子**：除非有非常特殊的约束，不应自研协议
5. **联邦化协议栈是未来**：一个协议不可能通吃所有场景，想好多层设计
