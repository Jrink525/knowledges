# The End of Code Review: Coding Agents Supersede Human Inspection — 深度解读

> **论文：** arXiv:2606.13175 [cs.SE]
> **作者：** Martin Monperrus (KTH Royal Institute of Technology)
> **日期：** 2026 年 6 月 11 日
>
> **一句话总结：** 本文论证编程 Agent（LLM-based autonomous systems）的能力已跨过临界点，**传统的强制人工代码审查（mandatory human code review）已成为冗余环节**，应当被 Agent 驱动的自动化验证取代。

---

## 问题背景（Problem Statement）

**传统认知：** 代码审查是现代软件工程中不可动摇的质量门禁。

自从 Fagan 在 1976 年形式化代码审查以来，将近半个世纪里，"任何代码合入前必须经过另一个人审查"是软件团队的金科玉律。它完成四个核心目标：缺陷检测、风格/规范执行、知识传递、团队共知。**没有严肃的团队会省略它。**

**但代价被严重低估：**
- 大公司开发者把 **10–15% 的工作时间**花在读和评论别人的代码上 [Sadowski et al., Google 2018]
- PR 审查延迟通常超过 **24 小时**，可达数天，成为持续交付的**结构性拖累**
- 审查产生社交摩擦：语气升级、资深偏见、首次贡献者因负面反馈放弃项目

**转折点：** Coding agents（Claude Code、Codex、Devin、GitHub Copilot）现在能自主读/写/测试/修复代码。SWE-bench 上顶级 Agent 已解决 **>80%** 的真实 GitHub issue。这个能力曲线在过去两年的提升——从不到 2% 到超过 70%——在自动软件工程史上前所未有。

这是一篇**论证（argument）论文**，不是实证研究。Monperrus 明确声明：

> "We do not present a new empirical study; instead, we synthesise existing capability evidence, and enumerate the implications."

    │

    ├→ Claim 3: 成本收益已经反转

## 核心论证（Arguments）

### Claim 1: 审查的每个目标都可以由 Agent 以更低的成本、更高的吞吐量来满足

| 目标 | 人类审查的现实 | Agent 的能力 |
|------|---------------|-------------|
| **缺陷检测** | 表面问题有效，深层逻辑缺陷不稳定 [Czerwonka et al., 2015] | 穷尽式数据流推理 + 全测试套件交叉引用 + 百万级开源模式学习 |
| **风格/规范** | Linter/formatter 已自动化，人类只是重复 | Agent 扩展到**语义风格**：命名一致性、惯用 API 用法、文档规范 |
| **安全审查** | 人类注意力最过载的领域 | 系统性枚举漏洞类别 [Pearce et al., 2022]；AI 安全扫描已超越手动审查基线 |
| **知识传递** | 随机、随意的评论 | Agent 在合并时**主动生成按需解释、架构总结、更新文档** |

**结构性优势：**
- 人类读 diff；Agent 同时持有完整文件 + 测试套件 + git 历史 + 项目文档
- 人类发现问题的极限是写一条评论；Agent 发现问题 → 生成修复 → 应用 → 跑测试 → 闭环
- 人类有日历、时区和有限注意力；Agent 持续运行，凌晨 3 点的检查标准和周一早上完全一致

### Claim 2: "Agent 写代码 + 人类强制审查"是死胡同

这是本文最尖锐的论点。大多数组织对 AI 的第一反应是"安全方案"：Agent 写，人审。Monperrus 说这在**两个独立维度上失效**：

**① 审查提供不了真正的保证**

当 Agent 生成代码时，传统审查的前提假设垮塌了：

> "在人类写代码、人类审查的模型下，审查者提供了独立检查。
> 当 Agent 生成代码，这个假设崩溃了。"

LLM 能产生成百上千行看起来很合理、内部一致的代码，但包含微妙的语义错误——这些错误不跑完整测试套件或不做穷尽数据流分析是看不见的。人类在屏幕上读 diff，**处于极差的位置**去捕捉 AI 生成代码中的错误。

现实中，对 Agent 生成代码的审查变成了**橡皮图章**：代码看起来正确 → 测试通过 → 真正审查的认知成本太高 → 批准。

> "审查本应提供的保证是虚幻的。"

**② 审查无法扩展**

AI 工具增加了开发者的吞吐量。一个被 Agent 辅助的开发者每天产生更多的 commit、PR、代码行。**人类审查能力不会缩放**——结果就是审查队列成为交付流水线的绑定约束。

> "Agent 写得多快，队列就涨得多快，审查退化为在时间压力下完成的形式流程。"

保留强制人工审查不会保住质量；它同时丢弃了 AI 的生产力收益和真正审查的质量收益。

**逻辑上一致的步骤：** 让**另一个独立的 Agent** 审查 Agent 生成的代码。人类仅在 Agent 标记不确定性或变更跨越明确风险阈值时介入。

### Claim 3: 成本收益计算已经翻转

| 维度 | 人类审查 | Agent 审查 |
|------|---------|-----------|
| **时间成本** | 10–15% 工作时间 | 瞬时 |
| **延迟** | 24h+ turnaround | 每个 commit 即时 |
| **一致性** | 因疲劳/时区/偏见而异 | 确定性可重复 |
| **审计性** | 非正式评论线程 | 结构化报告，可版本化查询 |
| **社交成本** | 语气升级、偏见、新人不友好 | 无社交摩擦 |

核心逻辑：随着 Agent 的扫描能力持续提升，**能逃脱 Agent 审查却被人类抓住的缺陷集合在缩小**，而人类审查的时间/延迟/社交成本不变。**交叉点已经到了。**

## 五组反方观点的回应（Counter-arguments）

### 1. 幻觉与漏报（Hallucination & false negatives）

**反方：** LLM 会漏掉训练分布之外的缺陷，还可能静默批准。

**回应：**
- **主要缓解：集成审查（ensemble review）**——运行多个独立 Agent（不同模型、不同 prompt 策略），达成共识后才签出
- **次要缓解：校准不确定性报告**——审查 Agent 应学会弃权（"I don't know"）并输出置信度

### 2. Agent 生成代码的安全漏洞

**反方：** Copilot 本身就会以可观概率引入 CWE 漏洞 [Pearce et al. 2022]，让同一个模型家族既写代码又审查代码，生成和审查的盲点可能**相关**。

**回应：** 对安全签出使用**网络安全专用的前沿模型**，最新基准显示前沿模型在漏洞识别任务上已超越传统静态分析工具。

### 3. 对抗输入与 Prompt 注入

**反方：** 敌对方可以通过 PR 提交精心构造的代码，在被审查 Agent 读取时通过内嵌的自然语言指令操纵其推理。

**回应：** 这确实是人工审查不存在的新攻击面。部署 Agent 审查的团队必须将 **Agent 审查者的 prompt 注入**视为一等威胁模型。

### 4. 架构连贯性需要人类判断

**反方：** 人类审查者能评估变更是否与系统长期设计一致；Agent 无法以足够保真度持有系统架构的心智模型。

**回应：** 这个论点**错误地将 PR 级 diff 审查和架构审查混为一谈**。架构连贯性最应通过设计文档、架构决策记录（ADR）和专门的架构审查来保证，而不是通过每 commit 的 diff 检查。实证研究一再表明，人类 PR 审查的重心是**表面缺陷和风格**，而非战略性架构有效性。

### 5. 伦理问责需要人类判断

**反方：** 代码变更可能涉及隐私、算法公平性和环境影响——Agent 优化的是技术质量指标，无法可靠检测这些。

**回应：** Agent 已经能标记安全敏感模式（PII 记录、不安全随机数、过度权限 API 调用），且合规性覆盖在增长。更根本地：**强制 PR 审查不是伦理审查的正确场所**——这个责任属于需求工程和上线后监控。

## 启示（Implications）

### 对工程实践

| 领域 | 变化 |
|------|------|
| **合并工作流** | 从"人类批准复选框"变为"结构化 Agent 签出"——测试覆盖率阈值、安全扫描、风格合规、推理痕迹。仅高风险变更才需要人工 |
| **团队角色** | 审查员角色消失。开发者从逐行审阅变为 **specifier 和 orchestrator**——精确描述需求、在高层评估 Agent 产出、在不确定性时介入 |
| **知识传递** | Agent 按需生成解释和架构总结，比匆忙注释更可靠。风险：非正式的双向对话减少——需用结对编程和指导计划补充 |
| **开源生态** | 维护者带宽是开源的结构性瓶颈。Agent 审查直接压缩反馈循环，降低首次贡献者门槛 |

### 对工具链

| 领域 | 变化 |
|------|------|
| **CI/CD 集成** | 审查从异步社交活动变为一等工程检查——像编译一样在每个 commit 上自动运行，产生 JSON/SARIF 结构化输出 |
| **IDE 集成** | 开发者在写代码时 Agent 就审查嵌入的代码——反馈循环从小时/天压缩到秒。交互模式从评论列表变为自然语言对话 |
| **版本控制平台** | 需要扩展身份/权限模型支持 Agent 角色。Agent 审查者需要加密签名身份、审计日志权限、批准/请求变更/阻止合并的能力 |
| **审查历史** | 从退化的非正式评论线程变为**机器可读的工程制品**——缺陷趋势仪表板、安全态势追踪、代码库健康评分 |

## 论文定位（Positioning）

这是一篇**立场论文（position paper）**，其力量不在于新实验数据，而在于：
1. 对现有证据的**系统综合**
2. 对当前 AI 热潮下"最安全选择"（Agent 写 + 人审）的**尖锐解构**
3. 对反方观点的**诚实面对**——尤其 5 组 counter-argument 的讨论

Monperrus 的身份（自动程序修复的先驱者）让这篇论文有独特份量——他是该领域最有资格说"审查可以被取代"的人之一。

**关键局限（论文自身承认）：**
- 没有新实证数据
- 没有讨论不同规模/类型的团队差异性
- 没有讨论 Agent 审查对非代码制品的适用性（设计文档、配置）

## 读后反思（Aftertaste）

### Keep（值得保留的洞察）

1. **"Agent 写 + 人审 = 橡皮图章 + 瓶颈"**是最锋利的论点。当审查者在认知上无法真正审查 AI 生成代码时，整个质量保证机制失效于系统层面
2. **Agent 的结构性优势被清晰阐明**：同时持有完整文件+测试+历史+文档；发现问题后可自动修复+验证+闭环；无疲劳/时区/偏见
3. **没有把技术潮流推到极端**——承认高风险路径需要人工介入、架构审查应该独立进行、伦理问责不属 PR 层面
4. **Prompt 注入作为新攻击面的识别**是负责任的处理——这个威胁存在于 Agent 审查的每一天

### Problem（值得质疑的假设）

1. **引用的"审查不找 bug"证据年代偏老**——Czerwonka et al. 的结论来自 2015 年 Microsoft 的研究。近十年的工程实践已有变化
2. **"每个审查目标都可以被 Agent 满足"——知识传递目标最薄弱**。Agent 生成的"按需解释"和人与人之间非正式的知识传递可能有本质差异
3. **SWE-bench 结果的推广性问题**——>80% 的解决率是否代表了真实软件工程的复杂性？Monperrus 自己也提到 benchmark 不能完全反映现实
4. **"成本收益已翻转"缺乏量化框架**——具体什么时候 crossover？100 人的团队是多少？1000 人的呢？没有给出可操作的阈值
5. **忽略了法律/合规/审计的刚性约束**——在许多受监管行业（金融、医疗、航空），人工审查是合规要求，不是技术选择

### Try（如果基于此文做研究或产品）

1. **测量 crossover 点**：设计实验系统性地测量人类审查在 AI 辅助工作流中的边际缺陷检测价值。什么时候低于 1%？低于 0.1%？
2. **构建 ensemble review 系统**：多个不同 Agent 审查同一变更，用 consensus + uncertainty 决定是否通过。测量 vs 人类 baselines
3. **Agent 审查者 prompt injection 防护**：设计检测和防御机制，特别是针对"嵌入式代码操纵"场景
4. **研究"从审查员到 specifier"的角色转变**：开发者花多少时间写 spec？Agent 理解 spec 的 fidelity 如何？最佳实践是什么？
