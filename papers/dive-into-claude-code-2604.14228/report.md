## 1. 摘要与核心论点

### 1.1 一句话总结

这篇论文通过对 Claude Code 公开 TypeScript 源码的深度逆向工程，揭示了**一个生产级 AI 编码智能体的完整架构设计空间**，并首次将其与开源多通道助手 **OpenClaw** 进行六维度架构对比，最终提炼出未来智能体系统的六个开放方向。

### 1.2 核心发现

| 维度 | 发现 |
|------|------|
| **代码构成** | 约 **1.6%** 为决策逻辑，**98.4%** 为操作基础设施（权限门控、工具路由、上下文管理、恢复逻辑） |
| **核心循环** | 简单 while-true 循环（queryLoop），用 async generator 实现，非复杂状态图 |
| **安全模型** | 7 层深度防御 + 7 种权限模式 + ML 分类器 |
| **上下文管理** | 5 层渐进式压缩管线，从预算削减到 Auto-compact 摘要 |
| **扩展机制** | 4 种不同上下文成本的机制：MCP、插件、技能、钩子 |
| **架构哲学** | 模型做决策，基础设施做护航——"操作系统"范式而非"状态图"范式 |

### 1.3 论文独特性

- 这是**第一篇**对 Claude Code 进行源码级架构分析的学术论文
- 采用**设计空间分析法**而非传统系统描述：识别重复出现的架构问题，展示每个问题下 Claude Code 的选择和替代方案
- 以 **OpenClaw 作为校准基准**——两个完全不同的部署场景下对相同设计问题的不同回答

## 2. 研究框架与方法论

### 2.1 证据三级体系

本论文在论证质量上建立了清晰的层级制度：

| 层级 | 标识 | 来源 | 可信度 |
|------|------|------|--------|
| **A (产品文档)** | ✅ | 官方 Anthropic 文档、工程出版物 | 反映产品意图，但不一定代表实际实现 |
| **B (代码验证)** | 🔬 | 提取的 TypeScript 代码库 (v2.1.88, ~1,884 文件, ~512K 行) | 最强证据，直接引用具体文件和函数 |
| **C (重构推理)** | 🤔 | 社区分析、OpenClaw 结构对比、代码模式推断 | 用模糊语言陈述，明确标注 |

### 2.2 设计空间分析流程

1. **识别设计问题**：检查每个子系统，找出存在替代方案的选择点
2. **追踪 Claude Code 的回答**：通过具体源码文件和函数实现
3. **提取价值框架**：从官方文档和创建者声明中提取五值框架
4. **映射到架构决策**：通过 13 个设计原则将价值观与架构选择关联
5. **引入 OpenClaw 对比**：展示相同问题在不同部署场景下的不同回答

### 2.3 方法论局限

- **静态快照**：仅反映 v2.1.88 版本，功能标志创建了构建时变异性
- **逆向工程认识论**：源码揭示实现结构但无法确认设计意图、运行时行为或未发布功能
- **单系统分析**：不能代表所有编码智能体的设计空间
- **OpenClaw 快照**：反映特定开发状态，可能不代表当前能力

## 3. 五大核心价值与十三设计原则

### 3.1 五大价值观

论文识别出塑造 Claude Code 架构的**五种人类价值观/理念**：

#### 🏛️ 价值观 1：人类决策权威性 (Human Decision Authority)
人类保留对系统行为的最终决策权。通过等级化主体层级（Anthropic → 操作者 → 用户）形式化谁对什么拥有权威。

**关键数据**: 用户批准 93% 的权限提示 → 架构响应不是增加更多警告，而是通过沙箱化和自动模式分类器重构问题边界。

#### 🛡️ 价值观 2：安全、安保与隐私 (Safety, Security, and Privacy)
即使在人类疏忽时，系统也必须保护人类及其代码、数据和基础设施。与"决策权威性"不同——权威关注人类的**选择权**，安全关注系统的**保护义务**。

**风险类别**: 过度积极行为、诚实错误、提示注入、模型失调。

#### ⚙️ 价值观 3：可靠执行 (Reliable Execution)
智能体忠实执行人类的真实意图，长时间保持连贯性，并在声明成功前支持验证其工作。

**跨越多个跨度**: 单轮正确性 + 长时程可靠性（跨越上下文窗口边界、会话恢复、多智能体委托）。

#### 🚀 价值观 4：能力放大 (Capability Amplification)
系统显著提升人类单位努力和成本下的产出。内部调查显示约 **27% 的任务**是在没有该工具的情况下不会尝试的工作。

**关键架构决策**: 投资确定性基础设施而非决策脚手架——越强的模型越需要丰富的操作环境，而非约束性的框架。

#### 🔄 价值观 5：上下文适应性 (Contextual Adaptability)
系统适应特定用户上下文，并随时间改善关系。

**实证数据**: 自动批准率从少于 50 次会话时的约 20% 增加到 750 次会话时的超过 40%，表明人-智能体关系存在**信任轨迹**而非固定信任状态。

### 3.2 长期能力保持——评价性透镜

论文引入第六个关注点作为**评价性透镜**而非设计价值：

> **核心问题**: 短期能力放大是否以长期人类理解、代码库连贯性和开发者管道为代价？

**实证依据**:
- 16 名经验开发者 RCT（246 个任务）→ AI 工具使开发者慢 19%（尽管感知提升 20%）
- Cursor 在 807 个仓库上的因果分析 → 代码复杂度增加 40.7%
- 54 名参与者的 EEG 研究 → LLM 用户显示减弱后的神经连接，在 AI 移除后仍持续
- AI 辅助条件下开发者的理解测试分数低 17%
- 2023–2024 年入门级技术招聘下降 25%

> ⚠️ 论文明确指出：这些证据并非特指 Claude Code 架构，而是适用于**任何具有有限上下文和工具使用循环的智能体系统**。

### 3.3 十三设计原则

| # | 原则 | 服务价值 | 设计问题 |
|---|------|---------|---------|
| 1 | **默认拒绝 + 人工升级** (Deny-first with human escalation) | 权威性, 安全 | 未识别的行为应允许、阻止还是升级？ |
| 2 | **渐进信任谱** (Graduated trust spectrum) | 权威性, 适应性 | 固定权限级别还是随时间演变？ |
| 3 | **分层深度防御** (Defense in depth) | 安全, 权威性, 可靠性 | 单一安全边界还是多层重叠？ |
| 4 | **外部化可编程策略** (Externalized programmable policy) | 安全, 权威性, 适应性 | 硬编码策略还是外部化配置？ |
| 5 | **上下文是稀缺资源** (Context as scarce resource) | 可靠性, 能力 | 单次截断还是渐进式管线？ |
| 6 | **仅追加持久状态** (Append-only durable state) | 可靠性, 权威性 | 可变状态、检查点快照还是仅追加日志？ |
| 7 | **最少脚手架，最多操作装备** (Minimal scaffolding, maximal harness) | 能力, 可靠性 | 投资脚手架侧推理还是操作基础设施？ |
| 8 | **价值优于规则** (Values over rules) | 能力, 权威性 | 刚性决策程序还是上下文判断？ |
| 9 | **可组合多机制可扩展性** (Composable multi-mechanism extensibility) | 能力, 适应性 | 统一扩展API还是分层机制？ |
| 10 | **可逆性加权风险评估** (Reversibility-weighted risk assessment) | 能力, 安全 | 所有操作同等监管还是可逆操作更轻？ |
| 11 | **透明文件式配置与记忆** (Transparent file-based config & memory) | 适应性, 权威性 | 不透明数据库还是用户可见的可版本控制文件？ |
| 12 | **隔离的子智能体边界** (Isolated subagent boundaries) | 可靠性, 安全, 能力 | 子智能体共享父级上下文还是隔离操作？ |
| 13 | **优雅恢复与韧性** (Graceful recovery & resilience) | 可靠性, 能力 | 错误时硬失败还是静默恢复？ |

### 3.4 与三大替代设计家族的对比

| 设计家族 | 代表系统 | 核心特征 | Claude Code 对比 |
|----------|---------|---------|-----------------|
| **基于规则编排** | LangGraph | 显式状态图、类型化边 | 放弃决策脚手架，选择最小化操作装备 |
| **容器隔离执行** | SWE-Agent, OpenHands | Docker 隔离 | 分层策略执行而非单纯容器隔离 |
| **版本控制即安全** | Aider | Git 回滚 | 默认拒绝 + 深度防御而非事后回滚 |

## 4. 架构总览

### 4.1 高层的七组件结构

论文从源码中解构出 Claude Code 的七个核心组件：

```
┌──────────────────────────────────────────────────┐
│                 Surface Layer                      │
│  (CLI, IDE, Headless, 子智能体, Coordinator)       │
├──────────────────────────────────────────────────┤
│                Backend Layer                       │
│  (QueryEngine, 会话存储, 文件系统, Bash, MCP)       │
├──────────────────────────────────────────────────┤
│                 Core Layer                         │
│  (queryLoop - async generator + while-true)        │
├──────────────────────────────────────────────────┤
│            Safety/Action Layer                     │
│  (权限系统, 钩子, 可扩展性, 工具, 沙箱, 子智能体)    │
├──────────────────────────────────────────────────┤
│          State Layer                               │
│  (上下文组装, 运行时状态, 持久化, 记忆, Sidechain)  │
└──────────────────────────────────────────────────┘
```

### 4.2 设计问题集

论文识别出 Claude Code 必须回答的 7 个重复设计问题：

| # | 设计问题 | Claude Code 的回答 |
|---|---------|-------------------|
| 1 | **推理在哪里？** | 在 LLM 调用中（非 QueryEngine），QueryEngine 是最小化调度器 |
| 2 | **迭代循环结构？** | 简单 while-true + async generator，非状态图 |
| 3 | **默认安全姿态？** | 默认拒绝 + ML 分类器 + 渐进信任谱 |
| 4 | **扩展表面如何划分？** | 4 种不同上下文成本的机制：钩子→技能→插件→MCP |
| 5 | **上下文如何构建和管理？** | 5 层渐进压缩管线 + CLAUDE.md 懒加载 |
| 6 | **子智能体如何委派？** | 隔离上下文窗口 + limited 工具集 + worktree 隔离 |
| 7 | **会话如何持久化？** | 仅追加 JSONL + 读时链修补 |

### 4.3 QueryEngine 澄清

论文特别澄清了一个常见误解：**QueryEngine 不是核心推理引擎**。

- QueryEngine 是一个**轻量调度器**，管理子智能体委托和交互式/后台查询路由
- 实际的推理循环是 `queryLoop`——一个 async generator 函数
- 交互式 CLI 直接调用 `query`，绕过 QueryEngine 完全

## 5. Turn 执行：Agentic Query Loop

### 5.1 queryLoop 管道

```
用户输入 → [Pre-model Context Shapers] → [Model Call] 
  → [Tool Dispatch] → [Permission System] → [Tool Execution]
    → [Result Processing] → [Loop or Stop]
```

#### Pre-model Context Shapers（预模型上下文整形器）

在每次模型调用前，应用 5 个上下文整形器：

1. **工具池组装** (`assembleToolPool`): 根据环境、功能标志和空值检查条件合并内置工具和 MCP 工具
2. **工具模式过滤**: 如果启用 ToolSearch，初始上下文只包含工具名称，完整模式按需加载
3. **CLAUDE.md 接入**: 将层级指令文件内容注入为用户上下文（非系统提示）
4. **记忆检索**: LLM 基于记忆文件头部扫描检索最多 5 个相关文件
5. **压缩边界**: 在上下文窗口边界附近创建压缩摘要

### 5.2 恢复机制

```
Error Types:
├── ToolExecutionError (可恢复 → 继续循环)
├── Transient API Error (自动重试 3 次)
├── Permission Denied (提供拒绝原因 → 模型重试更安全方案)
└── Fatal Error (优雅关机)
```

### 5.3 停止条件

循环继续直到满足以下任一条件：
- **模型自愿停止** (stop_reason = end_turn)
- **达到最大 turn 数** (可配置)
- **工具通过 stop 参数请求停止**
- **用户中断**

## 6. 权限与安全架构

### 6.1 七层深度防御

Claude Code 通过七个独立的安全层级实现"默认安全"原则：

| 层级 | 位置/文件 | 功能 | 绕过条件 |
|------|----------|------|---------|
| **① 工具预过滤** | `tools.ts` | 从模型视野中移除全局拒绝的工具 | N/A |
| **② 默认拒绝规则评估** | `permissions.ts` | 拒绝规则始终优先于允许规则 | 显式允许规则 |
| **③ 权限模式约束** | `types/permissions.ts` | 当前模式决定未匹配请求的基线处理 | 模式切换 |
| **④ 自动模式分类器** | `yoloClassifier.ts` | ML 分类器评估工具安全性 | 功能标志关闭 |
| **⑤ Shell 沙箱** | `shouldUseSandbox.ts` | 批准的 shell 命令在沙箱中执行 | 沙箱配置 |
| **⑥ 恢复时不恢复权限** | `conversationRecovery.ts` | 会话级权限不跨会话边界持久化 | N/A |
| **⑦ 钩子拦截** | `types/hooks.ts` | PreToolUse 钩子可修改权限决策 | 无钩子注册 |

### 6.2 七种权限模式

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `plan` | 用户批准所有计划 | 最高安全性 |
| `default` | 默认交互审批 | 默认 |
| `acceptEdits` | 自动批准编辑操作 | 编辑密集型 |
| `auto` | ML 分类器决策 | 平衡模式 |
| `dontAsk` | 不询问，自动允许 | 高信任度 |
| `bypassPermissions` | 绕过大部分但保留安全检查 | 风险偏好 |
| `bubble` | 内部模式 | 沙箱内操作 |

### 6.3 自动模式分类器生命周期

```
启用条件: TRANSCRIPT_CLASSIFIER 功能标志
评估内容: 建议的工具调用 × 对话记录 × 权限模板
输出: allow / deny / request manual approval

与钩子的交互:
  PreToolUse → 可返回 deny/ask + 修改参数
  PermissionDenied → 可提供 retry 指导
  PermissionRequest → 可提前 resolve（在 Coordinator 路径中）
  PostToolUse → 可注入附加上下文
```

### 6.4 安全假设突破

论文发现了一个关键结构性问题：**深度防御的独立性假设可能被违反**。

安全层共享共同约束——性能和 token 成本。当性能压力推动削减成本时，多个层可能同时退化：

- 自动模式分类器：额外 LLM 调用，直接 token 成本
- `bashSecurity.ts`：顺序 AST 检查，有解析延迟
- 默认拒绝规则：基于命令结构评估

**实证脆弱性**: 安全研究人员发现超过 50 个子命令的命令会回退到单一通用批准提示（因逐子命令解析导致 UI 冻结），证明深度防御在独立性假设被违反时失效。

### 6.5 预信任初始化漏洞

两种独立验证的漏洞共享根因：**预信任初始化顺序**——在项目初始化期间执行的代码（钩子、MCP 服务器连接、设置文件解析）运行在交互式信任对话框之前。

- CVE-2025-59536 (CVSS 8.7)
- CVE-2026-21852 (CVSS 5.3)
- 均由 Check Point Research 发现，在披露后数周内修补

## 7. 可扩展性设计

### 7.1 四种机制 + 上下文成本梯度

Claude Code 不提供统一扩展 API，而是提供四种不同上下文成本的机制：

```
上下文成本:  低 ─────────────────────────────────→ 高

钩子 (Hooks)    技能 (Skills)    插件 (Plugins)    MCP 服务器
  0 tokens       低 tokens        中 tokens         高 tokens
```

### 7.2 钩子系统

- 27 个钩子事件定义在 `coreTypes.ts` 中
- 无上下文成本——运行时注册，不进入上下文窗口
- 5 个直接参与权限流：
  - **PreToolUse**: 权限决策 + 参数修改
  - **PostToolUse**: 附加上下文注入 + MCP 输出修改
  - **PostToolUseFailure**: 错误特定指导
  - **PermissionDenied**: 重试指导
  - **PermissionRequest**: 提前解析决策

### 7.3 MCP (Model Context Protocol)

- 最高上下文成本：工具名称 + 工具模式 + 服务器指令都占用上下文
- 工具匹配：通过 `mcp__server__tool` 全限定名
- 安全上下文：服务器级规则匹配该服务器的所有工具

### 7.4 工具池组装

```
Tool Types:
├── Always included: AgentTool, BashTool, FileReadTool, FileEditTool, 
│                    FileWriteTool, SkillTool, WebFetchTool, WebSearchTool
├── Environment: GlobTool/GrepTool, ConfigTool (ant-only), PowerShellTool
├── Feature flag: TaskCreate/Get/Update/List, EnterWorktreeTool, TeamTools
└── Null-checked: SuggestBackgroundPRTool, WebBrowserTool, RemoteTriggerTool, 
                  MonitorTool, SleepTool
```

## 8. 上下文构建与记忆管理

### 8.1 上下文窗口组装

```
上下文源（按组装顺序）:
1. 系统提示（含输出风格 + --append-system-prompt）
2. 环境信息（git 状态等）
3. CLAUDE.md 层级
4. 路径作用域规则（懒加载）
5. 自动记忆（异步预取）
6. 工具元数据：技能描述、MCP 名称、ToolSearch 定义
7. 对话历史（受压缩约束）
8. 工具结果：文件读取、命令输出、子智能体摘要
9. 压缩摘要（替换较旧的历史片段）
```

### 8.2 CLAUDE.md 四级层级

| 级别 | 位置 | 范围 | 加载时机 |
|------|------|------|---------|
| **Managed** | `/etc/claude-code/CLAUDE.md` | OS 级策略 | 启动时 |
| **User** | `~/.claude/CLAUDE.md` | 用户全局私有指令 | 启动时 |
| **Project** | `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md` | 代码库检入 | 启动时/懒加载 |
| **Local** | `CLAUDE.local.md` | 私有项目特定 | 启动时 |

**关键架构决策**: CLAUDE.md 内容作为 **用户上下文**（用户消息）而非系统提示内容传递。这意味着：
- 模型对这些指令的遵从是 **概率性的** 而非有保证的
- 权限规则（默认拒绝）提供 **确定性执行层**
- 形成了清晰的分离：**指导（CLAUDE.md，概率性）** vs **强制（权限规则，确定性）**

### 8.3 五层压缩管线

```
Layer 1: Budget Reduction (始终激活)
  → 单工具结果大小限制
Layer 2: Snip (HISTORY_SNIP 标志)
  → 轻量较旧历史裁剪
Layer 3: Microcompact (CACHED_MICROCOMPACT)
  → 细粒度缓存感知压缩
Layer 4: Context Collapse (CONTEXT_COLLAPSE)
  → 读时虚拟投影（不修改物理记录）
Layer 5: Auto-compact (默认启用)
  → 完整模型生成摘要
```

**设计哲学**: 惰性降解——先应用破坏最小的压缩，仅在更便宜策略不足时升级。

**成本**: 五个交互层 + 功能标志 = 用户难以预测的行为。简单替代方案（单次截断）易理解但牺牲信息。

### 8.4 记忆系统设计

| 特性 | Claude Code | 对比：数据库方案 |
|------|-------------|----------------|
| 透明度 | 纯文本 Markdown，用户可读可编辑 | 不透明的嵌入索引 |
| 检索 | LLM 扫描文件头，选择最多 5 个文件 | 向量相似度检索 |
| 粒度 | 文件级 | 条目级（更精确） |
| 版本控制 | Git 友好 | 需额外基础设施 |
| 编辑能力 | 可直接编辑 | 不透明 |

**透明文件式记忆** 的设计权衡：
- **代价**: 更少的检索灵活性
- **收益**: 用户完全可见、可审计、可版本控制

## 9. 子智能体委派机制

### 9.1 四种子智能体类型

| 类型 | 用途 | 工具集 |
|------|------|--------|
| **Explore** | 代码探索和上下文收集 | 受限（主要是读取工具） |
| **Plan** | 多步任务分解 | 受限 |
| **General-purpose** | 通用委派任务 | 完整（但可配置） |
| **Custom** | 通过钩子创建的自定义类型 | 可配置 |

### 9.2 Worktree 隔离

- **文件系统级别**: 每个子智能体使用 `git worktree` 创建隔离副本
- **上下文隔离**: 子智能体在自己的受限上下文窗口中操作
- **摘要返回**: 子智能体**仅返回摘要文本**给父级，而非完整对话历史
- **限制**: 子智能体隔离限制了跨智能体一致性（五大价值张力之一）

### 9.3 Sidechain 持久化

- 每个子智能体有独立 `.jsonl` + `.meta.json` 文件
- 层次结构保持可审计：父级可查看任何子智能体的完整记录

## 10. 会话持久化与恢复

### 10.1 三条持久化通道

| 通道 | 位置 | 内容 | 范围 |
|------|------|------|------|
| **会话记录** | `{projectDir}/{sessionId}.jsonl` | 完整对话（含压缩标记） | 项目级 |
| **全局提示历史** | 配置家目录 `history.jsonl` | 仅用户提示 | 全局 |
| **子智能体 Sidechain** | 独立 `.jsonl` + `.meta.json` | 子智能体对话 | 子智能体级 |

### 10.2 仅追加设计原则

**关键设计选择**: 会话记录是仅追加 JSONL 文件 + 读时链修补。

- 压缩从不修改或删除以前写入的记录行——仅追加新的边界和摘要事件
- 恢复和分叉从磁盘重建上下文，但不恢复会话级权限
- 代价：结构化查询需要事后重建而非直接查找

### 10.3 压缩后的记录结构

```
[boundaryMarker, ...summaryMessages, ...messagesToKeep, ...attachments, ...hookResults]

边界标记包含:
- headUuid, anchorUuid, tailUuid → 支持读时链修补
```

## 12. 设计哲学讨论

### 12.1 核心哲学：模型判断 + 确定性装备

论文的最深刻洞见：Claude Code 的设计哲学可以总结为：

> **"模型做决定，装备做保障"**

- 1.6% 决策逻辑（LLM 调用） vs 98.4% 操作装备
- 装备创造条件让模型能做好决定，而非约束其选择
- 工程复杂度不是为了限制而是为了**使能**

### 12.2 与主流框架的反方向

Claude Code 与 LangGraph 等框架形成鲜明对比：
- LangGraph：显式状态图 + 类型化边，**约束模型路径**
- Claude Code：最小决策框架 + 最大操作装备，**释放模型路径**

**结论**: 编码智能体正在趋同于类似操作系统的抽象——核心循环作为内核，其余一切构成 OS。

### 12.3 五大价值张力

| 价值对 | 张力 | 证据 |
|--------|------|------|
| 权威性 × 安全 | 批准疲劳 vs 保护 | 93% 批准率削弱人类警惕性 |
| 安全 × 能力 | 性能 vs 防御深度 | 50+ 子命令回退跳过安全管理 |
| 适应性 × 安全 | 可扩展性 vs 攻击面 | 预信任初始化 CVE |
| 能力 × 适应性 | 主动性 vs 干扰 | 高频率时偏好下降 |
| 能力 × 可靠性 | 速度 vs 连贯性 | 上下文限制阻碍完整代码库感知 |

### 12.4 四个架构权衡

1. **安全 vs 自主性**: 渐进性权限梯度，但不恢复权限是安全意识设计
2. **上下文效率 vs 透明度**: 5 层压缩 × 功能标志 = 难以预测但高效的上下文管理
3. **简单性 vs 可扩展性**: 每种扩展机制在不同上下文成本层操作，增加复杂度换取灵活性
4. **权限模型在对抗条件下**: 预信任执行窗口落在默认拒绝评估管线之外

## 14. 对 AI 智能体系统设计的启示

### 14.1 对智能体构建者的指导

1. **投资确定性基础设施优先于决策脚手架**——越强的模型越需要丰富的操作环境而非约束

2. **深度防御的独立性假设可能是假的**——当安全层共享性能和成本约束时，一个层的退化可能串联到其他层

3. **透明 > 不透明**——文件式、用户可编辑的记忆和配置增加了可审计性和信任

4. **没有"统一扩展"银弹**——不同上下文成本的多种扩展机制比单一 API 更灵活

5. **安全是设计问题，不是后加功能**——预信任初始化和权限恢复语义必须从第一天开始设计

### 14.2 设计空间的核心洞见

> **"架构质量的主导因素不是模型能力，而是围绕简单智能体循环的装备质量。"**

当前沿模型收敛到类似的编码能力时，**操作装备**（上下文管理、安全分层、恢复机制）成为主要的差异化因素。

### 14.3 论文的独特贡献

- ✅ **首个对 Claude Code 的源码级架构分析**
- ✅ **设计空间分析法**——超越系统描述，揭示设计空间中的选择点
- ✅ **OpenClaw 对比**——用完全不同的部署上下文校准发现
- ✅ **证据三级体系**——为 AI 系统分析建立严格标准
- ✅ **五个价值 + 十三个原则 → 架构**——从人类价值到具体代码的完整链路
- ✅ **长期能力保持作为评价性透镜**——提出关于 AI 对人类技能影响的深刻问题
