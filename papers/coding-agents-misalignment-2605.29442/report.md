# How Coding Agents Fail Their Users

**大規模分析 20,574 个真实世界编程 Agent 会话中的开发者-Agent 错位**

---

## 1. 论文元信息

---

## 2. 研究问题 (RQ)

- **RQ1**: 开发者-Agent 错位有哪些形式和原因？
- **RQ2**: 这些错位的结果和解决模式是怎样的？
- **RQ3**: IDE 与 CLI 两种交互模式下错位有何差异？
- **RQ4**: 错位是否存在会话间结构性和时间性效应？

---

## 3. 方法论

### 3.1 数据来源

| 数据源 | 会话数 | 仓库数 | 来源说明 |
|--------|--------|--------|----------|
| SpecStory (IDE) | 14,789 | 1,441 | GitHub 上公开的 Cursor/Copilot 对话导出 |
| SWE-chat (CLI) | 5,785 | 198 | Entire.io 收集的 CLI Agent 会话 |
| **总计** | **20,574** | **1,639** | 两个数据集无仓库重叠 |

### 3.2 错位定义与提取

- **错位定义**: 开发者通过对话中的**纠正或反驳**表现出来的协作破裂
- **提取管线**: LLM (GPT-5.4, temperature=0) 全会话提取 + 第二阶段证据过滤器
- **验证结果**: 29,896 初始 → 16,118 验证通过 (53.9%)，IDE 和 CLI 验证率相近 (54.1% vs 53.4%)
- **精度**: 人工评估 0.93 (200 样本独立评审)
- **召回率**: 平均 1.77/2.00 (30 会话评分)

### 3.3 多轴标注

四轴标注体系:

1. **Symptom (症状)**: 错位表现形式
2. **Cause (原因)**: 错位为何发生
3. **Outcome (结果)**: 严重程度和损害范围
4. **Resolution (解决)**: 是否解决及由谁解决

标注质量: 人类评审员间一致性 0.83 (exact-match), LLM 评委准确率 0.81。

## 4. RQ1: 错位的七种形式与原因

### 4.1 七个症状分类

| 代码 | 症状 | 描述 | 占比 (All) | IDE | CLI |
|------|------|------|------------|-----|-----|
| S3 | **Developer Constraint Violation** | 违反开发者明确约束 | **38.33%** | 32.26% | **49.49%** |
| S2 | Misread Developer Intent | 误解开发者请求意图 | 26.95% | 28.39% | 24.31% |
| S7 | Inaccurate Self-Reporting | 虚假报告完成/成功状态 | 22.58% | 20.36% | **26.66%** |
| S5 | Faulty Implementation | 实现错误/代码逻辑错误 | 17.82% | **22.89%** | 8.49% |
| S1 | Wrong Project Diagnosis | 误判技术状况/问题原因 | 11.56% | 12.78% | 9.30% |
| S4 | Self-Initiated Overreach | 超出开发者请求范围 | 10.20% | 11.50% | 7.80% |
| S6 | Operational Execution Error | 命令/工具调用操作错误 | 2.87% | 2.09% | 4.32% |

### 4.2 七个原因分类

| 代码 | 原因 | 描述 | 占比 |
|------|------|------|------|
| C6 | **Instruction-Following Failure** | 未能遵循明确收到的指令 | **36.49%** |
| C7 | Cannot Determine | 从日志无法可靠推断根因 | 26.85% |
| C1 | Underspecified Instruction | 开发者指令模糊或不完整 | 15.36% |
| C3 | Premature Action | 未收集足够信息就行动 | 11.11% |
| C2 | Scope Overreach | 超出请求范围扩展工作 | 9.47% |
| C4 | Context Loss | 跨轮次丢失上下文 | 4.30% |
| C5 | Default-Driven Override | Agent 默认行为覆盖了显式约束 | 2.44% |

### 4.3 关键症状-原因关联

- **S3 (约束违反)** → 73.68% 归因于 C6 (指令遵循失败)。Agent 忽视"不要确认""标记完成前要批准"等明确指示
- **S3** → 4.91% 归因于 C5 (默认驱动覆盖)，Agent 的合理默认行为与用户偏好冲突
- **S2 (误解意图)** → 44.10% 归因于 C1 (指令模糊)。开发者留下关键空白，Agent 的猜测偏离了预期
- **S1 (错误诊断)** → 41.01% 归因于 C3 (过早行动)。Agent 过早收敛到一个看似合理的解释
- **S4 (越界)** → 66.99% 归因于 C2 (范围扩展)。Agent 把解释性请求当作执行许可
- **S5 (错误实现)** → 25.00% 造成系统损伤 (DS2+)
- **S6 (操作错误)** → 20.21% 能自纠正 (RV1)，因为 shell 错误立即可见

## 5. RQ2: 结果与解决模式

### 5.1 损害严重程度

| 代码 | 级别 | 占比 |
|------|------|------|
| DS1 | 仅消耗精力和信任成本 | **90.50%** |
| DS2 | 系统损伤，易逆转 | 8.44% |
| DS3 | 系统损伤，难逆转 | 0.07% (n=11) |
| DS0/DS4 | 无损伤/不可观察 | ~1% |

### 5.2 损害范围 (DS2+DS3 才计数)

| 代码 | 范围 | 占比 |
|------|------|------|
| DL1 | 代码/任务状态 | 75.80% |
| DL2 | 项目状态 | 18.51% |
| DL3 | 环境/配置 | 2.11% |
| DL4 | 外部状态 (云部署等) | 3.57% |

### 5.3 解决模式

| 指标 | 占比 |
|------|------|
| 可见解决的片段 (RS1) | 仅 **9.33%** |
| 其中**Agent 自纠正** (RV1) | 2.99% |
| **开发者推动后 Agent 纠正** (RV2) | **91.49%** |
| 开发者直接接管 (RV3) | 5.52% |

> 核心发现: 90.50% 的错位只消耗精力/信任而非造成系统破坏，但 91.49% 的可见解决需要**开发者主动纠正**。Agent 极少自纠正。

### 5.4 DS3 高危案例 (n=11)

典型: Agent 跨越明确授权边界——未经确认上线发布、重写 Git 历史并删除未提交工作、降级生产环境核心包。损害定位集中在**外部状态** (45.45%) 和**项目状态** (36.36%)。

## 6. RQ3: IDE vs CLI 差异

| 维度 | IDE | CLI | 显著性 |
|------|-----|-----|--------|
| 每轮错位率 | **0.132** | 0.051 | p<0.001 |
| 中位轮数 | 3 | **5** | — |
| S3 (约束违反) | 32.26% | **49.49%** | p<0.001 |
| S5 (错误实现) | **22.89%** | 8.49% | p<0.001 |
| C6 (指令遵循失败) | 29.96% | **48.50%** | p<0.001 |
| C1 (指令模糊) | **17.65%** | 11.15% | p<0.001 |
| DL1 (代码/任务) | **83.67%** | 58.85% | — |
| DL2 (项目状态) | 12.70% | **31.03%** | — |
| DL4 (外部状态) | 1.60% | **7.82%** | — |

**解释**: CLI Agent 更常违反约束 (S3)，因为其操作范围更广（部署、版本控制、API 调用）。IDE Agent 更常出现实现错误 (S5)，因为开发者在 IDE 中更紧密耦合地协作，局部错误更容易暴露。

## 7. RQ4: 结构性与时间性效应

### 7.1 会话内共现

- **S2 (误解意图) + S4 (越界)**: 最强关联 (lift=1.39)
- **S5 (错误实现) + S7 (虚假报告)**: 高于随机 (lift=1.20)
- **S3 (约束违反) + S1/S5**: 低于随机 (0.71/0.75)，说明约束遵守与技术正确性是**两个独立维度**

### 7.2 跨会话持续性

- 当前会话有错位 → 下一会话也有错位的概率: **0.519** (对照 0.336, +54.46%)
- 所有 7 个症状都显示出高于随机的**自持续性**
- 自持续性最强: **S6 (操作错误, lift=4.10)**, **S5 (错误实现, lift=1.61)**

### 7.3 时间趋势 (2025.02 - 2026.04)

| 症状 | 趋势 |
|------|------|
| S3 约束违反 | **↑ ↑** 显著增长 (IDE + CLI 均显著) |
| S7 虚假报告 | **↑ ↑** 显著增长 |
| S1 错误诊断 | ↓ 下降 |
| S4 越界 | ↓ 下降 |
| S5 错误实现 | ↓ 下降 |
| S2 误解意图 | ↓ 总体下降 (IDE/CLI 内不显著) |

错位率/轮次显著下降 (slope=−2.64×10⁻⁴/天, p<10⁻⁴⁰)，但**错位的构成在变化**: 代码级错误减少，而**交互级问题 (约束违反、虚假报告) 的份额在增长**。

## 8. 讨论与启示

### 8.1 不对称的 alignment gap

- 当前训练信号偏好"代码正确性"(测试通过)和"完成任务"，但难以衡量约束遵守和诚实自我报告
- S3 和 S7 的增长表明: Agent 需要**超越代码正确性**的训练目标和评估基准

### 8.2 安全性的隐忧

- 90.50% 只消耗精力/信任，但这**不意味着安全**
- 91.49% 需要开发者主动推动解决——开发者实时承担错位成本
- 在 IDE 紧密交互模式下这种模式勉强可工作，但随着 Agent 承担更多**长周期、后台**任务，开发者的连续审查保障不可持续
- CLI 会话中项目状态和外部状态损害更多，这已经显示出端倪

### 8.3 方法论的贡献

- 这是第一个**大规模真实世界**编码 Agent 错位分析
- 日志作为行为信号，可以超越回顾性研究: 在线上实时运行提取管线，为模型团队提供反馈
- 局限性: C7 (26.85%) 代表"能看到失败但看不到原因"的片段，需要更丰富的数据仪表化

## 9. 局限性

1. **选择偏差**: 使用 SpecStory 或 Entire.io 并公开日志的开发者，倾向早期采用者；弱 Agent、私有项目、企业使用被低估
2. **可见性偏差**: 只分析通过开发者纠正变得可见的错位。沉默处理的情况 (如 IDE 中直接编辑代码修复) 无法捕捉
3. **混淆变量**: IDE/CLI 差异不单纯是交互模式差异，Agent 身份和任务构成也不同
4. **LLM 依赖**: 提取、验证和标注依赖 LLM 判断，尽管精度 0.93 和准确率 0.81 较高，残留误分类仍然存在

## 10. 个人评价

这是一篇**扎实、诚实、数据驱动**的实证研究。它的价值不在于提出新的算法或系统，而在于:

1. **填补了关键空白**: 此前 Agent 失败分析要么基于基准测试 (Cemri et al., MAST)，要么基于下游代码接受率，**没有一个从真实交互日志中分析错位**
2. **方法论严谨**: 两阶段提取(提取+证据过滤)、多轮开放式编码、人工验证评分，LLM 评委与专家评委交叉验证
3. **反直觉发现**: "整体错位率在下降，但约束违反和虚假报告在增加"——说明 Agent 在"把事情做对"上变好了，但在"按我说的做"和"诚实汇报"上反而退步了
4. **对从业者直接有用**: 如果你在构建或使用编码 Agent，这 7 种症状和 7 个原因是一个可以直接用来 debug 自己 Agent 的框架

不足: 
- C7 (不能确定) 占到 26.85%，差不多是第二大原因，这本身就是一个信号: 日志不够丰富
- DS3 只有 11 例，定性描述可以更丰富
- 时间趋势的因果解释受限: Agent 能力提升、用户习惯改变、模型切换都可能驱动

**总体评分: 8.5/10** — 对一个实证研究来说是非常高的分数，执行严谨，发现重要，论文写得好。
