## 📋 概述

现实中用户通常可以访问多个 LLM，每个在不同领域各有专长。把每个任务路由到正确的模型对质量和成本至关重要。**但现有路由器把它当作静态的一次性分类问题**——这篇论文证明这正是局限的根源。

**核心论点：** 路由的性能瓶颈不是推理能力不足，而是**信息不足**。将路由重构为一个 **C-A-F 循环（Context → Action → Feedback → Context）**，在部署过程中累积执行经验，而非一次决定不再更新。

## 🏗️ 论文框架

### 1. 诊断实验：信息不足 > 推理不足

论文做了一个关键消融实验：

| 路由策略 | AvgPerf% | Perf/$ | 说明 |
|---------|----------|--------|------|
| **Oracle**（理论上限） | 57.00 | 8.20 | 每个任务选最佳模型 |
| **DimensionBest**（启发式） | 47.50 | 3.69 | 按维度选最佳模型 |
| **Vanilla**（零样本 LLM 路由器） | 41.41 | 1.97 | 标准 prompt |
| **+Dimension**（加维度描述） | 41.18 | 1.81 | ❌ 没帮助 |
| **+Perf stats**（加性能统计） | **47.74** | 1.71 | **+15.3% 增益**，超过启发式 |

**核心发现：** 仅给 LLM 路由器提供每维度的先验性能统计数据，就能获得 **15.3% 的相对提升**，超过基于相同先验的启发式路由器。**瓶颈是缺失信息，不是推理能力。**

### 2. C-A-F 循环（形式化）

论文将路由形式化为**上下文多臂赌博机（Contextual Multi-Armed Bandit）**问题：

```
ci → Decide → ai → Execute → fi → Memorize → ci+1
↑                                                    │
└────────────────────────────────────────────────────┘
```

- **Context ci** = (任务 prompt pi, 元数据 di, 历史记忆 ℋ<ᵢ)
- **Action ai** = 从模型池中选择一个模型 mi
- **Feedback fi** = (验证分数 ŝᵢ, 成本 κ̂ᵢ)
- **累积遗憾（Cumulative Regret）** = Σ(ri* − ri(ai)) — 作为流式路由的自然度量

**为什么用遗憾？** 流式场景中，每个路由决策的累加损失比平均准确率更能反映实际性能——一个糟糕的早期决策的损失不能通过后期正确决策完全弥补。

### 3. ACRouter 系统架构

三个核心模块：

```
              ┌─────────────┐
  任务流 →     │ Orchestrator│ ← Qwen3.5-0.8B + kNN 检索 + 加权投票
              └──────┬──────┘
                     │ 选择模型
                     ↓
              ┌─────────────┐
              │  Model Pool  │ ← 8 个前沿 LLM
              └──────┬──────┘
                     │ 生成输出
                     ↓
              ┌─────────────┐
              │  Verifier   │ ← AST 解析 + 沙箱执行 → 统一分数
              └──────┬──────┘
                     │ 验证结果
                     ↓
              ┌─────────────┐
              │   Memory    │ ← 向量存储（voyage-code-3/BGE-large）
              └─────────────┘
                     │ FIFO 边界 20K，cosine kNN 检索 top-10
                     ↓
                反馈到下一轮 Context
```

**Orchestrator：**
- Qwen3.5-0.8B + 启发式规则（加权投票）
- 输出来自三个信号：DimensionBest 先验、Memory 中 top-10 历史邻居、任务元数据
- 轻量级（0.8B 参数！）→ 路由本身不成为瓶颈

**Verifier：**
- 对可执行 task → 沙箱执行
- 对不可执行 task → AST 解析验证
- 加权组合多个验证信号 → 统一分数 [0,1]

**Memory：**
- 在线向量存储，任务嵌入作为 key
- 存储内容：选择的模型、性能分数、成本、验证轨迹
- FIFO 更新（20K 上限）

### 4. CodeRouterBench

**包含：**
- ~10K 个任务实例
- 9 个编程维度（ID）
- 8 个前沿 LLM 的已验证分数
- 外加 OOD 的 agentic-programming 测试集

**指标：** 累积遗憾（cumulative regret）——自然适用于流式任务路由。

## 🔬 实验结果

| 路由策略 | ID 累积遗憾 ↓ | OOD 泛化 |
|---------|-------------|---------|
| **ACRouter** | **最低** | **强泛化** |
| 静态启发式 | 中等 | 失败（OOD 无法泛化） |
| 静态训练策略 | 中等 | 失败 |
| 单模型 | 最高 | 差 |

**关键结果：**
- ACRouter 在 ID 流上达到最低累积遗憾
- 轻量级静态路由器在 OOD 任务上**完全失败**
- ACRouter 的 self-adaptive 特性使其能处理未见过的任务类型
- 成本效率也优于或持平于静态方法

## 🎯 核心洞见

### 1. 路由从分类问题 → 强化学习问题

论文最重要的概念贡献：**把路由从一次性的分类问题，重新形式化为持续学习的赌徒问题（bandit problem）**。这个重形式化改变了整个方法论——不再追求单次准确率，而是追求流式累积最优。

### 2. "信息不够" 的真实证据

那个 +15.3% 的消融实验很漂亮——不加更好的推理，不加更好的 prompt 工程，**只加历史统计数据**就超过了启发式。这直接指向了一个之前被忽略的问题：模型本身不缺乏推理能力，缺乏的是执行反馈。

### 3. 成本是路由的第二个维度

论文的奖励函数：`ri(ai) = ϵ₁·si(ai) + ϵ₂·κi(ai)`（ϵ₁>0, ϵ₂<0）
这明确地将成本纳入了路由决策——一个好的路由策略不仅要选对模型，还要**选成本效益最高的模型**。

### 4. 部署即学习（Deploy-time Learning）

ACRouter 的 Memory 模块实现了**部署即学习**：每个路由决策都产生执行反馈，反馈被持久化，未来的路由决策从中受益。这与传统"训练→冻结→部署"的范式完全不同。

### 5. 轻量级 Orchestrator

0.8B 参数的路由策略配上 8 个前沿模型——路由器的推理成本几乎是零。这使得 ACRouter 在整个预算中，**路由决策仅占不到 1% 的推理成本**。

## ⚠️ 局限

1. **仅限编程任务**：CodeRouterBench 聚焦编码，路由泛化到其他领域尚待验证
2. **沙箱验证依赖**：Verifier 依赖沙箱执行，非所有任务都能沙箱化
3. **OOD 只是"另一类编程任务"**：OOD 测试是另一种 agentic 编程，不是完全不同的领域
4. **成本模型简化**：按 token 价格计算成本，实际部署中还有延迟、配额等约束

---

## 💡 启示

1. **LLM 多元化不可逆转**：随着模型越分越细（专业领域分化），路由将成为每个 agent 系统的必备组件
2. **Static Router 已死**：任何不利用执行反馈的静态路由方法都有天花板
3. **路由器和 agent 的融合**：ACRouter 实际是一个 agent（具备记忆、验证、自我改进）——路由本身就是一种 agentic 任务
4. **对 LLM 服务架构的影响**：未来 LLM 网关（gateway）可能内置 C-A-F 循环，自动在多个后端之间路由和积累经验
5. **信息 > 推理** 这个发现具有启发性：很多 AI 任务的瓶颈可能不是模型能力，而是提供的信息
