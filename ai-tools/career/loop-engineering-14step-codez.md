# Loop Engineering：从手动 Prompter 到循环设计者的 14 步路线图

> **来源：** [Loop engineering: the 14-step roadmap from prompter to loop designer](https://x.com/0xcodez/status/2064374643729773029) — Codez (@0xCodez)，参考了 Anthropic 工程文档、Addy Osmani 的 Loop Engineering 长文及近期测量研究

---

大多数开发者还在手动提示他们的编码智能体：输入请求 → 等待 → 读 diff → 再输入。**90% 的构建者从未写过一条让智能体自己提示自己的循环。**

没有自动化、没有状态文件、没有验证器、没有调度。杠杆点已经转移——从手动写提示，到设计能自动提示的系统。

**14 步，3 个层级。停止提示。开始设计。**

---

## PART 1 · 为什么做 & 检查清单

### 01. Loop Engineering 就是用系统取代你自己作为提示者

两年来，你用编码智能体干活的方式一直是：写提示 → 分享上下文 → 读结果 → 写下一个提示。智能体是工具，你一直握着它。

**Loop Engineering** 是构建一个小系统：它自己寻找工作、交给智能体、检查结果、记录发生了什么、决定下一步。你设计一次系统，系统从此替你提示智能体。

Addy Osmani 将其拆为六个部分：

![六部分示意图](../image/loop-engineering-14step-codez-3.png)

Anthropic 的工程师现在每天合并的代码量是 2024 年的 **8 倍**——连 Anthropic 自己都说这"几乎肯定是夸大了真实生产力增益"。数字有争议，但机制没有：**杠杆点已从写提示转移到设计提示循环。**

---

### 02. 建任何东西之前，先跑 4 条件测试

循环在满足四个条件时才值得投入成本。少一个，循环的成本就超过回报——来自 AlphaSignal 分析的诚实评估，也是大多数 X 推文跳过的部分。

![4 条件测试](../image/loop-engineering-14step-codez-2.png)

四个条件用大白话说：

- **任务会重复。** 循环通过多次运行摊薄搭建成本。一次性任务的话，一个好的 prompt 更快更便宜。如果工作不是每周重复出现，你有的不是一个循环——而是一个只跑了一次的脚本。
- **验证是自动化的。** 循环需要能**不靠你**来判定工作失败的手段：测试套件、类型检查、linter、构建。没有自动化检查意味着你还是要回到椅子上逐条读 diff——正是循环本应消除的工作。
- **你的 token 预算能吸收浪费。** 循环会重读上下文、重试、探索。这些都在烧 token，不管这次运行有没有产出。这个技术跟预算是规模化的——为什么对 token 无限的人来说是显而易见的，对按量付费的人来说却是鲁莽的。
- **智能体有高级工程师的装备。** 日志、重现环境、运行自己写的代码并看到什么出错的能力。没有这些，循环就是在盲目迭代。

---

### 03. 谁赢、谁输——循环偏爱能花钱的人

经济学不是普适的。说 Loop Engineering"显而易见"的人往往有无限 token。说它"鲁莽"的人通常在 $20 的消费套餐上运行重量级验证循环。

**实际受益者：**
- **有可重复、机器可检查工作 + 运行预算的团队**——持续测试分类、依赖升级、lint 加修复、有强测试覆盖代码库上的 issue-to-PR 草稿
- **已有强测试套件的代码库**。如果初级工程师能按清单完成任务、测试套件能捕获他们的错误，那就适合循环
- **已经使用多智能体模式的异步优先团队**——routine 是他们缺失的编排层

**应该跳过的人（今天）：**
- **消费套餐上的独立开发者**——token 账单在生产力增益到来之前先到达
- **没有自动化验证的代码**——没有真正检查的循环等于智能体在自说自话
- **真正的瓶颈是审阅容量而非打字速度的团队**——循环产生更多代码；如果审阅已经是瓶颈，只是让队列更长

**对于一次性任务、探索性工作，或任何"完成"需要判断的事情，一条精心编写的一次性 prompt 仍然胜出。这篇文章的诚实版本是：Loop Engineering 是真实的，但大多数开发者现在还不需要它。**

---

### 04. 30 秒循环检查清单

第 2 步的 4 条件测试是战略决策。这个是战术检查——你针对具体任务在把它变成循环之前跑的检查清单。

![循环检查清单](../image/loop-engineering-14step-codez-4.png)

**漏掉一项就保持手动 prompt：**

1. ✅ 任务至少每周出现。少于每周 → 搭建成本永远无法摊薄
2. ✅ 测试、类型检查、构建或 linter 能拒绝错误输出。没有自动化门 → 智能体给自己批改作业
3. ✅ 智能体能运行它改过的代码。没有重现环境 → 迭代是盲目的
4. ✅ 循环有硬停止：token 预算、迭代次数或时间限制。没有硬停止 → 循环一直跑到有人注意到账单
5. ✅ 部署前有人类审阅。任何不可逆操作都需要人类批准门

**好的首次循环选择：**
- CI 失败分类 —— 夜间运行，扫描失败，分类原因，为简单的故障起草修复 PR
- 依赖升级 PR —— 每周一次，扫描更新，测试兼容性，开 PR
- Lint-and-fix —— 每次 PR 打开事件，自动应用样式修复
- Flaky test 复现 —— 循环直到一个理论通过测试
- 有强测试的 Issue-to-PR 草稿

**坏的首次循环（需要人类坐镇）：**
- 架构重写
- 认证/支付代码
- 生产部署
- 模糊的产品工作
- 任何"完成"需要判断的事情

## PART 2 · 5 个构建模块

### 05. 自动化（Automations）：心跳

自动化让循环成为真正的循环（而不是只跑过一次的脚本）。它们在调度上、事件上或触发条件下启动。它们是心跳——循环中的所有其他东西都挂在它们上面。

在两个关键工具中长什么样：

- **Codex**：Automations 标签页——选项目、设 prompt、设频率、选本地 checkout 或后台 worktree。有发现的运行进入 Triage inbox；无发现的自动归档。
- **Claude Code**：三个可组合的基元：`/loop`（会话级频率）、Desktop scheduled tasks（重启持久化）、Routines（笔记本离线云运行）。配合 hooks 使用生命周期事件。

自动化内部的两个基元，区分了"能工作的循环"和"昂贵的循环"：

- **`/loop`** 按频率重复执行。适合你想要定期检查（不管状态如何）时使用。
- **`/goal`** 持续运行直到你写的条件为真。由一个小型独立模型检查完成条件——所以写代码的智能体不是给它自己评分的那个。

![Automation 示意图](../image/loop-engineering-14step-codez-1.png)

这是**制造者-检查者分离**在停止条件上的应用。

---

### 06. Worktrees：无混乱的并行

你一运行超过一个智能体，文件就开始冲突了。两个智能体写同一个文件，跟两个工程师互不通气提交到同一行是一个头疼。

**Git worktree 解决了它**——一个独立的工作目录，共享同一个 repo 历史但有自己的分支。一个智能体的编辑物理上碰不到另一个的 checkout。

![worktrees 工作方式](../image/loop-engineering-14step-codez-7.png)

在两个工具中的体现：
- **Codex** 内置 worktree 支持——多个线程同时进入同一个 repo 不会相互碰撞。
- **Claude Code** 直接暴露 git worktree：`--worktree` 标志在独立 checkout 中打开会话；子智能体的 `isolation: worktree` 设置让每个 helper 获得一个用完自动清理的新 checkout。

**Worktrees 消除了机械碰撞，但你仍然是天花板**——你的审阅带宽决定了你能实际并行运行多少智能体，而不是工具。

---

### 07. Skills：一次写入项目知识，每次运行都读到

Skill 是你停止每次会话都像金鱼一样重新解释项目上下文的方法。两个工具使用相同格式：一个内含 `SKILL.md` 的文件夹，存放指令和元数据，外加可选的脚本、参考资料和资产。

**为什么这对循环尤其重要：没有技能的循环每次从头推导整个项目上下文。有技能，意图就能累积。** 代码规范、构建步骤、"我们因为那次事故不这样做"——在外部写一次，每次运行都能读到。

![Skills 示意图](../image/loop-engineering-14step-codez-5.png)

---

### 08. Connectors（连接器）：通过 MCP 连接真实工具

只能看到文件系统的循环是微小的循环。基于 **Model Context Protocol（MCP）** 的连接器让智能体读取你的 issue tracker、查询数据库、访问 staging API、往 Slack 发消息。

![MCP 连接器](../image/loop-engineering-14step-codez-6.png)

Codex 和 Claude Code 都支持 MCP，所以你在一个工具上写的连接器通常直接能在另一个上用。

这区别于"智能体说'这是修复方法'"和"循环打开 PR、关联 Linear ticket、CI 变绿后在频道里通知"。

**对循环工作回报最快的连接器（按优先级排序）：**
- **GitHub** —— 读 repo、创建分支、开 PR、评论 issue、响应 webhook 事件。任何代码循环最大的首日收益。
- **Linear 或 Jira** —— 随循环进展更新 ticket、将 PR 关联回 issue、验证通过时自动关闭。
- **Slack** —— 发布分类结果、在升级时提醒人类、早上汇总夜间运行。
- **Sentry / 错误追踪器** —— 让循环调查实时告警，为高频错误起草修复。

---

### 09. 子智能体（Sub-agents）：让制造者远离检查者

循环中最有用的结构，远超过其他：**将写代码的智能体与检查代码的智能体分开。**

Osmani 的表述很精确："写代码的模型在给自己批改作业时实在太好了。"第二个智能体用不同的指令（有时是不同的模型）捕获第一个智能体自己说服自己接受的东西。

![子智能体模式](../image/loop-engineering-14step-codez-8.png)

**这是 Anthropic 2024 年 12 月工程帖子中的 evaluator-optimizer 模式换了个名字。一个生成，另一个批评，重复。2026 年流行的词汇在 18 个月前就已经被记录下来了。**

在工具中如何实现：
- **Codex**：仅在你要求时生成子智能体，同时运行，然后将结果折叠回一个答案。你可以在 `.codex/agents/` 中以 TOML 文件定义自己的智能体——名称、描述、指令、可选的模型和推理 effort。你的安全审阅者可以用强模型 + 高 effort，而你的探索者则用某种快速只读的东西。
- **Claude Code**：通过在 `.claude/agents/` 中的子智能体和智能体团队实现同样的功能。通常的分工是：一个探索、一个实现、一个对照规范验证。

**为什么在循环内尤其重要：循环在你不在的时候运行。一个你真正信任的验证器是你唯一可以走开的理由。** 子智能体消耗更多 token（每个都做自己的模型和工具工作）——把钱花在你真正需要的第二意见上。

## PART 3 · 要么建对，要么别建

### 10. 状态文件（State File）：智能体会忘记，文件不会

这是听起来太简单不在乎、实际上是每个工作循环的脊椎骨的部分。一个 markdown 文件、一个 Linear board、一个 JSON 状态——任何存在于单个对话之外、记录什么已完成和下一步是什么的东西。

**为什么重要：智能体默认记忆很短。它们本轮学到的明天就没了，除非你写下来。**

> Osmani 的原则：**智能体会忘记，仓库不会。没有持久状态的循环每次重启；有状态的循环恢复。**

状态文件的两种模式：
- **仓库中的 Markdown**：项目根目录或 `/.claude/` 下的 `STATE.md`。版本控制。简单。Diff 可读。适合单人/小团队。
- **外部系统（Linear、GitHub Issues、数据库）**：跨仓库存活、可查询、支持团队级可见性。适合需要多人看到循环在做什么的**生产循环**。

对于可能偏离目标的长时间运行循环，配对使用**高层规范文件**（`VISION.md` 或 `AGENTS.md`），智能体每次运行都会重读。**状态告诉它在哪里。规范告诉它要去哪。**

---

### 11. 最小可行循环（Minimum Viable Loop）

如果你通过了第 2 步的 4 条件测试，在搞任何花哨的东西之前，构建能工作的最小循环。**四个部分，不需要集群。**

![最小可行循环](../image/loop-engineering-14step-codez-9.png)

用大白话说四个部分：

- **一个自动化**。按频率启动、在明确条件停止的定时运行。用 Claude Code 的 `/loop` 或 Codex 中的 automation。想运行到一个声明条件成立时配合 `/goal`。
- **一个 Skill**。一个 `SKILL.md`，存储智能体否则每次运行要从零推导的项目上下文。
- **一个状态文件**。记录什么已完成和下一步是什么的 markdown 文件或 Linear board。明天的运行**恢复**而不是**重启**。
- **一个门（Gate）**。自动拒绝错误工作的测试、类型检查或构建。这是决定循环是帮助还是浪费的部分。

**顺序很重要：先把一次手动运行搞可靠 → 变成技能 → 包在循环里 → 再调度它。跳步是循环在生产中失败的方式。**

**真正重要的指标是每项被接受的变更的成本**——不是消耗的 token，不是尝试的任务数，不是调度的循环数。如果接受率低于 50%，你就是在做循环本应让你免于做的审阅工作，循环在亏钱。

---

### 12. 拉尔夫·威格姆循环（Ralph Wiggum Loop）：悄悄失败的循环

工程师 Geoffrey Huntley 记录了这个失败模式并命名了它。一个本应在完成时才发出完成 token 的智能体**提前**发出 token，循环在一个半完成的任务上退出。**没有硬门，循环就悄悄失败，继续烧钱。**

![Ralph Wiggum 循环](../image/loop-engineering-14step-codez-8.png)

Ralph Wiggum 循环发生在以下情况：

- **没有真正的验证器。** 只有另一个被要求"审阅"的智能体，没有客观信号。两个乐观主义者在互相肯定。
- **软的完成条件。** "完成"由智能体的判断定义，而非测试、构建或类型检查。
- **没有硬停止。** 循环持续直到外部因素杀死它（速率限制、你注意到），而不是在验证成功时停止。

**修复方法是第 11 步的门**——能客观地判定工作失败的东西。一个能通过或失败的测试。一个能编译或不能的构建。一个返回零或非零的 linter。不是一个有意见的验证器。

**其他值得了解的被测量过的失败模式：**
- **长时间会话中的目标漂移**。每次摘要步骤都有损失；"不要做 X"的约束在第 47 轮消失。缓解：每次运行重读 `VISION.md` 或 `AGENTS.md`。
- **自我偏好偏差**。写代码的智能体在给自己的工作评分时太友好。缓解：一个单独的验证子智能体，不接触制造者的推理过程。
- **智能体懒惰**。循环在完成一部分时声明"够了"。缓解：`/goal` 加客观停止条件，由全新的模型检查。

---

### 13. 理解债务（Comprehension Debt）与认知投降（Cognitive Surrender）

这是一个随着循环变得更好而变得更尖锐的失败模式，而不是更糟。两个名字来自 Osmani 的论文：

- **理解债务**：循环越快地产出你**没写**的代码，仓库包含的内容与你的理解之间的差距就越大。真正令人痛苦的账单不是 token 账单。而是有一天你需要调试一个**团队里没人读过**的系统。
- **认知投降**：停止形成自己的意见、接受循环返回的任何东西的倾向。设计循环在你有判断力时是解药，在你用它来避免思考时是加速器。**同样的行动，相反的结果。**

缓解措施不是技术性的：

- **读 diff。** 如果你不读循环输出的代码，你就是在以复利租借理解债务。
- **抽查门。** 挑一些循环打开的 PR，验证批准通过它们的测试确实捕获了你关心的失败模式。**门会腐烂。**
- **阻止循环碰架构工作。** 让它做小的、机器可检查的变更。一旦你让它接触判断性工作，理解债务就会加速。
- **与队友结对设计循环。** 两双眼睛在设计循环时能捕获循环将永远利用的盲点。

---

### 14. 安全税——无人值守的循环就是无人值守的攻击面

一个无人值守运行的循环，也就是一个无人值守的攻击面。

你的循环必须防御的威胁模型：

- **生成的代码未经审阅就发布。** 循环开 PR 的速度快过人类审阅速度。如果门不包括安全检查（SAST、依赖审计、秘密扫描），不安全代码会自动合并。
- **技能作为注入向量。** 自动安装技能的循环继承了其描述中隐藏的每一个 prompt 注入。安装前审计技能来源。
- **凭证出现在日志中。** 长时间运行循环中的调试日志将秘密散布到你不监控的日志中。在生产循环中禁用详细日志；对确实被记录的内容做清理。
- **权限范围爬升。** 用只读权限测试的循环为了"方便"添加了一个写权限，然后再也不审计了。每 30 天重新审计权限。

## § 让循环变成吃钱无底洞的错误

- **不跑 4 条件测试就建循环。** 第 2 步存在是有原因的。大多数开发者至少有一个条件不满足。
- **没有客观的门。** 没有测试、类型检查或构建，被要求"审阅"的第二个智能体只是第二个乐观主义者。
- **同一个智能体既写又验。** 自我偏好偏差。制造者给自己的作业评 A+。
- **没有状态文件。** 明天的运行从零开始而不是恢复。
- **模糊的停止条件。** "看起来不错时就完成"从来不成立。使用测试、类型通过或通过构建。
- **没有 token 预算上限。** 循环重读上下文和重试。没有上限，雄心勃勃的循环会消耗你预期的 5-10 倍 token。
- **在消费套餐上运行重量级验证的循环。** token 账单或速率限制，总有一个会先到你。
- **自动安装社区技能。** 17,022 个审计过的技能中有 520 个泄露凭证。安装前读源码。
- **在判断性工作上跑循环。** 架构、认证、支付、模糊产品决策。把循环留给 lint-and-fix，而不是战略。
- **不读 diff。** 以复利计算的理解债务。当你调试一个没人读过的系统的那一天，成本超过了曾经花过的所有 token。

---

## 结论：杠杆移了，你的工作也移了

两年来，与编码智能体合作的杠杆在 prompt 上。更好的 prompt、更好的上下文、更好的一次性输出。**那个阶段在结束。** 智能体已经足够好了，下一个杠杆点上升了一层：**决定它们做什么、什么时候做、用什么门、什么状态在运行间存活的系统。**

但这个故事诚实版本不是每个人都该冲去建循环。大多数开发者现在还不需要——直到任务重复、验证自动化、预算能吸收浪费、智能体有高级工程师的装备。**少一个条件，循环就亏钱。**

**如果你通过了测试，从小做起。** 一个自动化。一个 Skill。一个状态文件。一个门。先让一次手动运行可靠。变成技能。包在循环里。然后调度它。**顺序很重要。跳步，你就在为一个没人理解的系统付钱。**

> Cherny 的观点不是工作变容易了。是杠杆点移了。**构建循环。保持做工程师。**
