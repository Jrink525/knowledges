# Agent Loop 架构：什么在运行循环？

> **来源：** [The Agent Loop Architecture](https://x.com/djfarrelly/status/2067677007140278630) — Dan Farrelly

![cover](../image/agent-loop-architecture-cover.jpg)

每个人都在问「WTF 是个循环？」但没人问：**是什么在跑这个循环？**

AI 讨论已经收敛到「循环是 agent 系统的核心原语」。Matt Van Horn 追溯了 agent loop 的谱系：ReAct → tool-use → 编排循环 → 循环监督循环。Addy Osmani 拆解了循环内的构建块：自动化、worktrees、skills、连接器、子 agent。

Van Horn 落脚在**持久性（durability）**——无法在重启后存活的循环不是循环。Osmani 的关键线索是**编排（orchestration）**——设计系统提示 agent，而不是你自己打字。

我想把他们的论点推进一步。持久性不仅仅是循环的一个属性——它是整个底层执行层。**持久的编排是构建 agent loop 架构的基础。**

---

## 循环在哪里断裂

`/loop` 和 `/goal` 模式处理单 agent、单会话工作很好。agent 循环直到任务完成。

但下一阶段（Van Horn 框架中的第 5 阶段）就出问题了：

- 循环监督其他循环
- 循环按定时运行，而非仅由人类触发
- 循环能在进程重启、部署和崩溃后存活
- 循环生成子 agent 并等待结果（有时几小时后）
- 循环需要事后可观测

**这不是 prompt 问题。这是基础设施问题。**

Van Horn 引用了 @runes_leo：「AI 编程中最贵的不再是写代码，而是管理 agent 循环。」终端里的 `while true` 给不了你这些。VM 上的长时进程也给不了。

想想当你在服务器上运行 agent 循环时会发生什么。进程会死或重启。一次部署、一次 OOM、一次竞价实例回收。循环重启了。但它之前在做什么？它在哪一步？它已经发了那条 Slack 消息吗？它已经调用了子 agent 吗？

**你不知道。它重新开始。** 重新获取已有的数据。重新调用 LLM 做已经做过的决策。发送重复通知。生成重复子 agent。你醒来看到三条相同的 Slack 消息和困惑的团队。

修复方案不是「更好的错误处理」——而是一个执行模型，其中**每一步都有检查点、每个决策都被持久化、恢复意味着从最后成功的一步继续。**

## Agent Loop 架构：三层

每一层映射到一个具体原语。

### 第一层：循环（The Loop）

循环是一个 **cron + 一个决策者**。它按时（或按触发）运行，评估状态，决定下一步做什么。

这就是 Van Horn 的定义具体化了：cron 从来缺的就是中间的**决策**。Agent 做决策，不是你。Cron 是心跳。LLM 是决策者。步骤是持久化执行，检查进度。

> 每周一早上 9 点，循环触发。它获取数据，问 LLM 是否需要出报告，如果需要就调用一个 skill。如果进程在步骤之间重启，已完成步骤不会重新执行。

这就是循环。不是 LLM，是围绕 LLM 的循环。

### 第二层：技能（The Skill）

在这里，skill **不是一条 prompt——它是一个持久的工作流**。多步骤、可重试、可组合、可独立部署。

Van Horn：「循环是管道。资产是它调用的 skill。」这部分是复利的来源。系统学会的每个新 skill 都让每个循环更强。

> 这个 skill 获取、分类、路由。它是一个内置容错的工作单元。skill 可以是带 LLM 的 AI 工作流，也可以是确定性代码。

### 第三层：编排器（The Orchestrator）

编排器是运行一切的引擎：定时 crons、执行步骤、管理重试、强制执行并发限制、存储运行历史、热部署新函数/工作流而不中断正在运行的进程。

**这一层没人讨论——因为它本该是隐形的。但它是基础。**

大多数人认为 agent = **「LLM + 工具」**。Agent loop 架构将其重构为 **agent = 「循环 + skill + 编排」**。LLM + 工具在循环内部。LLM 和工具可以随意切换，而架构保持不变。**编排使架构成为可能。**

## 出问题时怎么办

你的事故分类 skill 触发，但 metrics API 超时了。调用此 API 的步骤重试并再次命中 API。数据现在部分缓存了，API 完成。skill 继续下一步，像什么都没发生过一样。

有时候没那么简单。API key 过期了，或者你的托管服务商宕机 30 分钟。所有重试都耗尽了。然后呢？

`onFailure` 处理器在所有重试用尽后触发。它发帖到运维频道，让人知道。事件被保留，什么都不会丢失。下一次定时运行从失败的地方继续。

**持久的编排必须给你：**
- 针对临时错误的步骤级重试
- 针对不可恢复错误的失败处理钩子

临时错误也很贵。如果你的 skill 或 agent 从头重试，你就是在多次调用 LLM，不必要地燃烧 token。LLM 调用可以被检查点化。现在乘以系统中 10 或 30 个 agent——很贵。

**步骤级检查点不仅仅是正确性特性——它是省钱利器。**

## 自己写技能的 Agent

这是更有意思的部分。系统不是静态的——它旨在自我演化。

**Agent 不仅在循环内运行——它编写新的循环并在编排引擎中注册它们。**

每个部署的函数都是一个独立的持久 skill：可从循环或 agent 触发，或按定时运行，有自己的重试逻辑。**Skill 复利增长。**

这是一个**编排感知的 agent**（orchestration-aware agent）。

它的工作方式：AI agent 有一个编排 SDK 作为工具。它可以写新函数，在引擎中注册它们，然后它们立刻开始运行。Agent 进程热加载新函数，无需重启或中断正在运行的工作。

**具体示例：**

1. **人类表达需求** —「我们的服务经常半夜延迟飙升，没人注意到直到早上。」
2. **Agent 写 skill** — 一个健康检查循环每 30 分钟跑一次 + 一个事故分类 skill
3. **Agent 部署 skill** — Agent 写函数代码，被 sidecar 进程拾取，自动注册。立即生效，无需部署流水线、无需 PR
4. **Skill 自主运行** — 每 30 分钟引擎触发健康检查。出问题就调用分类 skill。无人值守。完全持久
5. **Agent 根据信号迭代** — 一个单独的 review 循环：cron 触发的函数每周运行一次，读取编排器的运行历史，评估绩效

> Review 是一个函数。它读取运行历史，检查事故是否与实际宕机关联，把信号喂给 LLM。不是魔法。一个 cron job，LLM 坐在决策位置。

**那验证呢？** Agent 写代码的好坏由它周围的护栏决定。代码可以被类型检查。Agent 可以本身调用函数来测试——因为它能直接与编排引擎交互。Review 循环捕获初始调试中没发现的問題。更进一步，agent 可以用 `onFailure` 钩子触发自己评估给定失败——这是一个持续改进的反馈循环。

**那冲突呢？** 并发控制（concurrency: `[{ limit: 1, key: "event.data.service" }]`）处理了简单情况——每个服务同时只跑一个事故分类。编排器排队。第二个等待第一个完成。没有重复告警，没有竞态条件。

Agent 不仅在执行任务——**它在为自己构建基础设施。** 每个 skill 在创建它的对话结束后仍然存在。杀死 agent 进程再重启——skill 继续运行。切换底层模型——skill 继续运行。Agent 是短暂的，它的产出是持久的。

## 开发者的视角

如果开发者看不到 agent 部署了什么、调试不了问题、审计不了凌晨 3 点跑了什么，那么整个架构就是一个大隐患。

**编排引擎存储每次运行、每一步、每个输入、每个输出、每次重试。** 上周二 agent 部署的一个 skill 在凌晨 4 点失败了？你能看到具体哪一步失败、输入是什么、抛了什么错误、重试了多少次才放弃。

这不是事后加上的仪表盘。它是持久执行固有的。每一个 `step.run()` 是一个检查点。**每个检查点都是可观测的。**

当写代码的不是人类时，可观测性不是锦上添花——它是信任层。

日常开发者的工作流：早上检查运行仪表盘。看哪些 skill 晚上跑了，哪些成功了，哪些失败了。如果 agent 写的 skill 表现异常，你可以直接读代码、编辑、删除，或者告诉 agent 修复它。Agent 创作了它，但**你拥有它**。Agent 和它的 skill 仍然是一个你应该照料的花园。

## 为什么持久性是基础

Van Horn：「这些东西必须能在重启后存活。」

「在容器里跑」给你的是 uptime，不是正确性。容器在崩溃后重启能把进程带回来，但每一个正在运行的循环重新开始。每一步重新执行。每一个 LLM 调用重新做。循环看起来在跑，但它在**盲目地跑**。

## 与现有工具对比

有些工具提供「漂亮」的 turnkey 方案，你也可以自己拼凑底层工具。选择没有对错，但正确的架构层应该允许你和你的 agent 随时间演化。灵活、动态、持久。

**持久执行的原语：** 适合 agent 使用、agent 容易编写、有可观测性和 API 让 agent 本身成为编排感知的。

## 自进化循环（The Compounding Loop）

Satya Nadella 最近的帖子给行业一直在感受的东西命了名：**护城河不是模型——是循环。**

他的框架：有两种资本。**人力资本**——团队多年建立的知识和判断。**Token 资本**——公司在基础模型上构建的 AI 工作流、决策模式和学到技能。

论点：**它们一起复利增长。** 每个改进的工作流产生更好的信号。更好的信号产生更锋利的 AI 行为。更锋利的行为释放人类的注意力做更高判断力的工作。

> 一台爬山机器。

Agent loop 架构具体实现了这一点：

- **每个 agent 部署的持久 skill 都是编码为可执行基础设施的制度知识。** 它持久存在。不管有没有人看都在运行。
- **一个 cron 触发的 review 循环**评估 skill 表现并迭代。这就是现实版的爬山机器。不是幻灯片里的飞轮图。一个带 cron 触发的函数。
- **如果你的 skill 在进程重启时死亡，复利重置为零。** 持久性是让投资持续存在的关键。

Nadella 的关键点：「一个公司应该能换掉一个『通才』模型，而不失去其学习系统中内置的『公司老将』专业知识。」这就是 skill 库模式。持久函数不在乎哪个 LLM 调用它们。

## 据此构建

讨论一直在围绕 agent 做什么：循环、工具、推理、上下文工程。**下一个讨论是关于什么在运行 agent。**

三层：**循环、技能、编排器。**

- **循环**是工作单元
- **技能**是资产
- **编排引擎**是让两者持久的基石
- **Sidecar 模式**是模型：agent 编写自己的持久技能、部署、审查它们表现、迭代

这不是思想实验。这是一个正在运行的模式。

> *Dan Farrelly 是 Inngest 的 CEO。Inngest 构建面向此架构的编排引擎——step.run()、step.invoke()、cron 触发器、事件驱动控制流、并发控制和完整步骤级可观测性。但架构模式大于任何单一工具。*

原语已经存在。据此构建。
