# Loop Engineering 清晰解读：循环本身很简单，难的在外面

> **来源：** [Loop Engineering Clearly Explained](https://x.com/akshay_pachaar/status/2069118430582866051) — @akshay_pachaar

![cover](../image/loop-engineering-explained-1.jpg)

你一半的信息流突然都在说同一件事：别再给 agent 写 prompt 了，开始设计循环吧。

Claude Code 的构建者 **Boris Cherny** 说得直白：

> **"我不再手动给 Claude 写 prompt 了。我有循环在跑。我的工作是写循环。"**

世界上最流行的编程 agent 之一的构建者都不再手动 prompt 了——那他到底在做什么？

这就是 Loop Engineering 的全部要义。

---

## 先看循环本身

Agent 不是魔法黑盒。它的核心就是一个很朴素的循环：

```python
while True:
    response = model(context)
    if response.has_tool_calls():
        results = run_tools(response.tool_calls)
        context += results
    else:
        break
```

**模型→工具→上下文→重复。** 模型读上下文，请求调用工具，你运行工具并把结果喂回去，模型再读，直到它不再请求工具。

**让人意外的是：这个循环本身早就被解决了。** 每个正经的 agent 框架最后都落在这六行代码上。没人会在 `while` 语句上竞争。

**所以，如果循环本身微不足道，大家到底在工程什么？**

---

## 工作已经从模型移到了外面

AI 的重心一直在远离模型本身——**一层层向外包**：

| 层次 | 说明 |
|------|------|
| **Prompt Engineering** | 你发送的文字 |
| **Context Engineering** | 模型能看到的一切，不只是你的指令 |
| **Harness Engineering** | 围绕模型的代码——运行工具、追踪状态、处理错误 |
| **Loop Engineering** | 推动整个系统朝目标前进的自主循环 |

每一层包裹着前一层。你并没有不再关心 prompt——你只是意识到 prompt 是更大系统的一个小部分。

> **LangChain 总结得很干净：Agent = 模型 + 框架。如果你不是做模型的，那你就在做框架。**

一个应该重新排序你优先级的研究发现：**框架现在比模型更重要。** 有些团队保持模型不变，只改了周围的代码，就把 benchmark 排名从中间提到了前五。**同一个大脑，不同的循环。**

Loop Engineering 就是构建那个大脑运行环境的工程学。以下是真正会出问题的四个难点。

---

## 难点 1：知道什么时候停止

这是没人提醒你的问题。

当 agent 不再请求工具时，**它只是结束了一次"回合"，不等于完成了任务。**

设想一个编程 agent：它写了一些代码，环顾四周，看到有进展，就宣布完成了。测试还在失败。它仍然宣布了胜利。

> **"终端消息"结束的是回合，不是任务。混淆这两者是最常见的循环出错方式。**

好的循环为正确的理由停止，所以你需要叠几层刹车：

- **最大迭代次数** — 硬上限，让卡住的 agent 没法无限跑下去
- **预算和时间限制** — token、费用和秒数的天花板
- **无进展检测** — 如果它重复相同的调用传入相同的参数，它在空转
- **真正的完成检查** — 证明任务完成的自动化条件

**最后一条最重。"完成了"应该意味着测试通过了，而不是 agent 对自己的工作感觉良好。**

---

## 难点 2：保持上下文清洁

长循环从内部开始腐烂。

agent 跑的轮次越多，上下文中堆积的垃圾就越多——旧工具输出、死胡同、过时的推理。随着堆栈增长，模型性能下降。这叫**上下文腐烂（context rot）**。

循环会让它螺旋恶化：腐烂的上下文产生更差的决策 → 增加更多噪声 → 进一步腐烂上下文。人们称之为**厄运循环（doom loop）**——你肯定感受过，agent 跑得越久就越蠢。

**应对方法：把上下文视为预算，而不是桶。**

| 策略 | 做法 |
|------|------|
| **压缩（Compaction）** | 对话太长时总结它，然后从总结继续 |
| **卸载（Offloading）** | 把庞大的输出推到文件里，只保留需要的那一截 |
| **子 Agent（Sub-agents）** | 把混乱的子任务交给独立的 agent，只让干净的返回结果回来 |

> **本能是保留一切，以防万一。技能是知道什么该扔掉。**

---

## 难点 3：agent 真正能用的工具

循环的质量取决于里面的工具。

堆一百个工具，agent 就不知道该用哪个了。**一组精练、不重叠的工具才是赢家。** Anthropic 的经验法则很尖锐：如果一个人类工程师都无法确定该用哪个工具，agent 更不可能。

有两个比人们预期更重要的点：

### 让写操作可安全重复

循环会重试。如果一次"创建客户"的重试真的创建了第二个客户，你醒来就会发现重复记录和双倍账单。**任何改变状态的操作都必须能安全调用两次。**（幂等性）

### 为 agent 写错误消息，而不是为人

好的错误告诉 agent 下一步该做什么。工具上线前问一个问题：一个 LLM 读到这条错误消息，知道下一步怎么走吗？

> **在循环中，错误不是死路。它是下一个指令。**

---

## 难点 4：能说"不"的东西

自主循环有一个安静失效模式：**被独自留下的 agent 倾向于同意自己。**

> **一个没有批评者的循环，只是一个对自己点头的 agent。**

该领域最尖锐的评论点出了真相：设计循环只是工作的一半，另一半是在循环中放入一个能说"不"的东西——一个测试、一个类型检查、一个真正的错误。

**修复方法：把制造者从检查者中分离出来。** 一个模型做工作。一个不同的检查——通常是另一个模型或硬测试——来打分。工人不给自己批改作业。

---

## 真正的转变

现在 Cherny 的话就说得通了。

**Prompting** 是你在每一步手把手引导 agent。**Loop Engineering** 是你构建一个能自己引导自己的系统，然后后退。

你的工作从"发指令"变成**设计三件事**：

1. **目标** — 写成一个 agent 能自我对照检查的成功标准
2. **循环** — 带着合理的刹车，让它好好停下来
3. **验证器** — 让"完成"是经过证明的，而不是声称的

**Andrej Karpathy** 的思维模式抓住了本质：别告诉模型该做什么，给它成功标准然后看着它跑。他运行的研究循环在夜间自动调整脚本、测试、保留有效的、丢弃无效的——他自己完全不在循环里。他安排一次，然后按播放键。

> **你不再是一双手，而是成为设计那台机器的人。**

---

## 从哪开始

你不需要第一天就有整夜自主运行的 agent。逐步构建：

1. **从基础循环开始，立刻加上最大迭代上限、超时和费用天花板**
2. **在开始之前就把"完成"定义为一个自动化检查，而不是结束后的感觉**
3. **保护上下文** — 压缩长运行、卸载大产出、隔离混乱的子任务
4. **审计你的工具** — 保持精练聚焦、让写操作安全重复、重写错误消息让 agent 能行动
5. **在循环中放入批评者** — 只有当你信任那个说"不"的东西时，才放手全自动

---

## 总结

Loop Engineering 不是一个框架或一个要安装的工具。它是一个**你该把精力投哪里的转变**。

> **模型正在变成商品。模型周围的循环才是真正的工程所在。**

最好的构建者们已经不再问"我该让 agent 做什么？"他们开始问：

> **"什么系统能不需要我就能完成这件事？"**

把这个问题回答好，你也会停止手写 prompt 的。
