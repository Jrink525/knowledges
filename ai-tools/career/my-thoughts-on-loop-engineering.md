# 我对循环工程的思考：验证器才是产品，其余都是管道

> **来源：** [Samuel McDonnell (@samueljmcd) on X](https://x.com/samueljmcd/status/2066524627585634765)

![封面图](../image/my-thoughts-on-loop-engineering-1.jpg)

循环工程（Loop Engineering）是个新标签。但最难的部分一直没变——**验证（Verification）**。

---

## 背景：循环工程的三阶段叙事

Boris Cherny（Claude Code 创建者兼负责人）在 Fortune Brainstorm Tech 大会上说：他不再自己写 prompt 了——是另一个 Claude 在替他写 prompt。他每天早上管理数百个 Agent，有时上千个。

围绕这个现象的框架叫"循环工程"（loop engineering）。三阶段叙事：

- **2024**：写好的 prompt
- **2025**：并行运行 Agent
- **2026**：构建替你运行 Agent 的循环

你不再敲 prompt，而是设计生成 prompt 的系统。

这个叙事没什么问题，但它**掩盖了决定你的循环能否交付东西的那个关键**。循环是一个生成器（generator）接上一个验证器（verifier）。生成器从来不是瓶颈。**验证器才是。**

---

## 循环到底是什么

脱去所有包装，循环就是用**自运行循环**替代"写 prompt → 读结果 → 再写 prompt"的人工流程：

> **发现 → 规划 → 执行 → 验证 → 重复，直到满足条件**

Agent 驱动自己的迭代。你设计它运行的轨道。

**简单版**：单个 Agent 循环自己的输出。研究→草稿→对照目标→修复弱点→重复直到达标。就像人在改稿，但不会厌烦。

**大规模版**：一个舰队。目标发给编排器 → 拆分成子任务分给专家 → 专家分派给子 Agent。每一层都跑发现→规划→执行→验证，直到目标达成。一个是单一作者，另一个是全栈项目团队。

这些在本质上都不是新东西——跟你自己手动运行 Agent 循环是一样的，只不过**人从内圈退到外圈去做设计了**。

---

## 开放与封闭：两条路径

### 开放循环（Open Loop）
- 给 Agent 广阔探索空间
- 设定条件和目标，中间自由发挥
- 优势：能产生你未指定的路径和原创输出
- 代价：token 消耗极高；标准宽松时沦为"垃圾生成器"
- **循环越自由，越依赖验证器"

### 封闭循环（Closed Loop）
- 提前固定路径：明确目标、定义步骤、逐级评估、终止条件或交给人手
- Agent 仍在循环，但在你构建的框架内运行，预算可控

**当前能产出结果的就是封闭循环。** 人们把功劳归于"自主性"——但自主性不是原因，**评估关卡（evaluation gate）才是。**

评估关卡阻挡了自信的错误答案传播到下一次迭代，再下一次。

> 这就是大多数循环文章闭嘴的地方。每个人都画发现→规划→执行→验证的图，几乎没人对"验证"这个方框说任何精确的话。**那个方框就是产品，其余都是管道。**

---

## 循环工程的两个理论根基

### ReAct（Princeton + Google）
交替推理和行动：**思考 → 行动 → 观察结果 → 再思考 → 重复直到完成**。在编程层面：理解目标→写代码→运行→读错误→推断原因→修复→重跑→直到测试通过。**循环本身就是重点。**

### Reflexion（Shinn et al.）
ReAct + 记忆。尝试失败时，Agent 用自然语言写下失败原因、存储笔记、下次尝试前先读取。在现代承载框架中，笔记存在文件里，不在上下文窗口中。这就是今天所有人所说的"持久化记忆"的种子。

**两个模式说的是一回事：会自我检查的 Agent 击败不会自我检查的 Agent。**

---

## 内层循环与外层循环

有用的循环有两层，而且经常被搞混：

### 内层循环（Inner Loop）
在单个任务内运行。Agent 在回答前验证自己的工作。
- **弱 Agent**：编辑文件，说"完成"
- **强 Agent**：编辑文件 → 写测试 → 跑测试 → 发现失败的边缘情况 → 修复 → 重跑 → 确认通过 → 然后说"完成"

同样的工具，唯一的区别是**模型是否选择了调用验证器**。这个选择就是演示和结果之间的差距。

### 外层循环（Outer Loop）
跨会话运行。Agent 失败后把教训记录到持久化文件中，下次会话读取文件，一开始就做对。`SKILL.md` 和 `AGENTS.md` 就是明显的存放位置。Agent 在上下文窗口重置后遗忘一切，但**仓库不会**。

- 内层循环已经成熟，大多数 Agent 现在都会做
- 外层循环还处于半成品状态——在正确的位置、以正确的粒度、持久化正确的教训，比听起来难得多，**这也是大量价值当前还在桌上的地方**

**两层都是验证：** 内层验证任务，外层验证你不会重复上周的错误。而如果看不见，两层的价值都不大。**在放大循环之前先仪表化评估关卡，否则你只是在更快地生成错误答案。**

## 标志性案例：Bun 的 Zig → Rust 移植

Jarred Sumner（Bun 作者）用 Claude Code 的动态工作流把 Bun 从 Zig 移植到 Rust。约 **75 万行 Rust**，**99.8%** 的现有测试套件通过。Anthropic 说 11 天，Sumner 自己说 6 天。

怎么做到的？
1. 第一遍：映射每个 struct 字段的正确 Rust 生命周期
2. 第二遍：行为等价移植每个文件，数百个 Agent 并行，每文件两个审查 Agent
3. 独立的"反驳"Agent 层：专门质疑其他 Agent 的输出
4. 修复循环：驱动构建和测试，直到两者都通过

**验证不是最后一步，它就是架构本身。**

### 最诚实的一句话

> "该移植尚未投入生产。"

——这整篇发布中最诚实的一句话，也是我要划重点的那句。

99.8% 的测试通过是**基准测试结果**——证明移植复现了旧测试已经覆盖的行为。**生产环境是"没人写过测试"的行为。** 两者之间的差距，就是整个行业不断绊倒的地方。**一个变绿的循环不等于一个正确的循环。它只是满足了**你给它的验证器 **。输出的质量被你给的验证器质量封顶，一丝不多。**

## 实际上应该构建什么

机械部件并不神秘：

- **定时触发器**：发现工作并启动 Agent
- **隔离的 Git worktrees**：并行 Agent 不互相踩踏
- **Skills 文件**：不必每次重新解释项目约定
- **连接器**：接上工作已经存在的工具
- **分离的生成器与验证器角色**：Agent 给自己的作业打分总是偏高的
- **记忆**：比会话活得更久的文件，带着教训前行

### 原生工具现状

Claude Code 的现有关键功能：

- **`/goal`**（v2.1.139+）：保持完成条件，跨回合持续工作直到条件满足
- **动态工作流**（v2.1.154+ 研究预览）：Claude 编写编排脚本，工作分发给多个并行 Agent，上限 16 并发 / 每次运行 1000 个 Agent

两者都减少同一件事：你不断下达指令、检查、再次下达指令的回旋。**只有当任务真正不适合单次完成时才用它们。** 它们消耗的 token 远多于正常会话。不是每个任务都是工作流任务，把小任务包装成工作流是另一种浪费。

---

## 瓶颈已经转移了

被称为"循环工程"的技能是真实的——只是它瞄准了系统的错误一半。设计编排现在已经是容易的部分，工具帮你做了大部分。**仍然困难、仍然需要手动、仍然产生真正结果的：是评估关卡。**

- Agent 检查什么？
- 对照什么标准？
- 失败如何在传播前被捕获？
- 什么被写下来，让下一次运行从更高起点开始？

> **Agent 时代的管理不是招聘有能力的工作人员。工作人员既有能力又廉价。它是关于设计它们运行的约束——跟管理人的时候从来都是同一个道理。**

**设计验证器，而不是 prompt。**
