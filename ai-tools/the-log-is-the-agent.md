## 核心论点

> **智能体不是模型、不是运行时、不是当前执行循环——智能体是它的数据，具体来说是它的事件历史，也就是"日志"。**

这个洞察的力量在于：当日志构建正确时，**仅凭日志就能恢复智能体的完整状态**。这一单一属性解锁了大规模部署性、可靠性、分叉能力、多人协作和提供商迁移等全部能力。

---

## 从 RPG 存档文件到智能体

想想你在《天际》或《艾尔登法环》里玩了一百小时的角色：

- 角色是游戏引擎吗？不是
- 角色是 PlayStation 吗？不是
- 角色是手柄吗？不是
- **角色是存档文件。**

如果你的 PlayStation 炸了，角色不会死——你买台新机器，从云端下载存档，角色精确地回到挥剑到一半的位置。

**AI 智能体正处在同样的转折点。**

---

## 定义智能体

智能体 = **日志**。

日志是事件历史：每一次用户输入、每一次模型输出、每一次工具调用和每一次工具结果——**仅追加的完整记录**。

日志还携带会话定义的引用：系统提示、工具描述、技能版本。这些不会每轮变化，更像是版本化的常量。

**运行时不包含智能体状态，模型不包含智能体状态，工具也不包含——它们只是解释器和追加器。** 它们读取持久化状态，执行操作，写下下一个事件。给你一个全新的执行器和同一份日志，它就能重建智能体并从那里继续。

---

## 实践中的日志

一旦你把智能体定义为日志，整个系统的其他部分就变得更容易推理了：

- **模型** 读取日志的一个视图，产出下一个动作
- **工具执行器** 执行工具调用，追加结果
- **UI** 读取日志，渲染时间线
- **追踪系统** 读取日志，渲染链路
- **审计员** 读取日志，重建发生了什么

这是数据库行业早已找到的模式：表、索引、缓存——全都是底层变更日志的投影。

### 简化版循环

```
while session_has_work:
    take_temporary_lease(session_id)
    state = reconstruct_from_log(session_id)
    response = model.next(state)
    append(response, to=session_log)

    if response.requests_tool:
        append(tool_call_started, to=session_log)
        result = run_tool(response.tool_call)
        append(result, to=session_log)
        continue

    finish_turn(session_id)
```

每轮 pass 获取临时租约、读取日志、前进一步、写回结果。循环是**幂等的、容错的**——只要每个有意义的转态转换都被持久化写入，任何执行器都能接起会话继续。

## 压缩不是日志

上下文窗口有限，需要压缩。但压缩是**有损的**，这削弱"日志即智能体"的主张了吗？

不，恰恰相反。完整日志是记录；压缩只是它的一个投影，就像物化视图不是数据库、摘要不是对话一样。

**保留原始日志，你随时可以生成新投影。丢掉原始日志只保留压缩，你就失去了一部分智能体。**

所以最干净的做法是：把压缩视为尽力而为的有损分支，作为一个新日志继续。

---

## 日志不是全世界

> **反论**: 工具会改变日志之外的状态啊——编辑文件、开 GitHub issue、发邮件——这不就破局了吗？

不，不会。你是从日志**重新推导**智能体状态，不是从世界推导。如果它已经发了邮件，回滚不会撤回那封邮件。日志不会让世界变得确定或可逆。

它保留的是一个忠实、可恢复的记录——记录了智能体做了什么、看到了什么——**这正是你不能失去的东西**。

存档文件也是一样：它不包含天际引擎或地图——只包含把你放回原位所需的玩家特定状态。

## 由此衍生的系统属性

### ✨ 可靠性
Claude Code 当前的问题：如果智能体走到权限审批时进程挂了，恢复后权限提示就没了。生产环境不可接受。

当日志即智能体时，执行器可以失败——新 worker 接过会话、重建状态，权限提示就停在原地。**进程死了，智能体没死。**

### ✨ 可扩展性
大多数 harness 一个进程跑一个智能体 → 智能体跟机器绑定。

当日志即状态时：**一个进程可以推进数千个智能体**，每个智能体每轮从日志重建状态。智能体不绑定任何机器。扩容就是加 worker——没有粘性会话、没有状态迁移、没有协调开销。

### ✨ 分叉
不是一条直线，而是可以分支日志。一个分支跑 Claude，另一个跑 GPT，第三个跑本地 Qwen——各自探索不同策略、不同沙箱、不同工具。探索多种方案变得简单多了。

### ✨ 多人协作
共享智能体不应该等于把聊天记录复制到 Slack——那好比截个数据库的图说是复制。

当日志即智能体时，共享意味着授权访问一个持久化历史，其他人可以审查、恢复或扩展。

### ✨ 迁移
如果智能体的身份困在特定提供商的假设里，迁移就很痛苦。当日志即智能体时，迁移变成适配器问题。不同模型可能需要不同的投影，但那是工程问题，不是身份问题。

---

## 日志是二等公民——这是一个问题

大多数智能体 harness 把日志当作事后补充：

- **Claude Code / Codex**: 写到本地磁盘 JSONL，Claude SDK 模式下是 fire-and-forget——任何错误都会导致数据丢失
- **OpenCode**: SQLite 本地存储，报告过损坏和数据丢失
- **Durable Objects**: 不同分片持有不同历史，重建困难

**日志是副作用，不是系统本身。**

当日志是一等公民——持久化、结构化、可回放且独立于机器时——所有这些属性都成为结构性的。你不是把它们"加上去"，它们自然涌现。

---

## 拥有日志 = 拥有智能体

> **最重要的教训**: 如果日志即智能体，那么谁拥有日志，谁就拥有智能体。

长期来看，最深的锁定不是模型锁定（模型可以换），不是 API 或工具锁定（这些可以封装和适配）——是**日志锁定**。如果提供商拥有你智能体的唯一持久化记录，那它就拥有你的智能体。

你可以导出一个投影（比如聊天记录）——但智能体不是它的最终输出，而是产生那个输出的**路径依赖历史**。

**Anthropic 有 Claude Managed Agents，Google 有 Gemini managed agents，每个主流提供商都在向拥有整个栈靠拢：托管智能体循环、托管记忆、沙箱、压缩、后台执行。** 他们想拥有你的智能体。

而智能体可能是你将运行的最亲密的软件。它需要访问你的个人信息、公司数据、工作流、决策。日志记录着这一切。如果它活在别人的基础设施上，受别人留存政策的约束，能被别人的系统查询——**那他们不只是托管你的智能体，他们拥有它。**

这不是反对托管基础设施——对大多数团队来说它有用且可能不可避免——但你应该知道你的日志在哪里、谁可以检查它、以及它能否被回放、分叉、迁移或导出。

## Omnara 的做法

在 Omnara（YC S25），智能体执行围绕持久化会话日志协调：

- **worker 不是智能体，只是当前执行器**
- 工具调用发起时，worker 写入日志、调度工具到执行环境、让它在别处完成
- 工具完成时，结果追加回日志，另一个 worker（可能不同）重建状态继续
- 关键区别：**智能体状态从不在当前执行的进程里**

这对处理真实世界的故障至关重要：worker 崩溃、机器重启、沙箱消失、工具运行过长、提供商故障、用户断连。如果智能体是进程，这一切都很可怕。如果智能体就是日志，这些都只是执行细节。

---

## 结语

一旦你用这种方式看智能体，很多东西各归其位：可靠性、可扩展性、压缩、分叉、迁移、多人协作、所有权。

它也改变了你的构建方式。你不再把日志当作系统的废气——而是把日志当作系统本身。循环变成日志之上的执行器。**这是数据库行业多年前经历过的反转：持久化历史是主数据，其他一切都是投影。**
