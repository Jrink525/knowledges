# 循环工程（Loop Engineering）：2026 年每个构建者必备的 AI 技能

> **来源：** [Loop Engineering: The AI skill every builder needs in 2026](https://x.com/0xwhrrari/status/2065539680146182270)

![cover](../image/loop-engineering-1.jpg)

大多数人还在手动给 agent 写 prompt。

他们打一个任务 → 等一个回答 → 自己审查 → 自己修错 → 再打一次 prompt。

**这意味着人类仍然是循环本身。**

下一步完全不同：

> 你不是给 agent 写 prompt。你设计一个**循环**——它来驱动 agent、检查结果、决定下一步、一直运行直到工作通过。

这就是 **Loop Engineering（循环工程）**。

> **"你不应该再手动给编程 agent 写 prompt 了。你应该设计循环来驱动你的 agent。"**

Anthropic Claude Code 的负责人 Boris Cherny 也说了类似的话：

> **"我不再手动给 Claude 写 prompt 了。我有循环在跑，它们驱动 Claude 并决定下一步做什么。我的工作是写循环。"**

---

## 为什么大多数人从来没真正构建过循环

循环听起来很酷，直到你看到 token 账单。

一个普通的 agent 循环能快速消耗大量上下文：

- **一个中等规模的编程循环** — 50K-200K tokens
- **一个舰队循环（一个编排器 + 多个专业 agent）** — 500K-2M tokens
- **一个每日定时循环** — 每周数百万 tokens

每次重试 = 花钱。每次自我纠错 = 花钱。每次验证 = 花钱。每个子 agent = 花钱。

> *这是没人真正谈够的隐痛。循环工程难，不是因为概念复杂——而是因为大多数人承担不起让 agent 长时间自由运行的成本。*

**"你当然说得轻松，你有无限 OpenAI 额度。"** —— 这个反应很公平。

这就是为什么便宜的长上下文模型如此重要。要让循环每天运行，你需要：

- **便宜的输入 token**
- **便宜的输出 token**
- **大上下文窗口**
- **工具调用**
- **JSON 输出**
- **高并发**
- **足够记住循环中之前发生了什么**

没有这些，循环就是昂贵的实验。有了这些，循环就是实用的工作流。

---

## 老方式 vs 新方式

过去两年，大多数人这样用 agent：

**老方式：**
- 你给一个 prompt
- agent 给一个输出
- 你审查输出
- 你修弱的部分
- 你手动重复

**新方式：**
- 你定义目标
- 循环发现需要什么
- 循环规划工作
- agent 执行
- 检查者验证结果
- 循环修复失败
- 系统在达到目标时停止

> Prompt 给 agent **一条指令**。循环工程给 agent **一份工作**。

---

## 循环工程到底是什么

循环工程是设计 AI agent 的**可重复反馈循环**的实践。

目标很简单：**从尝试到验证结果，不需要人类手动驱动每一步。**

### 基本循环的五个阶段

1. **Discover（发现）** — 弄清楚需要做什么
2. **Plan（规划）** — 制定执行方案
3. **Execute（执行）** — 动手做
4. **Verify（验证）** — 检查成果
5. **Iterate（迭代）** — 如果不合格，送回循环

> 通过就交付。失败就送回循环。这就是全部。

不是一条完美的 prompt。而是一个**持续改进输出直到达标**的系统。

## 单 Agent 循环 vs 舰队循环

### 单 Agent 循环

一个 agent 跑完整周期：发现、规划、执行、验证、改进。

就像一个人自己改写草稿。

**适用场景：** 专注的任务、小范围、简单目标、内容草稿、Bug 修复、研究总结。

> 一个大脑。一个循环。自我改进。

### 舰队循环（Fleet Loop）

舰队循环更大。你给一个**编排器 agent** 主目标，它把工作分解，分发给**专业 agent**。每个专业 agent 还可以再用更小的子 agent。

```
示例："构建一个生产力应用"

编排器掌控全局
    ↓            ↓              ↓
研究           工程            QA
专业 agent     专业 agent      专业 agent
    ↓            ↓              ↓
网页研究员     代码编写者      测试编写者
              + 调试器        + Bug 追踪器
```

这不是一个 agent 单干。更像一个小团队从头到尾跑一个项目。

---

## 开放循环 vs 封闭循环

### 开放循环（Open Loops）

探索性循环。你给 agent 一个宽泛目标，让它自己找路。

**强大之处：** agent 能发现你没指定的东西。
**代价：** 贵、杂乱。

**风险：**
- 试太多路径
- 烧太多 tokens
- 快速产出低质量内容
- 偏离真正目标
- 难以控制

开放循环很酷，但对大多数人来说，不是最好的起点。

### 封闭循环（Closed Loops）

有边界的循环。人类先设计路径，循环在清晰规则内自动运行。

**封闭循环有：**
- **明确的目标**
- **定义的步骤**
- **每步后的评估**
- **停止条件**
- **卡住时的移交点**

> 这是今天真正能落地见效的版本。更便宜、更可靠、输出更干净。
> **从封闭循环开始。等检查机制强壮了再开放它们。**

## 好循环的 6 个构建模块

概念上循环有五个阶段，但实际中需要六个构建模块才能让循环工作。

### 1. 自动化（Automations）

这是心跳。自动化启动循环，不需要你记得手动运行。

**示例：**
- 每天早上运行
- PR 打开时运行
- 文件变更时运行
- 新 ticket 出现时运行
- 持续运行直到所有测试通过

> 如果你还要手动启动一切，循环就没真正做够工作。

### 2. 工作树（Worktrees）

多个 agent 同时编辑代码时，没有隔离就会冲突——两个 agent 改同一个文件，一个覆盖另一个。

工作树（Worktree）给每个 agent 一个干净的工作空间和独立分支，让多个 agent 并行工作而不搞乱仓库。

### 3. 技能（Skills）

技能是可复用的项目知识。不用每次解释项目，你写一次重要上下文。

**好的技能文件包含：** 愿景、架构、规则、构建步骤、测试步骤、agent 绝对不要做的事。

> 没有技能，每个循环从零开始。有技能，每个循环带着积累的上下文开始。

### 4. 插件和连接器（Plugins & Connectors）

只看到文件的循环是受限的。连接器让循环触达你的真实工具。

**示例：** GitHub、Slack、Linear、Jira、Gmail、Google Drive、数据库、Staging API。

> 这区别就是"这是建议的修复"和"我开了 PR、关联了 ticket、盯着 CI、还发了更新"。

### 5. 子 Agent（Subagents）

制造者和检查者不应该总是同一个模型。

写代码的那个 agent 审查自己代码时往往太宽松。写文章的那个会漏过自己的薄弱部分。

**用独立的 agent 来做：** 探索、实现、审查、测试、事实核查、最终总结。

> 质量在审查者不是制造者时显著提升。

### 6. 记忆（Memory）

记忆让循环跨运行保持连续。模型会忘，但仓库不会，笔记不会，项目日志不会，Obsidian vault 不会。

**记忆可以存在：** Markdown 文件、项目日志、Linear tickets、GitHub issues、Obsidian vaults、数据库、Claude Projects。

> 一个长时间运行的循环需要知道试过什么、什么通过了、什么失败了、还需要做什么。
> 没有记忆，每次从零开始。

## 真实循环示例

### 编程循环

```
读 VISION.md + ARCHITECTURE.md
↓
规划下一个变更
↓
编辑代码
↓
跑测试
↓
如果测试失败 → 读错误 → 修复 → 再测
↓
如果测试通过 → 总结变更
↓
停止
```

没有人需要推动每一步。agent 自己写、测、修、验证。

### 研究循环

```
定义研究问题
↓
搜索资料
↓
总结发现
↓
对照来源验证声明
↓
对比冲突信息
↓
综合最终答案
↓
达到置信阈值时停止
```

这比要一个快速总结好太多了。

### 内容循环

```
主题 + 受众 + 目标已定义
↓
创建草稿
↓
批评 agent 审查草稿
↓
基于批评重写
↓
对照成功标准打分
↓
分数通过 → 发布
↓
分数不通过 → 再重写
```

一个想法变成一个内容系统。

### 销售外联循环

```
定义理想客户画像（ICP）
↓
找到匹配的潜在客户
↓
用公司数据丰富信息
↓
按标准筛选
↓
个性化消息
↓
质量审查
↓
发送或升级给人
```

同样的骨架：目标 → 行动 → 检查 → 修复 → 重复直到完成。

## Prompt 工程师 vs 循环工程师

这是 2026 年正在拉开的技能差距。

### Prompt 工程师

关注更好的指令。改进措辞。拿到更好的单次输出。

但**人类仍然在每次运行后审查一切。人类仍然是反馈循环。**

### 循环工程师

设计反馈系统。

**他们决定：**
- 什么启动循环
- agent 需要什么上下文
- 能用什么工具
- 什么算成功
- 谁来检查工作
- 循环何时停止
- 结果存在哪里

> Prompt 工程师说："给我写个函数。"
> 循环工程师说："写它、测它、修到通过、然后总结变更。"

同样的工具。不同的思维模式。

> **最高杠杆的 AI 构建者不是写更好的英文 prompt。他们在设计能正确发现、规划、执行、验证和停止的系统。**

## 一句话总结

循环工程是从**手动 prompt** 到**自动化反馈循环**的转变。

| 维度 | 说明 |
|------|------|
| **转变** | 老方式：一次一个任务地给 agent 写 prompt → 新方式：设计跑完整周期的循环 |
| **6 个模块** | 自动化、工作树、技能、连接器、子 agent、记忆 |
| **2 种规模** | 单 agent 循环（自我改进）、舰队循环（编排 + 专业 agent + 子 agent） |
| **2 种类型** | 开放循环（探索、昂贵）、封闭循环（有界、可靠、可负担） |
| **5 个阶段** | 发现 → 规划 → 执行 → 验证 → 迭代 |
| **成本问题** | 循环烧 token 很快；便宜的长上下文模型让循环变得实用 |

> **循环工程师的思维转变：**
> - Prompt 工程师问 AI 要输出
> - 循环工程师设计产生**已验证结果**的系统
>
> 这才是真正的解锁。
>
> 别试图写一条完美的 prompt。开始构建那个让不完美的输出变更好的循环。
>
> **一个可靠的循环，胜过一条完美的 prompt。**
