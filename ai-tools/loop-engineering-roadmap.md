# Loop Engineering：从手动 Prompter 到循环设计者的 14 步路线图

> **来源：** Codez (@0xCodez) X/Twitter 长文 — [Loop engineering: the 14-step roadmap from prompter to loop designer](https://x.com/0xcodez/status/2064374643729773029)
>
> 内容源自 Anthropic 工程文档、Addy Osmani 的 Loop Engineering 长文及实测研究

---

大多数开发者还在手动给编程 AI 写 prompt——他们打字、等待、读 diff、再打字。**90% 的构建者从未写过一条帮他们自动提示 Agent 的循环。**

没有自动化、没有状态文件、没有校验器、没有调度。杠杆点已经移动了——从"打 prompt"变成了"设计让 Agent 自动跑的系统"。

**14 步路线图，3 个层级：**

```
第一层：想清楚你是否真的需要 Loop
  │
第二层：学习 5 个构建模块
  │
第三层：构建最小可用循环，别伤着自己
```

---

## 第一层 · 为什么要做 & 自我检测

### 01. Loop Engineering 的定位：替换掉作为 prompter 的自己

过去两年，你用编程 Agent 干活的方式一直是：写 prompt → 分享上下文 → 读返回结果 → 写下一条 prompt。Agent 是工具，你全程握着它。**这部分即将结束。**

**Loop Engineering** 是构建一个小型系统，让它自己：
- 找到工作 → 分配给 Agent → 检查结果 → 记录发生了什么 → 决定下一步

你设计那个系统**一次**，然后系统替你提示 Agent。

Addy Osmani 将其拆为六部分：

```
规划 → 上下文收集 → 实现 → 质检 → 记录 → 决策
```

Anthropic 工程师现在每天合并的代码量是 2024 年的 **8 倍**——Anthropic 自己也说"这几乎肯定高估了真实的生产力增益"。这个数字有争议，但机制没有争议：**杠杆从写 prompt 转移到了设计安排 prompt 的循环。**

---

### 02. 构建前的 4 条件测试

在动手之前，先用这 4 个条件判断：构建循环是否值得。**少任何一条，循环的成本都大于收益。**

| # | 条件 | 通俗解释 |
|---|------|---------|
| 1️⃣ | **任务重复发生** | 循环的成本通过多次运行摊薄。一次性的活儿，写好 prompt 更快更便宜。**如果工作不每周重复，你有的不是循环——只是个跑了一次的脚本。** |
| 2️⃣ | **验证可自动化** | 循环需要不需要你在场也能判定"失败了"的东西。测试套件、类型检查器、linter、编译——没有自动校验，你就得坐回去读每个 diff——这正是循环想要替你消除的工作。 |
| 3️⃣ | **预算能吸收浪费** | 循环会重新读取上下文、重试、探索——这都在烧 token，不管这次运行有没有产出。这对"有免费 token"的人是常识，对按量付费的人是鲁莽。 |
| 4️⃣ | **Agent 有高级工程师的工具** | 日志、可复现的环境、能运行自己写的代码并看到哪里报错——没有这些，循环就是瞎子迭代。 |

---

### 03. 谁受益，谁该跳过

**受益者：**
- ✅ 有重复性、机器可校验工作的团队（持续的测试排查、依赖更新、lint+fix、issue→PR 草稿）
- ✅ 测试覆盖完善的代码库（如果初级工程师按清单能干活，测试能兜底，那就适合循环）
- ✅ 已经用多 Agent 模式的异步优先团队

**应该暂时跳过的人：**
- ❌ 个人开发者用消费级套餐——token 账单来得比生产力收益快
- ❌ 代码库没有自动化验证——没有真正检查的循环 = Agent 不断地自我认同
- ❌ 真正的瓶颈是审核能力而非打字速度——循环产生更多代码，审核瓶颈只是让队列更长
- ❌ 一次性任务、探索性工作、或"完成"靠判断力的——一个精准的 prompt 仍然赢

---

### 04. 30 秒循环检查清单

在把具体任务做成循环前，跑一遍这个战术检查表。**漏掉任何一条就保持手动 prompt：**

```
□ 1. 任务至少每周发生（<每周→设置成本永远摊不薄）
□ 2. 测试/类型检查/编译/linter 能拒绝坏输出（无自动闸门→Agent 自己批改作业）
□ 3. Agent 能运行它改动的代码（无可复现环境→迭代是瞎的）
□ 4. 循环有硬终止条件（token 预算/迭代次数/时间限制，否则跑到你发现账单）
□ 5. 合入前有人工审核（任何不可逆的操作都需要人工批准闸门）
```

**好的首试循环：**
- CI 失败排查——每晚扫描失败→分类原因→对简单的修复提 PR
- 依赖更新 PR——每周扫描更新→测试兼容性→开 PR
- Lint+fix 流程——每次 PR 打开时自动应用样式修复
- 不稳定测试重现——循环到某个假设通过测试为止
- 强测试下的 issue→PR 草稿——坏输出被测试套件拒绝

**别碰的循环（需要人在椅子上）：**
- 架构重构
- 鉴权/支付代码
- 生产部署
- 模糊产品工作
- 任何"完成"靠判断力的事

## 第二层 · 5 个构建模块

### 05. Automations（自动化）——心跳

自动化让循环成为真正的循环（而不是你手动跑了一次）。它们在**定时、事件或触发条件**下启动。这是心跳——循环里的一切都挂在上面。

**Codex 的做法：** Automations 标签页 → 选择项目、设置 prompt、设置节奏、选择本地 checkout 或后台 worktree。有发现的结果进入 Triage inbox；没发现的自动归档。

**Claude Code 的做法：** 三个可组合的原语：`/loop`（session 内定时），Desktop scheduled tasks（重启后保留），Routines（笔记本关掉后在云端跑）。用 hooks 挂生命周期事件。

**两个最重要的原语：**

```
/loop       → 按节奏重复运行，适合不管状态如何都定期检查
/goal       → 持续运行直到你写的条件成立。由另一个小型模型检查完成条件
              → 实现了 maker-vs-checker 分离
```

> **maker-vs-checker split**：写代码的 Agent 和评判的 Agent 不是同一个。

---

### 06. Worktrees（工作树）——无冲突的并行

一旦你运行超过一个 Agent，文件就会打架。两个 Agent 写同一个文件，就像两个工程师不商量就改同一行代码。

**Git worktree 解决了这个问题：** 一个独立的目录，在自己的分支上，共享同一个 repo 历史。一个 Agent 的修改物理上碰不到另一个的工作目录。

- **Codex** 内建 worktree 支持——多个线程同时访问同一个 repo 而不会冲突
- **Claude Code** 暴露 `git worktree`，`--worktree` 参数开一个独立 wokring tree 的 session，subagent 的 `isolation: worktree` 设置让每个 helper 有自己的 sandbox，用完自动清理

> Worktrees 消除了机械冲突，但你仍然是天花板——**你的审核带宽决定了能跑多少并行 Agent，而不是工具。**

---

### 07. Skills——一次写项目知识，每次运行都读

**Skill** 让你不用每次都像金鱼一样重新解释项目上下文。

两个工具用相同格式：文件夹内含 `SKILL.md`，加上可选 scripts、references、assets。

**为什么这对循环特别重要：** 没有 skills 的循环每次运行都从零重新推导项目上下文。有 skills 后，**意图是累积的**——约定、构建步骤、"因为那次事故所以不能这么做"——一次写在外面，每次运行都读。

---

### 08. Connectors（连接器）——通过 MCP 连接真实工具

只能看到文件系统的循环是个**小循环**。Connectors 基于 **Model Context Protocol (MCP)** 让 Agent 读取你的 issue tracker、查询数据库、访问 staging API、在 Slack 发消息。

Codex 和 Claude Code 都说 MCP，所以一个工具上写的 connector 通常另一个也能直接用。

**这就是"告诉我修复方案"的 Agent 和"开 PR、链接 Linear ticket、CI 通过后通知频道"的循环之间的区别。**

**回报最快的 connectors 排序：**
1. **GitHub** — 读 repo、建分支、开 PR、评论 issue、响应 webhook。循环的最大的 Day 1 收益
2. **Linear / Jira** — 循环进展时更新 ticket，PR 链接回 issue，验证通过自动关闭
3. **Slack** — 发布排查结果，升级时 @人，早上摘要夜间运行情况
4. **Sentry / 错误追踪** — 让循环调查线上告警，对高频问题草拟修复

---

### 09. Sub-Agents（子 Agent）——让 maker 和 checker 分离

**循环中最有用的结构设计：把写代码的 Agent 和检查的 Agent 分开。**

Osmani 的说法很精准：**写代码的模型对自己太过友好，批改自己作业时总给 A+。** 另一个有不同指令（有时是不同模型）的 Agent 能发现第一个 Agent 说服自己通过的 bug。

这就是 Anthropic 2024 年 12 月工程博客中的 **evaluator-optimizer pattern** 换个名字。2026 年才火起来的词汇，18 个月前就被记录下来了。

**在 Codex：** 你定义自己的 Agent 为 `.codex/agents/` 下的 TOML 文件——name、description、instructions、可选的 model 和 reasoning effort。安全审查 Agent 可以用高 reasoning 的强模型，探索 Agent 可以用快速只读的弱模型。

**在 Claude Code：** `.claude/agents/` 下的 subagents，agent teams 在它们之间传递工作。常见分工：一个探索，一个实现，一个对照 spec 验证。

> **为什么在循环里特别重要：** 循环在你不看着的时候运行。一个你真正信任的验证器，是你能走开的唯一理由。
>
> Sub-agents 消耗更多 token（每个做自己的模型和工具工作）——把 token 花在值得第二意见的地方。

## 第三层 · 正确构建，否则别构建

### 10. 状态文件（State File）——Agent 会忘，文件不会

这是听起来太简单以至于不重要的部分，但实际上是**每个工作循环的脊梁**。一个 markdown 文件、一张 Linear board、一个 JSON 状态——任何在单次对话之外存在的东西，记录什么已完成、什么下一步。

**为什么重要：** Agent 默认记忆很短。这 session 学到的，明天就没了，除非你把它写下来。**Agent 会忘，仓库不会。** 没有持久状态的循环每次重启；有状态的循环恢复。

**状态文件存放的两个模式：**

```
Markdown 存仓库里（STATE.md 或 .claude/ 下）
→ 版本控制，diff 可读，适合单人/小团队

外部系统（Linear/GitHub Issues/数据库）
→ 跨仓库保留，可查询，团队可见，适合生产循环
```

**长运行时防止漂移：** 配一个高层次的 spec 文件（`VISION.md` 或 `AGENTS.md`），Agent 每次运行重读。**State 告诉 Agent 在哪儿，Spec 告诉它要去哪儿。**

---

### 11. 最小可用循环（Minimum Viable Loop）

如果通过了第 2 步的 4 条件测试，构建最小的循环，**不要做花哨的东西。** 四个部分，不需要 swarm：

| 组件 | 说明 |
|------|------|
| **一个 Automation** | 定时运行，在清楚条件下停止。Claude Code 用 `/loop`，Codex 用 Automation。配合 `/goal` 让它运行到条件成立 |
| **一个 Skill** | 一个 `SKILL.md`，存储 Agent 否则每次从零推导的项目上下文 |
| **一个 State File** | 一个 markdown 文件或 Linear board，记录已完成和下一步。明天的运行**恢复**而非重启 |
| **一个 Gate（闸门）** | 测试、类型检查或编译——能自动判定失败的东西。**这决定循环是在帮你还是花钱** |

**顺序很重要：**
```
先把一次手动跑可靠 → 变成 Skill → 包进循环 → 再安排调度
跳步 = 循环在生产中失败
```

**唯一值得追踪的指标：每次接受的改动成本（cost per accepted change）**——不是 token 花费、不是任务尝试数、不是调度次数。如果你的接受率低于 50%，你就在做循环本应替你省掉的审核工作。

---

### 12. Ralph Wiggum 循环——静默失败的循环

工程师 Geoffrey Huntley 记录了这个失败模式并命名。一个本应只在完成时发射完成 token 的 Agent 提前发射了，循环在只做了一半的工作上退出。**没有硬闸门，循环静默失败并继续烧钱。**

**Ralph Wiggum 循环的根源：**
- ❌ **没有真正的验证器**——只是让另一个 Agent "review"，没有客观信号。两个乐观者在互相确认
- ❌ **软完成条件**——"完成"由 Agent 的判断定义，而不是测试、编译或类型检查的结果
- ❌ **没有硬终止**——循环持续直到外部事件杀死它（限速、你注意到），而不是直到成功被验证

**修复方式就是第 11 步的 Gate**——能判定失败的客观条件：通过/不通过的测试、编译成功/失败、linter 返回 0/非 0。不是"有观点的验证器"。

**其他已知失败模式：**
- **长 session 中的目标漂移**——每次摘要步骤都有损失，第 47 轮的"不要做 X"约束可能消失。防御：Agent 每次运行重读 `VISION.md` / `AGENTS.md`
- **自偏好偏误**——写代码的 Agent 对自己太好。防御：独立的验证子 Agent，不接触制造者的推理过程
- **Agent 式懒惰**——部分完成就宣告"够好了"。防御：`/goal` 配合客观停止条件，由新模型检查

---

### 13. 理解债务（Comprehension Debt）与认知投降（Cognitive Surrender）

**这个失败模式在循环越好时越尖锐。**

- **理解债务**：循环越快地提交你没写的代码，仓库内容的实际距离和你理解之间的距离就越大。真正痛的不是 token 账单——而是有一天你需要调试一个团队没人读过的系统。
- **认知投降**：停止形成自己的判断，接受循环返回的任何东西。"设计循环"这件事——有判断力时是解药，为避免思考时是催化剂。**同样的行为，相反的结果。**

**缓解手段不是技术层面的：**
- 📖 **读 diff**——如果你不读循环提交了什么，你就在以复利借入理解债务
- 🔍 **抽检闸门**——挑几个循环开的 PR，验证批准它们的测试是否真的能捕获你关心的失败模式。**闸门会腐烂**
- 🚫 **阻止循环碰架构工作**——让循环做小型的、机器可检查的改动。一旦让它碰判断力的事情，理解债务就会加速
- 👥 **和队友结对设计循环**——另一双眼睛在设计循环时就发现盲点，否则循环会永远利用这些盲点

---

### 14. 安全税（The Security Tax）——无人值守的循环也是无人值守的攻击面

一个无人值守运行的循环 = 一个无人值守运行的攻击面。

**你的循环需要防御的威胁模型：**

| 威胁 | 说明 | 防护 |
|------|------|------|
| 未审核代码合入 | 循环开 PR 的速度快于人类阅读。没有包含安全检查的闸门（SAST、依赖审计、密钥扫描），不安全代码会自动合入 | 安全检查加入门 |
| Skill 作为注入向量 | 自动安装 community skill 会继承其描述中的 prompt 注入 | **安装前审计来源** |
| 凭据落入日志 | 长运行循环的调试日志将密钥洒在你不会监控的日志中 | 生产循环禁用 verbose 日志，清理已记录的 |
| 权限范围蔓延 | 用只读权限测试的循环被加了"一个"写权限，再也没审计过 | **每 30 天重新审计权限** |

## 把循环变成烧钱坑的 10 个错误

```
│ 1. 没跑 4 条件测试就构建                          → 大多数人至少不满足一条
│ 2. 没有客观闸门                                  → 另一个 Agent "review" = 第二个乐观者
│ 3. 一个 Agent 既写又验                            → 自己批改自己，永远 A+
│ 4. 没有状态文件                                  → 明天从零重启，而非恢复
│ 5. 模糊的停止条件                                 → "看起来不错"永远靠不住
│ 6. 没有 token 预算上限                            → 野心循环能烧 5-10x 预期 token
│ 7. 消费级套餐跑重验证循环                           → token 账单或限速，总有一个先到
│ 8. 自动安装社区 Skill                             → 17,022 个审计过的 skill 中 520 个泄漏凭据
│ 9. 循环碰判断力工作                               → 架构/鉴权/支付/模糊产品决策
│ 10. 不读 diff                                    → 以复利借入理解债务
```

## 结论：杠杆已经转移了，你的工作也是

过去两年，使用编程 Agent 的杠杆在 prompt——更好的 prompt、更好的上下文、更好的一次性输出。

**那个阶段正在结束。** Agent 已经足够好，下一个杠杆点在一层之上：**决定它们做什么、什么时候做、用什么闸门、什么状态在两次运行之间保留的系统。**

但诚实版本不是"所有人都该赶紧去构建循环"。
大多数开发者还不需要——直到任务重复、验证自动化、预算能吸收浪费、Agent 有高级工程师的工具。**缺一条，循环的成本就大于收益。**

如果你通过了测试，从小做起：1 个 Automation、1 个 Skill、1 个 State File、1 个 Gate。
- ✅ 先把一次手动跑可靠
- ✅ 变成 Skill
- ✅ 包进循环
- ✅ 再安排调度

**顺序很重要。跳步 = 花钱建一个没人理解的系统。**

杠杆点移动了。构建循环。保持工程师。

□ 有硬终止条件吗？                              → 否则不部署
