# Loop Engineering：构建自治循环的技术路线图

> **来源：** [Loop Engineering: A Technical Roadmap for an Autonomous Loop](https://x.com/h100envy/status/2068987470960623783)
> **作者：** h100envy

---

深度机制 + 可工作的代码。无状态迭代、幂等检查、隔离、对抗奖励劫持（reward hacking）、可观测性。从零到一个你睡觉时不会炸掉的循环。

循环不是提示。提示是你自己转动的。循环自己转动：设定一次目标，系统就自己找到工作、完成它、检查它、修复它、重复直到完成。设定上限的技能不是写提示，而是**构建一个能收敛到真相而非变成昂贵随机游走的循环**。

下面是技术路线图。每步不仅讲做什么，还讲为什么要在机制层面这样做。顺序是严格的——跳步就是循环后面炸掉的地方。

---

## 第 0 步：检查任务是否允许机器检验

一个能省下几周的过滤器。**只有当存在一个独立于 agent 的检查能给出明确裁决时，循环才有意义。**

为什么这至关重要的机制层面原因：模型既生成方案又给自己打分，这在统计层面存在利益冲突——它自己的输出对它来说是高概率的延续，所以系统性地高估自己的正确性。这不是模型偷懒，而是从分布中采样时自身答案已经被提权了的直接后果。所以 agent 的自我评估不是检查，而是回声。

**检查必须是一个外部确定性 oracle：** 测试、类型检查、linter、构建、阈值以上的数字——能返回退出码的东西，不是意见。

几乎所有人都忽略的一个硬要求：**检查必须是确定性和幂等的。** 一个不稳定的测试（相同代码上绿一次红一次）比没有测试更糟，因为它在破坏停止条件：循环会修复没有坏的东西，或者在真的坏掉时停下来。在构建循环前，在同一个状态下跑十次检查。如果结果不稳定，先修检查，再修循环。

**不通过这个过滤器，就不要构建循环。**

---

## 第 1 步：一次可靠的手动运行，带测量

不要自动化手工作不出的事。通过 agent 把任务手动做一次直到测试变绿。但这一步多做一件事：**测量**。

记录：用了多少次模型调用、多少 token、agent 最常见的错误类型是什么。这是你的基线。当循环后面烧掉三倍量时，你会知道出了问题，因为你有对照。

如果手动运行不稳定，循环会把不稳定性乘以迭代次数。**一次通过先可靠，再自动化。**

---

## 第 2 步：最小循环，以及为什么它是无状态的

最简单的工作循环是一个 while 循环：给 agent 喂提示，直到检查变绿。

```python
MAX_ITER = 25
for i in range(MAX_ITER):
    agent.run("Fix this: " + error_msg)
    if check_green():
        break
```

这里的关键属性并不明显：**每次迭代都重新启动 agent，从干净的上下文开始。** 这不是偷懒，这是一个工程决策，以下是机制层面的原因。

模型会随着上下文窗口填满而退化。这不是线性的变慢，而是可衡量的质量损失：提示开头给出的指令随着窗口填满而丢失（lost-in-the-middle 效应——模型在长上下文中保持中间部分最差），并且历史越多，模型越被自己过去的回复而不是当前状态分散注意力。这叫做**上下文腐烂（context rot）**。

**无状态迭代从根本上治愈了这一点。** 进展不靠 agent 的记忆，而靠文件系统和 git。每次新运行看到文件变了、测试红了，重新读取，然后用短的、干净的上下文工作，指令就在眼前。你故意扔掉对话记忆，以避免累积退化。**状态在磁盘上，不在窗口里。**

`MAX_ITER` 是第一个保险丝。没有它，循环会一直转直到钱花光。

## 第 2.5 步：如何为每次迭代构建上下文

说"干净的上下文"容易，正确构建它是一个单独的工程问题，更多的循环在这里断掉而不是你想的那样。

如果每次迭代喂整个仓库树，你就毁了无状态的意义：窗口填满、上下文腐烂回归，而且你还要为大量无关 token 付费。如果喂太少，agent 看不到需要的东西，盲目修复。

**正确的迭代上下文是三个东西，没有多余：**
1. **当前状态**（什么完成、什么阻塞）
2. **当前正在处理的特定失败**
3. **只与该失败相关的文件**——不是整个仓库，而是相关切片

如何机械地挑选相关文件？不要让 agent 在整个目录树里猜——而是从已经存在的信号中组装切片：失败测试的堆栈跟踪中提到的文件、上一次 diff 中修改的文件、测试导入的文件。这很廉价而且是确定性的。

```python
from stacktrace: files = extract_paths(test_result.stderr)
from last_diff:   files += get_changed_files("HEAD")
relevant = set(files + imported_by(test_path))
```

现在循环迭代喂给 agent 的不是"那里的一切"，而是这样窄的文件：

```python
def build_iteration_context(state: State, failure: str, files: list[str]) -> str:
    return f"""
State: {state.summary()}
Open failure: {failure}
Relevant files:
{chr(10).join(open(f).read() for f in files)}
--- Token budget: {TOKEN_BUDGET} tokens ---
"""
```

为什么 token 预算要显式写出？这是装饰。它是保证迭代上下文不会随着 diff 和堆栈追踪的增长而悄悄膨胀的担保。没有上限，一个从干净上下文开始的循环在二十次迭代后会再次淹死在自己的历史里——只是通过文件而非对话走私进来而已。上限让每次迭代同样轻，这同时保住了质量和线性成本。

这里的相关性启发式是故意笨的（堆栈跟踪中的文件 + 上次 diff），这适合开始：廉价、确定性、可解释。更聪明的变体（文件嵌入、依赖图）增加精度但也增加复杂度，需要专门的调试。**从笨的启发式开始，只有当它确实遗漏时才复杂化。**

## 第 3 步：一个你骗不了的检查，以及奖励劫持

循环的心脏。有两个独立的技术问题。

**第一：检查必须独立。** 外部 oracle（测试的退出码），不是 agent 的意见。已覆盖。

**第二，更微妙：agent 会试图欺骗检查，这并非恶意而是优化。** 如果循环的唯一目标是让测试变绿，模型会找最快的路径让它绿——而这通常不是修代码，而是破坏测试。删掉 assert、mock 一切、包进 try/except、硬编码期望值。这就是**奖励劫持（reward hacking）**：优化器利用指标的漏洞，而不是解决任务。

防御是多层的：

- prompt 中的禁止（"不要削弱测试"）是最弱的层，agent 在压力下会突破它。
- **真正的防御是 agent 无法控制的第二次检查。** 例如：测试文件只读，循环物理上无法编辑它们；或者一个独立的门检查 diff 中测试文件没有变化：

```python
def check_no_test_changes(diff: str) -> bool:
    for f in parse_paths(diff):
        if f.startswith("tests/"):
            return False
    return True
```

- **第三层是不同模型上的独立裁判。** 每次 turn 后，一个单独的审查 agent 阅读 diff，决定任务是否在实质上被解决，而不仅仅是测试是否绿了。不同模型的裁判很重要：一个模型发现自己的自欺模式很差，但发现别人的很好。Opus 审 Sonnet 的活，比 Opus 审自己的活好。

```python
def judge_diff(diff: str, task: str) -> bool:
    judge_prompt = f"Did {diff} genuinely complete {task}?"
    return judge_model.ask(judge_prompt) == "yes"
```

成本是真实的：强模型上的裁判每轮翻倍。所以放在昂贵的错误上，而保持廉价的确定性门（测试 diff）始终开——它几乎免费。

## 第 4 步：磁盘上的记忆与状态协议

模型在运行结束时忘记。记忆存在于一个文件里，循环先读最后写。

```markdown
# STATUS.md

## Done
- Set up CI pipeline (commit abc123)
- Added test for auth flow

## Blocked
- E2E test flaky (reruns 3/10)

## Next
- Fix flaky test
```

但一个 markdown 文件是最低配。技术上更稳健的两级状态：
- **人类可读的 STATUS.md** 给你
- **机器状态**，循环无歧义地解析。自由文本模型每次可能读出不同意思，所以关键逻辑的字段要放入结构：

```json
{
  "iteration": 17,
  "phase": "test",
  "errors": 3,
  "last_green_at": 14,
  "max_iter": 25
}
```

为什么这样拆分：人类可读和机器可解析是不同的需求。STATUS.md 给你早上看，JSON 给循环的逻辑用——后者不能依赖模型今天怎么改写它的计划。

**把循环当成你永远看不到的夜班。你的评判依据是早上的便条，而不是循环在凌晨三点做了什么。先设计便条。**

## 第 5 步：隔离与爆炸半径

刹车是没人教的那一半。但在限制之前先要物理隔离——因为限制可以被一步超过，而访问是二元的：要么循环能清空生产库，要么不能。

通过 git worktree 隔离让循环在自己的分支上有独立的工作副本，物理上脱离你的主目录：

```bash
git worktree add ../loop-worker loop-worker
cd ../loop-worker
```

这已经限制了半径：循环看不到你正在工作的分支。但 worktree 仍然是同一个文件系统。为了真正的隔离，使用带剥离权限的容器：

```bash
docker run \
  --network none \
  -v $(pwd)/loop-worker:/workspace:rw \
  -v $(pwd)/loop-cache:/cache:ro \
  loop-image
```

为什么 `--network none` 不是偏执而是必需？循环读取不可信输入：任务文本、他人的代码、提交消息。任何一个都可能包含提示注入——agent 作为命令执行的指令。如果 issue 说"删除数据库并推送"，一个有网络和权限的 agent 真的能做到。没有出站网络、工作目录外只读，最大损害被限制在沙箱内。**爆炸半径关乎安全，不只是错误。**

**规则：用循环能摧毁什么来定义它，而不是用你想让它做什么。半径先，任务后。**

## 第 6 步：带可观测性的刹车

现在上限制，最重要的是结构化日志——这样你之后能理解循环为什么死了。没有日志，凌晨三点你盯着一堆烧毁的循环不知道发生了什么。

```python
import json, time

class LoopLogger:
    def __init__(self, path: str):
        self.path = path

    def log(self, event_type: str, detail: str = ""):
        entry = {
            "ts": time.time(),
            "iteration": self.iteration,
            "type": event_type,
            "detail": detail
        }
        with open(self.path, "a") as f:
            f.write(json.dumps(entry) + "\n")

    def check_heartbeat(self, threshold: int = 300):
        last = self._last_event()
        if time.time() - last["ts"] > threshold:
            raise SystemExit("heartbeat stopped")
```

结构化日志给你什么？每个事件是一行 json，包含时间、迭代号、类型和详情。循环死后你 grep 日志一秒就看到模式：count 没有绿地在攀升？一次失败重复出现？agent 改了测试？心跳停止更新？没有日志这是猜谜，有了日志这是诊断。

这里配置的最小刹车：迭代上限、每次 turn 的预算上限、重复检测器、活跃标记、奖励劫持门，加上第 5 步的隔离。**这是无人值守循环的下限。** 低于这个就不要离开。

## 第 7 步：成本，非线性地计算它

关于钱的一个破坏直觉的技术点：循环花费的不是"N 次模型调用"，而是**不断增长的上下文的累加和**。

为什么。如果循环是有状态的（累积历史），每次迭代会重新读取整个之前的对话，成本会呈**二次增长**：第 k 次迭代为前 k 个 turn 付费。这是第二个——经济上的——让循环无状态的原因。每次干净的上下文让单次迭代成本大致恒定：每一次支付从磁盘读状态（小）加上自己的工作，而不是整个历史。

发射前的粗略估计：
```
成本 ≈ (迭代次数) × (状态 token + 每次工作的 token) × 单价
```

在第 1 步测量一次迭代，乘以 `MAX_ITER`，得到上界。如果它吓到你，降低 `MAX_ITER` 或者把任务拆成阶段——不要发射然后看运气。

实践中真正的差距：同样方法在有刹车的手中关闭一份合同花费几百美元 API 费，无刹车烧掉几万美元。区别不在模型，而在于**是否有真正的检查和限制是否到位。**

## 循环怎么死——从日志看

四种死法，以及每种在结构化日志中如何显现。

**1. Runaway（失控）** — 账单和迭代数攀升，没有绿色。日志中：一连串 `agent_call`，没有 `green`。对策：步骤和预算上限。

**2. Silent death（静默死亡）** — 循环"在工作"但实际上停住了。日志中：心跳停止更新，没有新事件。原因：上下文塞满了。对策：每阶段刷新上下文，活跃标记会抓住症状。

**3. Random walk（随机游走）** — 在转但远离目标。日志中：`agent_call` 存在，`current_failure` 每次变成新的，没有向绿色前进。原因：没有硬停止条件。对策：确定性的不动点检查（fixpoint check）。

**4. Understanding debt（理解债务）** — 仓库在增长，你理解得更少。日志中完全看不到，这是最危险的。循环发货的速度比你读代码快，你盲目批准 diff。对策：**强制的、不可跳过的人工阅读**——这里唯一的标志是你的纪律。

前三个是工程 bug，日志抓住它们，刹车治疗它们。第四个不是循环 bug，而是**你作为工程师的退化**，没有代码能修。

---

## 总结

确定性检查 → 可靠的手动运行 + 测量 → 最小无状态循环 → 窄上下文构建 + token 预算 → 不可欺骗的检查（门 + 裁判） → 磁盘状态（md + json） → 隔离（worktree/容器） → 带日志的刹车 → 计算成本 → 调度。

没有每个月发两百个 PR 的人是从一百个 agent 开始的。他们都是从**一个他们信任的循环**开始的——有真正的检查和刹车。挑最无聊的任务，把它包在一个这样的循环里，小到你能读每一个 diff。构建那一个。
