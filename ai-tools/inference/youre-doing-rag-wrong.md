# 你的 RAG 做错了——用"问答数据包"替代文本块

> **来源：** [You're doing RAG wrong](https://x.com/akshay_pachaar/status/2052743644411765230) — 作者 Akshay 🚀

![Cover](../image/youre-doing-rag-wrong-1.jpg)

有一种新方法能做到：
- **语料库压缩 40 倍**
- **每次查询的 Tokens 减少 3 倍**
- **向量搜索相关性提升 2.3 倍**

它不需要动你的检索算法、重排序器或嵌入模型。它修复的是上游一个几乎没人检视的结构性问题。

## 为什么"文本块"是一个糟糕的知识单元

每个 RAG 管线都建立在同一个假设上：**一段文本（chunk）是嵌入知识的正确单元**。

这个假设几乎没人质疑过——而正是它导致了人们在下游拼命修补的那些检索失败。

### 文本块的三大结构性缺陷

**1. 没有"想法边界"**
文本块是一个结构上中性的容器。它不知道：
- 它的思想从哪里开始、在哪里结束
- 它来自文档的哪个版本
- 谁有权限看到它

文本切分器只是在 token 数满了就一刀切下去。结果你检索到的是：半张表格、没有论证的结论、或者一个被剥离了语境的声明。模型根本不知道丢了什么。

**2. 版本混乱**
大多数企业语料库中，同一个文档在 SharePoint、Confluence 和 Git 上有十几个近乎相同的版本。Top-K 检索一口气返回同一段落的五份拷贝，当前版本和废弃版本混在一起。LLM 把它们揉成一个"自信的错误答案"。

**3. 没有元数据附着点**
因为文本块不携带元数据，在数据层面就没有地方挂接访问控制。权限过滤、版本状态、安全级别——所有这些最终都变成挂在编排器（orchestrator）上的逻辑代码，和它们要管控的内容完全脱节。

LangChain、LlamaIndex 和 Haystack 都在上层编排，从向量库里检索你放进去的东西。但大多数技术栈在**文档解析器**和**向量库**之间——什么都没有。

这个空白地带，就是三个问题叠加的地方。

## 更好的知识单元：问答数据包（Question-Answer Packet）

文本块之所以失败，是因为它在结构上是"无意识"的。修复方案是让知识单元在结构上变得**显式**。

**不是嵌入一段文本窗口，而是嵌入一个声明：一个问题、它的已验证答案、以及治理字段——以类型化的 schema 组织。**

一个事实一个单元，仅此而已。

你的查询本身就是问题。当你的索引存储的是**问题的答案**时，匹配变成结构性的，不只是语义性的。你不再是"希望"正确的段落浮到顶部——你是在直接把问题匹配到它的答案上。

### Blockify 的实现

**Blockify**（来自 Iternal Technologies）作为一个预处理层，实现了这种结构，称之为 **IdeaBlock**：

```
IdeaBlock {
  问题（Question）
  已验证答案（Validated Answer, 2-3 句）
  治理字段（Clearance Level, Version State, Source...）
}
```

关键洞察：**当你嵌入的是一个问答数据包而不是一段文本窗口，你的嵌入向量代表的是一个原子的声明，而不是一段恰好包含它的叙述。**

### 数据效果

在 Blockify 的内部基准测试中（17 份文档，298 页）：

| 指标 | 传统文本块 | IdeaBlocks | 提升 |
|------|-----------|------------|------|
| 查询到最佳匹配的平均余弦距离 | 0.3624 | **0.1585** | **2.29 倍** |

**反直觉的发现：数据越少，精度越高。**

很多人以为压缩语料库会损害检索效果。语义蒸馏（semantic distillation）的结果恰恰相反：

- 原始文档 → 2,042 个原始 IdeaBlock
- 去重后（80-85% 相似度阈值，3-5 轮迭代）→ **1,200 个规范化 IdeaBlock**
- 词数从 88,877 降到 **44,537**
- 蒸馏后的数据集在向量准确率上比未蒸馏的 **高出 13.55%**

冗余拷贝为什么有害：同一段落的十五个近似副本在嵌入空间的同一区域创建了十五个竞争向量。检索把概率质量分散到它们中间，拉低了规范版本的匹配分。合并成一个规范化块后，信号就清晰了。

**你的向量索引不是用来填满的硬盘，它是一个检索面——冗余会劣化它。**

## 全面的管线：从文档到 IdeaBlock

Blockify 的管线在处理流程上分为七个阶段，每阶段有明确的输入和输出：

### 阶段 1：范围界定（Scoping）
在解析任何文档之前，先定义索引层级：**组织 > 业务单元 > 产品 > 角色**。这决定了哪些块被标记到哪个访问层级，也影响后续去重的方式。

### 阶段 2：导入（Ingestion）
支持格式：DOCX、PDF、PPT、PNG/JPG、Markdown、HTML。
解析器将原始内容交给 LLM 层（运行微调的 LLaMA 3/QWEN 3.5/Gemma4 等），后者将原始块转换为草稿 IdeaBlock。
每段输入限制在 1,000-4,000 字符，实际最佳约 2,000 字符。

### 阶段 3：切分与提取（Chunking and Extraction）
上下文感知的切分：LLM 将段落转换为草稿 IdeaBlock，而不是按 token 数暴力切割。每个块的输出是一个**问答对**，不是一段散文。

### 阶段 4：语义去重（Semantic Deduplication）
- 按余弦相似度（80-85% 阈值）聚类，3-5 轮迭代
- 近似副本通过另一台微调 LLM 合并为单个规范块
- 针对 GPU 做了优化，Intel Xeon 版本也可在 CPU 上运行
- 结果：索引中的每个向量代表一个**独特声明**，而不是十五个竞争同一检索槽的副本

### 阶段 5：自动标注（Auto-tagging）
每个块获得类型化元数据：
- **安全级别**：PUBLIC / INTERNAL / CONFIDENTIAL / SECRET
- **版本状态**：Current / Deprecated / Draft / Approved
- **产品线、出口管制标志、数据隐私标签**
- 由管线自动应用，而非文档作者人工标注

### 阶段 6：人工验证（Human Validation）
产品级语料库（2,000-3,000 个 IdeaBlock）分摊给 5-10 个领域专家，每人每季度花 1-2 小时审阅自己的部分。确认的是**带来源引用的结构化声明**，而不是原始文档。

### 阶段 7：导出（Export）
验证后的块通过 API 或 JSON-L 格式推送到向量数据库。
支持的向量库：Azure AI Search、Pinecone、Milvus、Vertex Matching Engine
支持的嵌入模型：OpenAI、Bedrock、Mistral、Jina、开源模型

**这层管线独立于你使用的向量库和嵌入模型——它总在文档解析器和向量库之间。**

## 应用层的变化

你嵌入的单元决定了应用层能做什么：

### 🔹 查询构建变简单了
你的查询本身就是问题。当索引存储的是问题的答案时，匹配是结构性的而非概率性的。你不再需要调优相似度阈值来弥补查询形状与文档形状之间的语义不匹配。

### 🔹 治理下沉到数据层
基于角色的访问、版本状态、安全级别——都作为类型化字段挂在每个块上，而不是缠绕在编排器上的逻辑代码。一个销售工程师和一位法务审阅员查询同一个索引，得到不同的数据集——不是因为检索层做了过滤，而是因为**块本身携带了访问边界**。

### 🔹 更新从单条记录传播
当一份规格说明书变更时，你更新**一个** IdeaBlock。所有查询这个块的应用程序下一次请求时自动获得修正后的答案。

传统文本块的做法：同一个事实散布在数十个近似相同的段落里，跨多个文档。更新意味着要把它们全部找到——在企业级规模下，这不是一个可处理的问题。

> **架构没有改变你查询的方式。它改变了你能信任答案的程度。**

## 底层的原理

文本块是一个**解析便利**，却变成了**检索假设**。

它没有想法边界、没有版本上下文、也没有访问状态。检索栈花了多年修补这个不匹配：重排序器、混合搜索、阈值调优、提示工程——全都是下游努力，都在真正的问题之外。

**修复方案不是更好的检索算法，而是更好的知识单元。**

RAG 技术栈正在开始在解析和向量化之间生长一层"蒸馏层"，就像 Web 技术在源站和浏览器之间生长出 CDN 层一样。

你可以自己构建：用聚类 + LLM 摘要 + schema 强制校验。也可以用像 Blockify 这样专门做这件事的工具。

> **不管哪种方式，以文本块为单元这个假设就是 Bug，在数据层修复它，比调优检索层的任何努力都更值得。**
