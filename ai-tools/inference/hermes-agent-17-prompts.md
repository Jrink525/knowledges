# 17 个 Prompt 让 Hermes Agent 在你睡觉时工作

> **来源：** [17 prompts that make Hermes run while you sleep](https://x.com/mnilax/status/2063697740526399833) — Mnimiy

![封面](../../image/hermes-agent-cover.jpg)

---

2026 年 2 月，Nous Research 发布了 **Hermes Agent**：一个开源自托管 Agent，不活在 IDE 里，关闭标签页也不会忘记。它以守护进程（daemon）方式运行在你的机器上，跨会话保持记忆，能用自然语言接受定时任务，还能从经验中自己编写可复用的 skill。

作者将 Hermes Agent 看作自己的常驻基础设施跑了 **5 周**，背后挂的是 Claude 模型，跑在一台 **$5 的 VPS** 上，累积了 17 个经过实战打磨的 prompt。

---

## 为什么常驻 Agent 不同

几乎所有的 AI 工具都是**会话式**的：打开 → 工作 → 关闭 → 上下文随会话死亡。但有一类工作不是会话形状的：

- 在你醒来前应该准备好的简报
- 即使你没在看的构建任务
- 你埋头工作时不断堆积的收件箱

Hermes 填补了这个空白：**持久（记忆跨会话存活）、定时（按时钟行动，不靠你的注意力）、可达（Telegram / Discord / Slack / 邮件，不是你忘记的标签页）**。

---

## 搭建前提（3 件事）

prompt 生效前必须满足三个条件：

1. **真正有大上下文的模型** — 小模型跑多步任务时会半途丢掉工具调用。前沿模型的余量才够。
2. **空闲时不收费的后端** — Serverless 后端空闲时休眠，任务唤醒，24/7 的 Agent 才不会变成 24/7 的账单。
3. **持久的主机** — 你的笔记本电脑不行。关闭盖子会杀死凌晨 3 点的任务。$5 VPS 才是正确选择。

## 17 个 Prompt

### 1. 7 点简报（替代通知堆）

> 以前每天早上的前 35 分钟人工处理 6 个仓库的 GitHub 通知，每周 3 小时。现在它在我坐下之前已经躺在 Telegram 里了。

```prompt
每天工作日 7 点，拉取我的未读 GitHub 通知和开放的 PR，
总结每项的变化和阻塞原因，用 3-5 条发到 Telegram
```

### 2. 仓库监控（只在重要时说话）

> 周五部署后一个红色 CI 直到周一才发现，三天坏掉的 main 背后只是一个十分钟的修复。现在 ~90 秒内就能收到通知。

```prompt
监控 [org/repo]。保持沉默，除非 CI 变红或新 issue 带 "bug" 标签。
发生时发我一个失败的 job 名称或 issue 正文，其他什么都不发。
```

### 3. 多平台收件箱分类

> 横跨 6 个平台每天约 120 条消息，一条客户 DM 因为埋在 Discord 噪音下躺了两天未回。

```prompt
每小时检查我的已连接频道，按发送者和紧急程度分组，
自动归档新闻通讯和通知，只升级提到截止日期、有人在等我或涉及钱的消息。
```

### 4. 周五研究摘要（自动去重）

> ~2 小时的周五 feed 滚动 → 5 条要点阅读。去重是关键：它记得上周的内容，所以看到的真的是新东西。

```prompt
每周五 18:00，搜索 [你的主题] 的新版本和严肃讨论，
去重掉上周发过的，把 5 条要点的摘要发到 Telegram。
```

### 5. 一句"搞懂这个仓库"

> 面对陌生代码库的冷启动日，压缩成约 4 分钟的地图。不是替代理解代码，是替代不知道东西在哪。

```prompt
clone [repo url]，用 5 条总结架构，
找到入口点和最危险的那个文件，起草一个清洁的贡献工作流。
```

### 6. 晚上托付的长任务

> "不要等我，做合理假设并列出来"——这条把凌晨 2 点的停滞变成早上的就绪结果。

```prompt
研究 [问题]，对比前 3 个选项的价格、限制和锁定程度，
完成后今晚就发结果。不要等我跟进，做合理假设并写最上面。
```

### 7. 监控竞品，不刷 feed

> 你在竞品功能上线当天早上就知道，而不是客户提起时才知道。

```prompt
每天 9 点检查 [产品A]、[产品B] 的更新日志和定价页，
只在实际有变化时通知我：新功能、价格变动、功能废弃。引用精确差异。
```

### 8. 自己代码的夜间审查

> 一次把一个带 token 的 console.log 带到了生产环境，一周后才注意到。夜间扫描现在能在天亮前抓到这类问题。

```prompt
每天 23:00 查看今天所有 commit，标记所有风险：
留下的 TODO、发出的 console.log、超过 80 行的函数、变更路径没有测试。
短列表形式。
```

### 9. 自动写站会（Stand-Up）

> 你走进站会时，报告已经写好了，不需要从记忆中拼凑昨天做了什么。

```prompt
每个工作日 9:55，从我的仓库和已连接频道整理站会内容：
昨天关闭了什么、进行中什么、阻塞了什么。三个短条目。
```

### 10. 名字雷达

> 愤怒的用户和安静的 bug 报告会找到你，而不是你三天后发现。

```prompt
每天一次，全网搜索 [我的项目或 handle] 的新提及，
忽略表扬，升级 bug 报告、投诉和未回答的问题。
```

### 11. 90 分钟 talk 变 5 条

```prompt
拿 [视频或播客 URL]，拉取转录稿，
用 5 条要点+时间戳给出核心论点。跳过开头和赞助商时间。
```

### 12. "解释这个错误并提出修复"

> 一个生产堆栈，正常需要一小时二分法调试。交给 agent 后，在读完 trace 之前就收到了失败的代码行、两句原因和最小补丁。

```prompt
这是一个堆栈 trace：[粘贴]。在我仓库里搜索原因，
用两句话解释实际失败原因，起草最小补丁，不碰其他任何东西。
```

### 13. 零收件箱但不发信（等审批）

```prompt
对常规邮件（日程安排、自我介绍、状态确认），
用我的语气起草回复并放在队列里等我一键批准。
绝不自己发送。升级任何需要真正决定的邮件。
```

### 14. 值班 Agent（诊断后再呼你）

> 凌晨 3 点的告警消息附带了诊断假设，而不仅仅是红灯。子 Agent 用自己的终端拉取日志，诊断不花你的时间。

```prompt
当监控告警触发时，不要只转发它。
拉取最近 50 行相关日志，检查最近部署了什么，
发送一段首猜原因+原始告警。
```

### 15. 指向 Claude（一条命令，无锁定）

> 作者最初用廉价本地模型降低成本。小模型在多步任务中途丢失工具调用，1-14 条全部悄悄失败，因为失败的不是 prompt 而是底层模型。

```prompt
hermes config set model anthropic/claude-opus-4.8
```

一条命令就能换回来，没有锁定。

### 16. Serverless 后端：空闲几乎不花钱

> 第一个月跑在常开后端上，23 小时空闲算力的成本超过了实际工作本身的成本。Serverless 后端空闲时休眠、按需唤醒。

```prompt
hermes config set terminal.backend daytona
```

### 17. 好的 run 变成永久技能

> 早上简报的格式，作者解释了 4 遍才意识到应该永久化。Hermes 能从 run 中自己写 SKILL.md，之后只需说"跑早晨简报"就自动完成。

```prompt
刚才的跑成功了。把它保存为可复用的 skill 叫 "[名称]",
这样下次你不用重新解释格式就能用同样的方式运行。
```

## 实际数字

作者对同一周的工作做了两次记录：一次完全人工，一次交给 Hermes + 上述 prompt：

| 指标 | 人工 | Hermes |
|---|---|---|
| 非无人值守小时 | 8.3h | 1.1h（仅审批和决策） |
| 最大单次连续注意力块 | 55min | 12min |
| 早起简报时花费 | 35min | 0（7 点已在 Telegram 等待） |
| 响应延迟（生产/客户事件） | 2-24h | ~90s |

> 核心不是你节省了多少小时，而是这些工作 **不在你需要醒着的时间里发生**。

## 踩过的坑（3 个教训）

1. **模糊的调度** — "给我更新我的仓库"产生了消防水。没有升级规则，agent 报告一切。每条定时 prompt 现在都带"只在 X 时告诉我"条款。
2. **没有 token 预算** — 每小时一次的聊天分类悄悄花掉了一周的预算，因为没有任何上限。范围要匹配你实际会读的内容，首周检查花费。
3. **用廉价模型省钱** — 小模型在多步任务中途丢失工具调用，失败的样子看起来像 prompt 的 bug。**模型不是省钱的地方。**

---

## 诚实的代价

- **你是管理员了** — 更新、正常运行时间和权限模型都是你的。永远记住的 agent 必须比聊天框管得更紧。
- **它拿你的权限跑 shell 命令** — 在给它真正访问权限之前，先用 Docker 或 serverless 设置隔离。沙箱是第一天就要做的决定。
- **夸大宣传很多** — Star 数在博客文章之间剧烈波动，很多"部署指南"是联盟推广噪音。用它对**你的周实际做的事**来评判。

## 完整配置（直接粘贴用）

```bash
# 配置（一次性设定）
hermes config set model anthropic/claude-opus-4.8
hermes config set terminal.backend daytona
```

然后粘贴 14 条 job prompt + skill 固化 prompt：

```prompt
# 1. 早晨简报
每天工作日 7 点，拉取我的未读 GitHub 通知和开放 PR，总结变化和阻塞，3-5 条发 Telegram

# 2. 仓库监控
监控 [org/repo]。沉默除非 CI 变红或新 issue 带 "bug" 标签。发失败 job 名或 issue 正文。

# 3. 收件箱分类
每小时检查已连接频道，按发送者和紧急程度分组，归档通知，升级提到截止日期/人在等/钱的。

# 4. 研究摘要
每周五 18:00 搜索 [主题] 新版本和讨论，去重上周，5 条摘要+链接发 Telegram。

# 5. 仓库快速入门
clone [url]，5 条总结架构，找到入口点和最危险文件，起草贡献工作流。

# 6. 异步研究
研究 [问题]，对比前 3 在价格/限制/锁定程度上，今晚发结果。不等，做假设写最上面。

# 7. 竞品监控
每天 9 点检查 [A][B] 更新日志和定价，只有东西变了才通知。引用精确差异。

# 8. 夜间代码审查
每天 23:00 看今天所有 commit，标记 TODO/console.log/80+行函数/无测试变更。短列表。

# 9. 站会
每工作日 9:55，从仓库和频道整理：昨天关闭什么/进行中什么/阻塞什么。三个短条目。

# 10. 名字雷达
每天搜索 [项目/handle] 全网新提及，忽略赞美，升级 bug 报告/投诉/未回答问题。

# 11. Talk 转要点
拿 [视频/播客 URL]，拉转录稿，5 条要点+时间戳给核心论点，跳过开头和赞助。

# 12. 错误诊断
堆栈 trace：[粘贴]，搜仓库找原因，两句话解释真实失败，最小补丁不碰其他。

# 13. 收件箱草稿
对常规邮件，用我语气起草回复排入待批队列。绝不自己发。升级需决策的。

# 14. 值班诊断
告警触发，拉最近 50 行日志，检查最近部署，发一段首猜原因+原始告警。

# 17. 固化技能（每次好 run 后）
存为可复用 skill 叫 "[名称]"，下次不用重新解释格式。
```

## 核心思维方式

> 一个 prompt 给聊天窗口是一句**提问**。一个 prompt 给常驻 Agent 是一份**职位描述**：它需要一个触发器（定时或事件）、一个执行体（做什么）和一个升级规则（什么情况下打扰你）。缺任何一个，prompt 要不就不触发，要不做错事，要不淹没你。

> 你不再想"我想问什么"，而是想"我想把什么常驻工作从盘子上移走，以及我以什么条件听到它的消息"。
