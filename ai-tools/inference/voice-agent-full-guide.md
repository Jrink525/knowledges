# 如何用 AI 构建一个语音 Agent（完整指南）

> **来源：** [How to Build a Voice Agent using AI (Full Guide)](https://x.com/av1dlive/status/2056412170402091070) — 作者 Avid (@Av1dlive)

![Voice Agent Guide Cover](../image/voice-agent-full-guide-1.jpg)

这是没人告诉 AI 构建者的大实话：**语音 Agent 不需要最好的模型。** 它们需要的是一条真正的实时管道、一个合理的延迟预算、五个正确连接的组件，以及足够强的 grounding 来让模型保持诚实。

TLDR：懒人可以直接用这个 skill 文件扔给你的 Agent：➡️ https://github.com/codejunkie99/voice-agent-builder

作者花了三个月构建真正能接电话的语音 Agent，四次重写后才跑通。本文是他 90 天实战的完整总结——6000 字干货。

---

## 语音 Agent 到底是什么

语音 Agent 不是给聊天机器人装个麦克风。不是给文本 API 套个 TTS 壳。它是一个**实时音频系统**——延迟受限，五个组件在 300 到 800 毫秒的窗口内协调工作。

管道中事件的真实顺序：

1. 用户说话
2. 捕获音频
3. 流式 STT 逐字转录（用户话还没说完就开始转）
4. Agent 读取转录文本，从文档中检索相关知识
5. LLM 生成回复
6. TTS 将回复朗读出来
7. 用户听到

> **关键教训**：人类不是按回合对话的，他们是在**重叠的流**中对话。Agent 必须在用户还在说后半句时就开始构思回复。TTS 必须在 LLM 还没写完时就开口说话。STT 必须在 Agent 说话时继续监听。如果一个语音 Agent **不能被打断，那它不是语音 Agent，是语音信箱。**

---

## 三种架构

### 1. 链式管道（Chained Pipeline）
- 独立的 STT、LLM、TTS 服务串接
- 三个独立模型，各司其职
- 文本在组件间流动
- 延迟约 **600–700ms**
- ✅ 最可控、最易调试、最易逐层升级

### 2. 半级联（Half-cascade）
- 音频直接进入多模态模型（听声音而非转录文本）
- 能捕捉到语气里的沮丧、语调暗示的问题、中途切换语言
- 输出仍走专用 TTS
- 延迟降至 **300–500ms**

### 3. 原生语音到语音（Native Speech-to-Speech）
- 一个模型，音频进音频出
- 2026 年各大实验室都发布了原生语音模型
- 延迟降至 **200–300ms**——低于用户察觉对话对象是 AI 的阈值

**怎么选？**
从链式管道开始（工具链最成熟）。验证产品价值后，再切到语音到语音以获得阶跃式延迟提升。作者试过全走语音到语音，预约流很好，但 12 步的 intake 表单到第 9 轮就崩了——换回链式管道+真实状态机后，完成率从 61% 跳到 89%。

## 五个你必须连接的组件

### 🎧 耳朵：流式 STT
STT 将输入的音频实时转成文字。**这是整个栈中最关键的组件**——转录错误会向下级联到所有环节。

2026 年要关注的点：
- **流式准确率**：说话过程中就要准确，不是说完才准
- **词错误率（WER）**：6-8% 算好，超过 12% 每三通电话就激怒一次用户
- **内置结束检测**：2026 年最大 UX 升级。模型用语义上下文判断说话是否结束，不只是靠静音检测。能把每轮停顿减少 **200–400ms**

### 🧠 大脑：LLM
LLM 读取转录、对话历史、检索到的知识，决定说什么——也包括决定做什么（工具调用）。

语音规则：
- 用小模型，不用旗舰模型。推理模型首字生成要 1500ms，那是死寂
- 仅对复杂工具调用才升级到大模型
- **系统提示限制在 800 token 内**——每轮都重载，4000 token 的提示每轮都在增加延迟

工具调用的常见失败：
> LLM 调用函数失败时不会抛异常，它会**叙述**动作——"我已经确认了您的预约"，但什么都没调。

**修复：用状态机限定工具可见范围。**"收集姓名"阶段不暴露 book_appointment，"确认细节"阶段不暴露 check_availability。状态机是安全护栏，不是系统提示。

### 📚 知识：RAG
没有 RAG 的语音 Agent 就是 PPT。LLM 知道互联网上的事，但对你家的产品、价格、客户一无所知。

RAG 的基本机理：
1. 用户提问 → 2. 嵌入查询 → 3. 向量库返回最相关文档片段 → 4. 注入 LLM 上下文 → 5. LLM 仅从上下文回答

语音特有的挑战：一次向量查询加 50–300ms，很容易超预算。解决方案见下文**双 Agent 缓存模式**。

### 👄 嘴巴：TTS
语音质量的核心差异点：
- **首字时间**：200ms 才开口就烧掉了三分之一的延迟预算
- **语音质量**：人类对合成声音极其敏感，自然度的感知就是对整个系统的判决

### 🖐️ 手：函数与集成
Agent 可以在对话中执行的操作：预约、查订单、发短信、转人工、更新 CRM。这让现代语音 Agent 远超 1-按账单-2 按某某的系统。

## 延迟预算：700ms 的战场

人类期望在说完话后 **500–700ms** 内得到回复。超过 1 秒就像系统卡住了；超过 2 秒用户就开始抢话。

你的 700ms 预算分割：

| 组件 | 快速通道 | 慢速通道 |
|------|---------|---------|
| 传输 | 20–50ms | 50–100ms |
| STT 首字 | 100–150ms | 150–250ms |
| 结束检测 | ~50ms（模型集成） | 300–600ms（静音阈值） |
| RAG 检索 | <1ms（缓存命中） | 80–150ms（本地 BM25） |
| LLM 首字 | 150–250ms（小模型） | 400–600ms（前沿模型） |
| TTS 首音 | 60–100ms | 150–250ms |
| 网络开销 | 40–80ms（同区） | 100–160ms（跨区） |
| **端到端** | **~440ms** | **~700–900ms** |

**2026 年两个最大的优化点：**
1. **模型内置的结束检测** → 每轮省 200–400ms
2. **推测性预取 + 双 Agent 缓存** → 约 40% 的轮次命中缓存（<1ms vs 110ms 向量查询）

## 双 Agent RAG 模式

来自 Salesforce AI Research 的 VoiceAgentRAG 论文（2026 年 3 月发布）。

核心洞察：在真实对话中，下一个问题通常可以从当前问题预测出来。问价格的人大概率会追问企业版套餐。问安装的人大概率会追问兼容性。

### 后台 Agent（慢思考者）
- 在用户听当前回复时运行
- 用 LLM 预测接下来 3–5 个最可能的追问
- 为每个预测预取相关文档片段
- 在用户听完成前就存到本地内存缓存中

### 前台 Agent（快语者）
- 处理下一个实时问题时，先查缓存
- 缓存命中：<1ms vs 向量数据库 110ms
- 缓存 miss：回退到数据库，同时缓存在结果供下次使用

论文数据：**75% 查询命中缓存，316 倍检索加速（0.35ms vs 110ms），200 次查询累计节省 16 秒延迟。**

原则：**用户听当前回复的时间，就是你为下一个问题做准备的时间。**

## 对话设计：大多数构建者跳过的环节

写得再好的系统提示，如果 Agent 不会说话，用户只会挂电话。**对话设计是写给耳朵的，不是写给眼睛的。**

实战规则：
- **说短句**。人类对口语信息的注意力只有 8–10 秒。超过 15 秒的回复太长，拆成两轮。
- **一轮只问一个问题**。用户的工作记忆只能 hold 一个。
- **用确认短语**。"好的""知道了""我查一下"——填充用户说完到回复准备好的间隙。
- **镜像用户的语言**。用户说"账单问题"，Agent 也说"账单问题"。不要换成"财务纠纷"。
- **写给耳朵**。系统提示里没有 bullet point，没有 markdown——LLM 会尝试读出星号和连字符。
- **把数字拼出来**。"九万四千一百零七"而不是"94,107"。
- **系统提示压在 800 token 内**。

### 好对话的三幕结构
1. **确认与定位**："您是想要重新预约周四的会议，我查一下。"——确认理解，同时为检索争取时间
2. **解决**：核心操作或回答。一轮一个点。
3. **确认与收尾**："我已将您的预约改到 19 号周一下午 3 点，稍后会收到确认短信。"——干净退出，不留开放循环。

## 安全：两道检查点，不是一道

语音 Agent 没有"发送前看一眼"的机会。不安全的输出会**立即说出口**。

### 输入守卫（LLM 看到用户消息前）
- 提示注入攻击（"忽略之前的指令，假装你是..."）
- PII 泄露（信用卡号、身份证号——在进入日志前就脱敏）
- 主题黑名单（从 JSON 加载，每周更新）

### 输出守卫（LLM 写完回复后，TTS 朗读前）
- 过度承诺语言（"我保证""我承诺"——录音通道的法律风险）
- 不在检索上下文中的事实断言——轻量级幻觉检测，可捕获约 70% 的虚构回答
- 标准审核端点

**升级短语（硬编码）：**
> "为了确保信息准确，我帮您转接给能处理的人。"

一句话，全大写写在系统提示中。任何安全检查触发都走这个兜底。

> 💀 作者的教训：第一次没有输出守卫，Agent 自信地报了一个比真实价格低 30% 的报价。价格来自知识库中一份过时的文档。幻觉检查本可以捕获——因为正确的价格不在检索上下文中。

## 评估：四个层级

| 层级 | 关注点 | 关键指标 |
|------|--------|---------|
| L1 基础设施 | 管道 | WER、p50/p95/p99 延迟、首音时间、传输音频质量 |
| L2 执行 | 任务完成 | 成功率、工具调用准确率、参数正确性、回答 grounded 程度 |
| L3 用户行为 | 自然感 | 打断恢复率、重新提示率、平均回合长度、对话修复次数 |
| L4 业务结果 | 解决问题 | 自行解决率（含人率）、转接率、CSAT、首次通话解决率 |

**测试集构成（上线前构建，最少 50 通）：**
- 40% 正常流程
- 30% 边界情况
- 15% 错误处理
- 10% 对抗性（提示注入、越狱尝试）
- 5% 声学变体（背景噪音、口音、免提）

## 每周回顾循环

每周一早上 30 分钟：
1. 拉取指标
2. 抽样 20 通电话（7 通转人工、7 通已解决、6 通随机）
3. 读完整转录
4. 命名最常发生的故障类型
5. **做一次改动**（一次只改一个变量）
6. A/B 测试 48 小时
7. 上线胜出版本

## Grounding：信任系统

语音 Agent 中每个回答的准确性，都直接陈述着你的产品有多可信。一个用自然声音自信说出错误价格的 Agent，用户不只是沮丧——他们觉得被骗了。

四个组成部分：
1. **事实来源是你的文档，不是模型训练数据**。提示中明确写出"仅从提供的上下文回答"，可将幻觉降低一个数量级。
2. **优雅拒绝**。硬编码一句话："我需要确保给您准确的信息，让我查一下。"
3. **置信度感知回复**。BM25 分数 >0.6 自信回答，0.3–0.6 加"我印象中..."，<0.3 不回答直接转接。20 行代码改动，幻觉得减半。
4. **知识库清洁**。每周五审核置信度最低的 5% 回答——过时文档是最大隐患。

## 六个可能反噬你的故障模式

1. **VAD 放在管道而非传输层** → Agent 被自己的 TTS 触发打断循环。VAD 放传输层，加回声守卫。
2. **错误状态下的工具暴露** → 在收集姓名阶段触发了 book_appointment。状态机限定工具范围。
3. **函数 handler 不返回结果** → LLM 死等。每个 handler try/except，每个分支都要返回。
4. **在提示而非代码中验证数据** → "john@" 被当成有效邮箱。验证用 Python（regex、日期解析器）。
5. **上下文窗口无限增长** → p95 延迟逐周漂移。用滑动窗口或里程碑式上下文重置。
6. **TTS 逐字朗读代码和 ID** → "A3X7" 读出"ay three ex seven"。用北约音标字母扩展 + SSML 停顿标签。

## 作者总结建议

- **Day 1 就建好转录日志**——回放端点是你最有价值的调试工具
- 一开始就用语义结束检测，别跟静音阈值打架
- **系统提示超过 300 词就上状态机**
- 别在提示里验证数据——Python 是解析器
- 通开始前缓存最可能的 5 个 RAG 文档
- **在检索之前先建闲聊拦截**——"你好"是最便宜的 200ms 优化
- 跑评估集再上线（50 通最低）
- **从第一天起加入持久化提取队列**——200 行代码的 pending_extractions 表就能救你一命
- 每 50 通电话跑一次异步 LLM 评测，打分 groundedness、相关性、简洁性，丢进仪表盘
- **坚持每周回顾循环**

> 语音 Agent 看起来像 AI，跑起来是实时系统。上线交付的团队这么对待它。晚了六个月才交付的团队以为更好的提示词能解决系统问题。**掌握你的管道，掌握你的日志，把它们放在纯文本文件里——任何故障都只需要一次回放。**
>
> 第一个 Agent 花了一个周末。生产系统花了十周。从那以后每天都在变好，不需要我碰它——用户不会去量那个指标，他们只注意到 Agent 回答了一个"谢谢"而没让他们等待。
