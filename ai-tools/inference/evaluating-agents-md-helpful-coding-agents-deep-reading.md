# Evaluating AGENTS.md: 论文深度解读

> **论文：** Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?
> **作者：** Thibaud Gloaguen, Niels Mündler, Mark Müller, Veselin Raychev, Martin Vechev（ETH Zurich）
> **链接：** https://arxiv.org/abs/2602.11988

---

## 一、研究问题

> 给代码仓库放一个 AGENTS.md / CLAUDE.md 告诉 coding agent「这个仓库怎么用」—— 这是当下最流行的 agent 配置实践之一。但它真的有效吗？

**核心研究问题：** 仓库级上下文文件（AGENTS.md, CLAUDE.md）是否真的能帮助 coding agent 解决真实世界的软件工程任务？

作者在两种互补场景下回答这个问题：
1. **SWE-bench Lite**（300 个任务，11 个流行 Python 仓库）— 使用 LLM 生成的上下文文件
2. **AGENTbench**（138 个任务，12 个小众仓库，均有开发者手写的上下文文件）— 比较无文件 / LLM 生成 / 开发者手写

---

## 二、AGENTbench — 新基准的构建方法

### 为什么需要新基准？

已有的 SWE-bench 仓库（django, flask, sympy 等）都是流行项目，**没有开发者写的 AGENTS.md**。要评估「开发者手写 vs 无」的效果，必须找个包含真实上下文文件的仓库集。

### 构建流程（5 步）

```
GitHub 搜索 → 筛选有 AGENTS.md/CLAUDE.md 的 Python 仓库
    ↓
筛选 PR（至少 400 个 PR，修改了 Python 文件，关联了 issue）
    ↓
环境搭建（为每个仓库生成 setup/run-test 脚本）
    ↓
任务描述标准化（用 LLM 将 PR 描述重写为标准格式：描述/复现步骤/期望/实际/规格）
    ↓
生成单元测试（LLM 生成 + 人工校验，保证测试在 base 上失败、在 patch 上通过）
```

### AGENTbench 统计

- 从 **5,694 个 PR** 中筛选出 **138 个实例**
- 覆盖 **12 个 Python 仓库**（均为小众但活跃的仓库）
- 平均每个仓库 **11.5 个实例**（分布均匀）
- 测试平均覆盖修改代码的 **75%**
- 上下文文件平均长度、代码文件数等统计与 SWE-bench Lite 相似

## 三、实验设计

### 3.1 被评估的 Agent

| Agent | 底层模型 | 上下文文件位置 |
|-------|---------|--------------|
| Claude Code (Anthropic) | Sonnet-4.5 (temp=0) | CLAUDE.md |
| Codex (OpenAI) | GPT-5.2 (temp=0) | AGENTS.md |
| Codex (OpenAI) | GPT-5.1 mini (temp=0) | AGENTS.md |
| Qwen Code | Qwen3-30b-coder (temp=0.7, top-p=0.8) | AGENTS.md |

### 3.2 三种设定

1. **None** — 无上下文文件（对 AGENTbench 手动删除开发者写的文件）
2. **LLM** — 用 LLM 生成上下文文件（按各 Agent 官方推荐方式）
3. **Human** — 使用开发者手写的上下文文件（仅 AGENTbench 有）

### 3.3 评估指标

- **成功率** — patch 能否让所有测试通过
- **步骤数** — 与环境交互的总次数（shell、文件读写、搜索等）
- **推理 token 数** — 模型在推理过程中消耗的 token（针对 GPT-5.2/5.1 mini 的 adaptive reasoning）
- **总成本** — LLM 推理费用（USD）

## 四、核心发现

### 发现 1：LLM 生成的 AGENTS.md 降低成功率、增加成本

| 指标 | SWE-bench Lite | AGENTbench |
|------|---------------|-----------|
| 成功率变化 | **-0.5%** (平均) | **-2%** (平均) |
| 步骤数增加 | +2.45 步 | +3.92 步 |
| 成本增加 | **+20%** | **+23%** |

**→ 5/8 的设定中，加 LLM 生成的上下文文件后效果更差。** 花更多钱，做更差事。

### 发现 2：开发者手写的比 LLM 生成的好，但也增加成本

| 指标 | 变化 |
|------|------|
| 成功率变化 | **+4%** (平均) |
| 步骤数增加 | +3.34 步 |
| 成本增加 | 最多 +19% |

开发者手写 > LLM 生成，但提升幅度很小（4%），而成本增长显著。

### 发现 3：上下文文件不能有效提供仓库概览

很多建议都让 AGENTS.md 包含仓库的目录结构和模块说明。论文测试了一个关键指标：**agent 花多少步数才能找到与 issue 相关的文件**。

结果令人失望——上下文文件**没有显著减少**找到相关文件所需的步数（见 Figure 4），甚至在某些模型上反而增加了（GPT-5.1 mini 因为额外读了多次上下文文件本身）。

### 发现 4：上下文文件是冗余文档（关键发现！）

论文做了一个精妙的实验：**把仓库中所有文档（如 README.md、docs/ 目录）全部删除，只保留 AGENTS.md**。

结果是：

- 在没有其他文档的情况下，**LLM 生成的上下文文件成功率提升了 +2.7%**
- 甚至 **超过了开发者手写的文档效果**

**结论：** 上下文文件中的信息与仓库已有的文档高度重叠。有 README 时，AGENTS.md 是冗余的；没有 README 时，它才真正起作用。这解释了为什么有人声称加了 AGENTS.md 后效果好——他们的仓库可能本身文档很差。

### 发现 5：Agent 确实遵守了指令

Trace 分析显示：

- uv 工具在上下文文件中被提到时使用频率 **1.6次/实例**，未被提到时 < 0.01
- 仓库专用工具被提到时使用 **2.5次/实例**，未被提到时 < 0.05
- Agent 跑了更多测试、搜索了更多文件、读了更多文件

**→ 不是因为 agent 不听话。它听了，然后做更多了，但做多不等于做好。**

### 发现 6：上下文文件让 agent 更「疑虑」

GPT-5.2/5.1 mini 有 adaptive reasoning 机制——任务越难，推理 token 消耗越多。

- LLM 生成的上下文文件 → 推理 token 增加 **+22%**（GPT-5.2）/ **+14%**（GPT-5.1 mini）
- 开发者手写的上下文文件 → 推理 token 增加 **+20%** / **+2%**

上下文文件原本应该让任务更容易，实际却让模型觉得任务更难了。

### 发现 7：更强模型不生成更好的上下文文件

用 GPT-5.2 生成上下文文件，然后在其他模型上测试：

- SWE-bench Lite：**提升 2%**（有帮助）
- AGENTbench：**又降低 3%**（反而有害）

**结论：** 更强的 LLM 不一定能生成更有效的上下文文件。

### 发现 8：不同生成提示词差异不大

比较 Codex 推荐的 prompt 和 Claude Code 推荐的 prompt 生成上下文文件的效果：没有一致的优劣差异。不同模型对不同 prompt 的敏感度都很小。

## 五、深入分析：为什么上下文文件反而有害？

### 5.1 冗余信息假说

论文的分析指向一个清晰的答案：

```
有 README + 文档时：
  上下文文件信息 ≈ 已有文档（冗余）
  → Agent 还得额外花 token 读冗余信息
  → 花钱更多，效果不变或恶化

无 README + 文档时：
  上下文文件信息 = 唯一文档
  → 确实有帮助
```

### 5.2 「更多 ≠ 更好」假说

上下文文件指导 agent 做更多探索（跑更多测试、搜更多文件）。但这些额外的探索：
1. 消耗更多 token（+20% 成本）
2. 可能引入更多错误路径
3. 没有实质提升对特定 issue 的理解

### 5.3 指令过载假说

开发者写的 AGENTS.md 往往包含大量规则和约束，而不是针对具体 task 的指导。agent 要遵守这些规则，同时解决 issue，等于加了认知负担。

## 六、对 Agent Engineering 的启示

### 对 AGENTS.md 作者的建议

1. **保持简短**：只放绝对必要的信息。一句「用 uv 装依赖」比长篇架构说明更有用
2. **按需加载**：不要一次性把所有规则塞到单个文件。层级化组织（主文件引用子文件）
3. **不要替代 README**：AGENTS.md 不应该复制 README 已经提供的内容
4. **任务是关键**：好的 issue 描述比好的 AGENTS.md 重要得多

### 对 Agent 框架开发者的建议

1. **默认不使用 LLM 生成的上下文文件**（论文明确建议在改进之前不要用）
2. 上下文文件应该被「按需查阅」，而不是「随 prompt 一起注入 context」
3. Agent 应该能判断何时需要查上下文文件，而不是每次都从头到尾读完
4. 需要研究更好的自动生成方法（论文留给 future work）

### 对我们之前的 knowledges 体系的印证

这篇论文的结论和之前我们收集的几篇文章高度一致：

| 文章 | 核心观点 | 与本文关系 |
|------|---------|-----------|
| AgentForge (Mohit) | 渐进式披露 skill，按需加载 | 直接支持：一次性全注入效果差 |
| 本文 | 上下文文件增加成本、降低效果 | - |
| Agentic Engineering Hacks (vanhorn) | thin harness, fat skills | 一样：上下文越精确越好 |
| MCP Core Architecture | 工具需声明 + 按需选择 | 一样：不是把所有工具都暴露给模型 |

**三者共同指向一个结论：** 一次性把所有上下文塞给模型是有害的。更好的方式是关键上下文按需加载、渐进披露。

## 七、论文局限性

1. **仅限 Python**。Python 是 LLM 训练数据中占比最大的语言，模型对 Python 工具链已经有大量参数化知识。对 Rust、Go、C++ 等更小众语言，上下文文件的作用可能完全不同。
2. **仅任务解决率**。没有评估安全性、代码效率、可维护性等其他维度。可能上下文文件在防安全漏洞方面有作用。
3. **Agent 版本已过时**。论文在 2026 年初完成，现在（2026 年中）Agent 框架迭代很快，结论可能很快就不适用。
4. **AGENTbench 规模小**（138 个实例）。统计显著性可能有限。
5. **仅评估单次交互**。没有评估 agent 跨 session 的持续学习能力——理论上上下文文件在跨 session 时最有价值。
6. **没有考虑迭代优化**。开发者会持续更新 AGENTS.md，论文只用了某个时间点的版本。

| 指标 | 无上下文 | LLM 生成 | 开发者手写 |
|------|---------|---------|-----------|
| SWE-bench 成功率 | 基准 | **-0.5%** | N/A |
| AGENTbench 成功率 | 基准 | **-2%** | **+4%** |

| 无文档时的效果 | - | **+2.7%** | 不如 LLM 生成 |

## 九、一句话总结

> **AGENTS.md 的核心矛盾：它让 agent 更忙了，但没让它更聪明了。** 如果仓库文档已有，它是冗余的；如果仓库没有文档，它才有用。LLM 生成的更差，开发者写的好一点但有限。目前最务实的建议是：**LLM 生成的别用，开发者写的写短点。**
