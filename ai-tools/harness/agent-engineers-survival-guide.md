# AI Agent 领域生存指南 —— 什么该学、什么该建、什么该跳过

> **来源：** @rohit4verse 的 X 长文  
> **作者背景：** 两年 Agent 领域构建经验，获得多个 $250k+ 的 offer，现任职一家 stealth 公司的技术负责人

---

## 核心论点

> 每天都有一个新框架、一个新基准、一个"10x"发布。问题不再是"怎么跟上"，而是：**这里面哪些是真正信号，哪些是穿着紧急外衣的噪音？**

每个路线图发布一个月后就过时了。上周季的框架现在成遗产了。你优化的基准被人玩坏了然后被替换了。

我们习惯了走传统路径：一套栈 + 主题 + 级别，一系列工作和任期，慢慢爬。**AI 重写了那张画布。** 任何人只要有正确的 prompt 和 taste，现在就能发出过去一个 2 年经验的工程师花一个 sprint 才能做出的成果。

> **专业知识仍然重要。** 没有什么能替代亲眼看过系统崩溃、凌晨 2 点调试过内存泄漏、为一个无聊的选择 vs 一个聪明的选择争辩过而且你是对的。那种 taste 会累积。**但"知道这周框架的 API 接口"不再会累积。** 六个月内就完全不一样了。

---

## 第一部分：真正的过滤器（五条测试）

你不能跟上每周的发布。你也不应该试。你需要的是一个**过滤器**，而不是信息流。

以下五条测试在 18 个月来的验证中被证明有效。在让一个新发布接触你的栈之前，先过一遍：

### 第 1 条：它两年后还重要吗？

- 如果是模型包装、CLI flag、"Devin but for X" → 答案几乎总是不
- 如果是**原语**（协议、记忆模式、沙箱方法）→ 答案更可能是是
- 包装的半衰期很短。原语的半衰期是数年

### 第 2 条：你有没有尊敬的人在之上构建过真实的东西，并且诚实写过它？

- 营销帖不算。**事故分析帖（postmortem）算。**
- 一篇"我们在生产环境中试了 X，这里是什么坏了"值十个发布公告
- 这个领域的好信号永远是由那些为此失去过一个周末的人写的

### 第 3 条：采用它需要你扔掉你的 tracing、retries、config、auth 吗？

- 如果是 → 这是一个试图成为平台的框架。**框架试图成平台有 90% 的死亡率**
- 好的原语能插进你现有的系统，不强求你迁移

### 第 4 条：跳过六个月你付出什么代价？

- 大多数发布：什么都不付出。六个月后你会知道更多。优胜版本会更清晰
- **这条测试让你跳过 90% 的发布而不焦虑**——但大多数人不愿跑，因为跳过让人感觉落后
- 然而不是

### 第 5 条：你能衡量它是否真的帮到你的 Agent 吗？

- 不能 → 你在猜
- 没有 Evals 的团队凭感觉运行并发回归
- 有 Evals 的团队可以让数据告诉他们这周 GPT-5.5 还是 Opus 4.7 在特定工作负载上更好

> **如果整篇文章你只养成一个习惯：** 当新东西发布时，写下你需要看到什么才能六个月后相信它重要。然后回来看。大多数情况下问题会自我回答。

## 第二部分：什么值得学（持久原语）

概念。模式。结构。这些是付复利回报的想法。它们生存于模型切换、框架切换、范式切换。

### 🧠 上下文工程（Context Engineering）

过去两年最重要的重命名： **"Prompt Engineering" → "Context Engineering"** 。这个转变是真实的，不是表面功夫。

模型不再是你给它一个聪明指令的东西。而是你在每一步为它**组装一个有效上下文**的东西。这个上下文是：系统指令、工具 schema、检索文档、之前工具输出、暂存区状态、压缩后的历史——全部同时存在。Agent 的行为是**你放进窗口的上下文的一个涌现属性**。

**牢记：上下文就是状态。每个不相干的 token 都在降低推理质量。**

一个具体的方式来感受这个：拿生产中的 Agent，打开完整 trace 日志。看第一步的上下文。看第七步的上下文。数一下哪些 token 还在做工。**你第一次这么做时你会尴尬的。** 然后你去修，同一个 Agent 会明显更可靠，不换模型不换 prompt。

> 📖 **必读：** Anthropic "Effective Context Engineering for AI Agents" + 多 Agent 研究 postmortem

### 🔧 工具设计（Tool Design）

工具是 Agent 见你业务的地方。模型基于名称和描述选择工具。基于错误信息重试。基于合同是否匹配 LLM 擅长的东西来成功或失败。

- **五个命名良好的工具 > 二十个普通的**
- 工具名应该像英文动词短语
- 描述应包括"何时用"和"何时不用"
- 错误信息应该是模型可以行动的反馈：
  - ✅ "Max tokens 500 exceeded, try summarizing first"
  - ❌ "Error: 400 Bad Request"
- 有团队报告仅改写错误信息就**减少了 40% 的重试循环**

> 📖 **必读：** Anthropic "Writing tools for agents" + 工具埋点数据分析

### 🔄 编排者-子 Agent 模式

2024 和 2025 的多 Agent 争论以所有人都现在在使的合成方案告终了。

- 朴素的多 Agent 系统（多个 Agent 并行写入共享状态）灾难性地失败——错误叠加
- 单 Agent 循环能扩展到比你预期更远
- **在生产中有效的唯一一种多 Agent 形态：** 编排者委托范围狭小的只读任务给隔离子 Agent，然后综合结果

Claude Code 的子 Agent 就是这个模式。Spring AI 和大多数生产框架现在正在标准化这个模式。

> 📖 **必读：** Cognition "Don't Build Multi-Agents" + Anthropic "How we built our multi-agent research system"（看似对立实则一致）

### 🎯 Evals 与黄金数据集

每一个有可靠 Agent 的团队都在用 Evals。没有的团队都没有。这是本领域中最高杠杆的习惯，也是我看到每家公司都严重投资不足的地方。

**什么有效：** 收集生产 trace → 标注失败 → 当作回归集 → 每次犯错添加 → 用 LLM-as-judge 处理主观部分，精确匹配/程序化检查处理其余 → 在任何 prompt、模型或工具变更前运行

**用这个心理模型：** Eval 是在一切基础都在变的前提下让 Agent 保持诚实的单元测试。模型升级、框架版本更新、供应商弃用端点——你的 Evals 是唯一能告诉你 Agent 是否仍然在称职工作的东西。没有它们，你写的系统的正确性依赖于一个移动靶的善意。

> ⚡ 瓶颈不是 Eval 框架（Braintrust / Langfuse / LangSmith）。**瓶颈是首先有一组标注数据。第一天就建它，在你的任何东西扩展之前。** 前 50 个样本一个下午手动标注就够了。

### 📁 文件系统即状态 + 思考-行动-观察循环

对于任何做真实多步工作的 Agent，稳定的架构是：**思考、行动、观察、重复。文件系统或结构化存储作为真相源。每个行动可记录、可回放。**

Claude Code、Cursor、Devin、Aider、OpenHands、goose——**它们都因为同一个原因汇集于此。**

模型是无状态的。Harness 必须是有状态的。文件系统是每个开发者都已经理解的、带有状态的基元。一旦你接受这个框架，整个 Harness 训练（checkpointing、可恢复性、子 Agent 验证、沙箱执行）都源自认真对待这个模式。

> 更深层的东西：**在任何值其计算费用的生产 Agent 中，Harness 做的事情比模型多。** 模型选择下一步动作。Harness 验证它、在沙箱中运行它、捕获输出、决定反馈什么、决定何时停止、决定何时 checkpoint、决定何时产生子 Agent。
>
> 把模型换成同等质量的另一个，好的 Harness 仍然能发货。把 Harness 换成更差的，世界上最好的模型仍然产生随机忘记自己在做什么的 Agent。

### 🔗 MCP（概念层面）

不要只学怎么调用 MCP 服务器。**学它的模型。** Agent 功能、工具和资源之间的干净分离，加上可扩展的 auth 和传输层。一旦你理解它，你看到的每一种其他"Agent 集成框架"都会看起来像 MCP 的差版本，你将省下评估每一个的时间。

> Linux 基金会正在管理它。每一个主要模型供应商都支持它。现在"AI 的 USB-C"比较比嘲讽更准确了。

### 🏖️ 沙箱作为基元

每一个生产 Coding Agent 都在沙箱中运行。每一个浏览器 Agent 都遇到间接 prompt 注入。每一个多租户 Agent 都有过权限范围 bug。

**把沙箱作为原始基础设施来处理，而不是等着客户问才加。**
- 进程隔离
- 网络出站控制
- 密钥范围
- Agent 和工具之间的认证边界

## 第三部分：用什么技术栈（2026 年 4 月版）

### 编排层

| 选项 | 适合场景 |
|------|---------|
| **LangGraph** | 生产默认。约 1/3 的大公司用。typed state、条件边、持久工作流、人在环 |
| **Mastra** | TypeScript 生态首选 |
| **Pydantic AI** | 类型安全优先 |
| **Claude Agent SDK / OpenAI Agents SDK** | 在 LangGraph 节点内用于供应商原生功能 |

### 协议

> **MCP。没有其他选项。** 把你的工具集成作为 MCP 服务器。消费外部集成也一样。

### 记忆

| 选项 | 适合场景 |
|------|---------|
| **Mem0** | 聊天风格个性化（用户偏好、轻量历史） |
| **Zep** | 生产对话系统（状态进化、实体追踪） |
| **Letta** | Agent跨多天/多周维持一致性 |
| 默认 | 先用上下文窗口 + vector store，遇到失败再加 |

### 可观测性 + Evals

| 选项 | 适合场景 |
|------|---------|
| **Langfuse** | OSS 默认，可自托管，MIT 许可 |
| **LangSmith** | 如果你已经是 LangChain 生态 |
| **Braintrust** | 研究风格的严格比较 eval 工作流 |
| **OpenLLMetry / Traceloop** | 供应商中立的 OpenTelemetry |

### 运行时 + 沙箱

- **E2B** — 通用沙箱代码执行
- **Browserbase + Stagehand** — 浏览器自动化
- **Anthropic Computer Use** — OS 级桌面控制
- **Modal** — 短生命周期执行
- **绝不运行无沙箱的代码执行。** 被 prompt 注入的 Agent 触发的爆炸半径是你不想讲的故事。

### 模型

| 模型 | 适合场景 |
|------|---------|
| **Claude Opus 4.7 / Sonnet 4.6** | 可靠工具使用、多步衔接、优雅失败恢复 |
| **GPT-5.4 / 5.5** | 最强 CLI/终端推理、OpenAI 基础设施 |
| **Gemini 2.5 / 3** | 长上下文/多模态任务 |
| **DeepSeek-V3.2 / Qwen 3.6** | 成本优先、狭义明确定义任务 |

> 把模型视为可替换的。如果你的 Agent 只跟一个模型工作，那是个坏味道，不是护城河。用 Evals 决定部署什么。每季度重新评估，不是每周。

## 第四部分：什么该跳过

| 技术 | 原因 |
|------|------|
| **AutoGen / AG2 用于生产** | 社区维护，发布停滞，抽象不匹配 |
| **CrewAI 新生产构建** | 实测简单，工程师已迁移出去 |
| **Microsoft Semantic Kernel** | 除非你锁定 Microsoft 企业栈 |
| **DSPy** | 哲学价值好，但受众小众 |
| **独立代码编写 Agent 作为架构选择** | 还不是生产默认模式 |
| **"自主 Agent" 推销** | 2023 年的概念，行业已转向"监督式、有界、可评估" |
| **Agent 应用商店/市场** | 企业不买。企业买垂直 Agent 或自建 |
| **水平"建任何 Agent"企业平台** | 目前混乱、慢、买 vs 建仍倾向自建 |
| **SWE-bench / OSWorld 榜单追逐** | 所有公开基准可被博弈 |
| **朴素并行多 Agent 架构** | 五个 Agent 在共享内存上聊天看起来震撼，生产中崩溃 |
| **按座定价** | 市场已转向成果/使用量计价 |
| **你在 Hacker News 上看到的下一个框架** | 等六个月。如果还重要，你会知道。如果不，你省了一次迁移。 |

## 第五部分：怎样实际前进

不要试图"跟上"Agent 技术。用这个序列来采用它们：

1. **选一个已经重要的成果。** 不是登月，不是水平"Agent 平台"项目。可衡量的、你的业务已经在意的。支持工单分流、第一版法律审查、营销线索资格认证……Agent 在它成功时成功。**这成为你 Day 1 的 Eval 目标。**

2. **在发任何东西之前设置 Tracing 和 Evals。** Langfuse 或 LangSmith。手动建一个小黄金数据集（50 个标注样本就够了）。你能改进的东西永远只有你能衡量的东西。

3. **从单 Agent 循环开始。** LangGraph 或 Pydantic AI。Claude Sonnet 4.6 或 GPT-5。三到七个设计良好的工具。文件系统或数据库作为状态。发给小受众。看 trace。

4. **把 Agent 当产品不是项目。** 它会以你预料不到的方式失败。那些失败就是你的路线图。通过真实生产 trace 建立回归集。

5. **只有当你挣到家时才增加范围。** 子 Agent？上下文是瓶颈时。记忆框架？单窗口上下文不够时。浏览器使用？底层 API 真的没有时。**让失败模式拉你进入，而不是预先架构。**

6. **挑无聊的基础设施。** MCP 工具。E2B 或 Browserbase 沙箱。Postgres 状态。你现有的 Auth 和观测栈。

7. **从第一天起看单位经济学。** 每次行动成本、缓存命中率、重试循环成本、模型调用分布。Agent 在 PoC 时看起来便宜，到 100x 规模会爆炸……除非你从一开始就工具化成本/成果。

8. **每季度评估模型，不是每周。** 锁定一季。季末用 Eval 套件跑一遍当前前沿，数据说话。

## 第七部分：反常赌注——在没有梯子的时候让作品自我介绍

你不需要学所有东西。你需要学那些能累积的东西，跳过不累积的。

我们大多数人是在一个假设世界非常稳定时被教育的。上学、拿学位、爬梯子。那个梯子假设了另一端的稳定行业。

Agent 空间目前没有稳定另一侧。你可能想为之工作的公司成立才 6 个月。他们构建的框架才 18 个月。底层的协议才 2 年。一半被引用最多的帖子是三年前不在这个领域的人写的。**没有梯子可爬，因为建筑在不断换楼层。**

当梯子失效时，剩下的是旧得多的办法：**做个东西，放上互联网，让作品介绍你。** 它是反常赌注，因为它绕过了证书系统。但它也是在一个移动的领域中唯一会累积的东西。

> **你需要在这个时刻发展的技能不是 "Agent"。** 是判断在一个表面不断变化的领域中哪些工作会累积的纪律。
>
> **累积的：** 上下文工程、工具设计、编排者-子 Agent 模式、Eval 纪律、Harness 思维方式。
>
> **不累积的：** 知道周二发布的框架的 API。
>
> 一旦你能区分它们，每周发布的潮水不再是压力，而是你可以忽略的噪音。
