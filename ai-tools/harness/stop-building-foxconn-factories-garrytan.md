# 停止为你的 Agent 建造富士康工厂

> **来源：** [@garrytan](https://x.com/garrytan) | Y Combinator CEO
> **阅读：** 264K+ 浏览 | 679 赞 | 1,462 书签

---

2026 年 1 月，Garry Tan（YC CEO）重新开始编程，用 Claude 写了 **54 万行 Rails 代码** 和配套的测试，构建了一个叫 Garry's List 的项目。他一度为此自豪——直到意识到真正值得自豪的不是那个应用本身，而是 **为了构建它而诞生的另一套东西**：GStack。

这篇文章是他用 54 万行代码换来的领悟：**大多数人在用构建富士康工厂的方式构建 AI 系统。你应该停下来。**

---

## 时间旅行者的陷阱

Garry 说他做的事情本质上是在完美扮演一个**时间旅行者**——一个 2013 年的 Web 2.0 工程师（他最后一次做真正的软件工程师）被传送到 2026 年，用 2026 年的 AI 工具写代码。

2013 年的工程师骨子里相信一件事：**能力 = 代码行数**。这个信念在过去几十年一直正确——直到现在。给他 Claude Code 或 Codex，他能做出 100-1000 个工程师的工作量。

问题在于：**工具升级了，但思维模式还停留在 2013。**

这几乎就是现在所有人的现状。它不像是陷阱，因为代码确实跑起来了——Garry's List 成功上线了。但那是**为过时理念服务的生产力**。

---

## 经济学翻转：LLM 变便宜了，代码变贵了

**旧经济学（直到 2025 年）：** LLM 调用贵，代码便宜。所以你写大量代码来约束模型、小心地控制每次调用。架构的设计目标是**减少模型调用**。

**新经济学（现在）：** 模型越来越便宜，且智能程度高到价值-成本比例已经翻转。同时模型**能自己写可用的代码**了。

> 你不再需要写代码来"看管模型"了。

这就是 **即时软件（Just-in-time Software）**，我们正在进入它的黄金时代。工件形态完全变了——54 万行的 Rails 应用被替换成基于 markdown 和少量代码构建的 agent，代码量只有原来的一小部分。

> 我们一直在写代码去看管一个现在比代码还聪明的东西。

---

## 富士康工厂内部：你代码库里的"安全网"

如果你最近在写 AI 代码，你很可能正在不知不觉地建造这种工厂。走一遍自己的代码库，数一数有多少行代码只因为你不信任模型而存在。

**Garry 的 54 万行分解：**

| 类别 | 行数 | 用途 |
|------|------|------|
| 应用代码 | 262,000 | 实际功能 |
| **测试** | **276,000** | 用来"看管"应用 |
| 后台任务 | 127 个 | 其中 33 个是 cron |

那个审计委员会比公司还大。一个 1,778 行的文件的唯一作用是**质疑模型生成的事实**——把每个声明分发给 5 个独立搜索，比较结果，如果不一致就标记。

> 当你用 2023 年的富士康工厂方式构建软件时，你建了一个**笼子**。如果不小心，你就会变成维护这个 AI 监狱的狱警。

## Markdown 就是新的程序

Garry 强调：他说的 **不是"提示词（prompting）"**。提示词是临时的——你打了些字，得到结果，然后就蒸发了。他说的是**构建**——版本化、可测试、可复用的构建。

新范式三层架构：

```
┌─────────────────────────────┐
│   Markdown 指令层            │  ← 意图、技能、判断（可测试的）
├─────────────────────────────┤
│   TypeScript 薄确定性层      │  ← 真正需要编码硬逻辑的部分
├─────────────────────────────┤
│   LLM 引擎                   │  ← 模型，驱动一切
└─────────────────────────────┘
```

最关键的是：**你要像测试代码一样测试 markdown。**

### Skillify 流程

当你用 agent 构建了某个功能，直到它正常工作后，说一句 **"skillify it"**。Agent 会自动产出：

1. **Markdown skill** — 技能的指令层
2. **最小化的代码** — 真正需要硬编码的部分
3. **代码的单元测试**
4. **Skill 的 LLM eval** — 用 LLM 判断 skill 是否按预期执行
5. **跨层的集成测试**
6. **Resolver** — 让 agent 在相关场景自动调用该技能
7. **Resolver 的 eval**

这个"技能包"（Skill Pack）就是一个**可复用的能力单元，持续叠加**。测试是魔法——对 skill 的覆盖率让它可以修改而不断裂。这是它和"vibe coding"的本质区别。

> 我们现在正在实时摸索 agentic engineering 的系统原语，就像早期 CPU 时代发明了栈、堆、寄存器和冯诺依曼架构一样。我觉得 Skill Pack 可能就是这个领域的类似物。

## 实际能做什么——不是一个理论

Garry 举了 Hackathon 评审的例子：
- 85 份参赛作品
- 上传 Google Drive 目录，说"go"
- Agent 分析了每个仓库的代码质量、做了深度代码审查、运行了评估、对视频进行了配音文字转写、看屏幕对比输出
- 在 4 分钟引擎轰鸣中产出了完整的排名

他没有写代码。他用 OpenClaw 完成任务并指导它，完成后说 skillify it，于是变成了一个 tarball——**任何人可以在任何 hackathon 上永远复用的能力。**

> 原本是一个真正的软件项目（爬虫+评分流水线+视频处理+研究模块+排名系统），变成了一次"对话→指导→skillify"的流程。

作为花絮：hackathon 的冠军后来建造的代码被 Garry 打磨后合入了主干——GStack 现在可以同时在模拟器和真实设备上测试 iOS 应用。

## Tokenmaxxing：入场费

几乎没有人愿意付出的入场费：**愿意在 token 上花钱。**

Peter Steinberger（OpenClaw 作者）说过他愿意每年花大约 100 万美元的 token 来做 OpenClaw。大多数人听了会退缩——但他们不应该。

OpenClaw 是一个"你自带扳手的法拉利"。模型是引擎，不是车。我们还在 Apple I 的时刻——焊接面包板。它出厂粗糙，你需要自己完成剩下的部分。

> 有人说 OpenClaw 不安全。他们不明白：**自由的粗糙也是它如此强大的原因。** 你不会在信任一个东西之前就给它装上安全护栏。

这就是 OpenAI 决定给每个 YC 公司提供 200 万美元的 **token credit 形式的 SAFE（不设上限）** 的原因。当你可以把原始智能转化为 token，再转化为软件，而成本曲线在一路向下的时候，神奇的事情就发生了。

**成本预测（Garry 的估算）：**

```
今天 $100K 的 token
明年 $10K
后年 $1K
再后年 $100
```

> 你可以花几十万美元一年（有些人更少），在 2026 年就活在其他所有人几年后被迫活的世界里。

唯一阻挡你的是 2013 年的那套本能——认为模型调用太贵了。它不再是了。

## Esalen，不是富士康

如果 54 万行控制代码给 worker 建了一座富士康工厂，解药是建一座**相反的东西**。

在 Big Sur 的悬崖上有一个叫 Esalen 的地方。人们去那里"被拆解和重建"，卸下盔甲后更真实地回来。没有流水线，没有工头，没有早上 6 点的哨声。**自由。**

> 建造 worker（无论是人类还是 AI）可以自由而不是被奴役的地方。

这就是整个精神的核心：**让 agent 能自由做事的地方，让人类能自由发挥。** 在知识工作中，工厂就是失败模式。解放人们的机构胜过囚禁他们的机构。

### 两种选择

| | 控制系统 | 自由系统 |
|--|---------|---------|
| 风格 | 精致的 | 粗糙的 |
| 原因 | 控制需要全面控制 | 信任你去完成它 |
| 比喻 | 富士康工厂 | Esalen 度假村 |
| 对 agent 的态度 | 怀疑、看管 | 信任、赋能 |

**选你要建哪一种。然后看看你的代码里有多少行在说这是哪一种。**
