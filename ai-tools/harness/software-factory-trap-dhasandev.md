# The Software Factory Trap — 软件工厂陷阱：Agent 时代最有深度的工程思考

## 一、核心命题：生成代码不难，难的是生成「对的」代码

生成代码已经不难了。一个好模型可以写 diff、搭建服务、写测试、解释 API、迁移组件，而且能一直干下去，比人类打字耐力强得多。这部分基本已经解决了。

**难点在于生成「对的」代码。**

对的代码不是指能编译通过的代码，而是：
- 符合需求的
- 尊重架构的
- 处理了业务约束的
- 通过了真正重要的测试的
- **换了下个月上下文之后依然可理解的**

这个差距就是**软件工厂陷阱**的藏身之处。要理解这个陷阱，首先要理解代码本质是什么，以及它省略了什么。

---

## 二、代码只是表面（Code Is The Surface）

Conal Elliott 在 Haskell Foundation 的访谈中提出了一个精妙的比喻：**程序是更大物体的表面。**

可执行程序不是完整的产物。完整的东西包含：规格说明、推导过程、意图、正确性论证、权衡取舍、以及走过概念空间最终产生代码的那条路径。

代码之所以高效，**正是因为它剥离了所有这些**。运行时不需要知道那个权衡是在什么会议上做出的。不需要 Slack 线程里客户澄清的那个边界情况。不需要让一个行为可接受而另一个不可接受的产品判断。运行时不需要这些。

**所以我们把它们拿掉了。**

但是，程序需要这些东西才能被理解、被扩展、被安全地修改——只是不需要它们来运行。于是，赋予代码含义的上下文悄无声息地消失了，只留下可执行的表面。

在大部分软件历史中，这还可以管理。有经验的工程师把缺失的上下文装在脑子里，或者写在没人更新的文档里。他们知道哪些对象重要、哪个源头说了算、哪种证明有效、流程在哪里会断、哪个捷径是假的。他们在私下完成本体论和认识论的工作，只是不这么叫而已。这就是「资深工程师」的含义。

**问题是：私有知识无法在 swarm 中规模化。** 部落知识不会因为存在于组织的某个角落就能被软件工厂使用。

> 必须在工作发生的时刻捕获——在决策、评审意见、设计文档、客户对话、trace 中——这样 agent 需要的时候才能读到。手动复盘文化同样无法规模化。捕获必须成为工作流的原生部分。

## 三、工厂不是 Agent（The Factory Is Not The Agent）

当一个 agent 完成一个任务后，你通常得到的是：

> 一个 diff → 一段总结 → 一个成功率声称

而实际上你需要的是：

> 这个输出依赖了哪些源头 → 通过什么证据证明 → 这些证据对应哪些需求 → 源头变化时什么会失效 → 下一步需要谁的判断

第二个记录不是装饰。它让任何人（人或系统）都能验证工作，而无需从头回放整条轨迹。

**没有它，review 就变成了重构。** 有人必须从输出中重建完整的意图-上下文-实现-证明链。这比 agent 节省的时间还要长。

错误在于认为工厂就是 agent。不是的。

一个新工程师不是仅仅因为聪明就有用的。他们需要入职、目标、仓库权限、所有权边界、权限、工单、评审循环、升级路径和「完成」的定义。**智能是必要的，但不充分。它必须被组织成劳动。**

Agent 也一样。当前这类 coding agent 能读文件、编辑代码、运行命令、检查失败、有时还能修复自己的错误。工具是真的。它们缺少的不是另一个工具，而是**周围的结构**：哪些上下文源头是权威的、什么权限适用、如何证明工作、以及输入变化时什么会变陈旧。

**没有这个结构，失败模式是可以预测的：** Agent 写代码，抽象层搞错了，实现通过了狭窄的测试但违反了产品意图，人类工程师花更多时间 review、回退、重新解释——比自己直接干还累。

> 工厂应该产出大于浪费的价值。很多自建工厂做的恰恰相反——它们自动化了代码生产，却把需求、证明、上下文和评审留作人类的「重构工作」。

## 四、团队形态正在改变（The Team Shape Is Changing）

这个问题还有第二个维度。

Agent-heavy 的工程不总是看起来像一个正常的工程师团队坐在经理下面、在同一个代码库上协同工作。在一个创始人的实际案例中，他描述的方式接近于**拿破仑式计划**：目标在顶层设定，每个工程师拥有一个领域，然后每个工程师带着自己的 agent 工人军团去完成自己那部分。

这个模式是有道理的。如果一个工程师能有效指挥多个 agent，这个工程师就不像是一个独立贡献者，更像是**一个拥有小军队的领域所有者**。执行变得廉价且本地化。每个领域都能快速进化。

**但没有共享的教义，这就变成了碎片化。**

共享层变得更重要，而不是更不重要：契约、API 协议、证明标准、源头权威、架构边界、评审规则。

更难的版本是**跨工厂协调**。两个工程师各自有 agent 在代码库不同深处操作，然后发现他们碰到了同一个抽象层或者做出了不兼容的假设。大多数 agent 系统通过人类来路由这种冲突：agent → 人，人 → 人，人 → agent。这对短循环有效。当 agent 已经运行了几个小时、相关上下文在计划下几层抽象时，它就不行了。

一个为这种环境构建的软件工厂需要一个**通信层**，让 agent 可以暴露冲突、比较源头权威性、提议共享契约、然后只升级那些真正需要人类判断的决策。

> 目标是节省人类注意力，不是从架构中移除人类。

## 五、证明也需要证明（The Proof Also Needs Proof）

跳出陷阱需要两样东西配合工作。大多数软件工厂设计两者都没有。

### 5.1 本体论（Ontology）

本体论告诉系统存在什么以及它们如何关联。在这个语境下，意味着工厂有一个关于重要对象的工作模型：

- **需求** → 来源 → 利益相关者 → 任务
- **产物** → 权限 → 测试 → 证明声明
- **产品表面** → 依赖 → 决策

它知道这些东西如何关联：需求可以支持任务，来源可以授权需求，测试可以支持证明声明，来源变化可以使产物失效。

> 这不是为了哲学而引入的哲学。这是资深工程师已经在隐式做的事情。他们在接触系统之前脑子里先画了一张地图。

### 5.2 认识论（Epistemology）

认识论告诉系统关于这些事物的**声称**如何变得可信。这是大多数工厂设计失败的地方：它们跟踪工作但不跟踪证明。

- 「API 返回了 200」是一个声明——可以通过运行时 trace、测试、日志或遥测记录来知道。**但它不能证明用户能完成 onboarding。** 那是另一个声明，需要不同的证据。
- 「这个输出已过时」是另一个声明——取决于产物是否使用了变化的源、该源是否被认定为权威上下文、以及变化是否足够实质性地使结果失效。

**一个声明直到证明类型匹配声明类型才有效：**
- 截图 ≠ 持久状态的证明
- 通过后端测试 ≠ 用户旅程的证明
- 干净的总结 ≠ 生成的工作仍匹配源头需求的证明

### 5.3 本体论 × 认识论

本体论给了工厂工作地图，认识论给了它证明规则。两者结合，把「agent 做了某事」变成：

> **在这个约束下，用这个证据，这个声明成为了真，以下情况会使它过时。**

当两者被编码到工作流中，工厂可以在人类看到之前验证更多自己的工作。

作者分享了内部教训：一次运行产生了数十个契约、依赖边、清单和可审查的收据。机械部分工作了。但产品成果比机器被迫证明的范围更广。后端契约通过了，端到端的用户声明仍未证明。工厂为**错误的对象**生成了令人印象深刻的大量证据。

> 这就是工厂陷阱内部的**证明陷阱**：一个系统看起来严格，却在证明错误的东西。

## 六、公共数据到底说了什么

作者整理了几项关键研究，说明它们共同反映的模式，而非选择性引用：

| 研究 | 发现 | 解读 |
|------|------|------|
| **Microsoft Research**（Copilot） | 完成有限 JS 任务快 55.8% | 边界清晰、证明面干净的任务——工厂机器问题尚未出现 |
| **GitHub 代码质量研究** | Copilot 用户通过全部 10 个单测的几率高 53.2% | 同上 |
| **METR 2025 初** | 16 位开发者、246 个真实 Issue，AI 使任务**慢 19%** | 真实代码库中的复杂问题——管理上下文增加了开销 |
| **Stack Overflow 2025 调查** | 更多开发者不信任 AI 准确性 | 最常见的挫败感：「几乎正确，但不太对」 |
| **Harness 2026 DevOps 报告** | 47% 高频 AI 用户说下游手工 QA/Code Review 变得更严重 | 代码快于验证能力时，瓶颈转移到下游 |
| **DORA 2025** | AI 是放大器：强队更强，弱流程更快变弱 | 组织维度 |

模式很清晰：**不是 AI 编程工具不行，而是更快的代码生产暴露了更慢的验证系统。** 衡量 time-to-diff 看起来很好，衡量 time-to-trusted-outcome 看起来更糟。

## 七、自建软件工厂的真实案例

作者与一位创始人的对话中看到了真正的实践。该团队已经跨越了「agent 能不能写代码」的争论阶段：

1. 大型需求文档 → 拆成技术规格
2. 规格 → 工作分解结构（WBS）
3. 工作交给递归 agent 工人
4. 输出包裹在属性和契约测试中
5. 结果对照原始需求评审
6. 源上下文变化时，**重写受影响的代码**而不是修补陈旧的部分编辑

这是一个真正的软件工厂。但它也是**很大的一个拥有成本**。

那位创始人传递的关键信号不是「我们喜欢 AI」，而是：**我们已经建好了内部机器，我们知道为什么它重要，我们宁愿为维护版付费也不长期拥有这套自定义机器。**

这打破了关于 AI 的一个流行叙事——每个公司都会生成自己的软件，停止为工具付费。现实更复杂：当机器变得痛苦、变化快速且非核心时，一些团队寻找维护层，而不是无限期拥有它。

## 八、旧流程，新运行时（Old Process, New Runtime）

这里有一个重要的历史平行。

很多老的工程纪律失败不是因为想法错了，而是因为**执行层是人类**。

HATEOAS（超媒体作为应用状态引擎）和 CORBA（公共对象请求代理架构）等理念源自一个真实问题：让分布式系统更可理解、可导航、一致。自描述接口、标准化的 API 契约、可审计的正确性论证。技术价值是真实的。

协调成本是残酷的。让大型组织同意 API 标准——要协调数百名工程师、数十个团队、不同的历史偏好和本地激励。

**Agent 改变了这个权衡。**

一个 coding agent 没有关于 API 风格的偏好。它不需要在架构会议上被说服。它不会厌倦维护 trace 链接。它只需要：
- 契约用可读形式给出
- 操作用可执行形式给出
- 证明义务用可检查形式给出

> 以前需要组织强制执行的东西，可以变成工作时刻的执行上下文。

这意味着一些以前流程繁重的想法，当被转变为**运行时输入**而不是人类强制策略时，会变得实际。如：自描述操作、Schema、接口契约、源头权威记录、需求与实现之间的 trace 链接、agent 可以在工作中消费的测试（而不是事后审计）。

## 九、更多上下文不是答案（More Context Is Not The Answer）

对以上所有问题的直观反应是：**给 agent 更多上下文。** 连接 Slack、Google Drive、Linear、仓库，给它每个会议记录、每个工单、每个 trace、每个文档、每个评论，让它自己搞定。

有时这有帮助。**但常常会创造一个新问题。**

- 如果 agent 使用一个小型、经批准的上下文包 → 系统能说清输出依赖了什么 → R17 变化时可以回答什么工作变旧了
- 如果 agent 读了半个仓库、40 个文档、旧 Slack 线程、随机 trace 和五个过时的规划笔记 → 系统只能回答：**「可能依赖一切」** → 等于无用

> **问题不是上下文数量，而是上下文的形状。**

代码库本身作为主要上下文来源是有问题的：它给出了之前拼图的最后一块，但没有可靠地解释为什么那些块是那个形状、什么权衡产生了它们、哪个客户承诺被保护、哪个捷径不可接受。

**这就是上下文图变得有用的地方——但只有当我们精确理解什么使它有价值：**
- 不是因为它包含一切
- 而是因为它记录了**什么重要、谁/什么授权了它、什么依赖它、以及什么在变化时会过时**

一个有上下文图不够，还需配合工作流设计。图坐在数据库里不会自动让 agent 变得更好。agent 需要知道读哪个切片、哪些边是权威的、哪些声明是过时的、两个源冲突时谁赢。

## 十、同步与异步工作

不是每个任务都应该通过相同的工厂流程运行。

### 异步任务（适合长时间自主执行）
- 后端迁移、API 变更、基础设施重构、内部系统变更
- 🟢 前提：需求完整、证明面强、对人 taste 的依赖最小
- 工厂的职责：让自主性可信，而不是监督每一步

### 同步任务（需要人工协作）
- 产品面 UI、交互设计、onboarding 流程、文案、用户体验
- 🟡 本体论和认识论仍然有用——让协作更精准
- 但证明不同：用户体验不能简化为「处理通过了」，需要人类对清晰度、信任和感觉的判断

**错误在于强迫每个任务进入一种模式。**

一个有用的软件工厂知道区别：用异步任务处理证明面强的工作，用同步协作处理证明依赖人类判断的工作。

## 十一、陷阱本身（The Trap）

> 软件工厂陷阱是：建了刚好够让 agent 有用的机器，然后发现机器本身变成了产品。

你要维护：
- 提示词
- 技能
- 上下文管道
- 文档
- 评审仪式
- 测试约定
- Agent 权限
- Trace 导出
- 把一切粘起来的脚本

每一个模型变化、工具变化、工作流变化、团队习惯变化，都迫使工厂也跟着变。

在小规模时，这很有趣。在公司规模时，它变成了**第二个没人预算过的代码库**。

---

## 十二、真正需要的是什么（What This Actually Requires）

回到 Conal Elliott 的框架。

> **程序是更大物体的投影。代码是影子。**

在它后面有规格说明、推导过程、正确性论证、意图、概念空间的路径。运行时不需要这些。但是扩展工作、纠正它、或者知道它是否应该存在——这些需要访问更大的物体，而不仅仅是它的影子。

这就是不同工程师接触同一代码库的区别：

| 层级 | 看到什么 | 做什么 |
|------|---------|--------|
| **初级工程师** | 文件 | 处理眼前的东西 |
| **高级工程师** | 能重构足够的上下文 | 理解代码在做什么以及为什么 |
| **Staff 工程师** | 问不同的问题 | 这是正确的东西吗？这连接了什么？改变这个会怎样？ |

Staff 工程师不是在代码上操作。他们是**在它背后的更大物体上操作**。

当前的 coding agent 主要在初级到高级的高度运作。它们不能可靠完成的是 **staff 工程师的移动**：退一步问框架是否正确、工作真正连接了什么、它是否应该存在。

这段高度差距就是保姆税的藏身之处。不是 agent 写不了代码，而是每个输出仍然需要一个人类来执行 agent 跳过的 staff 工程师检查。

**跑得最快的团队不是那些生成最多代码的，而是那些工厂能更接近 staff 工程师高度的。**
- 需要推导，而不仅是 diff
- 需要正确性论证，而不仅是通过的测试
- 需要产品意图，而不仅是一个规格切片
- 需要与业务现实的连接，而不仅是一个工单

当这些上下文被编码到工作流中，agent 就能推断之前需要人类推断的事情：发现需求与现有架构决策矛盾、标记证明面在证明错误的层次、注意两个 agent 工人正在计划下三层做出不兼容的假设、在代码还没写之前就问——**这个工作是否应该存在？**

> **那不是更多的上下文。那是正确高度的上下文。**

## 总结

| 核心观点 | 一句话 |
|---------|--------|
| **代码 ≠ 产品** | 代码是更大物体（规格、意图、权衡）的表面投影 |
| **工厂 ≠ Agent** | Agent 需要周围结构（权威源、权限、证明规则、过时检测）才能可信 |
| **团队变了** | 工程师变成领域所有者 + agent 军队，共享协调层比以往更重要 |
| **证明也需要证明** | 本体论（有什么 + 怎么关联）+ 认识论（怎么知道可信）= 可验证的工厂 |
| **数据警示** | AI 代码加速可能暴露更慢的验证系统；瓶颈转移到下游 |
| **旧流程新运行** | Agent 让以前人类组织无法维持的纪律（契约、trace、证明义务）变得可行 |
| **上下文量 ≠ 质量** | 上下文形状 > 数量；小型明确包优于大海捞针 |
| **同步 vs 异步** | 强证明面→异步；依赖人类判断→同步协同 |
| **陷阱** | 工厂变成第二代码库，你需要判断是否应该买而非自建 |
| **Staff 高度** | 让工厂能访问「更大的物体」，agent 就能做 staff 工程师的检查 |

> 软件行业一直知道人类会在工具处理了低级工作后向上移动抽象层。不太明显的是：向上移动需要更大的物体也跟上来。你无法可靠地把工作委托给只有影子的系统。
>
> 我的押注不是更多 agent 能解决这个问题。我的押注是：**围绕 agent 建立的工厂——那些保留了推导、正确性论证、产品意图和通常困在 staff 工程师脑子里的专家判断的工厂——才是持久的杠杆所在。**
>
> — danialhasan
