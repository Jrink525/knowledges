# AI Agent 学习路线图 2026：什么值得学、什么值得造、什么直接跳过

> **来源：** [What to Learn, Build, and Skip in AI Agents (2026)](https://x.com/rohit4verse/status/2049548305408131349)
> **作者：** Rohit (@rohit4verse) — 在 agent 领域深耕两年，年薪 250K+ 多份 offer，现为 stealth 公司技术负责人
> **日期：** 2026-04-29
> **数据：** 1686 ❤️ · 6571 🔖（社区书签数显示这是被疯狂收藏的经典文章）

---

## 核心论点

> **每周都有新框架、新 benchmark、"10x" 发布。问题不再是"怎么跟上"，而是：什么是真正的信号，什么是披着紧迫外衣的噪音。**

Rohit 基于两年实战提出的筛选框架：

**如果你只养成一个习惯：** 当新东西发布时，写下"六个月后我需要看到什么才能相信它重要"。然后回来检查。大多数时候问题会自己回答你。

---

## 一、真正有效的过滤器：5 道测试

在投入任何新东西之前，用这 5 个问题过一遍：

**1. 两年后它还重要吗？**
- 如果是前端模型的 wrapper、CLI 参数、"Devin 但换了个领域"→ 几乎不重要
- 如果是**原语**（协议、内存模式、沙箱方案）→ 更可能重要
- Wrapper 的半衰期很短，原语的半衰期是几年

**2. 有你尊重的人基于它构建了真实的东西并写下了诚实的体验？**
- 营销文章不算。**事故复盘算。** "我们在生产环境试了 X 然后哪里崩了"比十个发布公告都有价值
- 这个领域的真正信号，总是由**那些为它搭进去过一个周末的人**写的

**3. 采用它需要你扔掉 tracing、重试机制、配置系统、认证系统吗？**
- 如果是，它是一个**想当平台的框架**。想当平台的框架死亡率 90%
- 好原语是嵌入你现有系统的，不是逼你迁移的

**4. 跳过六个月的成本是什么？**
- 大多数启动：成本是零。六个月后你知道更多，赢家也更清晰了
- 这是让你无焦虑跳过 90% 启动的测试——也是大多数人拒绝做的，因为跳过感觉像掉队
- **它不是**

**5. 你能度量它是否真正帮助了你的 agent？**
- 没有 eval 的团队靠 vibe 运行，会不知不觉搞出 regression
- 有 eval 的团队可以让数据判断 GPT-5.5 还是 Opus 4.7 更适配你的特定工作负载

> **这 5 个测试背后是一个更难命名的技能：愿意对你**不采用**的东西保持"不酷"。本周在 Hacker News 上 viral 的框架会有 14 天的拉拉队——半年后一半已经停止维护，拉拉队早已转向。在发布热潮过去后保持冷静、等待、说"我六个月后再说"，是这个领域真正的职业技能。每个人都会读发布公告，但几乎没人擅长不反应。**

## 二、什么值得学（复合增长的原语）

这些概念经得起模型替换、框架替换、范式转换。深度理解它们，你能在一个周末上手任何新工具。

### 1. 上下文工程（Context Engineering）

过去两年最重要的重命名是"prompt engineering"变成了"context engineering"。

**模型不再是你精心撰写指令的东西——而是在每一步为其组装工作上下文的东西。** 这个上下文是：系统指令、工具 schema、检索文档、之前的工具输出、scratchpad 状态、压缩的历史——全部同时存在。Agent 的行为是你在窗口里放什么东西的涌现属性。

**内化这一点：上下文即状态。** 每一 token 的无关噪音都在消耗推理质量。上下文腐烂是真实的生产事故。到十步任务的第八步，原始目标可能已经被工具输出埋葬了。

**可以感受的一个具体做法：** 对生产环境中的任意 agent 打开完整 trace 日志。看第一步的上下文，看第七步的上下文。数一数还有多少 token 在发挥作用。第一次做你会不好意思。然后你去修，同一个 agent 会明显更可靠——**不换模型、不改 prompt**。

**必读：** Anthropic 的《Effective Context Engineering for AI Agents》

### 2. 工具设计（Tool Design）

工具是 agent 连接你业务的地方。**模型根据名字和描述选工具。模型根据错误消息做重试。模型成功还是失败，取决于工具的契约是否匹配 LLM 擅长的表达方式。**

- 5-10 个命名得当的工具 > 20 个平庸的工具
- 工具名应读起来像英语动词短语
- 描述应包括**何时用**和**何时不用**
- 错误消息应是模型可以据此行动的反馈——"超过 500 token 限制，试一下先总结"远胜"Error: 400 Bad Request"
- 有团队报告仅重写错误消息就**减少了 40% 的重试循环**

**必读：** Anthropic 的《Writing tools for agents》

### 3. Orchestrator-Subagent 模式

多 agent 设计的 2024-2025 年大辩论已经收敛到所有人都认可的答案：

- **天真并行多 agent**（多个 agent 对共享状态同时写入）灾难性失败，错误会复合
- **单 agent 循环**比你以为的更能规模化
- **唯一经得起生产验证的多 agent 模式**：一个 orchestrator agent 将狭窄范围的只读任务委派给隔离的 subagent，然后综合结果

**必读：** Cognition 的《Don't Build Multi-Agents》+ Anthropic 的《How we built our multi-agent research system》——看似对立，实则在用不同的词汇说同一件事。

**默认用单 agent。只有当你真正撞墙（上下文窗口压力、顺序工具调用的延迟、任务异质性需要聚焦上下文）时才升级到 orchestrator-subagent。**

### 4. Eval 与黄金数据集（Evals and Golden Datasets）

**这是本领域最高杠杆的习惯，也是最被低估的。**

- 每个能稳定交付 agent 的团队都有 eval。没有 eval 的团队都没有。
- 怎么做：收集生产 trace → 标注失败案例 → 做 regression 集 → 每次 prompt/模型/工具变更前跑
- Spotify 的 judge layer 在输出到达用户前拦截了约 25% 的坏结果
- **Eval 就是在一切都在变化时让 agent 保持诚实的单元测试**

### 5. 文件系统即状态 + 思考-行动-观察循环

对于任何做真实多步工作的 agent，可靠架构是：**思考→行动→观察→重复**。文件系统或结构化存储是事实源头。每一步都可日志、可回放。

Claude Code、Cursor、Devin、Aider、OpenHands、goose——它们全都殊途同归。

**更深层的洞察：在任何值得吃掉计算账单的生产 agent 中，harness 做的工作比模型多。** 模型选下一步动作，harness 验证它、在沙箱里运行、捕获输出、决定反馈什么、决定何时停止、何时 checkpoint、何时生成 subagent。换同等质量的模型，好的 harness 仍然能工作。换一个更差的 harness，世界上最好的模型也会产生一个随机忘记自己在做什么的 agent。

### 6. MCP（概念层面）

不要只学怎么调 MCP 服务器。**学那个模型**：agent 能力、工具和资源之间的清晰分离，配上可扩展的认证和传输层。一旦你理解这个模型，你看到的每一个"agent 集成框架"都会像 MCP 的劣化版。

### 7. 沙箱作为原语

每个生产环境中的 coding agent 都在沙箱里运行。每个浏览器 agent 都曾被间接 prompt injection 命中过。把沙箱当作**原语基础设施**，而不是客户问才加的 feature。

## 三、具体用什么工具（2026 年 4 月选型）

这些推荐会有变化，但变化得很慢。在这层尽量选无聊的。

| 领域 | 推荐 | 备注 |
|---|---|---|
| **编排** | LangGraph（生产默认）、Mastra（TypeScript）、Pydantic AI（类型安全） | 1/3 大企业用 LangGraph |
| **协议层** | MCP，别无选择 | 把你的工具集成做成 MCP server |
| **记忆** | Mem0（聊天个性化）、Zep（生产对话）、Letta（跨天/week 的 coherence） | 不要在没有记忆问题之前就用记忆框架 |
| **可观测性 + Eval** | Langfuse（OSS 默认）、LangSmith（LangChain 生态）、Braintrust（研究级 eval） | Tracing + evals 必须同时有，缺一不可 |
| **运行时 + 沙箱** | E2B（通用沙箱）、Browserbase + Stagehand（浏览器自动化）、Modal（短突发） | **永远不要跑无沙箱的代码执行** |
| **模型** | Sonnet 4.6（性价比之选）、Opus 4.7（复杂推理）、GPT-5.4/5.5（CLI/终端）、Gemini 2.5（长上下文）、DeepSeek-V3.2/Qwen 3.6（成本敏感） | 换模型就像换零件——如果 agent 只适配一个模型，那是臭味不是护城河 |

## 四、什么不值得学 / 直接跳过

**以下你会被告知必须学——但你真的不需要。**

| 跳过项 | 原因 |
|---|---|
| **AutoGen / AG2** | 微软转社区维护，发布停滞，抽象不匹配生产需求 |
| **CrewAI** | 演示效果好，工程师已转向。原型可以用，别承诺 |
| **Microsoft Semantic Kernel** | 除非你锁死在微软企业栈 |
| **DSPy** | 哲学上有价值，受众太小，不是通用 agent 框架 |
| **"自主 agent" 推销** | AutoGPT 时代已死，行业共识是"工程化 agent"——监督的、有边界的、有 eval |
| **Agent 应用商店/市场** | 从 2023 年吹到现在，从未获得企业 traction |
| **水平"造一切 agent"平台（Google Agentspace, AWS Bedrock, MS Copilot Studio）** | 最终会有用，现在还很混乱。买 vs 造的天平仍然偏向自己造 |
| **SWE-bench / OSWorld 排行榜追逐** | 几乎所有公开 benchmark 都可以被游戏而不解决底层任务 |
| **天真的并行多 agent** | 5 个 agent 在共享内存上聊天——演示好看，生产崩 |
| **本周 Hacker News 上的下一个框架** | 等六个月。如果它仍然重要，你会知道的 |

## 五、如何真正行动（不只是跟风）

**筛选序列：**

1. **选一个已经重要的业务结果。** 不是登月计划，不是"agent 平台"项目。可量化的、业务已经在意的：分流工单、起草法务初审、筛选入站线索、生成月度报告。Agent 成功了就是这个指标动了。
2. **在交付任何东西之前搭建 tracing 和 evals。** Langfuse 或 LangSmith。五十个手工标注的例子就够了。
3. **从单 agent 循环开始。** LangGraph 或 Pydantic AI。Claude Sonnet 4.6 或 GPT-5。3-7 个设计得当的工具。文件系统或数据库作为状态。交付给一小批用户。看 trace。
4. **把 agent 当产品，不是项目。** 它会在你没想到的地方失败。那些失败就是你的路线图。
5. **只在你已经赚到了"扩展"的资格之后再扩展。** Subagent？等上下文是瓶颈再说。记忆框架？等单窗口放不下再说。
6. **选无聊的基础设施。** MCP 管工具。E2B 管沙箱。你已经在跑的数据库。
7. **从第一天起关注单位经济。** Agent 在 POC 阶段看起来很便宜，在 100x 规模时爆炸。$0.5/次 的 POC，在中等规模下就是 $50K/月。
8. **每季度换模型，不是每周。**

## 六、识别信号 vs 噪音

**信号的特征：**
- 受尊重的工程团队写了带数据的复盘，不只是采纳率宣传
- 它是原语（协议、模式、基础设施），不是 wrapper 或捆绑包
- 能与现有系统互操作，而不是替换
- 描述的是它解决的**失败模式**，不是它启用的能力
- 存在时间长到有人写了"它哪没做好"的博客

**噪音的特征：**
- 30 天后仍无生产案例的演示视频
- 太干净而不真实的 benchmark 飞跃
- 吹"自主"、"Agent OS"、"造任何 agent"
- 文档假设你要扔掉现有 tracing、认证和配置
- GitHub star 涨得快但 commit、release、贡献者没跟上

**建议每周五 30 分钟习惯：** 读三样东西——Anthropic 的工程博客、Simon Willison 的笔记、Latent Space。快速阅读一到两篇复盘。其他东西跳过一周。你会知道什么重要。

## 七、非常规赌注

> 你**不**采用的每个框架，都是你**不欠**的一次迁移。
> 你**不**追逐的每个 benchmark，都是你**保留**的一个季度的专注力。

传统路径是：选一个栈 → 深耕多年 → 向上爬。这在栈稳定十年的时候有效。现在栈每季度都在变。

**赢的人不再优化"栈掌握深度"，而是优化品位（taste）、原语和交付速度。**
- 他们在公开场合造小东西
- 他们通过交付来学习
- 他们被已经做出的东西带入更高的房间
- **凭证（credential）就是作品本身**

> 这个领域没有稳定的"另一端"。你想加入的公司可能成立才六个月。他们用的框架诞生十八个月。底层协议才两年。这个领域一半最有影响力的文章是三年前还不在这行的人写的。没有梯子可以爬，因为楼在不停地变。
>
> 梯子失效后剩下的，是更古老的方法：**造一个东西，放到互联网上，让作品替你介绍自己。** 这是非常规路径，因为它绕过了认证体系。但这也是在一个移动的领域里唯一能复合增长的路径。

**你现在真正需要开发的技能不是"agent"——而是在表面持续变化的环境中，识别什么工作能复合增长的能力。**
- 上下文工程 → 复合
- 工具设计 → 复合
- Orchestrator-Subagent 模式 → 复合
- Eval 纪律 → 复合
- Harness 思维 → 复合
- 知道周二发布的框架的 API → 不复合

一旦你能区分这些，每周的发布潮就不再有压力感——它变成了你完全可以忽略的噪音。
