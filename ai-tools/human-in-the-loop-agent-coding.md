# Human-in-the-Loop：与 Agent 异步协作的编程模式

> **原文作者**: Eric Zakariasson (@ericzakariasson)  
> **原文链接**: [Human in the /loop](https://x.com/ericzakariasson/status/2070493377267646797)  
> **发布日期**: 2026-06-26  
> **核心概念**: Agent 异步协作 / 自动化验证循环 / Human-in-the-loop

---

## 📌 一句话总结

Eric 分享了一套与 AI Agent 异步协作的编程模式：定义可自动验证的"完成标准"→ 让 Agent 在循环中自主迭代 → 完成或需要决策时通知你 → 你同时跑多个这样的循环。**让 Agent 干它的活，你干你的活。**

---

## 1. 核心理念：让 Agent 跑起来，你自己也跑起来

> "我最喜欢用 Agent 编码的地方是：可以同时跑几个任务，自己该干嘛干嘛。当某个任务完成或需要决策时，我再出现。"

这不是让你盯着终端看 Agent 一步步工作，而是**并发委托 + 异步通知**的模式。Eric 的完整流程：

```
定义完成标准 → 写验证循环 → Agent 自主迭代 → 完成/卡住时通知你 → 你继续下一个
```

---

## 2. 定义完成标准（Definition of Done）

在启动长时间任务之前，先锁死一个 Agent **可以自行验证**的完成标准。

### 五种实战场景

#### 🎯 模型 / Eval 工作
- **目标**：一个评分（score）
- **循环**：改变方法 → 运行 eval → 仅当分数向正确的方向移动时才保留改动
- **本质**：最接近 Karpathy 的 autoresearch 理念，类似 ML 训练循环

#### 🎯 Web 应用 / UI 工作
- **目标**：QA 检查通过
- **循环**：加载页面或运行 Playwright → 截图 → 确认页面功能正常
- **本质**：视觉 + 功能验证

#### 🎯 后端 / 重构工作
- **目标**：测试套件通过
- **循环**：先写失败测试 → 实现直到全绿 → 保持绿色
- **本质**：TDD（测试驱动开发）的 Agent 版本

#### 🎯 性能 / 稳定性优化
- **目标**：一个数值（p95 延迟、某个基准分数）
- **循环**：修改 → 测试 → 直到压到目标线以下
- **本质**：持续的数值优化

#### 🎯 数据 / 内容清理
- **目标**：计数
- **循环**：直到零行验证失败，或每条记录都通过检查
- **本质**：批量数据质量保障

## 3. 循环怎么写

写验证循环 = **把你检查工作的方式写下来**。

### 两种风格

| 风格 | 适用场景 | 说明 |
|------|---------|------|
| **显式指令** | 新任务 / 复杂任务 | 每一步都写清楚，开始时比你以为的更详细 |
| **粗粒度目标** | 熟悉的模式 | 给一个目标和大致方向，让 Agent 填充中间细节 |

### 最佳实践

> "我从比我认为需要的更明确的指令开始，一旦看到 Agent 能推理出什么，就逐步放松。"

---

## 4. 循环的停止条件

循环需要一组明确的**停止条件**。Eric 使用的条件：

| 条件 | 说明 |
|------|------|
| ✅ **指标达标** | 目标数值已实现 |
| ⏹️ **无改进** | 尝试几次后不再有提升 |
| 💡 **用尽想法** | Agent 尝试了所有它能想到的方法 |
| 🛑 **卡住 / 不确定** | 停下来询问人类 |

核心思想：**有明确终点的循环才是可信任的循环。**

---

## 5. 被通知，而不是被监督

Agent 需要一个**通知通道**，而不是你需要盯着它。

### Eric 的方案

```
MCP (Model Context Protocol) + /notify → Slack
```

- Agent 通过 MCP 的 `/notify` 路径联系你
- **Slack 是主力**——因为其他一切都在那里
- 同样的设置也可以换成 iMessage 或其他通道
- **Agent 没有完整的 Slack 权限**，只是一个普通的通知渠道
- 状态更新和"需要决策"的消息像普通消息一样弹出来
- 你回复的消息就是循环的下一个输入

> **关键区别**：Agent 通知你，而不是你轮询 Agent。

---

## 6. 云端运行

大部分循环不跑在本地笔记本上，而是在云端：

- **云端持续运行**：一个循环可以跑几个小时，不需要你电脑开着
- **你的本地客户端作为编排器**（orchestrator）
- **工作任务分发**到云端 Agent

这意味着你可以关机睡觉，Agent 在云端继续跑。

---

## 7. 并发模式：同时跑多个循环

Eric 的实际工作流：

```
同时跑：
  - 3 个长期循环（通常）
  - 有时 5 个
  - + 其他短期任务（PR、一次性调研等）

负载管理：
  - 空闲时 → 再启动一个
  - 3 个都在等你 → 停止启动新任务，去 review
```

### 任务类型矩阵

| 维度 | 长期循环 | 短期任务 |
|------|---------|---------|
| 时长 | 数小时 | 分钟级 |
| 典型任务 | 重构、优化、迁移 | PR 审查、单次调研 |
| 并发量 | 3~5 个 | 不限 |
| 监督方式 | 异步通知 | 可实时 |

## 8. 关键 Prompt 模板

Eric 提供了他常用的结构：

```
/loop     → 驱动迭代循环
/notify   → 保持通知
```

`/loop` 负责自主迭代的逻辑，`/notify` 负责完成或卡住时通知人类的机制。这两个原语构成了完整的人机协作循环。

## 10. 总结：如何立即上手

### 第一步：选择一个验证型任务

选那种结果可以**量化衡量**的任务：
- 测试通过 / 不通过
- 延迟降到了目标值
- 某行数据清理到了零

不要选那种需要主观判断的任务（如"让这个 UI 更好看"）。

### 第二步：写 Definition of Done

写出你检查这个任务的方式——代码量可能就十几行。

### 第三步：套循环 + 加通知

- 循环逻辑：改 → 测 → 保留/退回 → 再来
- 通知通道：Slack、iMessage 或任何你能收到的渠道

### 第四步：在云端跑（或稳定的后台环境）

不要在必须关机的笔记本上跑长时间循环。

### 第五步：启动多个，然后去干别的

跑 3 个长期循环的同时，再做 PR review 或写新功能。
