# MCP 核心架构详解：Host、Client、Server、Tool、Resource、Prompt

> 原文：[Tech Fusionist @techyoutbe](https://x.com/techyoutbe/status/2063553215623577693)
> 中文整理：基于原文核心内容归纳

---

## 一、为什么 MCP 架构很重要

MCP 不只是连接器格式，它是**将 AI 应用与外部能力集成的系统设计模式**。

核心问题：如何让 AI 驱动的 Host 应用**安全地发现、访问和编排外部能力**，而不把 Assistant 紧耦合到后端系统？

没有正式架构的团队往往会：
- UI 直连数据库
- Assistant runtime 内嵌工具逻辑
- Prompt 模板埋入业务代码
- 每次新集成都让维护更难

**MCP 引入清晰边界**：Host 管交互和编排，Client 管协议通信，Server 暴露标准化能力，后端系统保持独立。

---

## 二、宏观心智模型

MCP 是分层系统：

```
用户 → Host应用 → MCP Client → MCP Server → 后端系统(API/DB/SaaS)
```

**MCP 把 AI 交互和能力访问分离，再把能力访问和后端执行分离。**

---

## 三、核心组件详解

### 1. 用户 (User)
意图的起点。用户不直接与 MCP 交互，而是通过 Host 应用发起请求。

### 2. Host 应用
AI Assistant 运行的**环境**：IDE 助手、团队聊天机器人、企业 AI 工作台等。

**Host 的职责：**
- 管理用户体验
- 理解 Assistant 工作流上下文
- 决定何时需要外部能力
- 将结果整合成最终响应

**Host 不应拥有：**
- 原始后端集成
- 服务端业务逻辑
- 协议暴露的能力定义

> 好的 Host 是编排者，不是单体。

### 3. MCP Client
Host 环境内部**协议感知组件**。

**职责：**
- 发现可用能力
- 发送结构化请求
- 接收结构化响应
- 管理与一个或多个 MCP Server 的协议交互

**不应过载：** 业务工作流、数据存储、自定义逻辑。

> 常见反模式：把太多智能塞进 Client → 协议边界模糊 → 可维护性骤降。

### 4. MCP Server
通过标准化接口向 Client **暴露能力**的层。

暴露三类能力：**Tools（工具）、Resources（资源）、Prompts（提示模板）**。

**职责：**
- 暴露结构化能力
- 校验输入
- 路由请求
- 与后端服务集成
- 封装响应
- 强制访问控制

> Server 不是系统本身，是前面的抽象层。

### 5. Tools（工具）
**可执行能力** — 当 Assistant 需要「做事」时用。

典型场景：创建工单、发送消息、触发部署、更新 CRM 记录、启动工作流。

工具是**面向操作的**，通常产生副作用。

### 6. Resources（资源）
**可检索的内容** — 当 Assistant 需要「读取」时用。

典型场景：获取文档、加载配置、检索日志、读知识库文章、查询仪表盘。

> 最常见的建模错误：把所有读操作都做成 Tool。小 demo 没问题，但会弱化语义。

### 7. Prompts（提示模板）
**可复用的交互模式**。

典型场景："用初级工程师的视角解释这段代码"、"从事件记录生成事后总结"。

Prompts 不是原始后端系统，也不是执行端点。它们是**结构化可复用的交互资产**，特别适合需要一致性和可控性的企业环境。

### 8. 外部/后端系统
数据和工作真正发生的地方：API、数据库、文件系统、SaaS 平台等。

这些系统应该保持在 MCP 能力层**后面**。直接暴露给 Host 会造成紧耦合。

---

## 四、MCP 的分层架构

```
1. 用户交互层     — 请求、问题、对话控制
2. AI 应用层      — Host 编排、上下文组装、响应合成
3. 协议层         — MCP 通信边界、结构化请求/响应、发现
4. 能力暴露层     — Tool/Resource/Prompt 定义、校验、封装
5. 集成/后端层    — 执行、存储、系统记录访问
```

---

## 五、最重要的设计原则：关注点分离

**不要混合 Host 逻辑、协议逻辑和后端逻辑。**

| 层 | 拥有 |
|---|---|
| **Host** | 用户交互、Assistant 工作流、决定何时需要能力、结果解释 |
| **Client** | 协议处理、能力发现、请求/响应结构、传输行为 |
| **Server** | 能力暴露、校验、后端集成、结果封装、访问控制 |
| **后端** | 实际执行、持久化数据、领域记录、运维 |

**错误示例：** Host 直调数据库、Client 嵌入业务规则、Server 处理 UI 状态。

---

## 六、数据流与控制流

1. **用户发送请求** → Host 接收
2. **Host 解析意图** → 确定是否需要外部能力
3. **Client 识别能力** → 找到正确 Server 和能力
4. **Client 发送结构化请求** → 经过 MCP 协议边界
5. **Server 解析请求** → 映射到正确的 Tool/Resource/Prompt
6. **后端交互** → 获取数据或执行操作
7. **Server 返回结构化结果** → 干净、不含后端复杂度
8. **Host 整合结果** → 转换为用户可读的响应

---

## 七、一个 Host，多个 Server

强大模式：一个 Host 连接多个 MCP Server，按领域拆分：

```
Host → Files Server, Docs Server, CRM Server, Metrics Server, Ticketing Server...
```

优势：更好的模块化、更清晰的归属、更易治理、可独立扩展。

---

## 八、常见部署模式

| 模式 | 适用场景 |
|---|---|
| **本地 Host + 本地 Server** | 本地工具、桌面助手、隐私敏感工作流 |
| **云 Host + 远程 Server** | 共享服务、SaaS 产品、集中式能力 |
| **企业多 Server** | 领域分离、大型组织、治理复杂环境 |

---

## 九、常见架构错误

1. **混合 Host 和后端逻辑** — 紧耦合，难演进
2. **Server 过载** — 开始涉及 UI 行为、多步 Assistant 流程
3. **一切皆 Tool** — 读操作应建模为 Resource
4. **直接暴露原始系统** — 后端复杂度应抽象为清晰能力面
5. **协议边界薄弱** — 请求模式、校验、响应结构不一致
6. **验证和权限归属不清晰**
7. **为 demo 设计而非为生产设计**

---

## 十、最佳实践

- 明确建模能力：清楚区分 Tool / Resource / Prompt
- 保持边界干净：每层有聚焦的职责
- 首选模块化 Server：按领域、信任边界、所有权拆分
- 使用结构化 Schema：可预测的请求/响应格式
- 设计可观测性：协议通信应可检查
- 抽象后端复杂度：Host 消费能力，而非实现细节
- 优化复用：可复用的能力面 > 一次性硬编码工作流
- 治理优先：访问控制、权限、审计

---

## 十一、总结

> **Host 编排，Client 通信，Server 暴露能力，后端系统干实际工作。**

这个心智模型是实现、调试和扩展 MCP 系统的基础。MCP 架构在小型原型上可能有额外开销，但一旦系统超出原型阶段需要 **规模、复用、可靠性、团队归属和长期可维护性**，分层 MCP 架构的价值就远远超过它的成本。
