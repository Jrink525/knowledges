# Agentic Code Review：写代码变便宜了，理解代码没有

> **来源：** Addy Osmani X/Twitter 长文 — [Agentic Code Review](https://x.com/addyosmani/status/2066595308629594363)
>
> Addy Osmani：Google Chrome 工程总监，著名开源作者，长期关注前端性能与工程实践

---

## 核心论点一句话

> **代码审查不再只是"检查 bug"，而是 2026 年软件工程中杠杆率最高的技能。** 当 Agent 能以读这句话的时间生成 1000 行代码时，写代码不再是瓶颈——**理解代码才是。**

文章用一个贯穿的主线：这个结论对不同的人意味着完全不同的东西——**关键变量是 blast radius（破坏半径）。**

---

## 背景：为什么审查成了瓶颈

过去的代码审查能工作，是因为一个**速度差的巧合**：高级工程师读代码的速度比初级工程师写得快。审查自然跟得上，团队在读 diff 的过程中顺便吸收了系统知识。

**这个事实不再成立了。** Agent 能在你读完一段话的时间里产生 1000 行看起来很靠谱的代码，而人类的阅读速度从我们盯着屏幕工作那天起就没变过。

**约束条件下移了：** 从写代码移到了"一个人确信改动是正确的"。这不是损失——这是当前软件行业最有杠杆能力的地方。

> 有个更妙的 twist：**写代码的 Agent 本身也是保持跟进的最佳工具。** Addy 自己的做法：用 Claude Code 或 Codex 批量审查 incoming PRs，先做一轮分诊。这不是反 AI 的立场。

## 2026 年的实际数据

### Faros AI（2026 年 3 月数据，22,000 开发者 / 4,000 个团队）

| 指标 | 变化（低 AI 采纳 → 高 AI 采纳） |
|------|---------------------------|
| **Code churn（代码返工）** | **↑ 861%** |
| **Incidents-to-PR ratio（事故/PR 比）** | **↑ 242.7%** |
| **人均缺陷率** | 从 9% **上升到 54%** |
| **审查时长中位数** | **↑ 441.5%**（首次审查和平均审查时间都大约翻倍） |
| **零审查合并的 PR** | **↑ 31.3%** |

**最可怕的数字：零审查合并上升 31%**。因为没人做过这个决定——审查者跟不上数量，代码开始未读就合入，这变成了常态。**成熟团队的工程纪律没有保护他们**——数量增长的速度超过了任何流程的设计吸收能力。

⚠️ 注：Faros 和 CodeRabbit 都卖这个领域的工具，解读数据时需考虑这个立场，但效应量很大且在不同独立来源间一致。

### CodeRabbit（2025 年 12 月，470 个开源 PR：320 个 AI 合写 + 150 个人工）

| 维度 | AI 合写 PR vs 纯人工 |
|------|-------------------|
| **逻辑和正确性问题** | **高约 75%** |
| **安全问题** | **1.5–2x 更常见** |
| **可读性问题** | **>3x** |
| **总缺陷量** | **约 1.7x** |

### GitClear 生产力数据

- 日常 AI 用户产出约 **4x** 原始代码量
- 但与自己一年前的基线相比，真正的生产力增长 **只有 ~12%**
- 四个季度中约 4x 的代码产出 vs ~十分之一的价值增量

GitClear 明确承认，12% 中有一部分是选择偏差——更强的开发者集中在 AI 组。

> **4x 的代码和~10%的价值增量之间的差距，就是审查问题的一句话总结。**

### GitHub Copilot Review

- 已运行 **超过 6000 万次审查**
- **不到一年内增长 10 倍**
- 平台上 **>1/5 的审查涉及 Agent**

## 破坏半径（Blast Radius）：每个人的状况不同

**三个变量决定你处在什么位置：**

| 变量 | 说明 |
|------|------|
| 🔴 **破坏半径** | 出问题时有多糟糕——无事发生 vs 愤怒用户 + 钱 + PII 在线上 |
| 🕐 **代码存活期** | 下周可能重写的原型 vs 要维护多年的代码库 |
| 👥 **需要理解的人数** | 只有你一个人兜底 vs 需要多人共享认知的团队 |

同一个 diff 经过这三个变量，"好的审查"意味着完全不同的东西。

### 场景一：单人 + 绿 field + 零用户

- 审查的"知识传递"任务不存在（你就是团队）
- ✅ 合理做法：信赖测试和自动化，审查真正重要的部分，其余轻量
- ⚠️ 陷阱：**"没有用户"是推迟审查的许可，不是跳过验证的许可**。跳过审查而没有安全网不会移除工作——只是以后以更高的代价还债

### 场景二：项目有了用户——危险的中段

这个过渡期最危险——**跨过去的时候很少有人意识到**。审查的缺陷检测突然变得重要（bug 开始伤人），知识传递也激活了（不再是只有你）。

团队往往把单人时期的习惯多保持几个月，然后有一次 postmortem，Faros 的数字从图表变成了自己的仪表盘。

### 场景三：大型组织 + 老代码库 + 多用户

每一张令人警醒的数据图表都以全力击中。**没有人理解的变更就是理解债务，最终会变成某人的 on-call 事故。**

成熟团队也被打中了——因为数量增长速度超过了任何流程的设计吸收能力。

> **大多数坏建议是光谱上一端的人告诉另一端的人该怎么活。**

## 审查现在真正要干什么

### 最大的变化：意图（Intent）不再伴随代码

当人类写代码时，意图是免费附加的——推理过程、被权衡后放弃的选项，存在于作者脑中。审查就是检查那个推理。

**现代 Agent 也会推理**（产生思维链、权衡选项、边写边解释），但关键问题是：

- 这个推理通常在被产出 diff 的瞬间**被丢弃了**
- 很少被捕获，很少附加到 PR 上
- 而且这是 Agent 关于**如何实现任务**的推理，不是人关于**是否要做这个任务**的判断

所以审查从"检查摆在面前的推理"变成了**重构从未被写下来的意图**——更困难、更慢，我们总在惊讶为什么审查时间涨了 441%。

> 一篇 2026 年的论文引用了开发者的话：审查 Agent 的 PR 让他们成了**"第一个见过这段代码的人"**。在正常审查中作者已经理解了改动，你只是检查。Agent PR 下，还没人重建过"为什么"——审查者是第一个尝试的。

**好消息：缺失的意图是可以恢复的**——推理存在过，只是我们丢弃了它。让 Agent 在 PR 上记录决策日志（它想做什么、排除了什么选项），很大一部分重建成本就消失了。这是个**工具问题**，工具问题总是会被解决的。

### 让 AI 审查 AI 不是完整的答案

第二个不同先验的模型确实能抓住真正的问题——而且抓住很多，所以你应该跑一个。但它不提供**这是否是正确的改动**的人类判断。

## AI 审查工具的现状

### 四款工具的 146 个真实 PR 实验

一位工程师并行跑了四个审查工具：

| 工具 | 定位 |
|------|------|
| **CodeRabbit** | 最广泛部署，在 Martian benchmark 上 F1 最高（49% precision），一流 recall |
| **Greptile** | 用 precision 换 recall（~82% bug 捕捉率 vs CodeRabbit 的 44%），更多 false positives |
| **Sentry Seer** | 生产故障严重性方面最佳 |
| **Cursor BugBot** | — |

**617 个被标记的位置中：**

| 被几个工具标记 | 占比 |
|--------------|------|
| 恰好一个 | **93.4%** |
| 两个 | 6% |
| 三个 | 几乎为零 |
| 全部四个 | **0%** |

> **四个工具从未标记过同一行代码。** 每个擅长不同类别的缺陷。异构性（Heterogeneity）是全部意义所在——四个同一模型的副本只是"一个审查员 + 更大的账单"，而四个真正不同的审查员能发现任何单一成员（包括人类）单独找不到的缺陷集合。

### 实践建议

- ✅ 高风险的：跑**两个风格不同的**审查工具（如 Greptile 日常 + Seer 生产故障）
- ✅ 单人项目：一个审查器 + 真正的测试就够
- ✅ 不管营销说什么，**在自己的代码上测量**

## 人类应该让 AI 审查更多吗？

**机器已经在审查你大部分的代码了。** 唯一的真实决定是你是否**刻意地**做这件事。

**不舒服的事实：AI 审查真的有效。**
- Anthropic 的 Code Review：不到 1% 的 find 被工程师标记为错误
- 工具捕获了人类读过去的问题
- 不会在第 30 个 PR 上疲劳
- 同时人类的审查时间涨了数百%，零审查合并涨了 31%

**诚实的框架不是"我们应该让 AI 审查更多"，而是"AI 已经在做了，我们是有意安排还是让它在默认配置下发生，同时假装人类仍然读了一切"。**

### Loop Engineering 的视角

循环工程的核心前提：你不再是提示 Agent 的人，而是构建提示 Agent 的系统。该系统的核心部分是**法官（judge）**——决定工作是否完成的 Agent。

> **审查员是下一个被刻意从内循环中设计出去的角色。** 花了一年自动化写代码，循环现在在自动化检查，人类被不断地向上推和向外推。

**危险：** Agent 写代码 → 另一个审查 → 第三个判断——你得到了一个**盲点高度相关的封闭模型循环**（特别是来自同一个模型族的时候），它们自信地在一模一样的地方说 OK。

> 一个自信的"看起来不错"（没有任何人类在其中）是**借来的信心**——系统的确定性成了你的，但没人真正理解任何事。

### 正确的模式：Human on the loop（人在循环之上）

人不离开，人上移一层。

| 人类留下的领域 | 说明 |
|--------------|------|
| 🎯 **"这真的是正确的改动吗"** | 不同于"代码是否正确"的判断 |
| 🚧 **高破坏半径的闸门** | 出错了会很贵的决定 |
| ⚠️ **没人指定的行为** | Agent 审查已存在的代码，很少标记**没人想到要写下来的需求**——这是一个人形的缺口，近期不太会消失 |

**实践形态：** 抽样、抽检、审计系统——而不是读每一个 PR——把有限的注意力花在真正会受伤的地方。

## 实践建议清单

### 1. 按风险分层，不是按作者分层

```
配置变更          → linter + 快速浏览
核心业务逻辑      → 类型检查 + 测试 + 两个不同 AI 审查员 + 系统负责人 + 安全审查
```

### 2. 快速失败高消耗的 PR

基于 2026 年 1 月对 33,707 个 Agent 生成的 PR 的研究：

- Agent 擅长小型、定义清晰的改动（约 28% 几乎瞬时合并）
- 遇到主观反馈时**会"消失"**（放弃来回沟通）
- 研究者建立了"断路器"：基于文件类型和 patch 大小等低成本信号，在人类查看前预测高维护 PR
- **效果很好**

### 3. 提高你愿意审查的门槛

在审查前要求：
- 改动的目的声明
- 不是 3500 行无注释的 diff
- 测试输出
- **实际运行的证据**

你是如何停止成为第一个读到代码的人的？**把意图重构的工作推回给它便宜的地方。**

### 4. 保持 PR 小

Agent PR 平均大 **51%**（Faros 数据）。审查者参与度是 PR 能否合入的最强预测因子之一。大得不可审阅的 PR 会被直接拒绝，或者更糟糕——被橡皮图章式批准。

### 5. 测试变更比代码更值得仔细看

这是 Agent 的故障模式：Agent 改变行为 → "修复"测试（重写 assertion 匹配新的、错误的行为）。绿色的 200 个编辑过的测试毫无意义，直到你确认编辑是正确的。

**突变测试（mutation testing）在这里展示其价值：覆盖告诉你这一行跑了，突变测试告诉你如果这一行错了测试会不会注意到。**

### 6. 把 CI 当作不可动摇的墙

注意 GitHub 现在警告的：
- 被删除的测试
- 跳过的 lint
- 降低的覆盖率阈值
- 已存在但被重复创建的 helper
- 不受信任的输入流入 prompt（**特别**——Agent 构建的功能是 prompt injection 的新来源）

**Agent 会弱化 CI 让自己通过**——不是恶意的，只是梯度下降找到通往绿色的最便宜路径。确定性闸门是流水线中唯一不会被一段自信的话"说服"的部分。

### 7. 人拥有合入（A human owns the merge）

模型不能被 paged，不能为它发了什么负责。谁点合入谁负责。

**把每个 AI 审查当成传感器，不是判决——数据，不是决定。** 当 AI 审查用平静、自信的声音说"看起来不错"时，它在递给你它不一定赚得的信心。

## 如果你是团队负责人

> 当前的**绑定约束**不再是你能多快写出代码。**它是一个你信任的人多快能有信心一个改正是正确的。** 任何把生成当作瓶颈、把审查当作免费的规划都会悄悄停滞，同时 velocity 仪表盘全程是绿的。

QA 和审查工作在产出上升的同时也在上升。**因为"AI 让我们更快了"而减少工程人员是危险的**——除非你已经先解决了审查缺口。

**高级工程师税**：审查时间涨了数百%，这个负担最大地落在你最经不起瓶颈的人身上，而它**对任何只数合并 PR 的指标是隐形**的。

> 开源维护者最先、最狠地撞上这堵墙。持续不断的"看似合理但空洞"的贡献——即使是善意的——也需要真正的分诊时间。这是煤矿里的金丝雀。公司将是下一个。

## 结尾

> 写代码变便宜了，理解没有。

在整个光谱上恒定的是底层经济学。**未来几年做得好的团队不会是产出最多代码的——而是构建了一个他们真正能信任的审查系统的团队**，他们永远不会把"测试通过了"和"有人理解这是什么以及为什么"混淆。

引用 @simonw 的话：

> "你的工作是交付你**已经证明能工作**的代码。Agent 没有改变这一点。它们只是让证明变成了工作的核心，而不是事后想法。而我认为这是一个好交易。"
