# 愚蠢的沙箱，聪明的主机：云端 Agent 架构原则

> **来源：** [Peter Pang (@intuitiveml) on X](https://x.com/intuitiveml/status/2066982286395068540)

![封面图](../image/dumb-sandbox-smart-host-1.jpg)

大多数云端 Agent 架构都在问同一个问题：**沙箱能做什么？**

能跑 shell 吗？能装包吗？能调 API 吗？能持久化文件吗？能调用 LLM 吗？崩溃后能恢复吗？

在搭建了这一层一段时间后，我认为更好的问题是反过来：**什么永远不该交给沙箱来做？**

---

## 桌面 Agent 与云端 Agent 的根本差异

**桌面 Agent** 把所有东西都坍缩在一个边界内。用户、机器、文件系统、凭据、运行时、进程——全在一起。Agent 可以读本地文件、用本地环境变量、直接调 API、靠用户在出错时重试。

**云端 Agent** 没有这种奢侈。

它在共享基础设施上运行。执行的是 LLM 写的代码。可能被人、cron 定时器、API 调用或其他 Agent 触发。用户可能不在现场。Prompt 可能有对抗性。执行环境内的代码可能已经沦陷。

这就改变了整个架构。

---

## 核心规则：主机聪明，沙箱愚蠢

**主机（Host）** 是受信任的、长生命周期的控制平面。它拥有：
- 身份认证
- 密钥管理
- 计费逻辑
- 持久化存储
- 重试策略
- 访问策略
- 可观测性
- 运行记录的真实源头（truth-of-record）

**沙箱（Sandbox）** 是可丢弃的执行边界。它运行模型选择的代码和 shell 命令。可以创建文件。可以通过窄通道调用工具。但——
- ❌ 不持有长期凭据
- ❌ 不直接写入业务状态
- ❌ 不决定计费
- ❌ 不以自己的身份调用内部服务

## 错误的设计诱惑

最诱人的错误是：**每次有新的功能需求，就让沙箱变得更聪明一点。**

| 新需求 | ❌ 错误做法 |
|--------|-----------|
| Agent 需要 Slack | 给沙箱一个 Slack token |
| Agent 需要持久化 | 让沙箱直接写数据库 |
| Agent 需要计费 | 在运行器内部扣减额度 |
| Agent 需要调内部 API | 给沙箱服务 URL + 密钥 |

每个决定单独看都挺方便的。但合在一起，它们就把一个不受信任的执行环境变成了**系统中权限最高的部分**。

这完全是反的。

---

## 正确的分界：桥接模式

沙箱只在一个维度上应该强大：**执行**。它能运行模型选择的代码。其他一切都应由主机中介。

### 🔑 认证服务调用
Agent 需要调一个需要 OAuth 的服务时：
- 沙箱不接收 OAuth token
- 沙箱通过**桥接（bridge）** 发请求
- 主机验证请求属于当前运行、检查策略、在服务端附加真实凭据、转发请求、记录一切、只返回响应

### 💾 持久化输出
Agent 需要保存结果时：
- 沙箱不成为数据库客户端
- 它发出结构化事件或产出 artifact
- 由**主机决定是否存储**
- 即使沙箱中途消失，持久化记录仍在主机端

### 💰 计费
Agent 需要花钱时：
- 沙箱不自行扣费
- 主机看到调用→应用速率限制→扣减额度→记录决策
- **任何依赖于不受信任代码自行上报用量来计算的计费系统，最终一定会出错**

### 📊 可观测性
- 沙箱内部的日志只用于调试，不是记录系统
- 主机需要看到每一个有意义的边界穿越点：工具调用、产出的 artifact、用户可见事件、额度扣减、重试、失败和最终状态

### 🤖 LLM 调用
- 沙箱仍然可以有有限的智能：跑辅助脚本、用本地工具、调 MCP 服务器、调用 LLM
- 但这些 LLM 调用**必须通过一个网关**，使用作用域限定的短期凭据
- 沙箱**永远不应看到真实的 provider key**

---

## 接口越窄越好

优秀的云端 Agent 运行时，主机与沙箱之间的接口应该很窄。**沙箱可以请求。主机做决定。**

那个接口往往看起来很无聊：stdout 标记、桥接调用、作用域令牌、过期凭据、结构化事件。

> 无聊是好事。无聊意味着可审计。无聊意味着可重放。无聊意味着——当沙箱被攻破时，你还能推算出发生了什么。

---

## 用故障场景做设计测试

### 如果 Prompt 注入诱使 Agent 倾倒环境变量，会泄露什么？

在"聪明主机-愚蠢沙箱"模型下，答案应该是：**没有任何长期有效的 provider key、数据库凭据、用户 OAuth token 或内部服务密钥。** 最坏情况下，攻击者只拿到了作用域限定的凭据——仅在此次运行中、从这个环境、在很短的时间窗口内有效。

### 如果沙箱崩溃，会丢失什么？

临时文件，可能。正在进行的执行，可能。**但不会丢失用户身份、计费状态、持久化事件日志、平台重试或解释发生了什么的能力。**

这就是保持沙箱"愚蠢"的意义。不是因为沙箱什么都不做，而是因为**它是可替换的**。

---

## 总结架构边界

| 需求 | 归属 |
|------|------|
| 需要密钥？ | **主机** |
| 需要持久化状态？ | **主机** |
| 判断用户是否有权限做某事？ | **主机** |
| 执行 Agent 写的代码？ | **沙箱** |
| 需要跨越回产品边界？ | **桥接（Bridge）** |

**云端 Agent 不是一个塞了 AI 的虚拟机。它是一个受信任的主机在协调一个不受信任的执行器。**

架构的重心不再是给沙箱添加更多能力，而是**设计沙箱周围的通道**。这才是让云端 Agent 在无人值守时安全运行的真正方法——不是更聪明的沙箱，而是更愚蠢的沙箱 + 更聪明的主机。
