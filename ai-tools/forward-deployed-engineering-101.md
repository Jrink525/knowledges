# Forward Deployed Engineering 101 — AI 时代最热门的工程角色

> **来源：** [Vasuman (Varick)](https://x.com/vasuman/status/2057177266984226892) — Varick（旧金山 Applied AI 公司）
> **核心观点：** 如果智能正在商品化，唯一的竞争优势就是**如何使用它**。

---

## 一、为什么 AI 公司需要 FDE

> "如果你相信智能正在变得像商品一样不值钱，那么唯一的竞争优势就是**在什么地方、以什么方式使用它**。"

Anthropic、OpenAI、Google 等 AI 公司都在疯狂招 FDE。原因很简单：企业知道需要用 AI，但没人知道怎么真正落地。

**FDE（Forward Deployed Engineer）** 是一个高度熟练的工程师，他能：
- 深入理解客户的问题
- 在一个从未见过的代码库里写代码
- 向非技术决策者沟通业务影响
- **关闭交易**

这被认为是一个**百万美元级别的招聘**。

> "Palantir 的 CTO 说，你不能在没有真正进入环境的情况下为那个环境构建产品。"

FDE 这个词来源于 Palantir。2010 年他们在阿富汗与特种部队合作——突击队白天执行任务，汇报反馈，FDE 夜间通宵部署代码。

---

## 二、FDE 的三步工作流

一个 Applied AI FDE 的工作可分为三个阶段：

```
Audit（审计） → Evals（评估） → Deployment（部署）
```

### Phase 1：Audit — 现场勘查

你到客户现场，逐个团队绘制工作流地图。比如：两周跟 Revenue Ops、一周跟采购、一个月跟财务。

**在每个团队你要搞清三件事：**
1. 他们的工作是什么样子
2. 瓶颈在哪里
3. 哪里可以建 Agent 来创造价值

#### 自动化决策三原则

**原则一：AI 的边界条件**

| 工作流特征 | 方案 | 
|-----------|------|
| 规则可归纳 + 输入格式多变（邮件/PDF/扫描件） + 需要调用工具 | ✅ **Agent** |
| 规则和输入都稳定 | ✅ **代码**（更快更便宜） |
| 需要人的判断 + 不同情况应对不同 | 保持人工 |

**原则二：ROI 要够大**

Agent 一个月只跑五次，不值得。找**高频、耗时**的自动化场景。

**原则三：别滥用 AI**

大部分自动化任务只需要一系列工具调用 + **一次 LLM 做编排层**。过多 AI 调用导致不必要的 token 成本（规模化后惊人！）和质量下降。

**审计阶段的最终产出：原型 (Prototype)。**

### Phase 2：Evals — 证明价值

客户花几百万部署 AI，他们需要知道系统**真的在干活**。

#### 如何构建好的 Eval

**1. 追踪人类的步骤，对 AI 每一步打分**

人不会一步解决复杂问题。画出人类的工作步骤，检查 AI 是否在每个检查点达到同样标准。

**2. 从小处开始，用优质样本做基准**

比如做客服 Agent：坐在人类客服旁边，搞清楚"最佳回答"长什么样。重复几次。现在你知道"优秀"的标准了。

> Evals 向客户证明价值。虽然人人都说想要 AI，但仍有不少人怀疑它是否真的有用。好的 Eval 是高管信任系统的基石。

### Phase 3：Deployment — 安全上线

**数据策略：** 避免大规模数据迁移。在现有数据层（SharePoint、数据库等）上构建 API，放一个模型做编排层查询即可。节省时间、金钱，最重要的是避免了把现有系统连根拔起的噩梦。

**沙箱测试：** 在客户的基础设施内创建执行环境，安全地运行、测试和调试 Agent。

**渐进上线：** 从小工作流开始，跑通了再叠加能力。

> **核心原则：从最小的自主单元开始。跑通了，再给它更多行动能力。**

示例路径：
1. Agent 捕获 Bug → 调查 → 写 ticket 总结原因
2. 跑通后 → 给它写修复代码的能力
3. 再跑通 → 给它部署修复的能力

## 三、30 天成为 FDE 的路线图

### Checkpoint 1（第 7 天）

| 能力 | 学习资源 |
|------|---------|
| 什么是 Agent？Agent Loop 怎么转？ | 读 Anthropic 的 Building Effective Agents，写一个 prompt → model → response 的循环脚本 |
| Agent 如何调用工具？ | 加两个工具调用（API 调用 + Web 搜索），用 Anthropic/OpenAI 的 tool use 教程 |
| 怎么建规范的安全护栏？ | 输入校验 + 最大步数限制 + 输出过滤 |
| 上下文窗口 vs 外部记忆怎么选？ | 默认用上下文，除非状态需要跨运行期持久化 |
| 什么是审计追踪？ | 记录每次 prompt、工具调用和响应的 timestamp |

### Checkpoint 2（第 14 天）

| 能力 | 细节 |
|------|------|
| 强制结构化输出 | 始终返回 JSON，读 OpenAI 开发者文档 |
| Demo 到生产 | 读 Agents 102，了解什么通常会崩 |
| 检查点 | 每 n 步保存 agent 状态到文件，可从最后检查点恢复 |

### Checkpoint 3（第 21 天）

| 能力 | 细节 |
|------|------|
| 重试逻辑与指数退避 | 每次外部调用都要重试：失败后等 1s → 2s → 4s → 8s，上限 16s |
| 部署成本优化 | 便宜子任务用便宜模型（Opus 只用于推理），缓存通用 prompt，限制 max tokens，追踪每次查询成本 |
| 构建黄金数据集 | 从 20 个真实查询开始，自己标出完美输出。读 Anthropic 的 Demystifying Evals for AI Agents |
| 多 Agent 流水线 | 当一个 Agent 搞不定时拆分任务：一个规划、多个执行、一个汇总 |

### Checkpoint 4（最后一周）

复盘以上所有内容，**用语言大声讲出来**。把每件事都与业务指标挂钩。

## 四、不同背景的人怎么做

### 咨询/PM 背景

优势：把数据翻译成 ROI 的能力
短板：工程经验不足

**补救：高质量作品集。选 2 个做完：**

| 项目 | 要求 |
|------|------|
| 生产级 Agent | 能执行你以前手动做的整个流程，能调 API、自主记录思考过程、有失败处理机制 |
| RAG 流水线 | 基于自定义数据集（选你想进入的行业：法律文档、病历、财务档案等） |
| Eval 框架 | 自己写一个，在多个维度（正确性、格式、成本、延迟）上对 Agent 输出评分 |
| MCP 服务器 | 把传统软件连上 LLM，实现 AI 集成 |

> **不要用 AI 外包你的理解。** 一步一步来，这些概念是可理解的。这是 30 天路线图，不是 30 分钟。

### 软件工程师背景

优势：工程能力
短板：沟通能力

> **"FDE 最重要的能力是沟通。你需要把 AI 能做什么、不能做什么，翻译成非技术 VP 能听懂的话。做不到，就不能做 FDE。"**

SWE 应该构建和前面同样的项目，但对**每个组件都要能解释**：技术栈、结果、迭代过程、业务价值。最重要的是，你**为什么**要构建这些 Agent。

### TLDR

1. **FDE 是目前科技界最热门的需求角色。** 每家公司都需要 AI，但没人知道怎么部署。
2. **工作有三个阶段：Audit → Evals → Deployment。**
3. **作品集和你的表达能力是决定因素。** 构建 Agent、RAG 流水线、Eval 框架、MCP，并且能自信地解释每件事背后的业务价值。
4. **沟通能力不行，一票否决。** 如果你不能向非技术决策者解释 AI 的能力边界，就没有部署。
5. **知道什么时候不该用 AI。** 这建立客户信任，也是生产环境 Agent ROI 的来源。

---

> **关联：** FDE 的最佳实践与 Harness Engineering 紧密相连——Eval 阶段对应 Harness 的验证循环，Deployment 对应沙箱 + 渐进式自主，Audit 对应 Context Engineering 的前期工作。
