# Claude Code 27 个隐藏功能、设置与快捷键

> **来源：** [27 Hidden Claude Features, Settings & Shortcuts That Most Users Don't Know](https://x.com/sairahul1/status/2070428662080618607) — Rahul (@sairahul1)

---

大多数人用 Claude Code 就像用高级自动补全工具：输入 prompt → 等待 → 接受第一个输出。**把 90% 的能力浪费掉了。**

这 27 个技巧将开发者从"2 倍产出"带到"10 倍产出"。分为 Beginner → Intermediate → Pro 三个等级。

> **核心思维模型：** Claude Code 不是一个自动贩卖机（放入 prompt 得到代码）。它是一个非常有才华的初级开发者——需要适当的入职引导、清晰的约束条件和定期 check-in。

所有技巧归根结底就是一件事：**给 Claude 更好的结构，让它能走得更远而不需要你。**

---

## 🟢 初级技巧（Beginner）

先掌握这些，它们能避免 80% 的浪费会话。

### 1. 在任何新项目中，先跑 `/init`

在每个新项目做的第一件事：运行 `/init`。Claude 读取你的代码库，生成一个 **CLAUDE.md** 文件。这个文件成为项目大脑——架构决策、编码规范、模式、命名规则。**后续每个会话都从这个已加载的上下文开始。** 没有它：Claude 猜测你的规范。有它：Claude 已经知道。**一条命令。不要跳过。**

### 2. 使用 `/statusline` 查看实时状态

配置 `/statusline` 把终端变成一个**实时仪表盘**：你能看到 token 用量、会话指标、模型信息——全部实时显示。不要猜你已经消耗了多少上下文，直接看数字。

### 3. 启用语音输入

免手编码让你保持在解决问题的模式。不用停下来打字，直接说出下一个指令。在审阅输出并想重定向时特别有用。设置中启用一次，然后频繁使用。

### 4. 用 `/context` 找到谁在吃你的 token

你的上下文窗口是有预算的。当它变重时，用 `/context` 检查什么在消耗它。你几乎总能发现死重——旧文件、无关上下文、过时的指令。**修剪它，输出立刻改善。**

### 5. 60% 时压缩，任务间清理

两条上下文卫生规则：
- 上下文达到 **60%** 时运行 `/compact`——Claude 压缩对话而不丢失关键信息
- 在完全不相关的任务之间运行 `/clear`——重新开始

大多数开发者等到上下文完全满了才做。**那时 Claude 已经在降级了。60% 是最佳节点。**

### 6. 开始之前，总是进入计划模式

在 Claude 写一行代码之前——让它先做计划。为什么有效：
- 迫使 Claude 先思考再行动
- 早期暴露不匹配的假设
- 在消耗算力之前给你一个检查点

以计划模式开始会话多花 3 分钟。没有计划的会话往往要花 3 个小时修复。

### 7. 呈现问题，而不是命令

大多数人说："给我建个登录页面。"

更好的方式："用户在用带特殊字符的密码时无法登录。这是错误信息。这是约束——我们不能改数据库 schema。"

**问题 + 约束 + 上下文 = 显著更好的输出。Claude 不是命令执行器，是问题解决者。**

### 8. 让 Claude 在开始前先提问

在每个复杂 prompt 中加上这句：
> 开始之前，列出你做这件事需要的所有澄清性问题，一次性问完。

第一个输出几乎从不是最好的输出。最好的输出出现在 Claude 理解了边界条件之后。让它问，一次性回答，然后看它跑。

### 9. 把自我验证编入待办清单

给 Claude 任务清单时，加上验证步骤：

```
完成每个步骤后，通过以下方式验证：
→ 运行测试
→ 检查类型错误
→ 确认没有副作用
```

没有这个：Claude 在第 2 步就宣布胜利。有这个：Claude 实际上完成了全部。

## 🟡 进阶技巧（Intermediate）

一旦基础到位，这些带来真正的生产力飞跃。

### 10. 部署子智能体进行并行工作

最被低估的功能之一。不要按顺序做任务，把它们分配到有隔离上下文的子智能体上：

"启动一个子智能体处理数据库 schema，另一个处理 API 端点，你来协调集成。"

并行工作，隔离上下文，无交叉污染。**生产力提升不是增量式的，是倍增式的。**

### 11. 把可复用的 prompt 保存为自定义 Skills

每次你重复写相同的指令，你都在亏损。把可复用的工作流保存为 skill 文件：

```
/claude add skill <name> <instruction>
```

加载 skill → Claude 遵循你的标准 → 每次如此。**永远不要再写相同的指令。**

### 12. 用更轻量的模型处理简单任务

- **Haiku** 用于：文件重命名、简单重构、样板代码、配置编辑
- **Sonnet/Opus** 用于：架构决策、复杂调试、多文件更改

大多数开发者用 Opus 做所有事。**这就像开法拉利去菜市场。让模型匹配任务。**

### 13. 随项目演进持续更新 CLAUDE.md

`/init` 生成的 CLAUDE.md 不是一次性的文件。随项目变更持续更新它。过时的上下文比没有上下文更糟糕——**过时的 CLAUDE.md 会让 Claude 自信地走向错误的方向。**

### 14. 从 CLAUDE.md 引用外部文件，而不是内联粘贴

不要把所有的东西都塞进 CLAUDE.md。引用外部文件：

```
阅读 docs/ARCHITECTURE.md 和 docs/CONVENTIONS.md
```

Claude 只加载需要的东西。**你的上下文保持精简，你的 token 预算保持健康。**

### 15. 一旦会话漂移，立刻终止

Claude 偏离轨道的那一刻——**停止它**。不要让糟糕的工作累积。累积的糟糕工作比重新开始更难修复。尽早捕获漂移，强硬地重新提示。

### 16. 质疑第一个输出

接受是高质工作的敌人。每次输出后：

"批判性地审查这一点。什么问题？什么边界情况被忽略了？"

**Claude 第一个答案很少是最好的。最好的答案在被挑战之后才出现。**

### 17. 设置通知 hooks

停止盯着终端。设置 hooks，在长时间任务完成时 ping 你：

```
claude hook add on_task_complete "notify-send 'Done!'"
```

去做别的事。**等你被叫回来。**

### 18. 给 Claude 喂截图进行布局验证

Claude 看不到你的运行中应用，但它能分析它的截图。直接把截图粘贴到会话中。**视觉调试快了 10 倍。**

## 🔴 高级技巧（Pro）

在这里 Claude Code 不再是编码助手，而开始像一整个团队。

### 19. Chrome DevTools 集成

让 Claude 通过 DevTools 与你的运行中应用交互。它可以实时读取控制台错误、网络请求和 DOM 状态。**调试变成了：Claude 看到页面 → Claude 修复页面。** 而不是：你复制粘贴错误 → Claude 猜测。

### 20. 恢复之前的会话

永远不要丢失上次会话的上下文。

```
claude --resume <session-id>
```

Claude 打开确切的会话——完整对话历史、完整上下文、你离开的确切位置。在 Claude Code 历史中找到你的会话 ID，把你要返回的会话加入书签。

### 21. 用 Git Worktrees 实现并行会话

这是严肃项目最大的生产力解锁。同时在不同的分支上运行多个 Claude 会话：

- 会话 1：构建认证
- 会话 2：构建支付
- 会话 3：构建仪表板

都在同时运行。**这就是你怎么在一周内交付一周的工作量。**

### 22. 用直接 API 端点替代 MCP 服务器

MCP 服务器增加开销。对于性能关键或高容量的工作，直接调用 API 端点：

```
// 不用 MCP 服务器
curl -X POST https://api.github.com/repos/.../issues
```

**更少开销、更少 token 消耗、更低延迟。昂贵的工具调用来一次直接 API 替换，立刻改善。**

### 23. 用 `/loop` 自动化重复任务

Claude 可以**无人值守连续运行最多 3 天**。设置它，走开。Claude 监控、记录、分类——持续数天。

### 24. 从手机远程控制

远程管理你的本地 Claude 会话。不在桌前时想起忘了启动某个任务？**打开手机 → 启动任务 → 你到家前它就完成了。**

### 25. 用自然语言查询数据库

对于一次性分析，完全跳过写 SQL：

```
"告诉我上周注册的用户中，哪个地域的转化率最高"
```

Claude 翻译成 SQL，运行它，返回答案。**不用写 SQL、不用记 schema，只是一个问题和答案。**

### 26. Ultrathink：用于真正困难的问题

当问题真的复杂时——分配最大 token 并明确告知 Claude：

```
这个问题的背景如下：[复杂的上下文]
这是一个核心问题，请花时间仔细思考，检查多方面可能性，
展示你的推理过程，不要仓促回答。
```

90% 的任务快速就好。**10% 真正困难的——放慢，思考预算在那里，用它。**

### 27. Ultracode：Opus 4.8 中的最高编码模式

为了最大的编码性能：

```
请以最高努力模式运行。这是关键系统更改。
```

这会激活 **ultracode**——Claude 的最高努力编码模式。保留给：
- 复杂的架构决策
- 安全关键实现
- 性能优化环节
- 多系统重构

它消耗更多 token，花更长时间。**但对于困难问题，它产生显著更好的代码。**

## 27 个技巧背后的模式

每一个技巧都是以下三件事之一：

| 类别 | 技巧 | 核心思想 |
|------|------|---------|
| **上下文纪律** | CLAUDE.md、/init、/compact、/clear、文件引用 | 给 Claude 更好的信息，就能做更好的决策 |
| **智能委派** | 子智能体、worktrees、并行会话、/loop | 当 Claude 能做很多事时，停止一次只做一件事 |
| **激进自动化** | hooks、远程控制、skills、数据库查询 | 任何手动做超过两次的事，都该让 Claude 做 |

**10 倍产出的开发者并不更聪明。他们更系统化。**

## 从哪里开始

不要试图今天就把全部 27 项都实现：

| 时间 | 重点 | 效果 |
|------|------|------|
| **第 1 周** | `/init`、`/compact`、计划模式、自我验证待办 | 这四项就能彻底改变你的会话 |
| **第 2 周** | 自定义 skills、轻量模型、CLAUDE.md 更新、质疑每个输出 | 不再重复写相同的指令 |
| **第 3 周** | 子智能体、worktrees、并行会话 | 同时交付多个功能 |
| **第 4+ 周** | `/loop`、ultrathink、DevTools、远程控制 | 完全不再盯着终端 |

**系统会叠加。每一层让下一层更强。从开头开始，掌握它，向上移动。**
