# Loops 详解：Claude、GPT、Mira 以及真正有效的方式

> **来源：** [Loops explained: Claude, GPT, Mira and what actually works](https://x.com/anatolikopadze/status/2068328135611822149)

![Cover](../image/loops-explained-cover.jpg)

---

AI 已经进入每个人的生活好几年了。但大多数每天都在用 AI 的人，用的仍然是最慢的方式：打字、等待、修改、再问，全都靠手动。

不是快的方式复杂，而是没人给他们展示过快的方式长什么样。

快的方式叫 **loop（循环）**，而这正是全世界最优秀的 AI 工程师眼下最关注的事情。这篇文章会补上没人解释过的那部分。

读完这篇文章，你会比 timeline 上绝大多数人更懂 loop：它是什么，底层到底怎么工作，什么时候值得用、什么时候是陷阱，如何自己在 Claude 或 ChatGPT 里搭一个基础的，以及那些值得你在生活中运行的简单 loop。

## 大多数人怎么用 AI？

仔细观察那个「一次一问」的习惯，因为这就是问题的全部。每一步都要经过你。你决定问什么，你判断答案，你决定下一步。AI 永远不会自己动，除非你推它。你一停，它就停。

这没什么不好，但它有天花板——**你是引擎**。AI 只是你手里的工具，工具自己什么也做不了。

还有另一种工作方式，这也是全世界最优秀的工程师正在改变开发方式的原因。你不再一步步牵着 AI 走，而是把目标一次性给它，让它自己跑完所有步骤：规划、执行、检查结果、修复弱项、重复，直到目标达成。**你退出来，工作继续。**

> （文章此处插入了两段来自知名工程师的推文视频，表述同一个观点——loop 是真正的变革）

两位最受尊敬的工程师，用不同的话说着同一件事。大多数人读到这些话时，其实不太清楚它们在实践中到底是什么意思。所以，我们来好好拆解一下。

## 什么是 Loop？

**Prompt（提示词）是一个单次指令。Loop 是一个目标，AI 朝着它持续工作直到达到目标。** 可以把它理解为一个递归目标：你定义一个目的，AI 不断迭代直到完成。

Prompt 给你一个答案就停下了，等你来决定下一步。而 Loop 自己跑完整个周期：

```
发现  →  弄清楚需要做什么
规划  →  决定怎么做
执行  →  做工作
验证  →  对照目标检查
迭代  →  没达标？把结果喂回去，重复
```

这五步里有三步才是真正干活的，也是人们最容易搞错的地方。

**验证（Verify）是 loop 的心脏。** 如果没有一个真正的检查机制，那你就不是有了一个 loop，而是让 AI 在自说自话地重复。检查才是把重复变成进步的东西。它可以是一个硬测试（"代码能通过吗"）、一个可衡量的条件（"数字是否超过 X"），或者一个评分标准。没有关卡，就意味着 AI 在给自己批改作业——而干活的那个模型对自己总是太宽容。

**状态（State）让 loop 学会进步。** 每一轮，AI 必须记住自己已经试过什么，否则它会永远重复同一个错误。一个真正的 loop 会在一旁维护一份小记录：什么做完了、什么失败了、下一步是什么。下次运行可以从之前断掉的地方继续，而不是从零开始。这也正是开销开始飙升的地方，后面会讲到。

**停止条件（Stop Condition）让它保持理智。** 没有退出条件的 loop 会一直跑下去，直到成功、崩溃、或者耗尽你的余额。每个正经的 loop 都有两种停止方式：成功，和一个硬限制（"试 8 次后停止并报告"）。跳过这个，你就造了一台可以白跑一整晚的机器。

**Prompt 给 AI 一条指令。Loop 给 AI 一份工作、一个判断工作完成的方法、以及一条何时放弃的规则。**

## 你真的需要 Loop 吗？

大多数文章在告诉你什么时候不该用 loop 之前，就先推销你 loop 了。以下是真正认真的人实际使用的判断标准。**Loop 只有在这四个条件都满足时才值得搭建：**

- **任务至少每周重复一次。** 低于这个频率，搭建成本永远回收不了。一次性任务还是用一个好 prompt 更合适。
- **有可以自动拒绝糟糕输出的东西。** 测试、类型检查、构建、linter、硬规则。**如果没有什么可以帮你判断工作不合格，loop 只是在空转。**
- **AI 确实可以自己端到端地完成工作，** 而不是把一半交回给你。
- **"完成"是客观的，不是见仁见智的判断。** 如果质量是口味问题，人类还是更强。

**缺一个条件，就继续用手动 prompt。** 这个话题诚实的版本是：loop 工程是真实存在的，但大多数人暂时还不需要那个重型版本。人人能用的轻量版后面会讲到。但你应该先知道界限在哪里。

## 为代码而生的 Loop

Loop 最先在软件领域起飞，因为代码是世界上最容易验证的东西。测试要么通过要么失败，没得争——所以 AI 始终知道自己做完了没有。

一个编码 loop 被赋予目标和一个严格的检查方式：

```
▸ LOOP 规格
目标：/tests/auth 里的所有测试通过，lint 干净，没有类型错误

每一轮迭代：
  1. 运行测试套件，读取每一个失败
  2. 挑出影响最大的那一个失败
  3. 写最小的改动来修复它
  4. 重新运行测试、lint 和类型检查

验证：全部测试通过 + 零 lint 警告 + 零类型错误
停止条件：验证通过，或达到 8 次迭代
停止时：总结改变了什么、还有什么失败
```

一个真正的 loop 由五个积木块组装而成。Claude Code 和 Codex 现在全部自带了这五个：

### 1. 自动化（心跳）
这是让它成为 loop 而不是一次性的触发器。你定义一个 prompt、一个节奏和一个目标，它会按计划自动运行，不需要你启动。在 Claude Code 里，`/loop` 按间隔重新运行 prompt，`/goal` 让会话持续到条件真正满足，hooks 在 agent 生命周期的特定时点触发命令。推到 cron job 或 GitHub Actions 上，关上笔记本它也会继续跑。结果来找你，而不是你到处去检查。

### 2. 技能（可复用的指令集）
不用每次都粘贴一大段指令，把它们存成一个文件，让 loop 每次都读它：规则、要遵循的模式、绝对不能碰的东西。现在自动化只需要通过名字调用技能，重复性任务保持可维护，而不会腐烂在没人更新的计划任务里。

### 3. 子 Agent（让干活的和检查的分开）
Loop 结构里最有用的一招：把干活的那个 agent 和检查它的 agent 分开。写代码的模型对自己太宽容。用不同的指令、有时用更强的模型和高 effort 来检查，能抓住第一个 agent 说服自己的那些问题。写的人可以用快又便宜的模型，审查的人用慢但严格的模型。**这种分离是大部分质量的来源。**

### 4. 连接器（让它行动，而不是建议）
这就是"我来修复"和"我帮你开了 PR、关联了 ticket、构建通过了还在频道里发了通知"之间的区别。连接器让 loop 在真实环境中行动，而不是只会描述它"如果能够"会做什么。

### 5. 验证器（关卡）
能够自动拒绝不合格工作的测试、类型检查或构建。**这一个模块决定了 loop 是帮你还是烧你的钱。** 其他都是管道。这才是让它真实的那个部分。

把这些组合起来，你就得到了大团队现在规模化运行的东西：一群 agent 同时循环做同一份工作，几十个甚至上千个。一位工程师用这种 loop 在 **大约六天内将整个代码库从一种语言重写为另一种**——手工做需要将近一年。这是正经软件开发的真正变革。而它有一个演示视频里永远不会展示的陷阱。

## 没人提到的成本

Loop 跑在 token 上，token 就是钱。问题不是每一步花一点钱，而是**成本如何复利增长**。

每次 loop 转一圈，agent 都要重新读取它的上下文：目标、代码、上次的结果、什么失败了。这一整堆东西每一轮都要重新发给模型，而且每一轮都在变大。跑了十次的 loop 不是花十个 prompt 的钱，而是花十个**越来越大的** prompt 的钱。maker-and-checker 那招提升了质量，但也让账单翻倍——因为现在两个模型都在读同一份工作。

```
▸ 一个 LOOP 的大致成本
单 agent、中等任务：         ~50,000 – 200,000 tokens
上下文每轮重发：             每轮增长
一队 agent 并行：             以上全部乘以倍数
```

**真正重要的指标，几乎没人追踪的，是每次验收的变更的成本（cost per accepted change）。** 不是 token 花了多少，也不是 loop 跑了多少次。如果 loop 给了你十个结果你扔了六个，那你相当于在做它原本想帮你省掉的审查工作。接受率低于 50%，它花的比省的多。

Loop 也会静默失败。工程师 Geoffrey Huntley 称之为 **"Ralph Wiggum loop"**（辛普森一家里的那个笨小孩）：agent 过早地认为自己完成了，在一个半成品上退出，而 loop 继续跑着，持续消耗却什么也产不出。没有能够判定工作不合格的硬关卡，loop 不会崩溃——它会**静默地扣你的费**。

这就是为什么重型版属于有钱和防护栏的团队使用：迭代上限、token 预算、无聊步骤上用便宜的模型、监控。如果你不在此列，你也没错过什么——核心思想用零头成本、零搭建就能工作。

## 真正有效的搭建顺序

如果你真的要建一个 loop，**顺序比工具重要**。能把 loop 推到生产环境里活下来的人，全是这么做的：

```
1. 先把一次手动运行做稳定。
2. 把它变成技能（保存指令）。
3. 把技能包进 loop（加上关卡 + 停止条件）。
4. 然后才排上计划。
```

跳过前面，把一个还没手动调可靠的东西直接排上计划——这就是 loop 在你睡觉时爆炸的方式。**先手动证明它能跑通，再加固，再自动化。**

## 自己搭一个基础的 Loop（任何 LLM 都行）

你不需要一个 coding agent 来感受 loop 怎么工作。你现在就可以在任意 LLM 里用手动方式跑一个简单的 loop，只需要一段 prompt。诀窍是一次性给模型 loop 的三个部分：目标、严格的成功标准、以及一个迫使它在允许自己停止之前先自我检查的协议。

```
▸ 自检 LOOP（粘贴到 Claude 或 ChatGPT）

你将在一个循环中工作，直到任务达到标准。

任务：
[准确描述你想要生成的内容]

成功标准（严格，不要软通过）：
- [标准 1]
- [标准 2]
- [标准 3]

循环协议，每轮重复：
1. 规划   - 说明下一步的唯一行动
2. 执行   - 生成或改进工作
3. 验证   - 对每个标准打分 1-10
           要极其诚实。准确列出仍然薄弱的地方。
4. 决定   - 如果每个标准都是 8+，打印 "FINAL" 然后停止
           否则打印 "ITERATING" 然后继续，优先修复
           上轮验证中最弱的那个点

规则：
- 每个标准未达到 8 或以上，绝不宣布完成
- 每轮必须修复上一轮验证中最弱的分
- 不要问我问题。做出合理的假设，记录下来，继续

开始。运行 loop 直到 FINAL。
```

看会发生什么。模型会起草、按你的标准给自己的作业打分、找到弱项、重写，一遍又一遍，直到它真正越过线，而不是把第一个差不多的结果塞给你。**这就是一个 loop。你用一段话就搭了一个。**

但注意还缺什么——这也是下一部分的重点。**你仍然是触发器。** 你打开了聊天窗口，粘贴了 prompt，你坐在那里看着它迭代。关了标签页它就不存在了。没有计划，没有"每天早上做这个"，没有"收到邮件时醒来"。它没法来找你，因为它只有在你看着它的时候才存在。

要让一个 loop 能自己运行、按计划、被真实事件触发、不需要你值守——你就得进入前面提到的重型世界了：工具、托管、代码、关卡，还有账单。这在你处理真正重量的任务时是合理的。但对于 99% 的日常任务，已经有了一个现成的、极其简单的方案。

## 同样的想法，用在实际生活里

剥掉代码和成本，剩下的就是一个简单、真正有用的概念：**一个能自我运行的任务**——按计划，或者在某件事发生的瞬间——不需要你记住它，也不需要你在场。你不需要是工程师。你只需要为生活搭建的 loop，而不是为代码库搭建的。

有一个免费的选择：用自然语言描述，就能创建。不需要代码、托管、密钥、保持打开的标签页、也不用搞错搭建顺序。它叫 **Mira**，就在 Telegram 里——你很可能已经打开了这个 App。像给朋友发消息一样跟它说话就行，它运行的 loop 叫 **Skills**。

每个 Skill 都安静地装着一个真正的 loop 所需要的全部部件——触发器、动作、自我运行的能力——只是你完全不需要把它们连接起来。你只需要说出你想要的：

```
▸ SKILL
"每个工作日早上 7 点，检查我的 Gmail 和 Google Calendar。
给我发一份简短简报：3 个最重要的会议、收件箱里的急事、
以及一件我说过要跟进但还没做的事。控制在 120 字以内。"
```

这就是一个真正的 loop。时间触发器，跨越两个连接的应用的多步操作，自己运行，结果来找你。你只写了一句话。

## Mira 真正能做什么

关键在这里：**Mira 不是一个更聪明的聊天机器人。** 它和 ChatGPT 的区别很简单：ChatGPT 回答，Mira 行动。你不是让它写邮件，你是让它发邮件。你不是拿到一个草稿 ticket，而是拿到一个真正在 Linear 里建好的 ticket，owner 已经分配好。**它真的做事，在后台，而且它记得每段对话之间的你。**

它通过 Composio 连接了 500+ 个应用（Notion、Gmail、Google Calendar、GitHub、Figma、Stripe 等等），有跨会话和群聊的长时记忆，而且它不依赖单一模型——会根据任务跑 GPT、Claude、Gemini。

### 工作上

这是 loop 概念零代码就能体现价值的地方。

```
▸ SKILLS
"每个会议前一小时，提醒我上次和那个人聊的内容和决定。"

"当我转发一条消息过来时，把它变成 Linear ticket，
设置正确的优先级，分配好负责人。"

"每周五下午 4 点，收集团队的任务状态和指标，
在群里发一份干净的一周摘要。"

"总结我在群里时错过的一切，用 5 条要点。"
```

它几秒钟就能让你跟上 200 条消息的讨论，在你继续说话的时候帮你建好 ticket，走进会议室时已经带好了简报。在群聊里，它能记住整个团队的决定和任务，不止是你自己的。

### 创作者

这是大多数人低估的部分。Mira 可以在聊天窗口内端到端地做内容。

```
▸ SKILLS
"我会发一条语音笔记说一个粗糙的想法。把它变成一条带文案
和话题标签的完善帖子。"

"拿这个点子写 X、Instagram、LinkedIn、邮件和 Newsletter
的版本，每种用合适的格式。"

"为这篇帖子生成 3 张图片选项。"

"把这张图片变成一段短视频发到我的 Telegram 频道。"
```

语音笔记进去，完成的帖子出来，大约 30 秒。一段简述变成六个平台的版本。它能在聊天里直接生成图片和视频，编辑照片、换背景、构建吉祥物和头像，甚至能唇同步和做动画。**整个内容管道都在一个窗口里。**

### 语音

Mira 把语音作为一等输入——这比听起来重要得多。

```
▸ SKILLS
"把我的语音消息转写成干净的文字。"
"把这篇文章读成音频给我听。"
"汇总这个群聊里的语音消息，提炼关键点。"
```

它能转录你的语音消息，把文字读给你听，理解群聊里的语音消息并能总结讨论内容，还能在你无法打字时当免提语音助手。

### 生活

同一个引擎，对准其他所有事情。

```
▸ SKILLS
"每天晚上 7 点，问我今天有没有训练。帮我记连续天数，
别让我偷偷断一天以上。"

"每天晚上问我 3 个关于今天的问题，记住答案，
每周告诉我有什么变化。"

"拍张照片帮我追踪卡路里。"

"盯着这个航班，价格降到我的目标线就买。"

"每天早上给我一份不搞标题党的资讯摘要，关于我关注的话题。"
```

一个会盯你打卡的教练。一本真的能记住你、渐渐变成每日陪伴的日记。拍张照片就能追踪热量，不需要独立 App。根据你自己的错误构建的语言练习。盯着航班价格降到目标线就自动买的票务 watcher。去掉标题党的每日摘要。

打开 Telegram。找到 Mira。给它发条消息。免费版马上就能用。先试试这些：

```
@mira, 帮我规划这周
```

**这篇文章里的任何例子，你打进去的那一刻就变成一个正在运行的 loop。**

## 这真正意味着什么

**Loop 不是一个趋势。它是谁在做工这件事上的一个转变。** AI 不再等你一步步推着它走，它开始自己跑完整份工作了。

但也别硬追，别硬塞到它不该去的地方——很多时候你只会白白烧钱而已。

我的建议：**先用现成的免费工具感受一下，等真正觉得不够用了，再开始考虑你真正需要什么。**
