# Loop Engineering: 构建一个在你睡觉时还在写代码的 AI — 完整翻译

> 原文作者：Phosphen (@phosphenq)，2026-06-20
> 62 万阅读，849 赞，1996 书签

---

## 前言

去年 12 月，Boris Cherny 一个月内提交了 259 个 PR。每一个都由 Claude 编写。他说那个月他一次都没打开过编辑器。

他的工作不是写代码——而是写那个写代码的东西。Peter Steinberger 用两行话概括了这个概念，这段文字已被浏览超过 800 万次：

> "你不应该再亲自 prompt coding agent 了。你应该设计循环（loops），让循环去 prompt 你的 agent。"

这就是 **Loop Engineering**。你构建一个小系统：它找到要干的活，交给 agent，检查结果，决定下一步——然后让它在你睡觉时一直跑。

但同一个月，有人暴露了另一面：某人写的循环无人看守跑了 11 天，烧掉 $47,000 才被发现。所以这其实是两种技能，但几乎所有人只教了第一种：**构建能交付工作的循环**，以及 **构建能防止它冲出悬崖的刹车**。

有一个问题决定了你构建的是哪种循环：

> **它是在收敛于某个真实的东西，还是只是一个昂贵的随机游走？**

---

## 1. 循环的六个零件和一个问题

剥开所有噪音，一个能工作的循环由六个部分组成。第一次看到这个列表会觉得太平淡了。但每个零件内部的细节，才是决定循环是稳固运行还是半夜漏钱的地方。

你已经不用再手写这些东西了。一年前，一个循环意味着一堆 shell 脚本，你得永远自己维护和盯着。现在这些零件都内置在产品里了，同样的六个零件，在 OpenAI Codex 和 Claude Code 上都能映射。一旦你看懂了这一点，工具就不再是争论的焦点——循环才是设计本身。

---

## 2. 状态（State）是下一轮唯一继承的东西

从最底层开始，因为其他一切都依赖它。

模型在运行结束后就会遗忘。对话结束、上下文清空，下一轮醒来时一无所知。聊天中的记忆随运行一同死亡。它必须活在磁盘上。

实践中这通常是一个文件。仓库里的 `STATUS.md`，或者通过 Connector 连接的 Linear 看板，记录着哪些已完成、哪些进行中、哪些是下一步，以及少数几件循环绝对不能碰的东西。

**把循环当作你从不看守的夜班。** 评判你的不是半夜 3 点它做了什么，而是早上 9 点桌上留了什么样的便条。先设计好那个便条，循环的大部分也就自动设计好了。

---

## 3. 自动化（Automations）是循环和一次性任务的区别

一个循环只有在它能自动触发时，才配得上这个名字。

**Codex 中**，你可以在 Automations 标签页里构建一个：选择仓库、写 prompt、设定节奏。找到东西的运行落入 Triage inbox；没找到的自动归档。OpenAI 用这个做代码库中那些不显眼的基础工作：每日 issue 分类、CI 失败总结、抓住上周溜进来的 bug。

**Claude Code 中**，通过几个原语实现，但流行的教程经常会搞错细节，所以值得精确说明：
- `/loop` 是一个 skill，不是内置命令——它在你的会话期间按节奏重跑一个 prompt
- 周期性云端任务是 **Routines**，用 `/schedule` 设置，最小间隔 1 小时，合上笔记本也会继续运行。没有本地 crontab
- 最值得学的是 **`/goal`**——它一直运行到你写的条件为真，**每一轮之后由另一个更小的模型检查是否真的完成了**。写代码的那个 agent 不能给自己打分

写出停用条件（stop condition），像写合同一样。"test/auth 目录下所有测试通过"是合同。"让它更好"是让循环一直跑到发薪日的写法。

---

## 4. Worktrees 让并行 agent 不互相踩脚

一旦你运行超过一个 agent，失败模式就变了——不再是模型的问题。而是两个 agent 写同一个文件，就像两个工程师在不同频道上往同一行代码提交。

**git worktree 是它们之间的墙**：一个独立的工作目录，在自己的分支上，共享仓库历史，这样 agent 的修改不会进入另一个 agent 的工作区。

但有一个工具不会告诉你的注意事项：Worktrees 消除了冲突，而不是瓶颈。**你仍然是天花板。** 无论你启动多少个 agent，你自己的审查带宽决定了你能信任多少个。

---

## 5. Skills：把你的意图写一次，放在外面

Agent 每一轮都是冷启动，任何你没有明确表达的意图，它都会用自信的猜测来填补。**Skill 就是写在外部、让模型每次都能读到的意图**，这样它就不再猜测了。

格式在两个工具上是一样的：一个文件夹，内含 `SKILL.md`（指令 + 描述），加上任何需要的脚本。当你按名字调用它，或任务匹配其描述时触发——**这正是为什么平淡、字面的描述比花哨的描述更好用**。

没有 Skills，每个周期都要从头学习你的项目。有了 Skills，循环每天早上都会锋利一点。

---

## 6. Connectors 让循环行动，而不只是说话

一个只能读写文件系统的循环是玩具。Connector 建立在共享的 MCP 标准上，让 agent 读取问题追踪器、访问 staging API、查询数据库、往频道发消息。

这就是"说'这是修复方案'"的 agent 和"打开 PR、链接 issue、报告全绿"的循环之间的分界线。因为两个工具都说 MCP，你为一个工具写的 connector 通常可以直接用在另一个工具上。

---

## 7. Sub-agents 让制造者和检查者分开

任何循环中价值最高的操作：**把写代码的 agent 和检查代码的 agent 分开。**

一个模型审查自己的代码，每次都会通过。第二个 agent——使用不同的指令，理想情况下使用不同的模型——能抓住第一个 agent 自己说服自己的那些东西。

因此一条实用的模式是：**一个探索，一个实现，一个对照规范验证。** 这就是 `/goal` 在底层已经做的事情——用全新的模型来判断工作是否完成。第二意见（Second opinions）要花 token，因为每个 agent 运行自己的模型和工具——所以把 token 花在犯错代价最高的地方。

---

## 8. 一个循环长什么样

把六个零件拼起来，一个单线程就变成了一台小机器。

一个定时自动化每天早上在仓库上运行。它调用你的 triage skill，读取隔夜的 CI 失败、打开的 issue、最近的 commit，把每个真实发现写进 `STATUS.md`。对每个值得做的发现，循环打开一个隔离的 worktree，派出一个 sub-agent 起草修复，第二个 sub-agent 对照你的 skills 和测试做审查。Connector 打开 PR、更新 issue。循环处理不了的东西留在你的 triage inbox 里。`STATUS.md` 记住了什么通过了、什么还开着，所以明天从今天停下的地方继续。

你醒来时看到的是一段简短的交接，而不是一堵日志墙。

你只设计了一次。你一次都没有 prompt 它。

---

## 9. 在你走开之前，先装好刹车

这是没人教的那一半，也是决定你的循环是资产还是负债的那一半。

那个 $47,000 的账单不是因为某个聪明的模型叛变了。**它是一个分析 agent 和一个验证 agent，彼此礼貌地要求对方多做一点工作——没有步数限制、没有预算上限、没有停止条件。** 跑了 11 天。任何一个上限就能在第一轮杀了它。

**在装引擎之前装好刹车。在装引擎踏板之前，先装好刹车踏板。**

用循环能破坏的东西来限定它的范围，而不是用你想让它做的事：哪些仓库、哪些分支、多少美元、多少步后强制退出。**先选爆炸半径，再选任务。**

并且诚实地计算成本。Steinberger 的 100-agent 舰队是真的，每月大约运行 $130 万。它有效是因为 OpenAI 现在雇佣了他并支付账单。前沿案例是有人赞助的。你的不是。在 $20 到 $200 的套餐上，能盈利的循环是：**小规模的、有上限的、指向一个乏味工作的——不是一个 swarm。**

## 10. 循环怎么死

每一个失败的循环都死于四种死法之一。现在记住它们的名字，以便凌晨 3 点能识别出来。

### 失控递归（Runaway Recursion）
两个 agent 永远喂养对方。解药就是上面说的步数上限和预算上限——没什么更聪明的办法。

### 静默死亡（Silent Death）
某个开发者的隔夜运行遇到了完整的上下文窗口，停止了，然后继续尝试恢复进入同一个死墙——同时看起来还活着。你的循环可能已经死了几个小时还在报告进度。解药是心跳检测和每个阶段用新的上下文——而不是一个无穷无尽的运行。

### 随机游走（Random Walk）
没有可验证的停止条件，循环漂移远离目标而不是接近它。一个通过的测试套件是一个它可以达到的固定点。而"看起来做完了"不是。

### 理解债务（Comprehension Debt）
循环越快地把不是你写的代码交付出去，你的仓库实际在做什么和你理解之间的差距就越大。让它无监督地运行太久，你就不再是工程师，变成了一个橡皮图章。解药是一个循环永远不允许跳过的**人工审查门（human-read gate）**。

## 11. 构建循环，保持做工程师

两个人构建完全相同的循环，可能得到完全相反的结果。一个人用它在他深入理解的领域上跑得更快。另一个人用它来完全停止理解工作本身。循环分不清他们。你能。

Cherny 的意思从来不是说工作变容易了。而是杠杆变了：**从 prompt 转移到了循环，从打字转移到了判断。** 这是一份比写 prompt 更难的工作，而不是更轻松的。

所以这是行动方案——故意很小，因为现在还很早，成本波动很大：

> 明天早上，拿起你仍在手动完成的最无聊的工作——分类 CI、关闭过时的 issue、追踪一个 flaky 测试——然后用一个有上限的循环包裹它。**先装刹车。小到足以让你还阅读每一份它交付的 diff。**

没有人是从 100 个 agent 开始，就月交付 200 个 PR 的。他们从**一个他们信任的循环**开始，并且全程保持做工程师。先构建那一个。

## 💡 核心洞见

1. **Loop Engineering > Prompt Engineering** — 循环设计是比 prompt 更高级的杠杆层，将人类从"操作员"变为"设计师"
2. **刹车比引擎更重要** — $47,000 的真实教训：任何循环都需要步数限制、预算上限和停止条件
3. **四种死法可预测** — 失控递归、静默死亡、随机游走、理解债务，每种的解药都已知
4. **理解债务最危险** — 代码写得越快，人不理解的部分越多，最终变成橡皮图章
5. **先构建一个可信的循环，而不是一百个** — Boris Cherny 的 259 PR 是从一个开始，不是从 swarm 开始
6. **工具已抽象** — Codex 和 Claude Code 共享相同的六零件架构，区别只在具体实现细节
