# AI Agent 的上下文工程：完整操作手册

> **来源：** [Context Engineering for AI Agents: The Complete Playbook](https://x.com/sairahul1/status/2067171101978071501) — Rahul (@sairahul1)

![cover](../image/context-engineering-for-ai-agents-cover.jpg)

你的 AI agent 前 10 步跑得很好。

然后大概在第 15 步左右，它开始变得敷衍了。错误的工具调用、忘记原始指令、输出质量下降。

大多数人怪模型。

但几乎从来不是模型的错。

——是模型看到的内容有问题。

**组织模型看到的内容，就叫上下文工程（Context Engineering）。**

它正在迅速成为构建 AI agent 最重要的技能。

下面是完整的操作手册。

---

## Prompt Engineering 已死，Context Engineering 才是现在需要的

你听过 prompt engineering——写清晰的指令、提供好的示例、告诉模型扮演什么角色。

这对聊天机器人来说完美有效。

但一旦你开始构建 agent，它就不管用了。

原因如下：

- **聊天机器人**回答一个问题就停止。
- **Agent** 则持续行动——浏览网页、调用 API、写代码、跑命令——一步一步又一步，有时几十步。

每一步的输出都会加入模型的上下文。

而上下文是有限的。

Anthropic 的工程团队这样定义：

> 「上下文是采样 LLM 时包含的 token 集合。上下文工程是优化这些 token 的效用，以稳定实现预期结果。」

简单说：确保你的 agent 在**正确的时间**看到**正确的信息**，以**正确的格式**。

Prompt engineering 是上下文工程的一个子集。上下文工程才是全部。

---

## Agent 的上下文窗口就是 RAM——它在被填满

LangChain 用了一个很好的类比。

把 LLM 想象成一种新型操作系统：
- **模型 = CPU**——负责思考
- **上下文窗口 = RAM**——工作内存，模型当前能看到并推理的一切都在这

就像电脑 RAM 满了会变慢，上下文窗口拥挤时 agent 的推理能力也会下降。

**这就是「上下文腐败」（Context Rot）。**

### 研究数据

Chroma 做了一个研究，评估了 18 个前沿模型——GPT-4.1、Claude 4、Gemini 2.5、Qwen3 等。

**所有模型的性能都随着输入长度增加而下降。**

不是在硬限制处，而是在之前就开始了。

一个拥有 200K token 窗口的模型，可能在 50K token 处就出现明显退化。

这种下降是**连续的**，不是悬崖式的。

为什么？因为 Transformer 的工作方式是每个 token 关注每个其他 token——产生 n² 的关系。上下文越大，模型维持所有关系的能力就越弱。

### 「Lost in the Middle」问题

LLM 呈现 U 型的注意力曲线：
- → 上下文开头：记忆良好
- → 上下文结尾：记忆良好
- → 中间部分：基本被忽略

研究人员发现，当相关信息从上下文开头移到中间时，准确率**下降 30+ 个百分点**。

你的原始指令——被 50,000 token 的工具输出埋没——实际上消失了。

Claude Code 用户发现，输出质量在上下文容量的 **40–60%** 时就开始下降，远在任何硬限制之前。

## 争夺上下文空间的 7 类内容

所有 7 类都在争夺同一个有限的窗口：

### 1. 系统提示（System Prompt）
Agent 的身份、行为规则、控制流逻辑、不同任务类型的指令。在 agent 中，这不仅仅是「要友好」——它可以定义整个架构。

### 2. 工具定义（Tool Definitions）
Agent 可能调用的每个工具都需要一个 schema，描述它的功能、参数以及何时使用。

### 3. 工具调用结果（Tool Call Results）
每次工具调用都会把输出加入上下文。一次网页检索：5,000–10,000 token。一次文件读取：类似。这些积累得很快。

### 4. 检索知识 / RAG
从向量数据库、搜索结果、API 响应中拉取的文档——任何用来通知 agent 决策的信息。

### 5. 对话历史（Conversation History）
已发生一切的完整记录：用户消息、agent 响应、推理过程、之前的决策。每次轮次线性增长。

### 6. 记忆（Memory）
当前会话的短期记忆。来自之前会话的长期记忆——用户偏好、之前的结果、学到的模式。

### 7. Agent 状态（Agent State）
当前计划、待办列表、进度标记、草稿笔记。追踪 agent 在多步骤任务中位置的无信息。

7 类内容争夺同一个窗口。上下文工程就是决定**谁赢**。

## 四大核心策略

LangChain 发布了一个框架，把所有上下文工程技术组织为 4 个桶。

你学到的每个技术，都归入其中之一：

**写（Write）· 选（Select）· 压缩（Compress）· 隔离（Isolate）**

---

### 策略一：写（Agents forget. Give them a way to remember.）

当 agent 的上下文被填满并压缩时，它会丢失信息。

如果 agent 在丢失之前没把信息写下来——那些信息就永远消失了。

**写（Write）意味着给 agent 提供在上下文窗口之外持久化信息的方式。**

三种形式：

**🖊️ 草稿本（Scratchpads）**
给 agent 一个工具，让它在任务中做笔记：中间发现、做出的决策、它知道之后会用到的信息。

Anthropic 做了一个「think」工具——一个专门的思考空间让 Claude 处理问题。在 tau-bench 基准测试中，这一改进在某些任务上提升了**高达 54%** 的性能。

**📋 规则文件（Rules Files）**
持久的程序记忆。

如果你用过 Claude Code，你就见过 CLAUDE.md。每次会话开始时加载的指令——项目架构、约定、如何跑测试、需要注意什么。

Agent 每次启动都读它。它永远不会忘记基础。

**🧠 记忆提取（Memory Extraction）**
Agent 保存事实、用户偏好和学到的模式，以便跨会话检索。

完全存在于上下文窗口之外。Agent 明天需要的信息，明天到了就在那等着。

---

### 策略二：选（Don't give the agent everything. Give it what it needs right now.）

一个有 40 个工具、大量知识库和几个会话历史的 agent，不可能一次全加载。

必须有人决定当前步骤什么相关。

**传统 RAG：系统决定。**
用户提问 → 检索文档 → 塞进 prompt → 完成。静态。一次性。模型没有发言权。

**🔄 Agentic RAG：Agent 决定。**
Agent 自己搜索需要的东西、优化查询、挑选工具、判断何时信息足够。

检索不再是一次性流水线，而是一个迭代过程。

这很重要，因为相关的东西在每一步都在变——只有 agent 知道自己下一步需要什么。

#### 工具选择问题

如果你的 agent 有 40+ 个工具，那可能还没开始干活，**10,000 token 的工具定义**就已经占据了上下文。

**解决方案：对工具描述做 RAG 搜索。**

不把全部工具定义塞进每次调用，而是用语义搜索只把当前步骤相关的工具放到上下文中。

一篇叫 RAG-MCP 的论文测试了这种方法：
- 工具选择准确率：**14% → 43%（3 倍提升）**
- Token 消耗：**大约减半**

Anthropic 称之为混合策略：前面加载核心上下文（像 CLAUDE.md），让 agent 按需检索其他一切。

**前加载基础。按需检索其余。**

---

### 策略三：压缩（Context accumulates. Keep the meaning, cut the tokens.）

即使做了好的选择，上下文还是会积累。

每次工具调用、每个检索到的文档、每个决策都留在窗口里。

想象你的 agent 已经做了 20 次工具调用。上下文：80,000 token 的积累——工具输出、对话历史、推理轨迹。

其中大部分已不再相关。Agent 已经对它采取过行动了。

但它还在那里——占空间、降低注意力、推高成本和延迟。

**可以在 3 个阶段做压缩：**

**🔽 信息进入上下文前：**
- 将大文档切分成连贯的块再检索
- 重排序，只让最有用的块进入
- 工具输出进入主上下文前，先做实时摘要

**🔄 Agent 工作中：**
- 对话历史的滚动摘要——持续更新
- 热门混合方案：保留最近 10 条消息原文 + 摘要所有更早的
- 硬裁剪：上下文达到大小阈值后移除旧消息
- Claude Code 自动压缩：在 95% 容量时触发，自动总结完整轨迹

**✅ Agent 已采取行动后：**
- 工具结果清理：一旦工具结果在 15 步之前使用过，就丢弃
- 替换为一行摘要或完全移除
- Agent 不需要 20 步前抓取的网页的完整文本

**目标：减少 token 数量。保留真正重要的内容。**

---

### 策略四：隔离（The most powerful strategy. Enables multi-agent systems.）（最强策略，实现多 Agent 系统的基础）

其实不只是空间问题。而是**污染**。

研究阶段的详细文件搜索还在上下文中，agent 已经进入写代码阶段了。

那旧的研究上下文现在是噪音。它在模型需要专注于干净实现时分散注意力。

**隔离（Isolation）意味着给工作的不同部分分配各自的上下文窗口。**

**🤖 子 Agent（Sub-agents）**
父 agent 把聚焦的子任务委托给子 agent——比如「搜索代码库中所有认证相关的文件」。

子 agent 在自己的干净上下文窗口中工作。它回报时只返回一个压缩后的摘要。

所有混乱的搜索操作都隔离在子 agent 的上下文中，从不污染父级。

**📦 状态 Schema 隔离（LangGraph 的方法）**
设计 agent 的状态，让不同字段存储不同类型的上下文。

LLM 只看到当前步骤相关的字段。工具结果放在「后台」字段——对模型不可见，直到显式地呈现出来。

这样可以在不启动独立子 agent 的情况下，精细控制 agent 每一步看到的内容。

**隔离让复杂的多步骤工作流真正可靠。不同的工作，不同的上下文窗口，无污染。**

## Agent 失败的 4 种方式

Drew Breunig 识别出 agent 上下文增长时的四种失败模式。

每一个你见过的出问题的 agent，都能归入其中之一。

### ❌ 失败 1：上下文中毒（Context Poisoning）
幻觉或错误进入上下文。Agent 在后续步骤中反复引用它。

第 5 步的坏数据在每个后续步骤中不断放大。

**✅ 修复：** 工具输出在进入上下文前做验证。从错误中恢复后，压缩失败尝试的历史记录。不要留 10 步的死胡同调试痕迹可见——只有最终解决方案才重要。

### ❌ 失败 2：上下文分心（Context Distraction）
上下文太长，模型开始过度依赖近期历史。

模型不再综合新的计划，而是重复最近做的事。它不再思考，而是开始复读。

**✅ 修复：** 积极做摘要和裁剪。即使你有大上下文窗口。窗口大不等于要填满它。

### ❌ 失败 3：上下文混淆（Context Confusion）
多余的内容导致模型做出低质量决策。

典型例子：给 46 个工具时模型在基准测试上失败——即使上下文远未超限——但只给 19 个工具时工作良好。

不是工具多到上下文装不下。而是工具多到模型无法清晰推理。

**✅ 修复：** 动态工具管理。用 RAG-MCP 只显示当前步骤相关的工具。让工具集匹配当前阶段。

### ❌ 失败 4：上下文冲突（Context Clash）
新信息与上下文中已有的信息矛盾。

系统提示说一套。检索到的文档说另一套。Agent 无法协调矛盾，产生不一致的行为。

**✅ 修复：** 建立清晰的权威排序。系统提示 > 检索到的事实 > 对话历史。在注入前，验证新信息是否与现有上下文冲突。使用 XML 标签和清晰的标题，让模型知道信哪个来源。

## 如何为 Agent 写系统提示（不是聊天机器人，是 Agent）

聊天机器人的系统提示设定语气：「你是个乐于助人的助手。请简洁友好。」

Agent 的系统提示**定义架构**。

它规定了控制流——如何处理不同类型的任务、何时使用哪些工具、出错了怎么办、有什么护栏。

这更像给自主员工写**职位描述**，而不是写人格提示。

Anthropic 称之为在「正确高度」写作：

- ❌ **太死板：**「如果用户提到账单 AND 提到退款 AND 金额超过 $100，调用工具 X。」脆弱，每个你没预见的边缘情况都会崩。
- ❌ **太空泛：**「要乐于助人，使用合适的工具。」没有给 agent 任何具体信号，它无法做出好的自主决策。
- ✅ **最佳点：** 足够具体以引导自主行为。足够灵活让模型在新颖情况下运用判断。强启发式。非死板规则。

### 实用技巧
- 用 XML 标签或 Markdown 标题组织——Background、Instructions、Tool Guidance
- 从最小化开始，在失败中迭代——不要试图预先预见所有边缘情况
- 最小化不意味着短——一个复杂 agent 的系统提示可以有数千 token，没问题，只要每个 token 都值得
- 用 few-shot 示例——向 agent 展示好行为是什么样子，而不是试图用文字描述每个规则

## KV-Cache：关心上下文顺序的经济原因

大多数 agent 构建者不知道这个存在。

当你向 LLM 发送 token 时，模型为每个 token 计算键值表示。

计算开销很大。

所以推理提供商会**缓存**这些表示。

如果你的上下文前缀——开头部分——在多次 API 调用之间保持不变，提供商会复用缓存的算力，只处理结尾新增的 token。

**快。便宜。**

但如果你在调用之间重排或改变上下文的前面部分——缓存就失效了。提供商会从头重新计算一切。

**Claude Sonnet 的成本差异：**
- → 缓存的输入 token：每百万 $0.30
- → 未缓存的输入 token：每百万 $3.00

**10 倍差价。**

对一个 agent 来说，每次任务 30–40 次 API 调用，这个差异累积得很快。

### 实际 KV-Cache 效率规则

- ✅ 稳定内容放在上下文 **顶部**——系统提示、工具定义，任何轮次之间不变的内容
- ✅ 动态内容放在 **底部**——对话历史、当前步骤、agent 状态
- ❌ **不要**在对话中动态添加和移除工具——会使缓存失效
- ⭐ **用工具屏蔽代替工具移除**——把所有工具定义稳定放在前缀中（缓存），只把不相关的标记为当前阶段不可用

## 7 小时产出 35,000 行代码的工作流

Dex Horthy（HumanLayer CEO）在 AI Engineer Code Summit 上展示了这个方法。

他的团队用它在一个大型 Rust 代码库中，**单次 7 小时会话交付了约 35,000 行代码**。

**方法：频繁有意的压缩（Frequent Intentional Compaction）。**

将 agent 工作分为阶段。每个阶段产生一个压缩后的人工制品。每个新阶段从包含该人工制品的**全新上下文窗口开始**。

始终保持上下文窗口**低于 40–60%** 的容量。

### 阶段 1：研究（Research）
子 agent 探索代码库。读文件。追踪数据流。绘制架构图。

> *所有混乱的 grep 结果和文件内容留在子 agent 上下文中。绝不触及父级。（隔离）*

> *输出：一个紧凑的 research.md——文件路径、函数签名、模式、陷阱。（写）*

> *上下文重置：原始研究使用了 60–80% 的窗口。研究人工制品将其压缩到 15–20%。（压缩）*

### 阶段 2：规划（Planning）
新上下文窗口。只包含：研究文档 + 问题定义。

Agent 生成详细的实施计划。

**这是最重要的**人工审查检查点**。** 在此处捕获逻辑错误，修复起来简单且免费。后面修复要花几小时。

### 阶段 3：实施（Implementation）
又一个全新上下文窗口。只包含：规划文档。

Agent 按规划一步步执行。对复杂任务：一个 progress.md 追踪已完成和待完成的事项。（写）

**结果：每个阶段都有干净、聚焦的 agent。无污染。无上下文腐败。没有「第 20 步变敷衍」。**

## 各大平台的处理方式对比

### Claude Code
- **混合检索。** CLAUDE.md 预先加载。Glob 和 grep 工具处理按需的代码库导航
- **95% 容量自动压缩**——保留架构决策和最近访问的 5 个文件
- 可生成子 agent 处理复杂子任务，各有自己的干净上下文
- 哲学：**「做最简单有效的事。」** 让模型自己思考需要什么，给它找工具

### Manus
- **KV-Cache 感知的上下文排序：** 稳定前缀、动态后缀。工具屏蔽而非移除
- **观测压缩流水线——** 每次工具输出进入 agent 上下文前都经过处理
- 持久的待办列表用于状态追踪
- **文件系统作为溢出内存**，用于被驱逐的上下文
- 为规模而生：服务数十万用户，效率是经营成本问题

### ChatGPT Agent
- **视觉优先方法。** Agent 与 GUI 浏览器交互
- 截图作为视觉快照加入上下文，模型在其上推理
- 视觉 token 很贵，所以 agent 对截图数量有选择性
- 使用**强化学习**在数千台 VM 上学习最优工具使用策略，而非显式编程

### Google ADK
- **最严谨的架构方法**
- 三大设计原则：
  1. **分离存储与呈现**——持久状态不等于每次 API 调用中出现的内容
  2. **显式变换**——命名、有序的处理器，以可测试、可组合的步骤变换上下文
  3. **默认限定上下文范围**——每次模型调用只看到最低必要信息
- **工程纪律胜过 prompt 雕琢**

## 通用的 Agent 单步流水线

每个严肃平台都收敛到同一个 5 步循环：

1. **收集（Collect）**——用户输入、对话历史、工具结果、检索文档、agent 状态
2. **选择（Select）**——在剩余 token 预算内，哪些对当前步骤相关
3. **压缩（Compress）**——摘要、截断或重组以适合上下文
4. **排列（Arrange）**——稳定内容在前（缓存），动态内容在后
5. **组装 + 调用（Assemble + Call）**——最终上下文 → API 调用 → 获取输出 → 循环

这是每个你用过的主流 agent 内部运行的循环。理解它，就是区分能交付可靠 agent 的构建者和纳闷「为什么 agent 在第 15 步变敷衍了」的人。

---

## 总结

- **上下文腐败是真实存在的，** 在你的上下文限制之前就开始了
- **四大修复策略：**
  - ✅ **写（Write）**——在上下文外持久化信息，agent 不会遗忘
  - ✅ **选（Select）**——只拉取当前步骤需要的内容
  - ✅ **压缩（Compress）**——减少 token，保留意义，主动而非被动
  - ✅ **隔离（Isolate）**——不同工作用不同上下文，无污染
- **四大失败模式：**
  - ⚠️ **中毒（Poisoning）**——坏数据在每一步放大
  - ⚠️ **分心（Distraction）**——长历史让 agent 复读而非思考
  - ⚠️ **混淆（Confusion）**——太多工具降低决策质量
  - ⚠️ **冲突（Clash）**——矛盾产生不一致行为
- **KV-Cache 带来 10 倍成本节约。** 稳定内容放前面
- **最佳工作流：** 研究 → 压缩 → 规划 → 压缩 → 实施。每个阶段重新开始上下文

上下文工程不是可选的严肃 agent 工作的一部分。

**它就是工作本身。**
