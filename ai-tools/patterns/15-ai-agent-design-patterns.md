# 每个工程师必知的 15 种 AI Agent 设计模式

> **来源：** [15 AI Agent Design Patterns Every Engineer Must Know](https://x.com/sairahul1/status/2068986018943156440) — @sairahul1

![cover](../image/15-ai-agent-patterns-1.jpg)

每个构建 AI agent 的团队都会撞上同一堵墙。

从一条 prompt 和几个工具开始。能行。然后需求增长。更多边界情况。更多团队。更多风险。突然你的 "agent" 变成了一篇 3000 词的 system prompt，试图同时做五件事。

**解决方案不是更好的 prompt——而是选对模式。**

---

## 选模式前先判断：这真的需要 Agent 吗？

**需要 Agent 的信号：**
- → 单次模型调用无法给出可靠结果
- → 模型必须在运行时从多个工具或数据源中选择
- → 任务需要规划、验证或迭代优化
- → 工作流存在无法硬编码的真实不确定性

**通常不需要 Agent 的任务：** 总结、分类、简单提取、模板生成——这些输入到输出的路径是可预测的，作为直接模型调用更快速、更便宜、更可靠。包一层 agent 只会增加延迟和故障点。

---

## 15 种设计模式

### 1. 单 Agent（Single Agent）

最简单的起点。一个模型、一条 system prompt、一组边界明确的工具。模型决定调哪个工具、观察结果、继续直到拿到足够的信息给出答案。

**真实案例：** 客服 agent 查订单状态、查物流、如果不解决则创建工单——2-3 个工具和一份清晰的工作描述。

- ✅ **适用场景：** 任务定义明确、工具集小、一个 agent 能保持完整上下文
- ❌ **失效场景：** 持续添加工具 + system prompt 超过一页 → 需要换模式，不是加长 prompt

### 2. 多 Agent 串行（Multi-Agent Sequential）

专业 agent 按固定顺序运行，每个的输出喂给下一个的输入。

**真实案例：** 合同审核管线——一个 agent 提取义务条款，下一个识别风险，第三个起草采购摘要。顺序从不改变。

- ✅ **适用场景：** 工作流有清晰可重复的阶段，每个阶段准确产出下一个所需
- ❌ **失效场景：** 实际顺序需要根据中途发现的结果动态变化

### 3. 多 Agent 并行（Multi-Agent Parallel）

独立子任务同时运行，然后合并成一个结果视图。

**真实案例：** 凌晨 2 点的生产事故——三个 agent 同时调查日志、指标和近期部署（不是串行），因为宕机时每一分钟都宝贵。

- ✅ **适用场景：** 子任务真正独立且速度重要
- ❌ **失效场景：** 任务依赖彼此结果时强行并行

### 4. 循环（Loop）

重复执行一系列步骤直到满足退出条件。

**真实案例：** 数据清洗 agent 分析脏 CSV → 提出清洗方案 → 检查是否达标 → 不达标重试，限定最多轮数。

- ✅ **适用场景：** 任务需要多次尝试，且有清晰可检查的停止条件
- ❌ **失效场景：** 没有可靠的退出条件——导致失控成本和永不终止

### 5. 审查与批评（Review and Critique）

一个 judge agent 审查另一个 agent 的输出，给出具体的可操作反馈。

**真实案例：** 生成的报告由独立的"批评"agent 检查——标记薄弱声明、缺失证据或不清晰的段落，再送到人类手里。

- ✅ **适用场景：** 质量比速度重要，想要内建复核机制
- ❌ **失效场景：** 批评 agent 有与生成 agent 相同的盲区

### 6. 迭代优化（Iterative Refinement）

带质量得分阈值的反馈循环。生成器持续优化直到越过门槛。

**真实案例：** 营销文案生成器根据品牌指南给自己打分，不断重写到最低质量分数——不是一次通过/不通过检查，而是渐进式改善。

- ✅ **适用场景：** 输出质量真正波动且"够好"有可衡量阈值
- ❌ **失效场景：** 评分函数模糊或可被操纵——模型能虚抬分数

### 7. 协调器（Coordinator）

中央路由 agent 根据实际请求内容分发到专业 agent。

**真实案例：** 工单路由到计费、技术、账号、物流或欺诈专家——每个有窄上下文而不是一个 agent 试图知道一切。

- ✅ **适用场景：** 有真正不同类型的请求，需要不同的上下文、工具或决策逻辑
- ❌ **失效场景：** 路由本身变得模糊——请求无法干净落入一个类别

### 8. 分层任务分解（Hierarchical Task Decomposition）

根 agent 将复杂目标分解为更小的子目标，分发给专业 worker，然后综合成一个答案。

**真实案例：** "明年应该扩展到哪 3 个国家？" → 分解为竞争分析、监管研究、物流可行性和市场体量估算——各由不同专家处理，然后合并。

- ✅ **适用场景：** 问题太宽泛无法一次推理完成，但可干净拆分为独立专业领域
- ❌ **失效场景：** 子任务不能独立——工作流需要实时互相反馈

### 9. 群（Swarm）

多个专业 agent 参与共享讨论，挑战彼此的假设，由 facilitator 综合最终建议。

**真实案例：** 公司是否应该推出订阅层级？研究、工程、财务和支持 agent 各自主张自己的视角，经过多轮讨论后 facilitator 权衡取舍。

- ✅ **适用场景：** 没有单一的"正确"答案——需要真正竞争性视角形成合理决策
- ❌ **失效场景：** 需要快速确定性答案——Swarm 故意慢和探索性

### 10. ReAct（推理与行动）

agent 交替进行推理和行动：决定调查什么、调用工具、观察结果、判断证据是否足够。

**真实案例：** "队列处理器似乎卡住了"——agent 查文档、查服务健康、关联发现、然后才建议修复。调查路径不预定义，取决于沿途发现。

- ✅ **适用场景：** 答案路径无法预先规划——取决于每一步的发现
- ❌ **失效场景：** 调查跑长不收敛——必须限制推理-行动循环次数

### 11. Human-in-the-Loop（人在回路中）

Agent 调查并推荐，但人类对有风险或模糊的事做最终决定。

**真实案例：** 退款审批——低风险、明确的情况自动化；大金额、欺诈信号或策略例外暂停等待人工审查。

- ✅ **适用场景：** 决策有真实财务、法律或声誉风险，完全自动化尚不可接受
- ❌ **失效场景：** 你把它只当作 UI 功能而不是架构功能——需要持久状态、审查者分配、超时处理和升级路径

### 12. 计划然后执行（Plan-and-Execute）

规划 agent 事先创建完整结构化计划——可审查和修改——然后再执行。与 ReAct 不同，路径在执行前完全可见。

**真实案例：** "将 worker 集群从 10 扩展到 20 台实例，验证队列排空，更新 runbook。"完整计划在执行前可见。

- ✅ **适用场景：** 希望计划在执行前可审查或可审批——对有真实后果的操作很重要
- ❌ **失效场景：** 环境变化比计划执行更快——执行过时计划比没有计划更糟

### 13. 反思（Reflexion）

Agent 评估自己的失败，反思哪里出错了，将这段"记忆"带入下一次尝试。

**真实案例：** 代码生成 agent 写脚本 → 运行时失败 → agent 分析实际错误 → 记录要修复什么 → 重试，每次尝试都比前一次更聪明，而不是重复相同错误。

- ✅ **适用场景：** 失败是有信息量的，自我修正确实能改进下一次尝试
- ❌ **失效场景：** 失败模式是随机的或互不相关

### 14. 自定义逻辑（Custom Logic）

混合模式：确定性代码处理绝对不能出错的事，模型处理判断、草稿和异常处理。

**真实案例：** 退款工作流——购买验证和欺诈检查作为硬确定性规则运行（绝不委托给模型），而起草客户回复和路由建议保持 agentic。

- ✅ **适用场景：** 工作流有真实的分支逻辑，涉及法律或财务后果
- ❌ **失效场景：** 团队模糊界限，让模型做应该硬编码的决定——资格判定、权限和资金流动绝不能只让模型决定

### 15. 事件驱动 Agent（Event-Driven Agent）

Agent 不等待被请求。它订阅事件流，条件触发时立即行动。

**真实案例：** 欺诈检测 agent 在可疑交易事件触发的那一刻就做出反应——而不是等工单最终浮出水面，那时损失已经造成。

- ✅ **适用场景：** 时机比什么都重要，等人来请求意味着错过行动窗口
- ❌ **失效场景：** 触发条件定义不清晰——嘈杂的事件流变成"狼来了"

## 按不确定性选模式

| 不确定性类型 | 推荐模式 |
|-------------|---------|
| 不确定用哪个工具 | 单 Agent 或 ReAct |
| 不确定往哪路由 | Coordinator |
| 不确定质量如何 | Review & Critique 或 Iterative Refinement |
| 不确定执行路径 | Plan-and-Execute 或 ReAct |
| 不确定如何自我修正 | Reflexion 或 Loop |
| 不确定业务风险 | Human-in-the-Loop 或 Custom Logic |
| 不确定问题结构 | Hierarchical Decomposition 或 Swarm |
| 等不起请求 | Event-Driven Agent |

> Swarm 不比单 Agent 更"高级"——如果任务只需要一次可靠的工具调用。
> Plan-and-Execute 不是 ReAct 的"升级"——如果计划到第三步就过期了。
> **最可靠的生产系统不是最自主的。它们把自主权恰好放在创造价值的地方——其他地方都加以约束。**

## 生产级 Agent 系统的 10 条铁律

1. **从最小的能工作的模式开始。** 一个干净的 agent + 清晰工具合约 > 弱的系统 + 弱的多 agent
2. **把工具描述写成契约。** 模型从描述知道工具是做什么的，不是从你的意图
3. **限制每次请求的迭代次数、工具调用和花费。** 没有预算上限的 agent 是负债
4. **记录完整的行动轨迹。** 工具调用、参数、输出、最终决策——没有这些，事故调查是猜
5. **不可逆操作放在确定性检查或人工审批后面。** 模型绝不能是资金流动或生产变更的唯一闸门
6. **用真实失败案例评估，不只是 happy path。** Happy-path 正确是原型，边界案例正确是产品
7. **按职责分离 prompt。** "但是当 X 时不要做 Y" 溜进你的 prompt → agent 在做两份工
8. **把多 agent 系统当作分布式系统对待。** 部分失败、超时、重试和可观测性不是可选的
9. **模型审查不能替代确定性验证。** 用 judge 改进质量，用测试+权限检查来确保正确性
10. **倾向更简单的模式。** 不是因为简单总是更好——而是你省下的复杂度预算可以花在更好的工具、prompt 和评估上

> 大多数团队失败不是因为选错了模式。他们失败是因为从未问过：**自己到底在解决哪种不确定性。**
