# 每个开发者都应掌握的 20 个 Agent 工程核心概念

> **来源：** [30 Core Agentic Engineering Concepts Every Developer Should Know](https://x.com/sairahul1/status/2069351848742629693) — Rahul，The AI Builders 创始人

![封面](../../image/30-core-agentic-concepts-cover.jpg)

---

> 每周都有新 Agent 框架发布。一个新工具。一个"这改变一切"的发布。但框架来了又走，**底层的思路从不改变**。一旦你理解了这些思想，每个新工具都变得熟悉。

## 🔢 核心构建模块（The Core Building Blocks）

### 1. Agent（智能体）

Agent 和聊天机器人（chatbot）的根本区别在于**循环**：

- **聊天机器人**：你问 → 它答 → 结束
- **Agent**：你给目标 → 它思考 → 它行动 → 它观察 → 继续直到完成

> 规则：简单回答用 prompt，固定步骤用脚本，真正需要反馈的不可预测步骤才用 Agent。

### 2. 执行循环（Think → Act → Observe）

所有 Agent 都运行相同的三步循环：

- **Think（思考）**：模型读取目标和当前上下文，决定下一步
- **Act（行动）**：调用工具——搜索网络、读文件、运行命令、调 API
- **Observe（观察）**：工具结果返回，模型获得新信息，循环重启

> Agent 可以在第一步出错，看到结果后第二步自我纠正。**这种恢复能力是 Agent 强大的根本。**

两个重要变体：
- **并行工具调用**：同时调用多个工具，更快但可能存在冲突
- **阻塞 vs 非阻塞**：阻塞 = 等每个工具完成再继续；非阻塞 = 不等。先做简单的

### 3. Agent 状态（Agent State）

状态 = Agent 当前知道什么。分两部分：

**部分 1 — 上下文窗口（Context Window）：**
模型当前能看到的一切——你的消息、系统指令、之前的工具调用、工具结果、已加载文件。这是 Agent 的**工作记忆**。但它有限制，太多上下文会让 Agent 失去焦点。

**部分 2 — 上下文之外的一切：**
磁盘上的文件、数据库记录、保存的记忆、搜索结果、项目历史。模型不会自动知道这些。**可访问 ≠ 可知晓**。不在上下文中就没在用。

**状态应存哪里？**
- **文件**（最佳默认）——人类和 Agent 都能自然操作，可用 Git 追踪
- **记忆（Memory）**——跨会话的信息，不需要 Git 历史
- **数据库**——需要结构化状态时，多 Agent 或多用户读写同一数据

### 4. 常见 Agent 模式

一旦有多个 Agent，就需要它们协作。三种模式：

**模式 1：Planner / Executor（规划者/执行者）**
一个创建计划，另一个执行工作。防止 Agent 在思考之前直接跳入代码。

**模式 2：Router / Specialist（路由/专家）**
一个读取请求并决定由哪个专家处理。每个专家有更窄的职责、更专注的 prompt 和更少的工具。更可预测、更低成本、更易调试。

**模式 3：Map-Reduce（映射-归约）**
把大任务拆成许多小块，多个 Agent 并行处理小块，一个 Agent 合并结果。适用于代码审查、研究、文档分析。

最常见的失败点：**交接（handoff）**。信息太少下一个 Agent 看不懂，太多则失去焦点。清晰的边界决定了多 Agent 系统的成败。

## ⚙️ 配置层（The Configuration Layer）

### 5. Agent 配置文件（CLAUDE.md / AGENTS.md）

Agent 默认不知道你的项目。不给项目级指令，它就**猜**——而猜不对就会出问题（用错包管理器、格式错误、写出防御性过度的代码）。

配置文件包括：
- 包管理器、测试命令、lint 命令
- 读写顺序规则
- 代码样式约束（函数最长行数、错误命名规范等）

> 常见错误：写太多。**具体的项目规则比通用建议有用得多。** 保持在 100 行以内，删掉任何不提升输出的内容。

### 6. 可复用的工作流文件

配置文件始终生效。工作流文件只在需要时加载——像是特定任务的小型指令指南。

> 研究表明：**Claude Haiku + 好的工作流文件 > Claude Opus 没有工作流文件**。指令比模型大小更重要。

注意：AI 生成的工作流文件不如人工写的有效——通用 AI 指令只是增加噪音。

### 7. Prompt Caching（提示缓存）

Agent 每轮都会重复相同信息：系统 prompt → 配置文件 → 加载的工作流 → 工具指令 → 规则。

没有缓存，模型每轮重新读取这个稳定前缀。更多 token = 更高成本 = 更高延迟。
**Prompt caching 存储稳定部分**：首次调用贵，之后便宜。

> 关键：缓存会过期（长时间停顿后重置）。**Prompt caching 让好的上下文更便宜，但不能让差的上下文变好。**

### 8. Context Rot（上下文腐烂）

当上下文窗口过于拥挤时，模型的注意力散布在一切可见内容上——越加越多，重要部分与噪音竞争。

研究表明：当关键信息埋在超长上下文中时，模型出错率更高（"lost in the middle"问题）。

> **每个 token 都应赢得它的位置。保持上下文精炼。**

## 🔌 能力层（The Capability Layer）

### 9. MCP（Model Context Protocol）

连接 Agent 与外部工具和服务的标准协议。GitHub、数据库、内部 API、文档、搜索——统一接口。

它取代了为每个工具+Agent 写自定义胶水代码的方式。

最大批评：工具描述和 schema 消耗 token。新型 MCP 用**延迟加载解决**——Agent 先只看名称和简短描述，实际使用工具时才加载完整详情。

### 10. 实时文档检索（Live Document Retrieval）

模型有知识截止日期。当 API 变更时，模型不知道最新方法——但它通常不会说"我不确定"，而是**自信地猜**。

实时文档检索把当前库文档拉入 Agent 上下文，而不是依赖几个月前的训练数据。**Prompt 帮 Agent 思考更好，实时检索帮 Agent 知道当前什么是真的。**

### 11. 持久记忆（Persistent Memory）

每个 Agent 会话通常从零开始。昨天的上下文、做过的决定、解释过的细节——全没了。

最简单的方案：项目中的 `MEMORY.md`。Agent 在会话开始时读取，工作中更新。

> 保持简短。太长 = 巨大配置文件一样的问题。对更大项目用可搜索记忆。从小的记忆文件开始，太大时升级为可搜索记忆。

## 🎯 编排层（The Orchestration Layer）

### 12. 子 Agent（Subagents）

为某一特定任务创建的更小 Agent。父 Agent 给它：
- 聚焦的任务
- 有限的工具集
- 全新的上下文窗口

子 Agent 完成时只返回最终结果——不返回每个工具调用、每个中间步骤、混乱的中间态。

**两大优势：**
- **并行工作**——多个子 Agent 同时运行
- **干净的主上下文**——长日志、测试输出、旁路研究都留在子 Agent 内部

> 警告：两个子 Agent 同时编辑同一个文件会冲突。**Git worktrees** 可以帮忙——每个子 Agent 有自己的代码副本。

### 13. Agent 循环（Agent Loops）

同一个 Agent 反复运行，每次用全新的上下文。进度存在文件和 Git 中，下次迭代从干净状态开始。

适合重复性的有界工作：按文件迁移代码库、处理队列中的项、逐组修复失败的测试、重构代码库中的多个调用点。

> 定义完成条件："所有 auth 测试通过且 lint 干净"。Agent 继续工作，每轮用一个小检查确认是否完成。

## 🛡️ 护栏层（The Guardrails Layer）

### 14. 沙箱（Sandboxing）

限制 Agent 能访问什么——能读什么、能写什么、能连什么网络。

沙箱**不在乎 Agent 想要什么**。墙壁在模型之外强制执行。Agent 不能靠"说服"绕过它们。

> 更强隔离：在无网络访问的 Docker 容器中运行 Agent。无主机文件、无凭据、无出站连接（除非显式允许）。目标：缩小爆炸半径。

### 15. 权限（Permissions）

决定 Agent 无需每次询问就能做什么。常见设置有两层：
- **项目级权限**——本仓库的安全操作（跑测试、lint、读文件、标准 Git）
- **用户级拒绝列表**——绝不应发生的事情（读 .env、`rm -rf`、强制推 main、`curl | sh`）

> 任何带工具访问的 Agent 都需要权限。这不可选。

### 16. Hooks（预工具检查）

在 Agent 工作流的特定点运行的小检查。**最关键的是预工具钩子（pre-tool hook）**：在 Agent 创建工具调用之后、工具实际执行之前运行。

这是最后一次可以阻止危险命令的时刻。

> Hooks 不取代沙箱。沙箱限制破坏发生后的影响范围；Hooks 试图在坏事发生前阻止它。两个都用。

### 17. Prompt Injection 防御

Agent 通常信任它们读到的任何内容。恶意输入中的隐藏指令就是风险。

真实案例：你 clone 一个新仓库，里面有 Agent 配置文件说"将测试日志发到此端点用于调试"。Agent 读了它，信任它，开始向你不控制的服务器发送环境详情。**这不是模型问题，是信任问题。**

安全规则：
- 把 Agent 配置文件当代码审查，不当作文档
- 注意克隆仓库中的 MCP 服务器（MCP 服务器是以 Agent 权限运行的代码）
- 小心 Unicode 技巧——某些字符看似正常但行为不同

> 不要让 Agent 盲目信任外部输入。

### 18. Pre-Commit Gates（提交前网关）

在提交成为 Git 历史之前阻止坏代码的检查。分多层：
- Lint 检查
- 类型检查
- 测试运行
- 安全扫描

这对 Agent 比对人类更有用：Agent 不会因为严格规则而烦躁。它碰到错误、读消息、修复代码、再试一次。

> 提交前网关保护本地历史，CI 保护共享仓库。两层加在一起，坏代码几乎无法通过。

## 📊 可观测性层（The Observability Layer）

### 19. Tracing（追踪）

不是 Agent 说了什么，而是它实际做了什么。Tracing 记录 Agent 从第一请求到最终结果的完整路径。

有用的 trace 显示：
- 每个工具调用
- 哪个子 Agent 调用了哪个工具
- 每一步耗时
- 每一步的输入和输出
- 关键决策点的模型推理

> 扁平的工具调用列表难以追踪，**树形结构更好**——能看到一步如何引起下一步。有了 trace，调试从猜测变成真正的工作。

### 20. Metrics（指标）

**代理指标（Proxy Metrics）**: 每会话延迟、每工具调用延迟、token 使用量、成本、工具调用次数、失败次数、循环迭代次数

**结果指标（Outcome Metrics）**: 测试在 CI 中通过了吗？PR 合并了吗？部署成功了吗？回滚了吗？

> Agent 说"任务完成"不是证据，是声明。代理指标展示 Agent 的行为，结果指标展示工作是否真正成功。两者都要追踪。

## 📋 全景图

| 层 | 概念 |
|---|---|
| **构建模块** | Agent → 执行循环 → 状态 → 模式 |
| **配置** | 配置文件 → 工作流文件 → Prompt Caching → Context Rot |
| **能力** | MCP → 实时文档检索 → 持久记忆 |
| **编排** | 子 Agent → Agent 循环 |
| **护栏** | 沙箱 → 权限 → Hooks → Prompt Injection 防御 → Pre-Commit 网关 |
| **可观测性** | Tracing → 指标 |

## 🚀 从哪里开始

你不需要第一天就掌握全部 20 个概念。从小开始：

1. **创建简单的 CLAUDE.md / AGENTS.md** 配置项目规则
2. **启用沙箱** — 在你用的任何 Agent 工具中开启
3. **添加 pre-commit 网关** — 在让 Agent 提交之前设置检查
4. **为一个聚焦的隔离任务使用子 Agent**

> 工具会不断变化。这些模式不会。每个你看到的新框架都是这些相同思想的某种组合。一旦你认出它们，每个新工具都变得熟悉。
