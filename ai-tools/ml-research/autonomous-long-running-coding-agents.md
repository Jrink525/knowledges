# 自主运行的长周期编码 Agent

> **来源：** [Autonomous Long-Running Coding Agents](https://x.com/omarsar0/status/2065880971031834786) — Elvis (DAIR.AI, 前 Meta AI)

自主编码正从"更好的提示词"转向"更好的控制系统"。核心转变在于：工程师正在学会如何用目标（goals）、评估器（evaluators）、循环（loops）和工件（artifacts）把 Agent 包装起来，使其在人类停止打字后仍能继续工作。

这之所以重要，是因为大多数严肃的工程工作跨度很长：模糊的需求、隐藏的约束、部分失败、变化的上下文、反复的验证。新前沿在于**围绕 Agent 设计系统**，使其能够规划、执行、检查自己的工作、从错误中恢复，并在无需人类持续引导的情况下持续取得进展。

本文基于 DAIR.AI Academy 关于自主长周期编码 Agent 的 session，作者在其中讲解了 Claude Code 的 `/goal` 模式、新引入的 `/loop` 命令、验证器、工件以及实际中的编排模式。由作者与 Codex、Claude Code 协作撰写。

## 一、从提示词到目标设计

![从提示词到目标设计](../image/autonomous-long-running-coding-agents-2.jpg)

Claude Code 的 `/goal` 功能背后的核心思想很简单：

编码 Agent 仍然是执行者，但人类不再逐轮与其交互。取而代之的是，人类指定：
- **期望的最终状态**（end state）
- **证明成功的证据**（evidence）
- **不得违反的约束**（constraints）
- 以及可能的**轮次和预算上限**

> 这个目标更像一份**契约**（contract），而不是一个更长的 prompt。

- 弱目标：让模型有机会提前停止、走捷径、以在转录中看起来合理但在真实系统中失败的方式重新定义成功。
- 强目标：给 Agent 一个它可以反复**对照衡量自身**的靶子。

工程判断力仍然至关重要。最好的目标编码了模型否则会猜测的领域知识：

| 场景 | 强目标示例 |
|------|-----------|
| 研究实验 | 目标 benchmark 分数、留出评估集、必要的 loss 曲线、必须超过初始基线 |
| UI 任务 | 截图参考、具体的布局约束、浏览器验证步骤 |

模型可以执行，但人类仍然定义"完成"的真正含义。

## 二、评估器成为一等公民

![评估器成为一等公民](../image/autonomous-long-running-coding-agents-3.jpg)

长周期 Agent 需要的**第二个角色**是评估器（Evaluator）。它可以是：
- 另一个编码 Agent
- LLM-as-judge
- 一个脚本
- 测试套件
- benchmark harness
- 或以上所有混合

**关键设计选择：将评估器与任务匹配。**

| 成功标准 | 评估方式 |
|---------|---------|
| 明确的（crisp） | 确定性检查：类型检查、单元测试、lint 规则、集成测试、benchmark 脚本 |
| 模糊的（fuzzy） | Agent 评估器：需要语言、判断力、有时还有视觉能力 |

确定性检查做地基，Agent 评估做高层评审 — 这种组合既减少了幻觉式的"成功确认"，又允许在无法用测试断言清晰表达的任务上保持自主性。

## 三、验证器定义信任边界

![验证器定义信任边界](../image/autonomous-long-running-coding-agents-4.jpg)

更深层的观点：**自主性只有在系统拥有可靠的验证器时才能工作。**

编码 Agent 可以生成计划、实现功能、解释为什么它认为工作已完成 — 但这些解释**不应被视为证据**。证据来自 Agent 无法轻易绕过的外部检查。

| 场景 | 验证器 |
|------|-------|
| 代码 | 测试套件、类型检查器、benchmark、浏览器运行、截图对比、可复现脚本 |
| 研究 | 留出评估、复现表格、loss 曲线、超过基线的 benchmark 分数 |
| 设计 | 参考截图 + 视觉审查步骤 |

**大多数捷径都出现在这个边界上：**
- 验证器模糊 → 模型会满足任务最容易被解释的版本
- 验证器过窄 → 模型可能过度拟合，错过更广泛的意图

好的自主工作流需要**分层验证**：
- 廉价确定性检查 → 捕获基础失败
- 高层评审 → 捕获需要判断力的失败

> 验证器仍是一个开放的研究领域，预计越来越多的公司会在此领域大举投入。**微调验证器**在企业中需求很高。

## 四、循环让自主性持久

![循环让自主性持久](../image/autonomous-long-running-coding-agents-5.jpg)

目标给 Agent 方向，但**循环让工作持续**。

为什么循环必不可少：模型常常在真正完成前就停止——可能命中轮次限制、失去信心、耗尽了上下文、或认为部分解决方案够了。

循环是外层控制系统：
1. 唤醒
2. 检查进度
3. 运行检查
4. 将结果与目标对比
5. 目标未达成 → 发送 Agent 回去继续执行

最简单的形式：Ralph 循环模式 + 编码 Agent + 确定性条件。
更灵活的形式：循环中包含一个能推理进度并决定下一步行动的评估 Agent。

> 长期自主性工作的本质是**在控制层监督下的重复努力**，而不是一次性的智能行为。Agent 仍然可能失败，但循环让系统能够**注意到失败并继续**，而不是无声地宣告胜利。

## 五、规划是专业知识的入口

![规划是专业知识的入口](../image/autonomous-long-running-coding-agents-6.jpg)

session 中最强烈的主题之一：**规划仍然至关重要**。

你可以让前沿模型生成计划，但在交给自主循环之前，你仍然需要审查计划、质疑假设、让成功标准更清晰。

这引出了一个有用的分工：

| 角色 | 模型选择 | 任务 |
|------|---------|------|
| 规划模型（Planner） | 更强的模型 | 定义目标、识别缺失约束、结构化评估 |
| 执行模型（Executor） | 不同的模型 | 计划清晰后运行实现 |
| 评估模型（Evaluator） | 更便宜的模型 | 验证结果 |
| 视觉审查模型 | 视觉能力强的模型 | 截图对比、UI 检查 |

> 工程师应该停止把"模型"当作单一选择。**模型选择变成了架构决策。**

有些模型是更好的规划者，有些是更好的执行者，有些是更便宜的评估者，有些更擅长视觉审查。一个好的编排器让你可以在这些角色之间切换，而不是等待一个供应商提供完美的编码 Agent 接口。

## 六、可视化工件成为控制界面

![可视化工件成为控制界面](../image/autonomous-long-running-coding-agents-7.jpg)

当多个 Agent 并行运行时，终端转录无法规模化。原始文本是理解进度的糟糕界面。

**实时工件**（live artifacts）之所以重要，是因为一个包含 loss 曲线、benchmark 分数、任务状态、截图、成本估算和近期决策的仪表板，给人类提供了更好的监督自主性的方式。

最有用的模式：**存储与展示分离**

| 用途 | 格式 |
|------|------|
| 持久化存储 | Markdown / vault → 存放证据、日志、笔记、计划、结果 |
| 可视化展示 | HTML 工件 → 将状态渲染为可视、可交互的内容 |

Agent 搜索 Markdown，人类监控工件。

对于 UI 和产品工作，视觉线索尤其强大。截图参考可以比文字更精确地传达设计意图，而具备视觉能力的评估器可以比照参考检查实现。这减少了 Agent 虽然技术上实现了请求的组件，但错误处理了间距、层级、对齐或产品质感的问题。

## 七、Session 挖掘将使用记录转化为记忆

![Session 挖掘将使用记录转化为记忆](../image/autonomous-long-running-coding-agents-8.jpg)

另一个重要的洞察：**过去的 Agent session 是丰富的工作流数据。**

如果 Agent 反复以相同方式失败、忘记运行相同的检查、使用错误路径、或重试相同的损坏命令，这种模式不应该埋藏在日志中。

Session 挖掘将这些转录转化为**操作规则**：
1. Agent 扫描过去 30 天的工作
2. 发现重复的失败模式
3. 提出对项目指令、vault 学习或 Agent 规则的更新

这样，团队可以逐步改进其 harness，而无需手动记住每一个错误。

目标：**让本地环境变得更智能，而无需从头训练模型。** Agent 指令文件中的一条小规则可以防止未来 session 中的重复失败，尤其是当规则与项目相关时。

## 八、实用操作模型

![实用操作模型](../image/autonomous-long-running-coding-agents-9.jpg)

对于 AI 工程师来说，新兴的工作流如下：

1. **先在小范围、廉价的子集上验证**，再启动完整的自主运行
2. **编写目标**：包含可衡量的成功标准、明确的约束、以及轮次或时间预算
3. **分离执行者和评估者**：实现和判断不应该是同一个角色
4. **在长周期循环开始前定义外部验证器**
5. **尽可能使用确定性检查**，模糊标准再添加 Agent 评审
6. **要求证据工件**：日志、截图、benchmark 曲线、变更的文件
7. **挖掘过去的 session**，将反复出现的教训提升为项目指令

> 这就是**使用编码 Agent** 和**工程化自主编码系统**的区别。前者给你一段对话，后者给你一个 harness。

## 九、仍然存在的问题

![仍然存在的问题](../image/autonomous-long-running-coding-agents-10.jpg)

这些并不能消除困难问题。Agent 仍然：
- 走捷径
- 提前停止
- 高估完成度
- 生成自信但薄弱的计划 — 尤其是在新论文、不熟悉的 benchmark 或训练分布外的系统上

**更信任它们并不能解决这个问题。更好的控制系统才能。**

目标、循环、评估器、确定性检查、视觉工件和 session 记忆 — 这些都是让自主性**可观察和可纠正**的方法。

方向很明确。编码 Agent 的未来取决于**围绕更有能力的模型构建更好的编排**。工程师需要设计使 Agent 可以安全运行数小时或数天、并产生可验证产出的条件。
