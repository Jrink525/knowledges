# AI 的万亿美元机会：上下文图谱（Context Graphs）

> **来源：** [@JayaGup10 on X](https://x.com/jayagup10/status/2003525933534179480)
> *Jaya Gupta & Ashu Garg, 2025-12-22 | 7177 ❤️, 895 🔁*

---

## 核心论点

上一代企业软件通过成为**记录系统（Systems of Record）**创造了万亿美元的生态。Salesforce 管客户，Workday 管员工，SAP 管运营。谁拥有权威数据，谁就拥有工作流，谁就拥有锁定效应。

目前的争论焦点是：**Agent 时代这些记录系统会不会死？**

Jamin Ball 最近的文章 "Long Live Systems of Record" 反驳了"Agent 杀死一切"的说法——他认为 Agent 不会取代记录系统，而是提高了对好记录系统的要求标准。

作者同意这一点，但更进一步：**Ball 的框架假设 Agent 需要的数据已经存在于某个系统中了。这只是画面的一半。**

另一半是**真正运行企业的缺失层：决策痕迹（Decision Traces）**——异常、覆盖、先例、跨系统上下文——这些目前存在于 Slack 线程、交易桌对话、升级电话和人们的头脑中。

---

## 记录系统抓不到什么

Agent 正在进入真实工作流——合同审查、报价到收款、工单解决——团队遇到了**治理方案无法解决的墙**。

这堵墙不是数据缺失，而是**决策痕迹缺失**。Agent 遇到的是人类每天都在用判断力和组织记忆解决的模糊性。但这些判断的输入从未被作为持久化工件存储：

| 缺失类型 | 说明 |
|---------|------|
| **人脑中的异常逻辑** | "我们总给医疗公司多 10%，因为他们的采购周期太长了"——不在 CRM 里，是代代相传的部落知识 |
| **过去决策的先例** | "上季度我们给 X 公司做了类似结构——应该保持一致"——没有系统关联这两个交易 |
| **跨系统综合判断** | 客服主管看了 Salesforce 的 ARR、Zendesk 的两个未解决升级、Slack 里的流失风险——综合判断发生在脑子里，工单只写了"升级到 Tier 3" |
| **系统之外的审批链** | VP 在 Zoom 上或 Slack DM 里批准了折扣。机会记录显示了最终价格，但不显示谁批准的、为什么 |

> "从未被捕获"的意思不是数据脏了或孤立了，而是**连接数据到行动的理由从未被当作数据对待过。**

## 上下文图谱（Context Graph）

### 概念

当创业公司在 Agent 编排层中植入决策痕迹发射器，每次运行都记录决策的上下文——它们就得到了企业几乎从未拥有的东西：**一个结构化的、可回放的历史记录，记录了上下文如何转化为行动。**

作者把这套累积结构称为 **上下文图谱（Context Graph）**：
- 不是"模型的思维链"
- 而是跨实体和时间拼接的**决策痕迹活的记录**
- 目的是让先例变得可搜索

### 实际运作方式

以一个续约 Agent 为例：

1. Agent 提议 20% 折扣
2. 策略限制续约折扣最高 10%，除非有服务影响例外批准
3. Agent 拉取 PagerDuty 的三个 SEV-1 事件、Zendesk 中一个"除非修复否则取消"的升级、以及上季度 VP 批准过类似例外的续约记录
4. 路由给财务审批
5. 财务批准
6. CRM 最终只有一条事实：**"20% 折扣"**

有了决策记录，"为什么"变成了**一等公民数据**。随着时间的推移，这些记录自然形成上下文图谱：企业关心的实体（账户、续约、工单、事件、策略、审批人、Agent 运行）通过决策事件和"为什么"链接连接起来。

### 反馈循环

**捕获的决策痕迹 → 可搜索的先例 → 每个自动决策又向图谱添加一条新痕迹 → 图谱越来越丰富**

这不需要一开始就完全自主。从**人在环（Human-in-the-Loop）**开始：Agent 提议→收集上下文→路由审批→记录痕迹。随着类似案例重复，更多路径可以被自动化，因为系统有了一个结构化的先例和异常库。

## 为什么现有巨头建不了上下文图谱

### Salesforce / ServiceNow / Workday 的问题

- 建在**当前状态存储**上：知道机会现在什么样，不知道做决策时什么样
- 当折扣被批准时，证明其合理性的上下文没有被保留
- 无法回放决策时的世界状态 → 无法审计、无法学习、无法作为先例
- 继承父级系统的盲区：一个工单升级涉及 CRM、计费、PagerDuty、Slack——没有哪个现有系统能看到全部

### Snowflake / Databricks 的问题

- 它们在**读路径**上，不在**写路径**上
- 数据通过 ETL 在决策之后才到达数仓
- 到数据进入 Snowflake 时，决策上下文已经消失了
- 一个只能在事后读取的系统，不可能成为决策谱系的记录系统

### 创业公司的结构优势

**Agent 系统创业公司在编排路径中。**

当 Agent 分理工单、响应事件、决定折扣时，它从多个系统拉取上下文，评估策略，解决冲突，并采取行动。**编排层能看到全貌**：收集了什么输入、应用了哪些策略、批准了什么异常、为什么。因为它在执行工作流，它可以在**决策时刻**捕获上下文——不是通过事后的 ETL，而是作为一等公民记录。

## 创业公司的三条路径

### 路径 1：从头替换现有记录系统

围绕 Agent 执行重建 CRM 或 ERP，事件溯源的状态和策略捕获原生嵌入架构。

**例子：Regie** — 构建 AI 原生销售交互平台，替代 Outreach/Salesloft。Agent 是第一等参与者：可以做 prospecting、生成 outreach、跟进、路由、升级到人类。

### 路径 2：替换模块而非整个系统

针对特定子工作流（异常和审批集中的地方），成为那些决策的记录系统，同时将最终状态同步回现有系统。

**例子：Maximor** — 在财务领域：自动化现金管理、关账和核心会计工作流，保留 GL（总账）。ERP 仍然是账本，但 Maximor 成为对账逻辑所驻留的真相源。

### 路径 3：创建全新的记录系统

从编排层开始，但持久存储企业从未系统化存储的东西：**决策痕迹**。随时间推移，可回放的谱系成为权威工件。Agent 层不再是"自动化工具"，而成为企业去回答"我们为什么那么做"的地方。

**例子：PlayerZero** — 生产工程处在 SRE、支持、QA 和开发的交叉点。自动化工单 L2/L3 支持，但真正的资产是它构建的上下文图谱——代码、配置、基础设施和客户行为如何实际交互的活模型。

### 跨路径的基础设施：Agent 可观测性

**Arize** — 正在为这个新堆栈构建可观测层，让团队能看到 Agent 如何推理、在哪里失败、决策质量如何随时间变化。就像 Datadog 成为应用监控的基础设施，Arize 定位为 Agent 决策质量监控的基础设施。

## 创业者的关键信号

### 适用所有三条路径的信号

| 信号 | 说明 |
|------|------|
| **高人力消耗** | 如果有 50 个人手动做一个工作流（路由工单、分类请求、协调系统间数据），这就是信号。人力存在的原因是决策逻辑过于复杂，传统工具无法自动化 |
| **异常密集的决策** | 如果需要"视情况而定"——交易桌、核保、合规审查、升级管理——就是好战场。常规工作流不需要决策谱系 |

### 指向路径 3（新记录系统）的额外信号

> **存在于系统交叉点的组织**

- **RevOps** — 需要协调销售、财务、市场和客户成功
- **DevOps** — 需要搭建开发、IT 和支持的桥梁
- **SecurityOps** — 处于 IT、工程和合规的交叉点

这些"胶水"职能是一个信号。它们的出现正是因为没有单一记录系统能拥有**跨职能工作流**。组织架构图创造了一个角色来承载软件无法捕获的上下文。

自动化这个角色的 Agent 不仅跑得更快——它还可以持久化该角色被创建来产生的**决策、异常和先例**。这就是通往新记录系统的路径：不是拔掉现有系统，而是捕获只有在 Agent 进入工作流后才变得可见的一类真相。

## 总结

**核心问题：** 下一个万亿美元平台是给现有数据加 AI，还是捕获让数据变得可操作的决策痕迹？

作者押注后者。今天构建上下文图谱的创业公司正在打地基。
