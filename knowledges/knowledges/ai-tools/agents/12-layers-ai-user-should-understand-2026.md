# 2026 年每个 AI 用户都应理解的 12 层架构

> **来源：** [12 Layers Every AI User Should Understand in 2026](https://x.com/voxyz_ai/status/2058222816474919343) — @Voxyz_ai
> **中文整理：** 翻译 + 解析 + 扩展

---

## 为什么要理解这些层？

你每天都在用 AI：ChatGPT 回答问题、Claude Code 写代码、Perplexity 做研究。越来越多的人有了自己的 AI agent——它检查邮件、分析数据、为你建立记忆。

但你可能遇到过这些困惑：

- 同一个工具今天回答精准，明天满口胡言
- 同一个问题问 ChatGPT 和 Claude，得到完全不同的答案
- 每周都有新工具 launch，听起来都很厉害，但你不知道哪个真的能用

**这些问题都可以追溯到 AI 工具是如何构建的。**

一旦你理解了下面的心智模型，agent 就不再那么神秘了。一个 AI agent 由 12 层组成，每层做不同的事，每层都有自己独特的失效方式。

> **阅读方法：** 对每一层，问一个问题——如果它缺失了，实际工作会在哪里断裂？

---

## PART 1：基础 —— 存在于哪、是谁、用什么脑子

### 1. 工作表面（Work Surface）—— 人与 AI 相遇的地方

人们发送任务、检查结果、编辑草稿、给出审批的地方。

- ChatGPT 聊天框
- Claude 桌面应用
- Cursor 编辑器
- Agent 在 Linear / Notion 里留下的评论

**不同的工作表面有不同的用途：**
- **聊天框** → 通用入口
- **编辑器** → 写代码的地方
- **工单系统** → 团队工作流
- **浏览器扩展** → 嵌入你已在做的事

> **如果这层缺失：** 看上去聪明的演示永远进不了你的日常工作。Agent 能否真正进入你的生活，从这里开始。

---

### 2. Agent 合约 / 规范（Agent Contract）—— 工作说明书

**最容易跳过的一层，也是最能决定 agent 是否值得信任的一层。**

它定义了 agent 是谁：
- **角色：** 研究员 / 编辑 / 操作员 / 审批人
- **拥有什么、不拥有什么**
- **输出应该长什么样**
- **什么时候必须交回给人类**
- **如何处理失败**

> **弱版本：** "保持专业，提供全面帮助。"
> **强版本（OpenClaw SOUL.md 风格）：** "直接回答。不要'很高兴帮助您'的开场白。一句话能说清楚就说一句话。早早指出坏主意。"

OpenAI Agents SDK 的文档也把"清晰地定义一个专家"作为 agent 设计的明确起点。OpenClaw 则更进一步——把 agent 的个性写进自己的文件 `SOUL.md`，明确定义说话方式（简短优先、要有立场、直接指出问题）和禁忌（废话开场、过度礼貌、模棱两可）。

> **如果这层缺失：** Agent 总是像个临时工——事情能做，但责任边界永远在漂移。**合约是从"有用一次"到"可靠反复"的桥梁。**

---

### 3. 模型（Model）—— 大脑

推理、编码、多模态、速度、成本。

- GPT-5 / GPT-5.5
- Claude 4.x
- Gemini 3.x
- Llama 4

> 更强的模型 = 更高复杂任务天花板
> 更快的模型 = 更低成本，可运行在更多地方
> 弱的模型 = 你会怪 AI，但真正的问题是你用它的方式

**但一个强模型并不意味着整个系统值得信赖。** Agent 如何运行、如何记忆、谁控制它——模型层决定不了这些。**大多数 agent 的失败藏在其他层里。**

---

## PART 2：如何运行 —— 循环 + 与外部世界的连接

### 4. 运行时 / 状态（Runtime / State）—— 循环与持久化

一次 LLM 调用只是一次生成。Agent 需要把多次生成、工具调用和持久化状态链成一个循环：

**思考 → 行动 → 观察 → 重复**

运行时就是那个循环：
- **Plan（规划）** —— 分解任务
- **Act（行动）** —— 调用工具
- **Observe（观察）** —— 查看结果
- **Retry（重试）** —— 失败时重来
- **Handoff（交接）** —— 传给另一个 agent
- **State（持久化）** —— 保存上述所有状态

> **如果这层缺失：** Agent 只是单次助手。能回答一次，但完成不了一个真正的任务。
>
> 示例：OpenAI Agents SDK、LangGraph、Claude Agent SDK

---

### 5. 工具与 Agent 互操作（Tool & Agent Interop）—— 协议层

Agent 不能自己做所有事。它需要调用外部工具、读取外部数据、与其他 agent 协作。

想想 USB-C 出现之前的世界：每个设备有自己的接口，你出门要带一包线。Agent 连接工具曾经也是这种感觉——每个集成就是一个一次性脚本。

**2026 年，这一层正在进入自己的 USB-C 时刻：**

- **MCP（Model Context Protocol）**：Agent ↔ 工具/数据/资源的协议。已归入 Linux 基金会下的 Agentic AI Foundation，有 10,000+ 公开服务器，被 ChatGPT、Cursor、Gemini、Copilot、VS Code 采用
- **A2A（Agent-to-Agent）**：Agent ↔ Agent 协作协议。Google 在 2025 年捐赠给 Linux 基金会，到 2026 年已有 150+ 组织参与或支持

> **如果这层缺失：** 每个集成都需要从头搭建胶水代码。规模化时，工具发现断裂、身份混淆、权限失控、agent 无法协作。
>
> MCP 让 agent 接入工具。A2A 让 agent 接入其他 agent。**两条不同的线，2026 年都不能跳过。**

---

### 6. 执行表面（Execution Surface）—— 行动的场所

Agent 真正采取行动的地方：
- **代码**（仓库、文件、命令行）
- **浏览器**（网页、Web 应用）
- **API**（调用外部服务）
- **桌面操作**（点击、输入）

> **如果这层缺失：** Agent 只能建议，不能行动——变成一个昂贵的聊天机器人。
>
> 但**模糊的边界比没有更糟糕**：能行动 + 没有回滚 + 没有日志 = 真实事故。
>
> 示例：Claude Code、ChatGPT Computer Use、Browser Use、Cursor Edit

> **⚠️ 任何 agent 采取行动时，都要和第 12 层（控制平面）一起看。**

## PART 3：记忆什么、查找什么、交付什么

### 7. 记忆（Memory）—— 自己保留的东西

LLM 本身没有长期记忆。ChatGPT 和 Claude 能记住你的偏好，那是产品层加上去的功能。

Agent 自己设计这层。记忆至少包含 **6 个子层**：

| 层级 | 类比 | 说明 |
|------|------|------|
| Hot session（工作记忆） | 今日便签 | 当前任务的临时状态 |
| Day-state（日间白板） | 今日待办 | 今天的行动记录 |
| Project memory（项目记忆） | 长期笔记 | 持续运行中积累的经验 |
| Retrieval index（检索索引） | 资料库 | 候选参考材料 |
| Canonical policy（规范策略） | 规章制度 | 长期规则 |
| Direct instruction（直接指令） | 老板刚说的话 | 用户最新输入 |

> **如果这层缺失：** Agent 每次都会失忆。
>
> 但仅仅有记忆还不够。当它出问题时更糟糕：**旧记忆覆盖新决策**——三个月前你提的一句偏好，即使上个月你已经改变了想法，它还在影响今天的输出。

**检验标准：能否分层、能否引用来源、能否过期。** 三条都满足，记忆就是 agent 的资产。少任何一条，它就是不可见的 bug。

> 作者上周写过这个话题：[A Framework for Agent Memory: Remember, Cite, Forget](https://x.com/voxyz_ai/status/2057419591618302029)（已整理至知识库）

---

### 8. 知识 / 检索（Knowledge / Retrieval）—— 可查找的东西

记忆是 agent **自己保留**的东西。知识是 agent **去查找**的东西。

- 文档、Wiki、笔记
- 代码仓库、PR、提交记录
- Slack、邮件、对话记录
- 公司数据库、Notion、Linear

> **如果这层缺失：** Agent 回答时充满自信，但没有任何东西可以支撑答案。

RAG、向量搜索、图搜索、关键词搜索——都是这层的工具。

**找到只是第一步。找到正确的东西 + 引用来源，才是让答案可信的关键。**

> 示例：Pinecone、Qdrant、Weaviate、pgvector，以及混合记忆+检索的开源项目如 GBrain。

---

### 9. 持久化工作流 / 编排（Durable Workflow）—— 可恢复的长时间流程

运行一次 agent 很容易。**可靠地运行很难。**

就像航班改签：恶劣天气导致备降，你不会从原出发地重新开始——你在最近的机场继续。Agent 运行长时间任务也需要同样的能力。

在实际工作中，agent 任务经常需要：
- **运行数小时或数天**（而非数秒）
- **中途暂停等待人工审批**
- **断线后恢复**
- **部分失败后重试**，而不是从头开始
- **产生具体制品**（代码 PR、报告、邮件草稿）
- **下次运行时拾取之前的状态**

> **如果这层缺失：** Agent 能完成一次任务，但难以调度、复用、恢复或审计。
>
> 示例：Temporal、Inngest、Restate、Trigger.dev、Cloudflare Workflows
>
> 2026 年 3 月，Temporal 宣布 OpenAI Agents SDK + Temporal Python SDK 集成 GA，强调 agent 工作流可以存活在 rate limit、网络问题和崩溃之上。

## PART 4：为什么可以信任 —— 决定一切的"无聊"层

### 10. 评估（Evals）—— 上线前健康检查

演示中表现完美的 agent 一旦接触真实工作就崩溃——差距就在 eval。

- **Offline eval（离线评估）**：上线前，固定测试集，衡量准确性、成本、错误模式
- **Online eval（在线评估）**：上线后，真实用户 A/B 测试、影子流量、回归监控
- **Policy tests（策略测试）**：精心设计的困难场景——禁止操作、异常输入、越狱尝试
- **Edge cases（边界场景）**：你担心的角落场景，自动运行

LangChain 2026 年 State of Agent Engineering 调查报告：在对 1,300+ 专业人士的调查中，**质量是生产环境部署 agent 的最大障碍**，约三分之一的公司将其列为首要障碍。

> **如果这层缺失：** 你不知道 agent 是真的稳定，还是演示碰巧成功了。
>
> 示例：Braintrust、LangSmith Eval、OpenAI Evals、Promptfoo

---

### 11. 可观测性（Observability）—— 安全摄像头

当 agent 出问题时，你能还原发生了什么吗？

Agent 自己可能不记得。LLM 在循环结束时就忘记了——所以你需要外部的摄像头。

- **Trace（链路追踪）**：这次运行调用了哪些工具，走了哪条路径
- **Logs（日志）**：每一步的输入、输出、成本、延迟
- **Artifacts（制品）**：它产生的文件、发送的邮件、触碰的 PR
- **Decisions（决策）**：哪里重试了、哪里交接了、哪里中止了
- **Source chain（来源链）**：每个结论来自哪里

> **如果这层缺失：** Agent 是一个黑盒。没人能调试它。没人敢用它。
>
> 在 LangChain 的同一调查中，**89% 的受访组织已经在 agent 上部署了某种形式的可观测性**。在 agent 构建者中，这层已经是基线。
>
> 示例：LangSmith、Arize、Helicone、Logfire、Phoenix

---

### 12. 控制平面 / 治理（Control Plane）—— 门禁卡

**最后一层，也是最被低估的一层。**

控制平面决定 agent 代表谁行动、能读什么、能写什么、能运行什么、能花多少钱、能发布到哪。

- **身份（Identity）**：agent 使用谁的身份
- **权限（Permissions）**：能调用哪些工具、能碰哪些数据
- **密钥（Secrets）**：API key 和登录凭证如何授予
- **预算（Budget）**：一次运行能花多少
- **租户边界（Tenant boundary）**：不能跨入其他客户的数据
- **审批（Approval）**：哪些操作需要人工介入
- **审计（Audit）**：事后，谁在何时做了什么
- **终止开关（Kill switch）**：一键停止

2026 年 Anthropic 为 Claude Code 增加了 auto mode，正是出于这个原因：审批每一步导致审批疲劳，完全跳过权限检查又不安全。中间路径是一个自动判断风险的层。

> **控制平面是 agent 产品本身的一部分。把它当成一个设置页面，你就会出事故。**
>
> > **如果这层缺失：** 能力跑到了责任前面。演示很漂亮。事故很频繁。

## 12 层速查表

```
PART 1: 基础
├── 1. 工作表面（Work Surface）—— 人在哪与它相遇
├── 2. Agent 合约（Agent Contract）—— 工作说明书
└── 3. 模型（Model）—— 大脑

PART 2: 如何运行
├── 4. 运行时（Runtime）—— 循环与持久化
├── 5. 工具互操作（MCP + A2A）—— 协议层
└── 6. 执行表面（Execution Surface）—— 行动的场所

PART 3: 记忆/知识/工作流
├── 7. 记忆（Memory）—— 自己保留
├── 8. 知识/检索（Knowledge）—— 可查找
└── 9. 持久化工作流（Durable Workflow）—— 可恢复的长流程

PART 4: 信任
├── 10. 评估（Evals）—— 上线前健康检查
├── 11. 可观测性（Observability）—— 安全摄像头
└── 12. 控制平面（Control Plane）—— 门禁卡
```

## 总结

**能力 ≠ 信任。** 一个能调用工具的 agent 并不意味着它应该进入你的实际工作流。

真实工作中的信任来自：来源链、评估、链路追踪、权限控制，以及人类的最终一票。

> 下次看到一个 AI 新工具，用这 12 层对照一下，你就知道它属于哪里。
