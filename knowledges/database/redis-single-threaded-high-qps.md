# Redis 单线程如何支撑百万 QPS

> 面试题：Redis 是单线程的，为什么能处理每秒数百万的请求？

---

## 1. Redis 的"单线程"是指什么

Redis **命令的执行**是单线程的：
- 6.0 之前：全部单线程（网络 I/O + 命令执行）
- 6.0+：网络 I/O 可多线程，但**命令执行依然是单线程**

同一时刻只有一个命令在跑。

---

## 2. 为什么能跑这么快

### 🚀 核心：I/O 多路复用

Redis 用 **epoll（Linux）/ kqueue（macOS）** 实现 I/O 多路复用：

- 一个线程同时监听**成千上万个 socket 连接**
- 只有有事件（可读/可写）时才去处理
- 没有事件时就挂起，不浪费 CPU

```c
// 事件循环伪代码
while (true) {
    events = epoll_wait(epfd, events, MAX_EVENTS, -1);
    for (event in events) {
        processCommand(event.fd);
    }
}
```

对比传统做法：每连接开一个线程/进程 → 上下文切换开销巨大。

### ⚡ 纯内存操作

Redis 所有数据在内存里，读写是 **ns 纳秒级**的。而传统数据库从磁盘读数据是 **ms 毫秒级**——差了百万倍。

### 🧩 数据结构极致优化

Redis 根据数据量自动切换底层编码：

| 场景 | 编码 |
|------|------|
| 小整数集合 | `intset`（紧凑数组） |
| 小字符串 | `embstr`（一次内存分配） |
| 小 hash 列表 | `ziplist`（连续内存） |
| 大集合 | `hashtable` / `skiplist` |

### 🔥 避免上下文切换和锁竞争

多线程的最大开销不是执行本身，而是：

- **锁竞争**：争锁、等待、释放
- **上下文切换**：线程切换导致 cache miss
- **CPU 缓存失效**：数据在不同核心间迁移

Redis 单线程直接**零锁**，所有请求顺序执行，不存在并发问题。

---

## 3. 为什么单线程不成为瓶颈

**Redis 的瓶颈几乎永远是「网络 I/O」和「内存带宽」，不是 CPU。**

实测数据参考（单机）：

| 操作 | QPS |
|------|-----|
| `SET/GET` 小字符串 | 10万 ~ 20万/秒 |
| `PING` | 可达 100万+ /秒 |
| Pipeline 批量 | 可达 数百万/秒 |

CPU 占用通常不到 50%，先打满的是网卡带宽。

---

## 4. Redis 6.0+ 的多线程改进

```
之前：单线程读网络 → 执行命令 → 单线程写网络
之后：多线程读网络 → 单线程执行命令 → 多线程写网络
```

- 网络读写（解析/序列化）是多线程的
- **命令执行依然是单线程的**

收益：网卡带宽高的情况下，多核分担网络 I/O 压力，QPS 提升约 2 倍。

---

## 5. 单线程的代价（面试追问题）

| 问题 | 影响 | 应对 |
|------|------|------|
| **慢命令阻塞** | `KEYS *`、大范围操作 → 整个 Redis 卡住 | 线上禁用 `KEYS`，用 `SCAN` 替代 |
| **大 Key** | 单次读写几 MB，耗时飙升 | 拆分大 key，`--bigkeys` 定期扫描 |
| **CPU 密集型操作** | `SORT`、复杂 Lua 脚本 | 避免在线执行 |
| **单核瓶颈** | 机器 64 核，Redis 只用 1 个 | Redis Cluster 分片，每分片用 1 核 |

---

## 💡 总结

> Redis 单线程能跑百万 QPS，靠的是 **epoll 多路复用 + 纯内存 + 零锁 + 极致的数据结构设计**。它不是不能多线程，而是单线程在它的场景下是最优解。
