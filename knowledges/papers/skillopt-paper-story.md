# 教会，再放手——训练冻住的模型

> 一只冻住的 agent 模型，能不能学会新本事？SkillOpt 说能——把教本事的那份"说明书"当参数训，模型本身一动不动。

![Cover](../image/skillopt-overview-1.jpg)

## 问题

你有个 agent，一个语言模型。它干活靠一份说明书——你放在系统消息里的那段话，告诉它"查这个来源""按这个格式答""先验证再回复"。这叫它的 *skill*。

问题是说明书是手写的。

你写一份，它不一定好用。改一改，也许就好点了，但改到什么程度很难说。你改的方向对不对，得看反馈——准确率上去了就是好改，下去了就是改坏了。

听上去像什么？

像训练。

你有个模型（说明书），有训练数据（任务批次），有损失函数（失败/成功分析），有参数更新（修改说明书），有验证集（held-out 检查）。唯一不同的，参数不是浮点数，是自然语言；梯度不是反向传播，是模型审模型。

但反过来说：既然说明书才是 agent 真正学到的本事，那为什么不把它当一个可训练的参数来优化？

这就是 SkillOpt 的起点。

*旧路*：之前的手艺是三类。第一类，手写说明书——质量靠写的人，改一次麻烦一次。第二类，把对话经验提炼成小技巧，一条一条加进去——但加多了互相矛盾，没人敢删。第三类，完全重新写——一改就全推倒，之前好的做法也丢了。都不像"优化"。

*转折*：微软的研究员看到——如果说明书才是 agent 的行为载体，那"训说明书"这件事本身就应该是可量、可控、可复现的。不能靠感觉改，要像训练参数一样训练它。

## 翻译

### 一只冻住的模型 + 一本本子在改的说明书

SkillOpt 的架构很干净。两套东西：

- *目标模型*：就是你要适配的那个 agent，冻住不动。它什么也不学。
- *优化器模型*：另一个更强的模型，专门看目标模型干活的结果，然后修改说明书。

目标模型用当前版本的说明书干活，干一批任务——这叫一次 *rollout*。干完了，结果交给优化器模型。优化器先分析：哪些干得好，哪些干得差。把好的和差的分开看——这叫 *minibatch reflection*。

优化器看了失败的那一堆，发现一个规律：模型总是查错地方。于是它提议：「在说明书写一条：查某来源之前先看日期」。

但它不直接写。它先把这条提议放进一个候选池，跟其他提议合并、去重、排序。然后，它只选排名最高的几条——*有限的*几条——加到说明书里。这叫 *bounded edit*，是 SkillOpt 最关键的设计。

为什么有限？因为一次改太多会乱加。要一条一条来，让效果可追溯。

### 门：改了不等于进步

改完的说明书不直接用。优化器拿一小批验证集数据，拿新说明书跑一遍。分数涨了？好，保留。没涨？扔掉，回到之前的版本。

这个 *validation gate* 很朴素，但效果显著。文本编辑不像梯度下降——你永远不知道这次改动是"往正确方向走了一步"还是"改歪了"。有门挡着，歪的就进不来。

### 被拒的编辑也是信息

改坏了的那份说明书，不被扔掉——它的内容被记录进一个 *rejected buffer*。以后优化器再提建议时，能看到：「这个方向试过了，把分数从 74 拉到了 69，别走这条路。」

这是文本空间的负反馈。没有梯度，但等价于梯度的符号：这个方向是坏的。

### 慢更新：多看几轮再说

还有一层。在每个 epoch 结束时（几轮快更新之后），优化器会回头看看整个轨迹：哪些改动活了，哪些被拒绝了，哪些问题一直没解决。它据此写一段长视角的回顾，加入下一轮的优化提示里，给优化器自己看，不给最终部署的说明书。

这叫 *slow/meta update*——像做实验的人回头看实验记录，不是修数据，是修方法。

你发现没有？这一整套——batch、minibatch、learning rate schedule、validation gate、momentum、rejected buffer——全是深度学习训练的对应物，但全在文本空间里实现。

## 核心概念

### 1. Bounded Edit（有界编辑）

这是整个系统最核心的旋钮。每次只改不超过 *L_t* 条内容，L_t 就是那个 *textual learning rate*。数字小 → 学得稳但慢；数字大 → 学得快但可能震荡。SkillOpt 默认用 cosine 衰减——一开始步子大些，后面逐步收窄做精修。

相比之下，手写 skill 是一下子写一大堆——没有版本控制，没有渐变，没有反馈闭环。

### 2. Validation Gate（验证门）

文本空间没有梯度告诉你"这次改的方向对不对"。所以必须有一条硬规则：改了之后分数不涨就别想进来。这个门把主观的"我觉得这样改应该有用"变成了客观的"用数据证明有用"。

缺了这个门，系统就退化回自我改写——今天改一句明天改一句，最后不知哪个版本有用。

### 3. Harness Adapter（支架适配器）

SkillOpt 不挑干活的方式。直聊、Codex 循环、Claude Code、具身环境——每一个只要写一个薄薄的 adapter 就行。adapter 只干三件事：把当前 skill 塞进 agent 的上下文、让 agent 干活、把结果打出来。剩下的优化逻辑一模一样。

这意味着训好的 skill 可以在不同"干活方式"之间迁移。作者证明了这个：Codex 上训好的 spreadsheet skill 直接放到 Claude Code 里用，涨了 +59.7 分。

### 4. Rejected Buffer（被拒缓存）

不是组件，是设计的精巧处。没有梯度，就用"这次改坏了"的负信号来近似。每次拒绝，系统多了一条"这里走过但下面是悬崖"的路标。这个缓存帮助系统避开重复错误——相当于文本空间的动量项。

## 洞见

**Skill 才是真的参数。模型是壳，说明书是魂。**

大多数团队把精力花在选模型、调参数、写 prompt。SkillOpt 提出一个更根本的视角：你写给 agent 的那份说明书，才是它适应新场景的主要载体。这份说明书可以像参数一样被训练——有 batch、有 learning rate、有 validation、有 momentum。

这意味着什么？意味着模型本身可以冻住不动（省钱省力），但行为在新领域上持续提升。意味着团队可以专心"训 skill"而不是"训模型"——前者是文本编辑，成本极低；后者是权重更新，贵还容易倒退。

## 博导审稿

*选题眼光*：好。Agent skill 优化是当前 agent 工程的实际痛点——大多数团队的手写 skill 效果不稳定，但又没有更好的方法论。SkillOpt 切中了真实缺口。

*方法成熟度*：干净。把深度学习训练的每一个概念翻译到文本空间，不是噱头，每一部分都有消融实验支撑。唯一的担心：优化器模型必须比目标模型强（他们用了 GPT-5.5 做优化器），如果优化器不够强，这个闭环还能不能成立？作者做了自优化实验（用小模型当自己的优化器），有提升但幅度小很多——这是个合理的限制。

*实验诚意*：52/52 全胜，覆盖面广（6 benchmark × 7 模型 × 3 harness），消融到位（有界编辑、验证门、被拒缓存、慢更新四个组件都做了 ablations）。转移实验（跨模型、跨 harness、跨 benchmark）做得很加分。

*写作功力*：技术细节清晰，但情绪叙事偏弱——作为学术论文足够好，但故事感不足。

*判决*：*strong accept*。方法新颖、实验扎实、落地潜力大。

## 启发

如果你维护一个 agent 系统，最值得做的事可能就是**开始"训"你的 skill 文档**。

- 把手写的 SKILL.md 当第一个版本
- 用 GPT-5.5 或同级别模型做优化器
- 收集几批失败/成功轨迹
- 让优化器提有界编辑，只用验证门的版本
- 几个 epoch 下来，你的 skill 会比手写的版强出一大截

SkillOpt 已经在 SearchQA 涨了 9.6 分、SpreadsheetBench 涨了 38.9 分、LiveMath 涨了 29.3 分。这些数字告诉我们一件事：**你写给 agent 的那份说明书，远比你想象的值钱——也远比你想象的值得被优化。**
