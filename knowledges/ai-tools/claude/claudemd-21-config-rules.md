## 前置：不用 CLAUDE.md 的真实成本

> 作者：0xDepressionn

一份叫 CLAUDE.md 的文件曾登上 GitHub Trending #1。82000 star，7800 fork。

故事起点是 Andrej Karpathy（特斯拉前 AI 总监、OpenAI 创始成员）。他识别出 Claude Code 失败的 4 个关键行为，写进一个文件。一位开发者把这 4 条规则扩展后发布，结果火了——**编码准确率从 65% 提升到 94%**。

大多数日常使用 Claude Code 的开发者至今没设置过这个文件。他们每次会话从零开始——重复解释同样的上下文、清理没人授权的范围蔓延、回退没人要求的重构。

**每次打开 Claude Code 都从空白的起点开始。**

它不知道你的技术栈、你的标准、你的项目上下文、你已经试过什么、三场会话前你明确决定不做的事。所以它猜——猜的时候就会重构你没让它碰的代码，推荐破坏你现有架构的框架，不问就删文件，推翻你已经做过的决策。

CLAUDE.md 就是一个放在项目根目录的纯文本文件。Claude Code 每次会话开始自动读取。一次设置，零重复解释。

### 不用 CLAUDE.md，你在烧多少钱？

**第一部分：DEFAULTS — 每周花 $375 重复解释自己**

开发者平均每天花 30 分钟向 Claude 重新解释上下文。技术栈、编码标准、项目背景、已经试过什么——没有一样在会话之间持久化。

30 分钟/天 × $150/小时（开发者时薪）= $75/天 = **$375/周/人**。
一个 5 人团队：**$1,875/周**——只在重复解释上下文上。

**第二部分：BEHAVIOR — 每周花 $450 撤销未授权的变更**

你让 Claude 修一个函数。它重构了三个文件，重命名了你的变量，重新组织了 import，重写了你精心打磨的注释——全都不问。

审核和撤销这些变更平均 1 小时/次，$150/小时。一周三次：$450/周/人。
一个 5 人团队：**$2,250/周**——只为清理没人授权的改动。

**第三部分：MEMORY + STACK — 每周花 $300 从遗忘中恢复**

Claude 在会话之间忘记一切。你做过的每个决策、尝试失败的每个方案、六个月前选 Prisma 而非 Drizzle 的原因、因特定客户需求存在的约束——全忘记了。然后它建议你早已排除的方案。

因此花 2 小时/周恢复：$300/周/人。来自错误的技术栈建议的额外修复：$225/周。

**总计：$975/周/人。一个 5 人团队：$4,875/周。**

而设置 CLAUDE.md 只需要 2 小时。仅 Karpathy 的 4 条规则就能把编码准确率从 65% 提升到 94%。

> 设置了的开发者用的 Claude 能记住决策、不越界、破坏前确认、从不推荐破坏架构的框架。
> 没设置的每周花 $975 重复自己。

## 上篇：CLAUDE.md 的底层哲学

> 作者：nainsidwiv50980

大多数开发者认为 Claude 输出不好是因为：
- 提示词不够详细
- 模型不够聪明
- AI "还没到那个地步"

但在观察了大量开发者使用 Claude Code 的方式后，我发现了一个真相：

**优秀输出和平庸输出之间的最大差距，通常跟提示词无关——它来自上下文。更具体地说：CLAUDE.md。**

大多数人用错了这个文件。他们把它当成随便写写的备忘录，甚至直接无视。

而高级用户把 CLAUDE.md 当作**整个 AI 工作流的操作系统**。

同款模型，完全不同的人机体验。

### Claude 为什么感觉"不稳定"

很多开发者不知道：**Claude 每次会话开始时都是上下文盲的。**

它不知道：
- 你的项目结构是怎样的
- 你遵循什么模式
- 做过什么架构决策
- 哪些错误最致命
- 你的工作流期待是什么

除非你明确定义这些，否则 Claude 只能即兴发挥——而不一致性正是从这里开始的。

### CLAUDE.md 的真正目的

大多数人认为 CLAUDE.md 是用来"给指令"的。

这只是表面。

**它的真正目的是：减少决策模糊性。**

好的 AI 工作流不是靠无休止地加提示词构建的，而是靠**消除不确定性**构建的。

最好的 CLAUDE.md 不是试图控制一切——它们只是防止最昂贵的错误反复发生。

这是一个巨大的思维转变：

> **CLAUDE.md 不是愿望清单，而是一个错误预防系统。**

### 为什么大多数 CLAUDE.md 在悄悄失败

#### 错误一：文件太长

更多人写 = 更好输出。事实恰恰相反。

每一行额外内容都在争夺注意力。文件越长：
- 指令遵循越弱
- 上下文质量越低
- 重要规则越被稀释

最高效的设置通常出奇地精简。因为好的系统优先关注**清晰度而非数量**。

#### 错误二：通用指令

像"写干净的代码"、"当资深工程师"、"一步步思考"——听起来有用，但几乎不改变行为。Claude 已经理解这些概念。

真正有价值的是**项目特定的约束**：
- 永远不要修改部署配置
- 始终运行类型检查
- 只使用现有工具库
- 避免触碰无关文件
- 改动架构前先问

这些指令阻止真正的错误。

#### 错误三：无结构

大多数人把工作流偏好、技术栈细节、架构笔记、个人意见、随机提醒全部塞进一个巨型文件。

高级用户有意识地将上下文分离：
- **全局规则** — 适用于所有项目
- **项目规则** — 当前项目特定
- **个人工作流偏好** — 你的习惯

更清晰的层次 = 更清晰的输出。

## 中篇：高质量 CLAUDE.md 的五部分框架

> 作者：nainsidwiv50980

分析大量优秀 Claude 工作流后，几乎每个好的设置都包含这五个方面。

### 1. 命令 Commands

Claude 在理解项目如何实际运行时表现要好得多。

没有命令，它就靠猜——猜就产生不必要的来回沟通。

关键内容：
- 如何运行开发服务器
- 如何测试
- 如何 lint
- 如何做类型检查

仅此一项就能消除几乎所有工作流中的摩擦。

### 2. 架构图 Architecture Map

Claude 不会神奇地理解你的代码库结构。

你需要定义基础：
- 业务逻辑在哪
- API 路由在哪
- UI 组件属于哪
- 状态管理在哪

不需要每个细节，只需要足够阻止糟糕的放置决策。

### 3. 硬规则 Hard Rules

这可能是整个文件中 ROI 最高的部分。

每条规则应该回答一个问题：

> "如果去掉这条，Claude 会犯昂贵的错误吗？"

- 是的 → 保留
- 不是 → 删除

最强规则的特点：
- **具体** — 不是"小心测试"，而是"运行 `npm test` 验证通过后再提交"
- **可执行** — 不是模糊的原则，是可操作的约束
- **项目相关** — 针对当前项目的实际痛点

关于否定规则：**告诉 Claude 不要做什么，和告诉它要做什么同样重要。**

### 4. 工作流行为 Workflow Behavior

这控制 Claude 处理任务的方式——不仅是最终输出，还有过程。

示例：
- 重大改动前先问
- 只做最小限度的修改
- 先解释权衡
- 按逻辑拆分提交
- 改完后验证变更

这些显著提升协作质量。

### 5. 边界 Out-of-Scope Boundaries

这一节被严重低估了。

提升 AI 一致性的最佳方法之一是**清晰定义禁区**：

- 不修改部署配置
- 不碰手动维护的文件
- 避免依赖变更
- 绝不自动修改生产基础设施

边界创造稳定，稳定产生复利。

### 关于 CLAUDE.md 最强大的洞察

> 一个好的 CLAUDE.md 是随着时间**进化**的。

每次 Claude 犯错，你就完善系统。

几个月后，它变成了：
- 机构记忆
- 工作流文档
- 架构指南
- 错误预防
- 运营上下文

这时候 Claude 开始感觉明显更智能——不是因为模型变了，而是因为你的系统进化了。

> 提示词是暂时的，系统会复利。

## 下篇：CLAUDE.md 21 条实战配置

> 作者：Anatoli Kopadze（@AnatoliKopadze）

一份 CLAUDE.md 文件曾冲上 GitHub Trending #1，82000 star，7800 fork。

**大多数人没用过它。用过的人不知道里面该写什么。**

每次打开新的 Claude 会话，它都是零记忆——不知道你的名字、工作、偏好。你花前几分钟重复解释，或者不解释——Claude 给出不适合你的东西。CLAUDE.md 永久解决这个问题。

### 怎么创建 CLAUDE.md（2 分钟）

1. 在项目文件夹创建文件，文件名必须是 `CLAUDE.md`（全大写，无空格）
2. 用任何编辑器打开
3. 把本文中相关指令复制进去——**不需要全部 21 条，先挑 3-4 条最痛点**
4. 保存。Claude Code 每次打开该文件夹的会话时自动读取

---

### 一、沟通方式 How Claude Talks to You（指令 1-4）

#### 1. 砍掉废话 ❌ "Great question!"

Claude 的默认开场：「Great question!」「Of course!」「Certainly!」「Absolutely!」——每天用几个小时时，这些废话不断累积。

**写入 CLAUDE.md：**
> 每次回答直接给答案。不需要开场白、不需要表演热情。我问什么，直接答什么。

#### 2. 行动前先给选项 🎯

Claude 默认选一个方向直接干。你让它改一段文字，它把全文语气都变了。你说重构文档，它改得不符合你的思维方式。

**写入 CLAUDE.md：**
> 重要的任务前，先给 2-3 种方案让我选方向。选完再动手。

#### 3. 不知道就直说 🔍

Claude 在承认不确定之前，会先给一个自信、详细、但完全错误的东西。它用听起来很合理的数据、日期、引用来填补空白。

**写入 CLAUDE.md：**
> 对不确定的内容，在回答前标注「我不确定这个」而不是藏在自信的答案里。

#### 4. 回答长度匹配需求 📏

简单问题写四段，复杂问题给个骨架——两个极端都没用。

**写入 CLAUDE.md：**
> 回答长度与实际需要匹配：短问题短答，需要深度的内容给足细节。

---

### 二、行为方式 How Claude Behaves（指令 5-8）

#### 5. 大改前先问 📋

你说修一段，它重写了整个文档。你说缩短，它删了你需要的部分。

**写入 CLAUDE.md：**
> 任何重大变更前，先停下来告诉我准备做什么，等我确认后再动手。

#### 6. 只改被要求的 🔧

修一个 bug 时顺手「优化」五个其他东西。

**写入 CLAUDE.md：**
> 只改我问的内容。其他东西如果值得改，提一句但别碰。除非我要求。

#### 7. 每次做完告诉我改了什么 📝

改完了但你看不懂到底改了哪——每次都要自己做 diff。

**写入 CLAUDE.md：**
> 每次任务结束时，给我一个简短总结：什么变了、什么没变、有什么需要我注意的。

#### 8. 绝不在没问的情况下替我操作 🚫

AI 工具越来越「能做事」——发消息、发帖、分享文件、安排日程。这些行为有实际后果。

**写入 CLAUDE.md：**
> 任何对外产生影响的动作，绝不允许不经我明确确认就执行。这不是「小心一点」，而是一堵硬墙。

---

### 三、你的上下文 Your Context（指令 9-11）

#### 9. 告诉 Claude 你是谁

Claude 不知道你是专家还是新手、创始人还是自由职业者。没有上下文它就猜——猜对一半错一半。

**写入 CLAUDE.md：**
> 我是 [你的身份]。[技术栈/专业领域]。[沟通风格偏好：喜欢技术深度 / 简单明了 / 等等]。

#### 10. 给出你在做什么的背景

每次会话 Claude 都不知道你在做什么项目、为谁做、什么最重要。

**写入 CLAUDE.md：**
> 我正在做的项目：[简述]。目标用户：[谁]。关键约束：[预算 / 时间 / 技术限制等]。

#### 11. 锁定你的声音和风格

Claude 有默认写作风格。可以，但不是你的。每次用它写东西你都要改回自己的语气。

**写入 CLAUDE.md：**
> 定义你的写作风格：正式/口语化？技术/通俗？简短/详细？爱用什么句式、避免什么用词？写出来，Claude 从此第一稿就是你的声音。

---

### 四、记忆与连续性 Memory & Continuity（指令 12-15）

#### 12. 让 Claude 维护一个 MEMORY.md 文件 🧠

Claude 会话之间完全失忆。但 **Claude 可以写文件，文件会持久存在**。

**写入 CLAUDE.md：**
> 在项目根目录维护一个 MEMORY.md 文件。记录每次的重要决策：做了什么、为什么、考虑了哪些被否定的方案。每次会话开始先读这个文件。

#### 13. 会话结束时写总结 📓

关掉会话，两天后回来。花 15 分钟读旧消息回忆做到哪了。

**写入 CLAUDE.md：**
> 每次会话结束前，把当前状态写入 MEMORY.md：完成了什么、正在做什么、下一步是什么。下次打开直接接上。

#### 14. 记录什么行不通 ❌

试了 4 次终于写出可用的东西。三周后做类似任务，Claude 又从头开始给同样不靠谱的建议。

**写入 CLAUDE.md：**
> 维护一个「没走通」的日志。记录：试了什么、为什么不行、最终什么方案有效。下次遇到类似问题先查日志。

#### 15. 给出永远不变的事实基础 📌

每个项目都有永久不变的事实：过去的决策带来的约束、有重要原因存在的规则、总是成立的前提。

**写入 CLAUDE.md：**
> 一个永久事实块。列出那些不会改变的东西：[技术限制]、[业务规则]、[过去的关键决策] 等。写到文件里，Claude 就不用每次都重新问了。

---

### 五、开发者专项 For Developers（指令 16-21）

以下这 6 条是 **Claude Code** 用户的专属配置——决定了 Claude 是精准的编码助手，还是代码库里的野马。

#### 16. 不越界 🎯

让修一个 bug → 重构三个文件、重命名变量、重新整理 import、顺手「优化」你用了几个月的代码。有些改动会引入微妙的 bug。

**写入 CLAUDE.md：**
> 只改被要求的文件。除非我明确说「也改一下 X」，否则不要碰其他任何文件。

#### 17. 破坏性操作前必须确认 ⚠️

Claude Code 在终端里跑，有文件系统权限。它会删文件、覆盖函数、删表——因为你说了，哪怕你并没完全意识到你说了什么。

**写入 CLAUDE.md：**
> 任何可能破坏已有代码的操作前，停下来，展示具体会改动什么，等我确认再执行。删除、覆写、重构都是。

#### 18. 硬止损 🚧

有些操作不是「小心一点」——是「绝对不允许不经你本人坐在桌前说 Yes 就执行」。

**写入 CLAUDE.md：**
> 以下操作绝对不允许不经明确同意执行：
> - 部署到生产环境
> - 在线上数据库跑迁移
> - 对外部服务发 API 请求
> - 发消息/邮件/社交平台内容

#### 19. 锁定技术栈 🔒

没有固定技术栈，Claude 会推荐它见的最多的框架，不是你用的、不是你团队懂的、不一定兼容你已经搭好的东西。

**写入 CLAUDE.md：**
> 固定技术栈：[语言/框架/包管理器/数据库/基础设施]。Claude 只在这个栈内建议，不要推荐替代品。

#### 20. 永远展示变更摘要

Claude 做完改动后，你不知道改了哪些文件、有没有碰其他东西、有没有留未完成的任务。

**写入 CLAUDE.md：**
> 每次代码改动后，给一个文件级别的变更摘要：改了哪些文件、怎么改的、还有什么是待办的。

#### 21. Karpathy 的 4 条铁律 🏆

Andrej Karpathy（特斯拉前 AI 总监、OpenAI 创始成员）识别出 Claude Code 在编码任务上失败的 4 个关键行为。一个开发者把这 4 条写进 CLAUDE.md，冲到 GitHub Trending #1，**编码准确率从 65% 提升到 94%**。

**写入 CLAUDE.md（开发者必加）：**
```
1. 大改前展示完整计划
2. 只修改我指定的文件
3. 每步改动都告诉我具体做了什么
4. 不确定时停下来问，别蒙
```

## 配置策略建议

| 角色 | 至少该加 | 不要加 |
|------|---------|--------|
| **所有人** | 砍废话 + 匹配长度 + 你是谁 + 永久事实 | — |
| **内容创作者** | 锁定声音 + 大改前问 + 改完说明 | 开发者专属 (16~21) |
| **开发者 (Claude Code)** | 不越界 + 破坏性确认 + Karpathy 四律 | 如果非创作可省略声音定制 |
| **重度使用 (>4h/天)** | MEMORY.md + 会话总结 + 错误日志 | — |
| **团队决策者** | 先读成本分析（$975/周/人浪费），再决定全员推行 | — |

## 团队篇：CLAUDE.md 如何在团队中应用

> 作者：KyrieCheungYep

> 从工程师转做出海业务，把 AI 塞进团队日常工作中。团队里那些反复出问题的地方，十有八九不是谁能力不行，而是该说清楚的规矩，从来没人好好写下来。CLAUDE.md 就是这件事在 AI 编程工具上的具象化。

![CLAUDE.md 团队指南](../image/claudemd-team-guide-1.jpg)

### 场景：同一个项目，Claude 在不同人手上表现完全不同

团队里只要有第二个人在同一个仓库上用 Claude Code，就会出现：在你这里它老老实实写测试、按规范来；到同事那台机器，它能把整个组件推倒重写；到实习生那边，它压根不知道你们的接口该返回什么结构。

谁都没做错。问题是给 Claude 定的所有"规矩"，全散在两个永远不会同步的地方：每个人的脑子里，和每个人本地那台电脑里。

拿一个 5 人做跨境电商的出海团队举例——前后端分离的 monorepo，主要服务东南亚。下面所有配置、所有困难，都在这个项目里发生。

---

### 一、分层：三个文件

很多人对 CLAUDE.md 的第一个误会，是以为它就一个文件。

它不是。Claude Code 启动的时候，会一口气读好几个文件，然后揉成一份完整的指令。

具体怎么读？它从你当前所在的目录出发，顺着目录树往上爬，一直爬到 git 根目录，沿途碰到的每个 CLAUDE.md 都收进来；同时它还会去你的 home 目录里，把那个全局配置也拿上。这些文件叠在一起，才是 Claude 真正看到的规则。

有三层需要分清楚：

| 层级 | 文件位置 | 作用域 | 管理方式 |
|------|---------|--------|--------|
| **个人级** | `~/.claude/CLAUDE.md` | 你所有项目 | 你自己维护，不进 git |
| **项目级** | `./CLAUDE.md`（仓库根目录） | 整个团队 | 进 git，走 PR + Code Review |
| **项目个人覆盖** | `./CLAUDE.local.md` | 仅你在当前项目 | 自动 .gitignore，不进 git |

**合并规则：越具体的越优先。** 同一条指令冲突时，具体程度从高到低：`CLAUDE.local.md` > `./CLAUDE.md` > `~/.claude/CLAUDE.md`。

举个例子。团队里有个工程师叫小李。他那三个文件长这样：

**个人级 `~/.claude/CLAUDE.md`：**
```
用中文回答我
讲解时尽量详细
缩进用 4 空格
```

**项目级 `./CLAUDE.md`（全队共用、进 git）：**
```
缩进用 2 空格
API 调用必须包 try/catch
数据库查询必须用 projection
```

**他的 `./CLAUDE.local.md`（只在他电脑上，不进 git）：**
```
动支付模块以外的代码之前先问我
```

合并后 Claude 实际听到的：
- 中文回答（个人级，无人覆盖）
- 讲解讲细点（个人级）
- 缩进用 2 空格——个人级说 4 空格，项目级说 2 空格。项目级更具体，赢。小李的个人习惯被盖掉，团队一致优先
- API 调用包 try/catch（项目级）
- 数据库查询用 projection（项目级）
- 动支付模块以外的代码先问他（他自己的 local）

**核心问题：一条规则该放哪一层？**

只问自己一句话：**这是团队的决定，还是我个人的偏好？**

- 技术栈、构建命令、编码规范、API 约定、PR 规则 → **团队决定，进项目级**
- 用什么语言跟 Claude 说话、喜欢它讲多细、本地数据库开在哪个端口、你负责哪个模块 → **个人偏好，进个人级或 local**

放错层的代价：有人手一抖把"用 4 空格"写进项目级 CLAUDE.md，全队的 Claude 都被他一个人的习惯带跑了。"Claude 像不同的人"的怪象，很多时候就是这么来的。

---

### 二、CLAUDE.md 是命令，不是描述

写 CLAUDE.md 时容易跑偏成项目介绍——"本项目是一个面向东南亚市场的跨境电商平台，采用前后端分离架构……"

**打住。这是 README 的口气。CLAUDE.md 不该这么写。**

CLAUDE.md 会被整段塞进 Claude 的系统提示里。它里面的每句话，都得是一条能直接照做的命令，不是一段给人看着舒服的介绍。

❌ **描述式（读者常写错的）：**
```
我们比较重视类型安全，尽量用 TypeScript 写。
异常处理方面，try/catch 用得比较严格。
```

✅ **命令式（CLAUDE.md 该有的样子）：**
```
禁止使用 any 类型
所有 API 调用必须包 try/catch
```

区别在哪？描述式全是软绵绵的形容词，Claude 读完没法做任何明确判断。命令式"禁止""必须"——边界硬邦邦的。

写的时候，你脑子里不该想着"我在介绍我的项目"。你该想的是：我面前坐着一个能力很强、但对我们团队规矩一无所知的新同事，我要给他列一份没得商量的规矩。**多用祈使句。多用"必须、禁止、总是、绝不"。少用"建议、尽量、一般来说"。**

---

### 三、可直接使用的团队模板

放仓库根目录，叫 CLAUDE.md，提交进 git。

```markdown
# Project CLAUDE.md

## 常用命令
- 启动开发: pnpm dev
- 运行测试: pnpm test
- 类型检查: pnpm typecheck
- 代码检查: pnpm lint

## 技术栈
- 框架: Next.js 14 (App Router)
- 样式: Tailwind CSS
- 数据库: PostgreSQL via Prisma
- 包管理: pnpm
- 部署: Vercel

## 编码规范
- 禁止使用 any 类型
- 所有 API 调用必须包 try/catch
- 数据库查询必须用 projection
- 组件默认导出
- 命名: 组件 PascalCase，工具函数 camelCase
- 文件命名: 页面 page.tsx，组件 ComponentName.tsx

## 关键陷阱（团队特有）
- 金额运算必须用整数分，禁止用浮点数
- 报错信息必须从常量文件引用，禁止硬编码字符串
- 支付模块代码必须经过 2 人 review 才能合入 main
- 所有外部 API 调用必须加超时和熔断

## 架构约定
- 业务逻辑放在 lib/ 下，禁止写在组件里
- API 路由按 domain 分组：app/api/[domain]/
- 共享类型放在 packages/shared/
- 环境变量统一走 src/env.ts，禁止直接 process.env
```

#### 模板的设计逻辑

**常用命令——** 因为 Claude 不知道你们用 pnpm 还是 npm，也不知道测试命令叫什么。你不写，它就猜。猜错了，命令就跑错了。

**关键陷阱——** 价值压过"编码规范"。编码规范是通用知识，Claude 本来就懂个七七八八。可"金额必须用整数分""报错信息必须来自常量文件"——这是团队拿事故换来的经验，Claude 再聪明也猜不到。一份 CLAUDE.md 里最值钱的，永远是这种**"只有你们才知道"**的部分。要是一场 CLAUDE.md 通篇都是"要写注释""要有可读性"这种正确的废话，那它基本等于没写。

**长度控制——** 好的 CLAUDE.md 尽量压在 **200 行以内**。它会被整个塞进系统提示。每多塞一条没什么价值的指令，那些真正关键的指令权重就被稀释一分。宁可少，别贪多。

#### 反面教材

```markdown
# 项目介绍
本项目是一个面向东南亚市场的跨境电商平台，采用前后端分离架构。
我们的团队非常注重代码质量和可维护性，希望大家能写出优雅、
可读性强的代码。在编写代码时，请尽量遵循行业最佳实践……
```

全是形容词，没一条能照着做；全是放之四海皆准的废话。Claude 读完它，对这个项目的了解跟读之前一模一样。

#### 两个硬性提醒

1. **别放敏感信息。** API key、密钥、数据库连接串、详细安全漏洞细节——统统别放。这文件会进系统提示，还会进 git。当它是一份随时可能被公开的文档来对待。

2. **可以用 `/init` 生成初稿，但它只是初稿。** 在项目里跑 `/init`，Claude 会扫一遍代码库，自动构建命令、关键目录、检测到的代码风格——这些通常能识别。但它抓不到你们的隐性约定（那些"金额用整数分"的坑）。把 `/init` 的产出当起点，然后人工补上它不可能知道的东西。

---

### 四、把项目 CLAUDE.md 当代码管

项目级 CLAUDE.md **是代码**。它的每一次改动，都得走 Pull Request，走 Code Review。

这文件影响所有人，你改一个字，全队五个人的 Claude 行为同时变。跟你改一个被到处调用的公共函数风险一个量级。你不会让谁不经 review 就直接 push 改公共函数吧？那你也不该让谁直接 push 改 CLAUDE.md。

**改 CLAUDE.md 的 PR，描述里得说清三件事：** 改了啥、为啥改、对全队啥影响。

**Review 四条检查清单：**
1. 它是命令，还是描述？（口气够不够硬）
2. 它是这个团队独有的，还是通用废话？（配不配占这一行）
3. 它会不会跟文件里已经有的某条规则打架？
4. 它该进项目级，还是其实是某个人的偏好、该塞进 local？

**往深了说：** 团队一旦开始用 CLAUDE.md，你们的自然语言就变成了一种"元编程语言"。你用它编程的对象不是机器，是 AI 的行为。既然是编程，它就该享受跟代码一样的待遇：版本管理、Code Review、改动留痕。一样都不能少。

---

### 五、规则之间打起来了怎么办

分两种：

**跨层级冲突**（个人级说"4 空格"，项目级说"2 空格"）：有明确的裁决规则——越具体越优先，项目级优先。这不是 bug，是设计。

**同层级冲突**（同一个项目级 CLAUDE.md 内，A 加了"所有函数都要写单元测试"，B 又加了"原型阶段代码先别写测试"）：没有自动裁决，只能靠人。这就是前面说 PR review 的价值——B 提交时 reviewer 会发现冲突，让 B 改清楚，比如改成：

```
packages/server 下的代码必须写单元测试
experiments/ 目录下的原型代码可以不写
```

矛盾被一条更精确、带了路径范围的规则化掉了。

---

### 六、用 .claude/rules/ 目录拆开

团队规则越攒越多，全堆在一个 CLAUDE.md 里，文件臃肿到没人想读的时候，就该拆了。

**什么时候该拆？** 粗略判断线：CLAUDE.md 超过 150-200 行，或者团队里开始有人分管不同子系统（前端、后端、数据）时，就值得拆。5 人以下、就一个模块的小项目，一个文件够用。

**拆法：** 拆进 `.claude/rules/` 目录：
```
.claude/
  rules/
    components.md    # 前端组件规范
    api.md           # API 设计规范
    database.md      # 数据库操作规范
    testing.md       # 测试规范
```

然后在主 CLAUDE.md 里用 `@` 语法引用：
```
@file .claude/rules/api.md
@file .claude/rules/database.md
```

被引用的文件只专注写它那一块。比如 `.claude/rules/database.md`：
```
## 数据库规范
- 所有查询必须使用投影，禁止 select *
- 联表查询超过 3 个 JOIN 必须拆分
- 写操作必须走事务
- 迁移文件必须 reviewer 确认后才能合并
```

**拆完的好处：** 文件短了、好读了、diff 清爽了。更重要的：规则有了明确的主人。谁管 API 层谁维护 `rules/api.md`，前端负责人管 `rules/components.md`。规则的归属跟代码子系统对齐了。

> ⚠️ `@` 引用语法在不同 Claude Code 版本里改过几回。正式落地前先在自己机器的当前版本里实测，确认引用能正确加载。

---

### 七、个人偏好放置区

团队规范管的是"大家一致"。个人的东西有两个去处，别搞混：

**`CLAUDE.local.md`**——你对当前这个项目的私人覆盖。跟团队的 CLAUDE.md 一起躺在项目根目录，但 Claude Code 会自动把它加进 .gitignore。适合放：涉及你本地环境的东西、你负责的特定模块的额外约束。

**`~/.claude/CLAUDE.md`**——你的跨项目身份。那些走到哪都跟着你的偏好。适合放：你用什么语言跟 Claude 交流、沟通风格偏好（喜欢详细还是简洁）、全局通用的代码习惯。

一句话分清：`CLAUDE.local.md` 是"我在这个项目里的偏好"，`~/.claude/CLAUDE.md` 是"我在所有项目里的偏好"。一个跟项目走，一个跟人走。

---

### 八、用 /memory 检查 Claude 行为

团队协同里报得最多的就是："新来那同事，他的 Claude 表现得跟我们都不一样。"

查这个工具就一个。在 Claude Code 里敲：

```
/memory
```

它会列出当前会话实际加载了哪些 CLAUDE.md 文件。让出问题的人和一个正常的老成员各跑一次，两份清单对比。

**一般三种情况：**

1. **新人少加载了项目 CLAUDE.md。** 八成是从项目根目录以外的子目录启动了 Claude Code。让他回根目录重启。

2. **个人级配置跟项目规则打架。** 他的 `~/.claude/CLAUDE.md` 里藏着跟项目规范冲突的指令。让他翻翻全局配置。

3. **子目录规则没生效。** 子目录里的 CLAUDE.md 是按需加载的（Claude 实际读了那个目录的文件才生效）。让它先读一个该目录下的文件再跑一次 `/memory`。这对 monorepo 特别要紧——每个 package 可以有自己规则的子 CLAUDE.md，但它们是"你进了那块地盘才生效"。

**养成习惯：** 任何"Claude 行为诡异"的报障，第一步都先 `/memory`。

---

### 九、让配置文件慢慢生长

一份从来不更新的 CLAUDE.md，会慢慢烂成一张废纸。

代码在变，架构在变，团队踩的坑也在变。一年前的某条规则今天可能早就不适用了，可它还赖在文件里继续把 Claude 往沟里带。

健康的做法是让 CLAUDE.md **跟着事故和决策一起长**。"金额必须用整数分"不是哪天有人拍脑袋想出来的——它是印尼站结算冒出 0.01 误差、一路查到浮点精度、最后才被沉淀进去的。

**CLAUDE.md 的演进节奏：**
- 团队踩了个会反复出现的坑 → 走 PR，把它固化成一条"关键陷阱"
- 确立了一条新约定 → 走 PR 补进去
- 发现某条规则过时了 → 走 PR 删掉它

**把"回顾 CLAUDE.md"塞进固定节奏：** 每月一次，或每个大版本跟着捋一遍。从 Claude Code 里持续榨出价值的团队，从来都不是"配一次就撒手不管"的那种，而是把这份配置当活文档、不停打磨的那种。

---

### 写在最后

一个团队真正的瓶颈常常不在谁的能力，而在那些只长在资深成员脑子里、靠口口相传的隐性知识，压根没被写下来过。新人靠悟，老人靠记，同一个坑换个人再踩一遍。

CLAUDE.md 真正的意义不在"配置一个 AI 工具"这么浅显。它做的是：**把那些隐性的、口头的、飘在空气里的团队约定，变成能版本管理、能 Review、能被一代代继承下去的显性资产。**

一个新人 clone 下你们的仓库，他拿到的就不只是代码了。他拿到的是一个已经配置妥当、忠实反映团队真实做法的 Claude。他上班第一天，就站在了团队全部经验的肩膀上。

---

## 自己的实践

这份 CLAUDE.md 是我翻译完这篇文章后直接更新的。以前我也说「CLAUDE.md 不重要」，现在看这些话句句扎心——确实每次都要重复解释背景。

**建议策略：先加 3-4 条痛点，逐步迭代。** CLAUDE.md 是活的——每次犯错都是完善的机会。几个月后，它会成为你的机构记忆、工作流文档和错误预防系统。那时候 Claude 不是变聪明了，是你的系统进化了。

### 快速开始

> **0xDepressionn 的建议：先加 Karpathy 的 4 条规则。就这 4 条。**
> 粘贴到一个叫 `CLAUDE.md` 的新文件，放在项目根目录。只需要 2 分钟。
> 剩下的规则每周加几条——发现缺什么就补什么。

```
1. 大改前展示完整计划
2. 只修改我指定的文件
3. 每步改动都告诉我具体做了什么
4. 不确定时停下来问，别蒙
```
