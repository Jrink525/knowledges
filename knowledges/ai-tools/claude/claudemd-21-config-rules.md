## 上篇：CLAUDE.md 的底层哲学

> 作者：nainsidwiv50980

大多数开发者认为 Claude 输出不好是因为：
- 提示词不够详细
- 模型不够聪明
- AI "还没到那个地步"

但在观察了大量开发者使用 Claude Code 的方式后，我发现了一个真相：

**优秀输出和平庸输出之间的最大差距，通常跟提示词无关——它来自上下文。更具体地说：CLAUDE.md。**

大多数人用错了这个文件。他们把它当成随便写写的备忘录，甚至直接无视。

而高级用户把 CLAUDE.md 当作**整个 AI 工作流的操作系统**。

同款模型，完全不同的人机体验。

### Claude 为什么感觉"不稳定"

很多开发者不知道：**Claude 每次会话开始时都是上下文盲的。**

它不知道：
- 你的项目结构是怎样的
- 你遵循什么模式
- 做过什么架构决策
- 哪些错误最致命
- 你的工作流期待是什么

除非你明确定义这些，否则 Claude 只能即兴发挥——而不一致性正是从这里开始的。

### CLAUDE.md 的真正目的

大多数人认为 CLAUDE.md 是用来"给指令"的。

这只是表面。

**它的真正目的是：减少决策模糊性。**

好的 AI 工作流不是靠无休止地加提示词构建的，而是靠**消除不确定性**构建的。

最好的 CLAUDE.md 不是试图控制一切——它们只是防止最昂贵的错误反复发生。

这是一个巨大的思维转变：

> **CLAUDE.md 不是愿望清单，而是一个错误预防系统。**

### 为什么大多数 CLAUDE.md 在悄悄失败

#### 错误一：文件太长

更多人写 = 更好输出。事实恰恰相反。

每一行额外内容都在争夺注意力。文件越长：
- 指令遵循越弱
- 上下文质量越低
- 重要规则越被稀释

最高效的设置通常出奇地精简。因为好的系统优先关注**清晰度而非数量**。

#### 错误二：通用指令

像"写干净的代码"、"当资深工程师"、"一步步思考"——听起来有用，但几乎不改变行为。Claude 已经理解这些概念。

真正有价值的是**项目特定的约束**：
- 永远不要修改部署配置
- 始终运行类型检查
- 只使用现有工具库
- 避免触碰无关文件
- 改动架构前先问

这些指令阻止真正的错误。

#### 错误三：无结构

大多数人把工作流偏好、技术栈细节、架构笔记、个人意见、随机提醒全部塞进一个巨型文件。

高级用户有意识地将上下文分离：
- **全局规则** — 适用于所有项目
- **项目规则** — 当前项目特定
- **个人工作流偏好** — 你的习惯

更清晰的层次 = 更清晰的输出。

## 中篇：高质量 CLAUDE.md 的五部分框架

> 作者：nainsidwiv50980

分析大量优秀 Claude 工作流后，几乎每个好的设置都包含这五个方面。

### 1. 命令 Commands

Claude 在理解项目如何实际运行时表现要好得多。

没有命令，它就靠猜——猜就产生不必要的来回沟通。

关键内容：
- 如何运行开发服务器
- 如何测试
- 如何 lint
- 如何做类型检查

仅此一项就能消除几乎所有工作流中的摩擦。

### 2. 架构图 Architecture Map

Claude 不会神奇地理解你的代码库结构。

你需要定义基础：
- 业务逻辑在哪
- API 路由在哪
- UI 组件属于哪
- 状态管理在哪

不需要每个细节，只需要足够阻止糟糕的放置决策。

### 3. 硬规则 Hard Rules

这可能是整个文件中 ROI 最高的部分。

每条规则应该回答一个问题：

> "如果去掉这条，Claude 会犯昂贵的错误吗？"

- 是的 → 保留
- 不是 → 删除

最强规则的特点：
- **具体** — 不是"小心测试"，而是"运行 `npm test` 验证通过后再提交"
- **可执行** — 不是模糊的原则，是可操作的约束
- **项目相关** — 针对当前项目的实际痛点

关于否定规则：**告诉 Claude 不要做什么，和告诉它要做什么同样重要。**

### 4. 工作流行为 Workflow Behavior

这控制 Claude 处理任务的方式——不仅是最终输出，还有过程。

示例：
- 重大改动前先问
- 只做最小限度的修改
- 先解释权衡
- 按逻辑拆分提交
- 改完后验证变更

这些显著提升协作质量。

### 5. 边界 Out-of-Scope Boundaries

这一节被严重低估了。

提升 AI 一致性的最佳方法之一是**清晰定义禁区**：

- 不修改部署配置
- 不碰手动维护的文件
- 避免依赖变更
- 绝不自动修改生产基础设施

边界创造稳定，稳定产生复利。

### 关于 CLAUDE.md 最强大的洞察

> 一个好的 CLAUDE.md 是随着时间**进化**的。

每次 Claude 犯错，你就完善系统。

几个月后，它变成了：
- 机构记忆
- 工作流文档
- 架构指南
- 错误预防
- 运营上下文

这时候 Claude 开始感觉明显更智能——不是因为模型变了，而是因为你的系统进化了。

> 提示词是暂时的，系统会复利。

## 下篇：CLAUDE.md 21 条实战配置

> 作者：Anatoli Kopadze（@AnatoliKopadze）

一份 CLAUDE.md 文件曾冲上 GitHub Trending #1，82000 star，7800 fork。

**大多数人没用过它。用过的人不知道里面该写什么。**

每次打开新的 Claude 会话，它都是零记忆——不知道你的名字、工作、偏好。你花前几分钟重复解释，或者不解释——Claude 给出不适合你的东西。CLAUDE.md 永久解决这个问题。

### 怎么创建 CLAUDE.md（2 分钟）

1. 在项目文件夹创建文件，文件名必须是 `CLAUDE.md`（全大写，无空格）
2. 用任何编辑器打开
3. 把本文中相关指令复制进去——**不需要全部 21 条，先挑 3-4 条最痛点**
4. 保存。Claude Code 每次打开该文件夹的会话时自动读取

---

### 一、沟通方式 How Claude Talks to You（指令 1-4）

#### 1. 砍掉废话 ❌ "Great question!"

Claude 的默认开场：「Great question!」「Of course!」「Certainly!」「Absolutely!」——每天用几个小时时，这些废话不断累积。

**写入 CLAUDE.md：**
> 每次回答直接给答案。不需要开场白、不需要表演热情。我问什么，直接答什么。

#### 2. 行动前先给选项 🎯

Claude 默认选一个方向直接干。你让它改一段文字，它把全文语气都变了。你说重构文档，它改得不符合你的思维方式。

**写入 CLAUDE.md：**
> 重要的任务前，先给 2-3 种方案让我选方向。选完再动手。

#### 3. 不知道就直说 🔍

Claude 在承认不确定之前，会先给一个自信、详细、但完全错误的东西。它用听起来很合理的数据、日期、引用来填补空白。

**写入 CLAUDE.md：**
> 对不确定的内容，在回答前标注「我不确定这个」而不是藏在自信的答案里。

#### 4. 回答长度匹配需求 📏

简单问题写四段，复杂问题给个骨架——两个极端都没用。

**写入 CLAUDE.md：**
> 回答长度与实际需要匹配：短问题短答，需要深度的内容给足细节。

---

### 二、行为方式 How Claude Behaves（指令 5-8）

#### 5. 大改前先问 📋

你说修一段，它重写了整个文档。你说缩短，它删了你需要的部分。

**写入 CLAUDE.md：**
> 任何重大变更前，先停下来告诉我准备做什么，等我确认后再动手。

#### 6. 只改被要求的 🔧

修一个 bug 时顺手「优化」五个其他东西。

**写入 CLAUDE.md：**
> 只改我问的内容。其他东西如果值得改，提一句但别碰。除非我要求。

#### 7. 每次做完告诉我改了什么 📝

改完了但你看不懂到底改了哪——每次都要自己做 diff。

**写入 CLAUDE.md：**
> 每次任务结束时，给我一个简短总结：什么变了、什么没变、有什么需要我注意的。

#### 8. 绝不在没问的情况下替我操作 🚫

AI 工具越来越「能做事」——发消息、发帖、分享文件、安排日程。这些行为有实际后果。

**写入 CLAUDE.md：**
> 任何对外产生影响的动作，绝不允许不经我明确确认就执行。这不是「小心一点」，而是一堵硬墙。

---

### 三、你的上下文 Your Context（指令 9-11）

#### 9. 告诉 Claude 你是谁

Claude 不知道你是专家还是新手、创始人还是自由职业者。没有上下文它就猜——猜对一半错一半。

**写入 CLAUDE.md：**
> 我是 [你的身份]。[技术栈/专业领域]。[沟通风格偏好：喜欢技术深度 / 简单明了 / 等等]。

#### 10. 给出你在做什么的背景

每次会话 Claude 都不知道你在做什么项目、为谁做、什么最重要。

**写入 CLAUDE.md：**
> 我正在做的项目：[简述]。目标用户：[谁]。关键约束：[预算 / 时间 / 技术限制等]。

#### 11. 锁定你的声音和风格

Claude 有默认写作风格。可以，但不是你的。每次用它写东西你都要改回自己的语气。

**写入 CLAUDE.md：**
> 定义你的写作风格：正式/口语化？技术/通俗？简短/详细？爱用什么句式、避免什么用词？写出来，Claude 从此第一稿就是你的声音。

---

### 四、记忆与连续性 Memory & Continuity（指令 12-15）

#### 12. 让 Claude 维护一个 MEMORY.md 文件 🧠

Claude 会话之间完全失忆。但 **Claude 可以写文件，文件会持久存在**。

**写入 CLAUDE.md：**
> 在项目根目录维护一个 MEMORY.md 文件。记录每次的重要决策：做了什么、为什么、考虑了哪些被否定的方案。每次会话开始先读这个文件。

#### 13. 会话结束时写总结 📓

关掉会话，两天后回来。花 15 分钟读旧消息回忆做到哪了。

**写入 CLAUDE.md：**
> 每次会话结束前，把当前状态写入 MEMORY.md：完成了什么、正在做什么、下一步是什么。下次打开直接接上。

#### 14. 记录什么行不通 ❌

试了 4 次终于写出可用的东西。三周后做类似任务，Claude 又从头开始给同样不靠谱的建议。

**写入 CLAUDE.md：**
> 维护一个「没走通」的日志。记录：试了什么、为什么不行、最终什么方案有效。下次遇到类似问题先查日志。

#### 15. 给出永远不变的事实基础 📌

每个项目都有永久不变的事实：过去的决策带来的约束、有重要原因存在的规则、总是成立的前提。

**写入 CLAUDE.md：**
> 一个永久事实块。列出那些不会改变的东西：[技术限制]、[业务规则]、[过去的关键决策] 等。写到文件里，Claude 就不用每次都重新问了。

---

### 五、开发者专项 For Developers（指令 16-21）

以下这 6 条是 **Claude Code** 用户的专属配置——决定了 Claude 是精准的编码助手，还是代码库里的野马。

#### 16. 不越界 🎯

让修一个 bug → 重构三个文件、重命名变量、重新整理 import、顺手「优化」你用了几个月的代码。有些改动会引入微妙的 bug。

**写入 CLAUDE.md：**
> 只改被要求的文件。除非我明确说「也改一下 X」，否则不要碰其他任何文件。

#### 17. 破坏性操作前必须确认 ⚠️

Claude Code 在终端里跑，有文件系统权限。它会删文件、覆盖函数、删表——因为你说了，哪怕你并没完全意识到你说了什么。

**写入 CLAUDE.md：**
> 任何可能破坏已有代码的操作前，停下来，展示具体会改动什么，等我确认再执行。删除、覆写、重构都是。

#### 18. 硬止损 🚧

有些操作不是「小心一点」——是「绝对不允许不经你本人坐在桌前说 Yes 就执行」。

**写入 CLAUDE.md：**
> 以下操作绝对不允许不经明确同意执行：
> - 部署到生产环境
> - 在线上数据库跑迁移
> - 对外部服务发 API 请求
> - 发消息/邮件/社交平台内容

#### 19. 锁定技术栈 🔒

没有固定技术栈，Claude 会推荐它见的最多的框架，不是你用的、不是你团队懂的、不一定兼容你已经搭好的东西。

**写入 CLAUDE.md：**
> 固定技术栈：[语言/框架/包管理器/数据库/基础设施]。Claude 只在这个栈内建议，不要推荐替代品。

#### 20. 永远展示变更摘要

Claude 做完改动后，你不知道改了哪些文件、有没有碰其他东西、有没有留未完成的任务。

**写入 CLAUDE.md：**
> 每次代码改动后，给一个文件级别的变更摘要：改了哪些文件、怎么改的、还有什么是待办的。

#### 21. Karpathy 的 4 条铁律 🏆

Andrej Karpathy（特斯拉前 AI 总监、OpenAI 创始成员）识别出 Claude Code 在编码任务上失败的 4 个关键行为。一个开发者把这 4 条写进 CLAUDE.md，冲到 GitHub Trending #1，**编码准确率从 65% 提升到 94%**。

**写入 CLAUDE.md（开发者必加）：**
```
1. 大改前展示完整计划
2. 只修改我指定的文件
3. 每步改动都告诉我具体做了什么
4. 不确定时停下来问，别蒙
```

## 配置策略建议

| 角色 | 至少该加 | 不要加 |
|------|---------|--------|
| **所有人** | 砍废话 + 匹配长度 + 你是谁 + 永久事实 | — |
| **内容创作者** | 锁定声音 + 大改前问 + 改完说明 | 开发者专属 (16~21) |
| **开发者 (Claude Code)** | 不越界 + 破坏性确认 + Karpathy 四律 | 如果非创作可省略声音定制 |
| **重度使用 (>4h/天)** | MEMORY.md + 会话总结 + 错误日志 | — |

## 自己的实践

这份 CLAUDE.md 是我翻译完这篇文章后直接更新的。以前我也说「CLAUDE.md 不重要」，现在看这些话句句扎心——确实每次都要重复解释背景。

**建议策略：先加 3-4 条痛点，逐步迭代。** CLAUDE.md 是活的——每次犯错都是完善的机会。几个月后，它会成为你的机构记忆、工作流文档和错误预防系统。那时候 Claude 不是变聪明了，是你的系统进化了。
