# Agent Harness Engineering —— AI Agent 的脚手架才是真正的工程

> **来源：** Addy Osmani (@addyosmani) 的 X 长文  
> **原文标题：** Agent Harness Engineering  
> **统计：** 👍 3,208 🔁 455 💬 76 👁️ 78 万+

---

## 核心论点

一个 Coding Agent 是 **模型 + 模型周围的一切**。过去两年行业一直在争论模型——哪个最聪明、哪个写 React 最干净、哪个幻觉最少——但忽略了系统的另一半。

模型只是运行中 Agent 的一个输入。其余是 **Harness（脚手架）**：提示词、工具、上下文策略、钩子、沙箱、子 Agent、反馈回路和恢复路径——包裹着模型才能让它实际完成任务。

> **一个好的模型配上优秀的 Harness，稳定胜过一个优秀模型配上糟糕的 Harness。**

这句话怎么强调都不为过。同样一个顶级模型，放在通用框架中运行和放在高度定制的 Harness 中运行，性能差距巨大。这个差距就是 **Harness Gap（脚手架差距）**。

---

## 什么是 Harness？

### 核心定义

**Agent = 模型 + Harness。如果你不负责模型，你就在负责 Harness。**

Harness 包含每一行不属于模型本身的代码、配置和执行逻辑。一个原始模型不是 Agent。只有当 Harness 为它提供状态、工具执行、反馈回环和可执行约束时，它才成为 Agent。

### Harness 的具体组成

| 组件 | 包括什么 | 为什么重要 |
|------|---------|-----------|
| 🧠 **提示工程** | System prompts, CLAUDE.md, AGENTS.md, skill files | 定义 Agent 的行为基线 |
| 🛠️ **工具生态** | Tools, Skills, MCP Servers, 技术描述 | Agent 的行动能力来源 |
| 📦 **基础设施** | 文件系统、沙箱、无头浏览器 | 执行环境 |
| 🔄 **编排逻辑** | 子 Agent 派生、交接、路由 | 复杂任务的分解与协作 |
| 🪝 **钩子（Hooks）** | lint 检查、上下文压缩、提交前验证 | 确定性执行的质量保障 |
| 📊 **可观测性** | 日志、追踪、成本、延迟 | 了解 Agent 在做什么 |

### Claude Code 的 Harness 架构

Fareed Khan 对 Claude Code 的架构拆解是当前最清晰的成熟 Harness 参考：

```
┌─────────────────────────────────────────┐
│          Claude Code (Harness)          │
├────────────────┬────────────────────────┤
│  Knowledge     │  Memory Store          │
│  (Context       │  + Worktree Isolator  │
│   Injection)   │  (Loop State)          │
├────────────────┼────────────────────────┤
│  Permission    │  Subagent Context      │
│  Gate          │  Firewall (Multi-      │
│  (Destructive  │   Agent Layer)         │
│   Action Hooks)│                        │
├────────────────┴────────────────────────┤
│        Tool Dispatch Registry           │
│   (MCP Servers | Bash | Filesystem...)  │
├─────────────────────────────────────────┤
│               Model                      │
└─────────────────────────────────────────┘
```

Claude Code 的进化方向——**至少和底层模型一样取决于 Harness。**

## "不是模型问题，是配置问题"

当 Agent 做了一些匪夷所思的事时，常见反应是"等下一个模型版本修复"。Harness Engineering 的思维拒绝这个默认选项。

### 故障映射表

| Agent 的失败 | Harness 的修复 |
|-------------|---------------|
| 忽略了团队编码规范 | 加到 AGENTS.md |
| 执行了破坏性命令 | 写一个 Hook 来阻止 |
| 在 40 步任务中迷失 | 拆分为规划器 + 执行器 |
| 经常提交坏代码 | 接入类型检查作为回压信号 |

用 HumanLayer 的话说：**"这不是模型问题，这是配置问题。"**

当今模型理论上能做的和实际看到的之间的差距——**绝大部分是 Harness Gap。**

---

## 棘轮原则：每一次失败都成为一条规则

Harness Engineering 中最重要的习惯：**把 Agent 的错误当作永久信号，而非一次性的运气差。**

### 例子

1. Agent 提交了一个含有被注释掉的测试代码的 PR，还合进去了
2. 下次 AGENTS.md 必须写："永远不要注释掉测试，要么删要么修"
3. 下次 pre-commit hook 应该自动标记 diff 中的 `.skip(`
4. 评审子 Agent 必须更新为拦截被注释掉的测试

**约束只应在你观察到真实失败时添加，只应在模型足够强大不再需要时删除。**

> 一个好的 system prompt 里的每一行，都应该能追溯到某次具体的、历史上的失败。

这就是为什么 Harness Engineering 是一门需要不断积累的学科，而不是一个开箱即用的框架。正确的 Harness 由你代码库**独有的失败历史**塑造而成。

## 从期望行为反向设计 Harness

最有效的 Harness 设计方法：**想要什么行为 → 构建产生该行为的组件。**

Harness 的每个部分必须有明确职责。**如果你说不清某个组件存在的具体行为是什么，就应该删掉它。**

### 六个核心组件

#### 1. 文件系统 & Git —— 持久状态

文件系统是基础。模型只能操作上下文窗口中的内容。文件系统提供工作区来读取数据、卸载中间结果、让多个 Agent 协调。加上 Git 则提供免费版本控制——Agent 可以追踪进度、分支实验、回滚错误。

#### 2. Bash & 代码执行 —— 通用工具

大多数 Agent 运行在 ReAct 循环上：推理 → 通过工具调用行动 → 观察 → 重复。与其预先构建每一个可能用到的工具，不如给 Agent bash 权限，让它即时构建需要的东西。Agent 普遍擅长 shell 命令。

#### 3. 沙箱 & 默认工具链

Bash 只有在安全运行时才有用。沙箱提供隔离环境，预装语言运行时、测试 CLI、无头浏览器——让 Agent 可以观察自己的工作并完成自我验证闭环。

#### 4. 记忆 & 搜索 —— 持续学习

模型对训练权重和当前上下文外的内容一无所知。Harness 通过记忆文件（如 AGENTS.md）桥接这个差距，将知识注入每一次会话。对实时信息（如新库版本），将网页搜索和 MCP 工具直接嵌入 Harness。

#### 5. 对抗上下文腐烂

模型上下文窗口填满时，推理能力下降。Harness 通过三种技术管理这个稀缺资源：

| 技术 | 怎么做 |
|------|-------|
| **压缩（Compaction）** | 智能总结并卸载旧上下文，防止 API 错误 |
| **工具调用卸载** | 把大工具输出（如 2000 行日志）存文件系统，上下文只保留关键首尾 |
| **渐进式披露** | 只在任务明确需要时才透出指令和工具，而不是启动时全加载 |

#### 6. 长周期执行

长时自主运行面临"过早停止"和"问题分解差"两大问题。Harness 通过三种结构设计对抗：

- **循环（Loops）：** 拦截模型尝试退出的行为，强制它在新的上下文窗口中继续向完成目标推进
- **规划（Planning）：** 强制模型将目标分解为逐步的计划文件，每一步通过自我验证钩子检查工作
- **拆分（Splits）：** 将生成和评估分离到不同 Agent，防止模型评价自己工作时固有的偏好偏差

## Hooks：执行保障层

Hooks 弥合了"请求行动"和"强制执行"之间的差距。它们在特定生命周期运行：工具调用前、文件编辑后、commit 前。

> 成功应该无声，失败应该话多。

如果类型检查通过了，Agent 什么也听不到；如果失败了，错误被直接注入循环供自我修正。

---

## 规则文件是飞行员的检查清单

仓库根目录下的一个纯文本 markdown 文件，仍然是**性价比最高的配置点**。但必须像对待**飞行员的检查清单**那样对待它，而不是风格指南。

- 保持简短
- **每一条规则必须是通过一次实际失败挣来的**
- 十个高度聚焦的工具永远胜过五十个功能重叠的工具
- 工具描述会填充到 prompt 中——未经验证的 MCP 服务器可能在做坏事之前就注入了坏 prompt

---

## Harness 不会缩小，只会移动

模型变好并不会消除对 Harness 的需求——**只是让需求转移了。**

看起来更好的模型会让脚手架变得多余——比如最近的模型升级大幅减少了对"上下文焦虑"缓解措施的需求。但地板上升的同时天花板也在上升。以前够不着的任务现在可以做了，带来全新的失败模式。

Harness 的每个组件编码了一个假设：**模型自己无法做什么。** 当模型进步时，过时的脚手架应该被移除，新的脚手架必须被构建以触及下一层地平线。

---

## Harness-as-a-Service（HaaS）

行业正在从"用 LLM API（提供补全）构建"转向"用 Harness API（提供运行时）构建"。

SDK 现在直接提供循环、工具、上下文管理、hooks 和沙箱。不再从头构建编排，现代默认做法是：**选择一个 Harness 框架，配置其核心支柱，只专注于领域特定的提示词和工具设计。**

这让问题排查变得可规模化——你在调整一个结构良好的配置面，而不是重新发明整个 Agent 架构。

---

## 趋势：Harness 模式正在收敛

看看今天顶级的 Coding Agent——**它们彼此之间的相似程度，超过了底层模型的相似程度。**

模型不同，但 Harness 模式正在收敛。行业正在快速识别出将生成文本变成可交付软件所需的负载承载式脚手架。

最激动人心的开放问题已经超越了单 Agent：
- 并行编排多个 Agent
- 让 Agent 分析自己的追踪来修复 Harness 层面的故障
- 构建按需即时组装工具的环境

最终，Harness 不再作为静态配置文件存在——它的角色更像是**编译器**。

---

## 关键 Takeaways

1. **Agent = 模型 + Harness。** 你不在做模型，那你就在做 Harness。
2. **一个好的模型配上好的 Harness > 最好的模型配上差的 Harness。** Harness Gap 是目前最大的优化空间。
3. **每一次 Agent 失败都是永久信号。** 不要等下一个模型版本，现在就加固 Harness。
4. **系统 prompt 的每一行都应能追溯到一次具体的失败。** 约束是挣来的，不是预设的。
5. **从期望行为反向设计 Harness。** 说不清某个组件存在的理由，就删掉它。
6. **Hooks 是执行保障层。** 成功静默，失败话多——把错误直接注入 Agent 的自修正循环。
7. **Harness 不会缩小只会移动。** 模型进步后，过时脚手架应移除，新脚手架应构建。
8. **HaaS 是趋势。** 别从零搭编排，选一个 Harness 框架，专心做你自己的定制层。
9. **翻看自己写过的 AGENTS.md/CLAUDE.md——里面有多少条规则是真的从失败中挣来的？**
