# Agent Complexity Ratchet —— AI 编程时代的测试棘轮机制

> **来源：** Garry Tan 的 X 长文（系列第 7 篇）

## 核心论点

过去一年我用 AI 写了两个开源项目：GStack（让 AI Coding Agent 更智能的框架）和 GBrain（AI Agent 的长期记忆系统）。总共约 **97 万行代码、665 个测试文件**——几乎全部通过 Claude Code 和 Codex 完成（多数时间 15 个 Conductor 会话并行）。

上周我在 72 小时内合入了 **14 个 PR，近 29,000 行新代码**。每个版本测试覆盖率都比上一个更好。

这本应不可能——速度和质量长期是反比关系。但现在你不用选了。

**90% 测试覆盖率就是解锁。而 AI Agent 让达到这个门槛的成本变成了零。**

---

## 系统已经发生了相变

### 以前：软件是脆的

过去 50 年，软件工程的一切都围绕一个理念：**预防错误，因为错误是灾难性的。**

你必须第一次就把代码写对。漏掉一个边缘情况→生产崩溃。写坏一个数据库迁移→丢客户数据。理解和维护一个系统的复杂度上限是：**一个团队能同时装进脑子里的事物数量。**

### 现在：软件是"可塑的"

我说的"可塑"不是指草率，而是一种以前不可能存在的韧性。

AI Coding Agent 可以读代码、理解上下文、诊断错误、写修复。不完美，但足够好到**改变软件的缺陷模型**：

- 迁移崩了？Agent 读错误信息，理解 45 个版本的数据库 schema 历史，写修复、写测试。
- 文件同步在百万个符号链接上挂死？Agent 诊断解析器超时，把边界设在 30 秒，带测试一起发。
- 提取管线的归因有 bug？跨模型评估捕获它，Prompt 被迭代，数据库层增加强制约束。

大多数代码级错误——逻辑 bug、解析失败、边缘情况——**Agent 可以在下一轮次诊断并修复。** 剩下的真正灾难性错误是那些破坏状态的错误（生产数据错误迁移、隐私泄露等），但这不影响一个事实：**代码库中绝大部分错误现在是可修复的。**

## Agent Complexity Ratchet（复杂度棘轮）

棘轮是一种只允许单向运动的机制。**在 AI 编程里，就是"质量只升不降"。**

每次 AI Agent 的编码会话都往代码库中添加三样东西：

| 组件 | 作用 |
|------|------|
| ✅ **测试** | 编码 "正确" 的定义，每次变更自动运行，失败时大声通报 |
| 📝 **文档** | 记录决策原因——不只是"做什么"，还有"为什么"和"权衡是什么" |
| 📊 **评估结果** | 设定质量阈值的结构化评估——有分数，所以你知道下一个版本是变好了还是变差了 |

下次 Agent 再次处理代码库时，它会把这三样全部加载到上下文窗口中。测试决定了它不能低于某个质量——因为测试会失败。文档就在上下文里——不能忽略它。评分有记录——低于基线的版本不会发出。

**质量地板随着每次操作只升不降。那就是棘轮。**

### 实例：GBrain 的"持有者混淆"

GBrain 有一个功能叫做认识论提取（epistemological extraction）：读取数千页内容，识别谁相信什么、置信度多少、随时间怎么变化。

第一轮提取了 100,720 条主张。跨模型评估（GPT-5.5 + Claude 独立打分）给出 **6.8/10**。

最大问题：**Holder Confusion（持有者混淆）**。比如"AI 将在 2027 年取代 80% 的软件工程师"——这个主张是原作者写的？是引用的另一个人？还是分析引擎从播客转录中推断的？V1 版本在 35% 的情况下搞错这个区别。

So：
1. 评估结果被记录下来
2. 六个具体的失败模式被识别出来
3. V2 Prompt 覆盖了全部六种
4. 权重四舍五入（置信度）在数据库层强制执行——不允许虚假精度
5. 17 个测试锁定了契约

从此任何未来版本的提取都不能在不通过这 17 个测试的情况下发布。没人需要记住为什么权重舍入重要、什么是持有者混淆。**测试在替你记住。**

**质量地板永久性提高了。这就是棘轮的一圈。**

## 为什么大部分 Vibecoded 项目会死

"Andrej Karpathy 的 Vibecoding"（用自然语言描述需求，让模型生成代码）很强大，我也在用。但据我观察 YC 申请项目和开源仓库——**跳过测试的 Vibecoded 项目一旦达到几千行代码、几个交互功能，就开始崩溃。**

它们跳过了棘轮。没有测试、没有文档、没有评估。Agent 不断添加复杂度，但没有任何东西阻止回归。每个新功能都可能破坏旧功能。没有测试，你要等到用户报告才发现。到 v0.5 代码库变成了鬼屋——每个改动都打破一些意想不到的东西。然后开发者写博客说"AI 编程不行"。

> **AI 编程完全可以。他们只是没建棘轮。**

没有测试，改进就是个有噪音的过程：Agent 试图把事情变好，但没有回归信号，好变更和坏变更同样不可见。稠密测试套件意味着在测试覆盖的表面区域上，质量只会上升。

## 什么是"一切可 harness 即可 test"

棘轮不仅适用于传统代码。它适用于计算机能观察到的一切：

| 层面 | 可测试内容 |
|------|-----------|
| **OS 级** | 进程树、文件系统状态、网络 socket、cron 调度 |
| **终端级** | 每个按键、每行输出、每个交互提示 |
| **浏览器级** | 渲染的页面、按钮状态、导航事件 |
| **API 级** | 结构化响应解析和校验 |
| **行为级** | Agent 说了什么、调了什么工具、顺序、是否先询问再行动 |

这些都是可 harness 的。可 harness 就能观察。可观察就能做断言。能做断言就能上棘轮。

### 实例：GStack 的 TTY 级行为测试

GStack 有一个「交互式计划审查」功能：要求 Agent 逐节审查你的架构，提问、探询边缘情况、挑战你的假设——就像一个真正读代码的工程经理。

问题是 Claude Code 有时跳过所有交互部分：读计划文件、一次性 dump 所有发现、退出——一个也没问用户。整个功能的核心是来回对话，跳过就废了。

你怎么测试"AI 有没有进行对话"？传统测试框架覆盖不到。

我用 Bun 的 TTY 功能构建了一个测试 harness（PR #1354）：在伪终端中生成 Claude Code，提供特定场景，触发审查技能，实时监控终端输出。测试观察 Agent 是否在完成前发出交互式问题。如果它 dump 发现后直接退出——测试失败。

**这不是在测试代码。这是在测试 AI Agent 是否遵循行为契约。在 TTY 级别。看着它工作。**

棘轮的响应是三层：

1. **Skill 指令中的 STOP 门**——显式规则："你在进入下一节之前必须询问用户"，附反合理化条款（命名具体失败模式）
2. **Anti-shortcut 子句**——"计划文件是交互式审查的 OUTPUT，不是替代品。"一句话封杀掉漏洞
3. **Gate 级地板测试**——TTY harness 在受控场景中生成 Claude Code，失败条件是 Agent 未提出至少一个交互问题

另一个例子 PR #880：测试不仅仅是检查代码编译。它构建开源插件 → 生成真实 OpenClaw 实例 → 通过 CLI 安装插件 → 运行 `plugins inspect` 验证运行时已加载 → 设置 config → 验证 config → 运行 `plugins doctor` 确认零诊断。两个独立程序之间的完整端到端链路。359 行测试代码。人类几乎永远不会手动写这种测试——设置太繁琐了。**Claude 花了大约 5 分钟写出来。这就是努力墙实时消失的证明。**

## 90% 覆盖率意味着什么

**Capers Jones** 研究了超过 10,000 个软件项目，测量了缺陷移除效率（DRE）：

| 覆盖率 | DRE（缺陷捕获率） |
|--------|-----------------|
| < 70% | ~65-75% |
| 85-95% | **92-97%** |
| MC/DC (Level A航空标准) | >99% |

曲线不是线性的。**在大约 85% 处有一个拐点**，缺陷逃逸急剧下降。

然而 **Mockus, Nagappan, Dinh-Trong** 对 Windows Vista 的研究也显示：达到 90%+ 需要的努力急剧上升。最后 20% 的覆盖比前 70% 花费不成比例的工作。这是几十年的老问题，也是大多数团队停在 70-80% 的原因。

**但 AI Coding Agent 不感知"努力"。**

它们不会在周五下午 5 点觉得写第 14 个边缘情况测试很无聊。它们不会看一个复杂的集成测试然后想"我等会回来看"。让人类停在 70% 的"最后 20% 的惊人工作量"——正是 AI Agent 最擅长做的工作。

> **到达 90% 曾经需要英雄般的努力。现在它是周二的事。这就是 game change。**

### 棘轮不等于虚荣指标

重要的是：棘轮说的不是行覆盖率的虚荣指标。它说的是编码行为契约的测试——持有者混淆测试、权重舍入测试、交互审查门——每个测试锁住一个具体的经验教训。

90% 意味着几乎所有行为变更都会触发测试信号。Agent 要么通过（可以发版），要么打破一个测试（立即发现）。

剩下的 10% 是集成点、基础设施管道和真正难测试的边缘情况。这可以。**90% 就是把混乱变成棘轮的关键。**

## 软件复杂度上限被大幅提高了

过去：上限是一个团队能把整个系统装进脑子里的人数。
现在：上限是**一个人 + 能加载完整代码库、schema 历史、测试套件和文档的 Agent**。

这个数字大得多。而且随着上下文窗口变大、模型推理代码能力变强，它还在增长。

**不采用这个模式（Agent + 人的 Taste + 只升不降的测试套件）的团队，已经在以比"一个拥有这些手段的独立开发者"更低的质量和更慢的速度发版了。**

## 关键 Takeaways

1. **棘轮三要素：测试 + 文档 + 评估基线**。每次 Agent 操作都完善这三者的合集。
2. **90% 覆盖率是魔法阈值**——对此处缺陷逃逸率曲线是非线性的。而 Agent 让达到这个阈值不再需要"英雄般的努力"。
3. **一切可 harness 即可 test。** TTY 测试、OS 级断言、行为契约——测试面远大于传统单元测试。
4. **测试是新团队的制度记忆。** 员工离职带不走。模型换版本不会丢失。凌晨 2 点你判断力下降时仍然有效。
5. **无测试的 Vibecoding 有硬天花板。** 几千行代码不建棘轮，最终变成鬼屋。不是 AI 不行，是没建棘轮。
6. **最精妙的技巧是 STOP gate + anti-shortcut clause + 命名失败模式**——不只是告诉 Agent "不要" 怎么做，而是给它能理解的、指向真实历史失败的具体约束。
