# Claude Code 记忆系统深度解析：五层架构与 Auto-Memory 机制

> **一句话总结**：Claude Code 没有单一的记忆系统，而是五层分层架构，每一层的目标、成本和消费者都不同。本文聚焦 Auto-Memory 层，揭秘其写入、检索、归档和安全设计的全部细节。

## 一、为什么要关注记忆系统？

记忆是 AI Agent 身份和性能的关键组成部分。

作者的工作背景：在 [@CamelAIOrg](https://x.com/CamelAIOrg) 从事大量与记忆相关的工作，因此对 Claude Code 的记忆实现特别关注——它与其他 Agent 框架的记忆方案是否有本质差异？

> **前置阅读**：@himanshustwts 写了一篇关于 Claude Code 记忆系统整体架构的文章，本文在此基础上深入到实现细节。

## 二、Claude 的五层记忆架构

Claude 没有单一的记忆系统——它有**五个**，各自服务于不同的目的、成本和终端用户。

### 第 1 层：对话历史（Conversation History）

- 基础层，本质是一个 jsonl 文件，记录用户提示和 Claude 的响应
- 可通 `/export` 命令导出
- **上限**：每个项目 100 条
- 大段粘贴的内容会被哈希后单独存储，保持日志精简
- **成本：零**

### 第 2 层：会话记忆（Session Memory）

- 结构化的 Markdown 记忆，由后台 Agent **在会话进行中**持续构建和更新
- 最终消费者不是用户，而是**压缩策略的一环**
- 当前机制：上下文窗口达到约 85% 时触发摘要。有了会话记忆后，这一过程将变成持续性的
- **⚠️ 状态**：受 `tengu_session_memory` 标志控制——不确定是否已发布功能，但在代码中已存在

### 第 3 层：CLAUDE.md（用户控制层）

- 用户手动控制的编程风格、偏好、需要记住的事项
- 启动会话时，`CLAUDE.md` 的所有内容立即注入
- 支持层级发现：Claude 会遍历目录层次结构自动加载多个 `CLAUDE.md`
- 等价于 Codex 的 AGENTS.md

### 第 4 层：Auto-Memory（自动记忆）⭐ 核心

- 这就是通常所说的"Claude Code 的记忆"
- 从多个会话中**自动学习**用户的角色、偏好、项目上下文、外部系统指针
- 存储位置：`~/.claude/projects/<slug>/memory/`

### 第 5 层：团队记忆（Team Memory）

- 用于团队协作场景，作者认为比较基础，不做深入探讨

> **本文聚焦 Auto-Memory**——所有记忆模块的用户侧都是本地且可见的。

## 三、Auto-Memory 的写入机制

### 3.1 总体流程

Claude 的 Auto-Memory 不是简单追加，而是经过**三个管理阶段**的精细系统：

1. **每次交互后提取**（Per-turn extraction）
2. **周期归档**（AutoDream）
3. **手动删除**（非自动）

### 3.2 阶段一：每次交互后提取

每次对话轮次结束后，一个后台 Agent 会：

1. 扫描最后 N 条消息
2. 判断哪些内容值得记住
3. 决定是**新建**记忆文件还是**更新**已有文件

**关键行为：**
- 执行提取前，Agent 会先读取所有已有记忆，确保**更新而非重复创建**
- 新主题文件创建时，会同时在 `MEMORY.md` 中记录索引，相当于目录 + 一行摘要
- 这个索引结构使得 Agent 后续能快速定位到正确的记忆文件

### 3.3 四种记忆类型

Claude 的记忆分为四种固定类型，作为检索时的标签：

| 类型 | 含义 | 示例内容 |
|------|------|---------|
| `user` | 你是谁 | 角色、专长、偏好 |
| `feedback` | 你希望 Claude 如何工作 | 纠正和确认过的方法 |
| `project` | 代码/Git 无法告诉你的项目信息 | 截止日期、决策、动机 |
| `reference` | 仓库外的查找位置 | Linear 看板、Slack 频道、仪表盘 |

### 3.4 去重机制

如果用户直接对 Claude 说"记住我们用 bun 而不是 npm"，Claude 在对话中**直接写入**了记忆，那么后台的提取 Agent 会检测到本次会话已有记忆写入（`hasMemoryWritesSince()`），直接跳过本轮提取。

**主 Agent 和后台提取 Agent 在每轮窗口中是互斥的**——它们永远不会双写。

## 四、Forked Subagent 模式

### 为什么用户不感知延迟？

提取 Agent 以 **Forked Subagent** 方式运行，与父会话**共享 Prompt Cache**。系统提示词前缀在父进程和衍生进程之间共用，因此大部分输入 token 是缓存命中。用户**永远不会等待**记忆操作完成。

### 沙箱限制

衍生子 Agent 被严格沙箱化：

| 权限 | 状态 |
|------|:----:|
| 读文件系统 | ✅ 可以 |
| 写记忆路径 | ✅ 仅限 `isAutoMemPath()` 校验通过的路径 |
| MCP 工具 | ❌ 禁止 |
| Agent 工具 | ❌ 禁止 |
| 写 Bash | ❌ 禁止 |
| 单次任务上限 | **5 轮**（防止钻牛角尖） |

### 系统提示约束

提取 Agent 的提示中明确写着：

> **"不要浪费时间验证内容。不要 grep 源文件，不要阅读代码。"**

## 五、AutoDream 定期归档

### 触发条件

这个功能叫 **autoDream**，受 `tengu_onyx_plover` 标志控制。触发器：

- **时间条件**：距离上次归档 ≥ 24 小时
- **会话条件**：距离上次归档 ≥ 5 个会话

**两者都满足**时触发。

### 四阶段处理流程

1. **读取索引**：读取 `MEMORY.md` 并浏览现有主题文件
2. **收集信号**：从日报和会话记录中获取近期信号（通过窄 grep，不会全量读取）
3. **合并信息**：
   - 将新信息合并到已有文件
   - 将相对日期转为**绝对日期**
   - 删除与当前代码库**矛盾**的事实
4. **修剪优化**：
   - 移除过时的指针
   - 缩短冗长的条目
   - **解决文件间的矛盾**
   - 提示明确："如果两个文件不一致，修正错误的那一个"

### 并发控制

AutoDream 也是 Forked Subagent 运行，但使用**锁文件**防止多个会话同时触发归档。

## 六、记忆的检索机制

### 6.1 不是全量加载

Claude 不会把所有记忆文件加载到上下文——那会本末倒置。

- `MEMORY.md`（索引文件，上限 200 行 / 25KB）**始终**加载到系统提示中
- 单个记忆文件**按需加载**，并非全部引入

### 6.2 Sonnet 作为相关性过滤器

系统使用 **Sonnet**（即使主模型是 Opus）作为快速相关性过滤器：

```
用户发起请求 →
  1. 扫描所有记忆文件 frontmatter（最多 200 个，最新优先）
  2. 格式化为清单：[type] filename (timestamp): description
  3. 清单 + 用户查询 发送给 Sonnet
  4. Sonnet 返回最相关的 5 个文件名
  5. 仅这 5 个文件加载到上下文
```

这就是为什么 `frontmatter` 中的 `description` 字段**极其重要**——它是 Sonnet 判断相关性时唯一能看到的内容。

### 6.3 选择器的两个关键行为

1. **不确定就不选**：提示明确要求"如果不确定一个记忆是否有用，就不要包含它"，减少误报
2. **排除已有信息**：对 Claude 已在使用的工具，选择性过滤其文档，只取**警告和陷阱**
3. **去重**：前几轮已经展示过的文件不再纳入，将 5 个插槽分配给真正有价值的新记忆

### 6.4 旧记忆的"怀疑注入"

对于**超过 1 天**的记忆，Claude 会在内容旁注入显式警告：

> **"这条记忆已有 47 天。记忆是时间点的观察结果，非实时状态。关于代码行为或文件:行号引用的信息可能已过时。在引用前请对照当前代码核实。"**

系统提示中还有专门的"基于记忆推荐前的检查"段落：
- 如果记忆提到某个文件路径，先检查文件是否存在
- 如果提到某个函数，先 grep 确认
- 如果用户即将按推荐行动，先做验证

## 七、记忆路径安全设计

### 7.1 配置层级限制

`autoMemoryDirectory` 设置允许自定义记忆存储路径，但**只能在全局设置** `~/.claude/settings.json` 中配置，不能在项目级设置 `.claude/settings.json` 中配置。

**原因**：如果恶意仓库将路径指向敏感目录（如 `~/.ssh`），将获得无提示的写入权限。

### 7.2 路径验证

路径会校验以下攻击向量：
- 目录遍历攻击（`..` 段）
- 根路径
- 空字节注入

### 7.3 提取 Agent 的双重保险

即使路径验证被绕过，提取 Agent 的沙箱还会再次拦截——它只能写入通过 `isAutoMemPath()` 校验的路径，其他写入全部拒绝。

## 八、总结：约束比模型本身更重要

剖析 Claude Code 记忆系统后，最值得关注的不是 Opus 4.6 的能力，而是**框架的约束设计**：

| 控制点 | 具体约束 |
|--------|---------|
| 记忆格式 | 强制 YAML frontmatter、一行索引条目、四种固定类型 |
| 检索方式 | 由更便宜、**主模型无法控制**的 Sonnet 处理 |
| 删除机制 | **无自动触发器**，只能由 Agent 判断后执行 |
| 过时处理 | 由框架注入"怀疑警告"，强制标注时间戳 |
| 写入沙箱 | 提取 Agent 只能写特定路径 |
| 去重 | 硬编码到提示中的逻辑 |
| 轮次上限 | 提取任务最多 5 轮 |

> **模型再强大，框架也不信任它能无监督管理自己的记忆。每一步都有约束。**

这种设计哲学值得所有构建 Agent 系统的开发者借鉴：**不要信任模型会自己"做得对"，用架构约束来保证正确的行为。**
