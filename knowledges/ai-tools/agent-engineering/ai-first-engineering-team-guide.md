# AI-First 工程团队构建指南

> **核心论点：** 将 AI"附加"到现有流程只能带来 10-20% 的效率提升。真正的 AI-First 团队，是围绕 AI 重新设计工程流程、架构和组织形态，让 AI 负责"构建"，工程师负责"判断"。

---

## 一、瓶颈已转移，流程必须重写

### 论据

传统工程管理的所有方法论都建立在一个隐性假设之上：**写代码很贵**。从瀑布到敏捷，从需求文档到代码评审，整个行业用了几十年时间设计了一套流程来最优化分配"写代码"这份稀缺资源。

这个假设已经失效了。

现在的瓶颈链条变为：**产品定义 → 设计探索 → 代码（已解）→ 验证测试 → 发布反馈 → 团队资源**。其中最棘手的是验证、评审、跨职能协作和安全性——这些环节还没有被 AI 充分渗透。

### 动作为主

- **诊断你的瓶颈：** 如果你的团队每天花最多时间在"等评审"、"等测试"、"等排期"上，说明流程还停留在旧时代。瓶颈在哪，资源就往哪调。
- **追问率最高的那条工作流：** 找出团队里最折腾人、最啰嗦的流程，问清楚"它到底还在为谁服务"。如果答案模糊，直接砍掉。流程不会自然消亡，只会不断叠加，需要主动清理。
- **指标看结果而非 AI 占比：** 用"新人上手时间"、"PR 生命周期"、"发布频率"来衡量变革效果，而不是"代码百分之多少是 AI 写的"。后者是虚荣指标，前者才是结果导向。

### 延伸：AI 速度悖论（AI Velocity Paradox）

Harness 联合创始人兼 CEO Jyoti Bansal 提出了一个极具穿透力的观察——**AI 速度悖论**，恰好解释了为什么瓶颈转移如此关键：

```
内循环（写代码）：    10x 加速  🔥
外循环（部署+运维）：  原地踏步  🐢
```

AI 编码工具让写代码这个**内循环**提速了 10 倍，但代码写完后的**外循环**——测试、安全检查、部署、合规审批、云成本管理——仍然严重依赖手工流程和断开的工具链。

> 当系统只有一部分变快时，你不会得到更快的软件交付——你只会得到一个越来越大的积压队列，等着被 review、部署、安全审查和管理。

**关键数据：** 大多数工程团队把 **60-70% 的时间和精力花在代码写完之后**——部署、安全、修 incident、管理基础设施、控制云成本。AI 编码工具不会减少这部分负担。实际上，随着代码生成加速，外循环面临的压力只会更大。

**对团队管理的启示：** 如果你只优化了写代码的速度，却没有同步优化外循环，你的团队会陷入"更快的输入，更堵的输出"的困境。瓶颈诊断不能只看开发生命周期，还要看代码被"交出去"之后发生了什么。

## 二、两种路线：AI-Assisted vs AI-First

### 论点

常见的做法是把 AI 当成现有流水线的"加速器"——工程师打开 Cursor，PM 用 ChatGPT 起草规范，QA 用 AI 生成测试脚本。工作流程本身纹丝不动，效率提升停留在 10-20%。

这是 AI-Assisted，不是 AI-First。

AI-First 的逻辑是反过来：**让 AI 做构建的主体，人做判断的主体。** 工程师的主要产出不再是代码量，而是给 Agent 补充读写能力、定义验收标准、评估决策质量。管理成本下降，争吵减少——同时不确定性上升。

### 动作建议

- **重构流程而非叠加工具：** 不要问"AI 如何帮助我们的工程师"，要问"我们如何重新构建一切，以便 AI 负责构建，工程师提供指导和判断"。这两个问题指向完全不同的组织变革路径。
- **明确授权砍流程：** 站会、周报、六个月路线图、马拉松式代码评审——如果它们存在的理由只是"一直都这么做的"，那正是第一批该砍的。Anthropic 的做法是先明确告诉大家：可以砍。

## 三、发布节奏与反馈闭环决定了组织形态

### 论点

AI-First 团队的发布节奏应该是：**早上一个版本，A/B Test，跟进数据，晚上再发一个版本。** 一天内完成一次从构想到验证的循环。这是 JIT Planning（即时规划）的自然结果——原型成本趋零后，"提前规划"的杠杆消失了。

这种节奏对组织形态有直接影响：

**扁平不是风格问题，是效率问题。** 当一个 team mission 变化时，每多一层级就多一层对齐损耗。Anthropic 的两条产品线共用同一个团队目标，不让子团队各自定独立 mission，就是为了保证当方向调整时，组织能像一体一样快速转向。

**经理必须从 IC 做起。** 这不是姿态问题——在 AI 时代，经理只有亲身写过代码、用过工具，才能理解什么该交给 AI、什么该保留给人做判断。Fiona 自己上一次生产环境提交是 2017 年，加入 Anthropic 后才重新写起代码："现在连 git 命令都不记得了，全靠 Claude 帮我搞定。"她把这当作重新理解工程本质的必经之路。

### 动作建议

- **尝试 JIT Planning：** 用一个 sprint 试验"不写六个月的 roadmap，只规划接下来一周最该做的事"。看看你的团队是否因为失去"提前规划"的杠杆而变差——很可能不会。
- **评估你的组织层级：** 如果一次方向调整需要超过一周才能传达到所有工程师，你的层级太多了。
- **工程师经理先写代码：** 如果你的技术经理超过半年没有提交过代码，让他们回到一线用 AI 工具做一次完整的功能开发。

## 四、代码评审的新分工

### 论点

当代码量因为 AI 辅助而暴增时，人工评审承受的压力不可持续。需要重新划分"什么交给 AI，什么保留给人"：

**AI 可以承担** 风格检查、lint 去重、常规 bug 检测、单元测试补全、安全性漏洞扫描、依赖供应链风险分析。这些是模式识别工作，AI 做得又快又稳。

**人工必须保留** 法律合规审核、安全敏感代码的边界确认、产品体验的 sense 和品味。最后这一点尤其微妙——Fiona 分享过一个例子：她让 Claude 把终端界面变成雪人主题，结果画出来像 Mr. Peanut（Planters 零食品牌的吉祥物）。设计师一句话就点出了问题，但这条判断很难写进自动化规则。

### 动作建议

- **让 AI 做"预审"，人做"终审"：** 所有 PR 先经过 AI 层扫描，打上分类标签（质量/安全/依赖），人只需要关注标记为"高风险"的改动。不在逐行正确性检查上消耗人工精力。
- **品味不能外包：** 需要产品直觉的判断——UI 布局是否自然、文案措辞是否准确、交互是否流畅——始终留给有产品 sense 的人。这是 AI 当前最难跨越的门槛。
- **技术辩论用 PR 代替白板：** 有架构分歧时，让 AI 同时生成多个版本的原型 PR，直接在代码层面讨论影响范围。写代码的成本已经降下来了，争论的成本反而显得更高。但注意——要明确规则防止"谁最后一个 commit 谁赢"的坏风气。

## 五、流水线重构：架构曝光给 AI

### 论点

AI-First 的工程流水线和传统的最大区别在于：**AI Agent 必须能端到端地感知和操作整个系统。** 它不能只是写代码的工具，还要能看到架构全景、调用 Infra、触发 CI/CD、分析日志。

这意味着几件事：

- **统一架构和代码库**，让 Agent 一次看到所有信息，不需要在多个 repo 之间跳转拼接上下文。
- **基础设施通过 API 暴露**，自动扩展、回滚、监控指标都变成 Agent 的可调用工具。
- **CI/CD Pipeline 确定化并暴露给 Agent**，让 Agent 可以自主触发验证、发布、灰度。

### 动作建议

- **检查你的 CI/CD 对 Agent 的可访问性：** 你的 Agent 能否自动触发一条完整的构建-测试-部署管道？如果不能，这就是第一个要重构的环节。
- **统一代码库：** 如果团队有多个 repo 且相互依赖，考虑逐步收敛。Agent 的上下文窗口是有限的，不要让架构碎片消耗它。

### 延伸：实现"架构曝光"的工程挑战

Harness（估值 55 亿美元的 AI 软件交付平台）的架构选择揭示了一个核心痛点：**现代软件交付的信息散落在几十个独立工具中**。部署流水线、云基础设施、服务依赖、安全策略、构建系统——所有数据互相影响，但 AI Agent 几乎不可能完整理解工程环境里正在发生什么。

Harness 的解法是 **Software Delivery Knowledge Graph（软件交付知识图谱）**——一个连接所有工程数据的集中知识层，让 AI 系统对"软件是怎么被构建、部署、安全防护和管理的"有完整的认知。

**最佳实例：2026 年 4 月发布的 Cursor 集成。** 开发者可以留在 Cursor 界面内，用自然语言触发部署、运行流水线、检查治理规则。后台连接到一个由 Harness 托管的 MCP 服务器，由 Knowledge Graph 驱动，且沿用现有 RBAC 权限——无需为 AI 工作流重建访问控制。

> Cursor 理解**内循环**（代码在做什么）。Harness 理解**外循环**（代码如何被测试、安全防护、部署和管理）。插件连接了两者。这就是**企业级规模下的上下文工程**。

**对团队的启示：**
- 不要只关注"给 Agent 多少代码上下文"，更要关注"给 Agent 多少系统上下文"——CI/CD 状态、依赖图、基础设施拓扑、安全策略
- MCP 协议（Model Context Protocol）或许是暴露工程数据给 Agent 的最优路径——它天然适合做 Knowledge Graph 的访问层
- 采用现有权限体系绑定 Agent 操作，而不是为 AI 另建一套安全层——前者安全、可控、可审计

### 企业正在验证

这不是纸上谈兵。2026 年的几个关键信号：
- **Workday** 选定 Harness 支持企业级 Agentic AI 软件交付
- **Infosys** 战略合作，自动化从代码到生产的整个多云路径
- **Google Cloud** Technology Partner of the Year（Application Development & DevOps）
- 客户包括 **United Airlines、Morningstar、Choice Hotels**——不是 POC，是真实生产环境

这些案例说明："架构曝光给 AI"已经从工程实践上升为企业战略。如果你的平台还没有考虑如何将工程全貌暴露给 Agent，下一轮竞争中就可能掉队。

## 六、自我修复——工程组织的新基石

### 论点

AI-First 团队最终应该追求的不是"更快地让人修 bug"，而是**让系统自己发现、分类、修复和验证问题**，把人的介入压缩到最小的审核环节。

这个闭环的工作方式：

> 错误发生 → AI 分析错误类型和内容，生成分类摘要（含日志、受影响用户、建议排查路径） → 自动创建工单 → 去重（比对已有工单，已关闭的重新出现则自动重开） → Agent 评估严重性和优先级 → 开发人员（结合 AI 诊断）推送修复 → CI/CD 重新走一遍 → 排查引擎验证 → 已解决则自动关闭工单 → 回到起点

注意这个循环里**人只在两个点介入**：确认修复方案的正确性（风险审查）和对长期反复出现的问题做根因分析。日常的 triage、分类、建议路径、回归验证全部由 Agent 完成。

### 动作建议

- **从错误分类开始搭建：** 不一定一开始就做全自动闭环。可以先让 Agent 每天早上自动分析昨夜错误日志，生成分类摘要和工单——这一步已经能大幅减少 oncall 负担。
- **去重是早期高回报环节：** 同类型错误反复出现是常见问题。让 Agent 自动比对新增问题与已有工单，认出老朋友可以直接挂回已有工单，不需要人工再 triage 一遍。
- **每个工具只做一件事：** CI/CD 管道的每个阶段只做一件事——检测、分类、修复、验证——像工厂流水线一样链接起来。这样可以单独升级每个环节而不影响整体。

## 七、角色重构：架构师定义规则，操作员做判断，AI 做执行

### 论点

AI-First 团队的工程师角色正在分化：

**架构师** 制定「标准操作程序」——教 AI 如何工作。决定架构与系统边界，建立测试基础设施和分类系统，定义什么是"好"的代码。他们的核心产出是决策规则和系统设计，而非代码行数。

**操作员** 接受 AI 分派的工单，进行调查、验证并批准修复。AI 生成代码 → 人审查风险边界。这类工作仍然需要技能和判断力，但不再需要旧时代要求的架构推理能力。

**产品经理** 的决策周期必须大幅缩短。Anthropic 的实践是让 PM 也直接提交代码，用 PR 替代设计文档和评审会，让产品决策发生在代码变更中而非文档里。

### 招聘重点变化

不再看重原始编码吞吐量——模型已把这条线拉平了。现在重点看两类人：

1. **有产品感觉的建造者：** 好奇心强，看到问题就想做产品解决，会反复打磨体验。
2. **深度系统专家：** 能在分布式系统、安全、基础设施等硬核领域做判断的人——这些领域 AI 还远不能独立处理。

### 动作建议

- **重新定义团队的工种描述：** 把 JD 里的"精通 Java/Go"权重降低，把"能评估 AI 输出质量"、"能定义什么是好代码"、"能做产品层面的判断"的权重提高。
- **非技术成员学写代码：** 设计师和 PM 能用 AI 工具提交 PR——这个能力会让整个团队的瓶颈从"跨职能沟通"变成"自动化的执行"。
- **招聘时问判断类问题：** 不再问"这个算法怎么写"，而是给一段 AI 生成的代码，问候选人"哪里有问题？改进方向是什么？"

## 八、还没答案的问题

三个真实存在的未解难题：

1. **跨平台团队结构还有必要吗？** 当工程师能借助 AI 在 iOS、Android、Web 之间自由切换，传统的按平台分队还有没有意义？
2. **自动化评审的边界在哪？** "信任但验证"的分界线会随模型升级不断移动。不是一次定下来的决定，而是需要持续校准的动态平衡。
3. **角色模糊后，怎样让每个人都感觉有产出感？** 当工程师做内容、PM 写代码、设计师修 bug，传统的产出归属变得模糊。公平感的设计是一个新课题。

这些问题还没有标准答案——但能意识到它们的存在，本身已经是组织成熟度的体现。

## 九、AI 模型不是制胜关键，模型周围的一切才是

### AI 浪潮的规律

每一次 AI 浪潮都遵循同一个模式：

1. **初期：** 所有人盯着最显眼的部分——哪个模型更好？哪个 benchmark 更高？
2. **中期：** 关注点开始转移——从"选哪个模型"到"模型怎么用"
3. **后期：** 人们意识到——**真正的价值来自模型周围的"基础设施层"**：上下文、安全、治理、可观测性、合规、成本控制

> 到 2026 年，大多数前沿 AI 模型已经足够好用了。选择 OpenAI 还是 Anthropic 已经不再是最大的挑战。
>
> **更大的挑战是模型周围的一切。**

### 这对 AI-First 团队意味着什么

- **选模型不再是战略决策：** 模型会持续迭代、快速商品化。绑定特定模型是危险的。让团队的核心能力围绕"如何用好模型"而非"用哪个模型"来建设。
- **上下文工程 > Prompt 工程：** 给 AI Agent 足够的工程上下文（架构、依赖、安全策略、部署状态）比精心调教 prompt 更关键。Harness 的 Knowledge Graph + MCP 就是这条思路的实现。
- **基础设施壁垒才是真壁垒：** 团队之间的差距不在谁能写出更好的 prompt，而在谁建了更完善的 AI 基础设施——数据接入、权限管理、可观测性、自动化外循环。

### 战略建议

> 2026 年在企业 AI 领域赢的公司，不是拥有最好模型的公司——而是那些自动化了**代码写完之后的 60-70% 工程工作**的公司。

**"AI for everything after code"**——Harness CEO 对这个品类的定义，比传统 DevOps 市场大得多。对于工程团队来说，这意味着：
- 你的核心竞争力不在写代码的速度（AI 已拉平），而在**代码产生后如何被安全、高效、可控地交付到生产环境**
- AI-First 不是关于 Cursor 或 Copilot 选型，而是关于**重新设计整个软件交付基础设施**
- 如果你还在纠结"用哪个模型好"，而竞争对手已经在自动化外循环，差距会指数级扩大

---

## 关键 Takeaways

1. **瓶颈已转移：** 代码不再是瓶颈，验证/评审/协作/安全才是。重新分配你的团队精力。
2. **AI-Assisted ≠ AI-First：** 附加 10% 还是重构 10 倍，取决于你是否敢于重写流程而非叠加工具。
3. **工程师的核心能力变了：** 不是编码速度，而是决策质量——评估、批评和指导 AI 输出的能力。
4. **自我修复是终极目标：** 检测→分类→修复→验证形成自动闭环，把人的介入压缩到风险审查。
5. **组织形态由节奏决定：** 日级别发布需要扁平结构、JIT 规划、经理从 IC 做起。
6. **代码评审分层：** AI 预审（质量/安全/依赖）+ 人终审（合规/风险/品味）。
7. **角色正在分化：** 架构师定规则，操作员做判断，AI 做执行。
8. **不确定性不是 bug 是 feature：** 流程减少不是混乱，是把管理成本转化为对产品和用户的关注。
9. **赢在模型周围，而非模型本身：** 模型会商品化，壁垒在上下文工程、基础设施自动化和外循环效率上。自动化代码写完后的 60-70% 工程工作，是 AI-First 团队的下一个主战场。
