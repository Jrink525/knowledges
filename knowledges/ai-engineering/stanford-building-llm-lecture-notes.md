## 📌 总览：训练 LLM 的五大组件

训练一个大语言模型，主要涉及五大组件。理解它们各自的角色，有助于看清学术界和工业界的分工。

| 组件 | 内容 | 谁在主攻 | 重要性评估 |
|------|------|----------|-----------|
| **Architecture** | Transformer 架构设计（层数、头数、激活函数等） | 学术界 🎓 | 实际影响小 |
| **Training Loss / Algorithm** | 预训练损失函数、后训练对齐算法 | 学术界 🎓 | 中等 |
| **Data** | 数据清洗、过滤、配比、去重 | 工业界 🏭 | **极高** |
| **Evaluation** | 评测方法与基准设计 | 工业界 🏭 | **极高** |
| **Systems** | 分布式训练、GPU 利用率、推理优化 | 工业界 🏭 | **极高** |

> 💡 **讲者核心观点**：学术界花了太多时间在架构和损失函数的小修小改上。但在实践中，**数据质量、评测方法和系统效率才是决定模型好坏的关键**。一个激活函数的小改进，可能还不如再多训 10 小时的效果。

# 第一部分：Pre-training（预训练）

## 1.1 什么是语言模型（LM）

语言模型本质上是**词元序列的概率分布模型**：
- 给定句子 "the mouse ate the cheese" → 模型输出这句话出现在互联网上的概率
- 语法错误的句子（"the the mouse ate cheese"）→ 概率更低
- 语义荒谬的句子（"the cheese ate the mouse"）→ 概率更低

### 自回归语言模型（Autoregressive LM）

利用**链式法则**分解联合概率：

$$P(x_1, x_2, ..., x_L) = P(x_1) \cdot P(x_2|x_1) \cdot P(x_3|x_1, x_2) \cdot ... \cdot P(x_L|x_1, ..., x_{L-1})$$

**训练**：预测下一个 token → 用**交叉熵损失（Cross Entropy Loss）**与真实 token 比较 → 梯度更新权重
**推理**：逐 token 自回归生成，每次将新生成的 token 加入上下文，再预测下一个

### ⚠️ 局限
- 生成长文本时必须逐 token 循环，速度慢
- 虽然是当前最佳范式，但仍存在根本性的效率问题

## 1.2 Tokenizer（分词器）

### 为什么需要 Tokenizer？

| 方案 | 问题 |
|------|------|
| 按单词分 | 拼写错误 → 找不到对应 token；非拉丁语言（如泰语）无空格分隔 |
| 按字符分 | 序列过长 → Transformer 的注意力复杂度 $O(L^2)$ 无法接受 |

### BPE（Byte Pair Encoding）算法

1. 从字符级别 token 开始（约 256 个基础 token）
2. 在语料中统计最频繁出现的 token 对
3. 将该对合并为一个新 token
4. 重复直到达到预设的词汇表大小（通常 32K ~ 100K）

**示例**："tokenization" → BPE 可能拆成 "token" + "ization" （取决于语料统计）

### 局限与争议

**① 数学能力差**
- 数字 327 可能是一个独立 token，而 328 是另一个 token
- 模型无法利用数字之间的数值关系进行分解式推理
- 有些模型甚至连一位数加法都做不好

**② GPT-4 的改进**
- 专门改进了代码的 token 化方式（如 Python 每层 4 空格缩进 → 编码成独立 token）
- 这让模型写代码时更快、更准确

**③ 未来方向**
- 学术界希望抛弃 tokenizer，直接基于字符或字节（如 Character Transformer）
- 但需要新架构解决序列长度带来的计算复杂度问题

## 1.3 预训练模型的评测方法

### Perplexity（困惑度）

$$\text{perplexity} = 2^{\text{avg\_loss}}$$

- 取值范围：1（完美） ~ 词汇表大小（完全随机）
- GPT-3 级别模型：~10（相当于在 10 个候选词之间犹豫）
- 2017 年模型：~70
- **不同模型间通过 tokenizer 不同，perplexity 不具可比性**

### 学术基准

- **MMLU**：覆盖各学科的多选题。评测方式存在歧义——只看 A/B/C/D 哪个 token 的概率最高，不同工具可能得到不同结果
- **HELM**（Stanford）：聚合多个可自动评测的任务
- **Open LLM Leaderboard**（Hugging Face）：类似 HELM 的社区版

### 评测的两大挑战

**① 评测方法不一致**
> "我跑 LLaMA 65B 在 HELM 上准确率 63.7%，但在另一个基准上只有 48.8%。两者差 15 个点——这就是不同的评测方式造成的。"

LLaMA 官方论文中的评测结果和社区复现结果经常有显著差异。

**② 数据污染（Train-Test Contamination）**
- 互联网数据集（如 Common Crawl）可能包含了 MMLU 等测试集的内容
- **检测技巧**：如果模型按原始顺序生成某段文本时的 perplexity 明显低于随机打乱顺序后的值 → 高度怀疑数据泄露
- 实际工业界还会用更多启发式方法检测

## 1.4 数据流水线

> "人们常说在互联网上训练，但互联网到底是什么？"

### 从 Common Crawl 到可用训练数据

```
Common Crawl (250B 页 ≈ 1PB 文本)
    ↓
① 文本提取
   - 从 HTML 中提取正文
   - 数学公式（LaTeX）提取很难，早期模型常常丢/错
   - 多语言编码处理
    ↓
② 过滤不良内容
   - NSFW、有害内容、个人身份信息（PII）
   - 每个公司都有很长的黑名单，但从不公开（防止被人针对性篡改）
    ↓
③ 去重
   - URL 去重、段落去重、整页去重
   - 反向文档频率（MinHash / SimHash）用于近似去重
    ↓
④ 基于规则的过滤
   - Token 分布异常（非目标语言占比过高）
   - 词长异常（全是超短或超长词）
   - 页面太短（<500 字符）或太长（>500K 字符）
    ↓
⑤ 基于模型的过滤
   - 训练一个二分类器：以维基百科引用的外部来源作为正样本，随机网页作为负样本
   - 用这个分类器给所有页面打分，保留高质量页面
    ↓
⑥ 领域分类与加权
   - 将数据分为代码、学术论文、书籍、新闻、娱乐等类别
   - 人工调整各领域采样比例：
     - 代码↑ → 提升推理能力
     - 书籍↑ → 提升语言质量与长文理解
     - 娱乐/社交媒体↓ → 减少无意义内容
    ↓
⑦ 高质收尾（High-Quality Annealing）
   - 接近训练结束时，降低学习率
   - 同时在维基百科、精选图书等高质数据上做额外的训练
   - 相当于在极高质量数据上"过拟合"一小部分
   - 对模型质量有明显提升
```

### 数据规模演化

| 模型/时期 | 训练数据量 | 对比 |
|-----------|-----------|------|
| 早期学术基准（2018-2020） | ~150B tokens | ~800 GB 文本 |
| GPT-3 (2020) | ~300B tokens | |
| LLaMA (2023) | ~1.4T tokens | |
| LLaMA 2 (2023) | 2T tokens | |
| LLaMA 3 / Gemini / 当前顶级 (2024) | **15T tokens** | 约 80-100× 学术时代 |

### 未解决的开放问题

- 如何高效处理 PB 级数据？每天新增网页比所有模型可处理的还多
- 各领域数据的最佳混合比例是多少？
- 能否用合成数据无限扩展训练数据？（尚无定论）
- **数据是大多数 AI 公司的核心竞争壁垒，所以极少公开细节**
- 讲者团队约 70 人，其中**约 15 人专门做数据相关工作**

## 1.5 Scaling Laws（规模定律）

### 核心发现

训练损失与计算量 / 数据量 / 参数量之间存在**幂律关系**（在对数坐标下是一条直线）：

$$L(x) \propto x^{-\alpha}$$

**关键结论**：
- 增加计算量 → 可预测的 loss 下降
- 增大模型 → loss 下降
- 增加训练 token → loss 下降
- **迄今为止未见平台期！**（意味着模型规模还可继续增长）

### 具体例子：Transformer vs LSTM

如果手上有 10,000 块 GPU，不确定用 Transformer 还是 LSTM：

1. **分别训练**不同参数量级的 Transformer 和 LSTM 模型
2. 拟合各自的 scaling law 曲线（横轴 = 参数量，纵轴 = 测试 loss）
3. **外推**到更大计算量时，两条曲线的差异会清晰显示 Transformer 胜出

**两个关键参数**：
- **缩放速率（slope）**：决定了随着规模扩大 loss 下降的速度
- **截距（intercept）**：决定了小规模下的初始表现
- 某些架构在小规模下较差，但缩放速率更高 → 大规模时反超

> 🤔 **讲者观察**：绝大多数架构改进，最终只是改变了截距（初始化优势）。随着算力增长，这些优势很快就被抹平了。真正改变斜率的是**数据质量**。

### 应用场景：实验设计新范式

> "假如给你 10,000 块 GPU 训练一个月，你该怎么设计实验？"

**❌ 老方法**（学术界常见）：
- 30 天 → 训 30 个不同配置的大模型各 1 天 → 挑最好的
- 最终模型只训了 1 天！浪费了全部算力

**✅ 新方法**（工业界做法）：
- 前 3 天：训不同规模的小模型（多个），调超参
- 中间：拟合 scaling law → 预测最优配置
- 后 27 天：按预测训最终大模型
- 最终模型训了 27 天，而非 1 天

### Chinchilla 法则（DeepMind 2022）

**核心问题**：给定固定计算预算，应该用更多参数还是更多数据？

**方法**：
1. 画**等 FLOPs 曲线**（isoFLOP curves）
2. 每条曲线上的不同点代表参数和数据量的不同组合，但总计算量相同
3. 找到每条曲线上最佳点
4. 拟合"最优参数量 vs 总 FLOPs"的 scaling law
5. 结论：**每 1 个参数需要 20 个训练 token**（1:20 比例）

**⚠️ 重要注意事项**：
- 这是**训练最优**（最低 loss）的比例
- 如果考虑**推理成本**，更倾向小模型
- 推理最优比例约 **1:150**（每参数 150 tokens）
- LLaMA 3 405B（参数量 405B × 40 tokens ≈ 15.6T）的训练比例介于两者之间

### 苦味教训（Bitter Lesson）

> **Richard Sutton (2019)**：70 年 AI 研究的教训——利用算力的通用方法（搜索、学习）始终战胜人类知识注入。算力持续增长是唯一确定的趋势，所以唯一重要的是**能利用算力的架构**。

**对 LLM 的启示**：
- 别花时间在激活函数、注意力变体等小架构改动上
- **Systems + Data + Scale** 才是真正的杠杆
- OpenAI 凭 ChatGPT 成功是因为把简单的事（自回归、数据、规模）做到了极致

## 1.6 成本估算：以 LLaMA 3 405B 为例

讲者提供的**信封背面估算**（order of magnitude，可能与实际有偏差）：

### FLOPs 计算

标准公式（前向 + 反向）：

$$\text{FLOPs} \approx 6 \times \text{参数量} \times \text{训练 tokens}$$

代入 LLaMA 3 405B：
$$6 \times 405B \times 15.6T = 3.8 \times 10^{25} \text{ FLOPs}$$

> 🏛️ **拜登行政令**：如果 FLOPs > $10^{26}$，模型将受到政府特殊审查。LLaMA 3 405B 刚好在这条线之下。

### 成本明细

| 项 | 数值 | 计算依据 |

| 参数量 | 405B | |

# 第二部分：Post-training（后训练 / 对齐）

## 为什么需要后训练？

> GPT-3（纯预训练模型）回答 "explain the moon landing to a six year old"：
> → 它会生成 "explain the theory of gravity to a six year old"
>
> 因为互联网上的数据格式是"问题列表"，而不是"用户提问→助手回答"的对话形式。

**后训练的目标**：让 LLM **从语言模型变成 AI 助手**。

## 2.1 SFT（Supervised Fine-Tuning / 有监督微调）

### 做法

- **使用完全相同的语言建模损失**（交叉熵），在高质量的人类问答数据上继续微调
- 本质上就是**行为克隆（Behavioral Cloning）**
- 数据量远小于预训练，但**用更高的学习率**（因为预训练结束时学习率已衰减到接近 0）

### 数据来源

**① 人工收集**（昂贵、慢）
- 示例：OpenAssistant 数据集（众包模式）

**② 用 LLM 生成（Synthetic Data）**

> **Alpaca（Stanford 2023）**：用 175 条人工种子数据 → 调用 GPT-3.5 生成了 52,000 条问答对 → 微调 LLaMA 7B → 效果惊人

从此开创了"**用 LLM 训练 LLM**"（自举式合成数据）的热门研究方向。

**③ 人机协作（未来方向）**
- 让模型生成答案 + 人类只做少量编辑（比从头写答案快得多）
- 信息论角度：编辑仍然在注入新信息，但速度更快
- 本质上是**主动学习（Active Learning）**：找到最重要、模型最不确定的样本，让人类介入

### LIMA 论文的核心发现

> **仅需 2,000 条 SFT 数据！** 2K → 32K，性能几乎没有提升。

**原因**：
- 预训练模型已经见过所有类型的"用户"行为：列清单的、直接回答的、写博客的、闲聊的
- SFT 的本质是告诉模型"你现在应该优化这一类用户的回答格式"
- **并没有教模型新知识**——预训练时知识已经在了
- 所以这就是为什么"数据不在于多，而在于精"

### 合成数据的局限

> 一直在同样的分布上生成数据 → 不会有新信息 → 最终饱和。你现在只是在做自我对抗（self-play）。

**现场 Q&A 补充**：
- 纯用模型生成的数据做多代训练，很可能不会持续提升
- 关键是人机协作：用 LM 生成初稿 → 人类做针对性修改 → 在关键点上注入新信息

### SFT 的三个问题

| 问题 | 说明 |
|------|------|
| **① 受限于人类能力** | 评判好坏的能力 ≥ 生成最优答案的能力。我能判断一本书好不好，但我写不出那本书。 |
| **② 可能加剧幻觉（Hallucination）** | 如果人类给了模型一个"看似合理但模型预训练中没见过"的回答，模型会在推理时编造类似内容。例：让模型写出关于某本书的评论 → 人类给了正确的参考文献 → 但模型在预训练时**从未见过这本参考文献** → 模型学会编造看起来合理的引用。 |
| **③ 昂贵** | 让人类写出完整的、最优的答案，远比简单的二选一标注要贵得多。 |

## 2.2 RLHF（Reinforcement Learning from Human Feedback）

### 核心思想

- **不再克隆人类行为**（SFT 的 behavioral cloning）
- 转而**最大化人类的偏好**（不管人类自己能不能写出最好的答案，重要的是他们能分辨好坏）

**流程**：
1. 对每条指令，由模型生成 **两个回答**
2. 人类标注员选择"更好"的那个
3. 用算法让模型多生成"好的"回答、少生成"差的"回答

### 2.2.1 PPO（Proximal Policy Optimization）

#### 第 1 步：训练奖励模型（Reward Model）

**为什么需要奖励模型？**
- 直接用"好/坏"二值奖励太稀疏（信息太少）
- 回答 A 比 B 好一点点 VS 好非常多 → 二值奖励无法区分

**Bradley-Terry 模型**：

$$P(\text{output}_1 \succ \text{output}_2 | x) = \frac{\exp(R(x, \text{output}_1))}{\exp(R(x, \text{output}_1)) + \exp(R(x, \text{output}_2))}$$

其实就是将两个回答的奖励分数转化为 softmax 概率。使用交叉熵损失训练：

$$\mathcal{L} = -\log P(\text{人类偏好的回答更好})$$

**本质**：训练一个能做偏好评判的分类器（也是个大模型），输入是 prompt + 回答，输出连续奖励值。

#### 第 2 步：强化学习

1. 将 LLM 视为 RL 中的 **policy**
2. 从当前 policy 采样回答
3. 用奖励模型给回答评分
4. 用 PPO 更新 policy 权重
5. **必须加 KL 正则化**（防止 reward hacking——因为奖励模型本身不完美）

#### PPO 的痛点

> **讲者原话**："RL 理论上很漂亮，但实践中用过的人都知道有多乱。"

- rollout（采样子回答）
- clipping（防止策略更新过大）
- out-of-distribution 问题
- 实现极难，超参数极多，文档不全

#### ⚠️ 重要后果

PPO 优化后，LLM 不再以最大似然为目标：
- 输出的似然值（likelihood）**不再有概率意义**
- 模型倾向于给出**单一最优回答**（熵极小化）
- **Perplexity 完全不可靠**

### 2.2.2 DPO（Direct Preference Optimization）

> 也是 Stanford 提出（2023），本质上是 PPO 的简化版。

**核心思想**：
- 不需要 RL，不需要奖励模型，不需要 rollout
- 直接做**最大化偏好回答的概率 + 最小化不偏好回答的概率**

$$\mathcal{L}_{\text{DPO}} = -\log \frac{\exp(\beta \cdot \log P(\text{preferred}|x))}{\exp(\beta \cdot \log P(\text{preferred}|x)) + \exp(\beta \cdot \log P(\text{dispreferred}|x))}$$

**简化理解**：
- 对偏好回答：增大其条件概率
- 对不偏好回答：减小其条件概率
- 加温度系数 $\beta$ 控制惩罚强度

#### DPO vs PPO 对比

| 方面 | PPO | DPO |
|------|-----|-----|
| 复杂度 | 很高（RL，rollout，clipping） | 很低（只需最大似然） |
| 需要额外模型 | 需要奖励模型 | 不需要 |
| 可用无标注数据 | ✅ 可用奖励模型给无标注数据打分 | ❌ 只用已标注偏好数据 |
| 效果 | ✅ 好 | ✅ 至少一样好 |
| 谁在用 | 有 RL 团队的公司 | 开源社区 + 越来越多工业界 |

> ⚡ **性能对比**：在摘要任务上，PPO 和 DPO 的效果相当，且都显著优于人类生成的参考摘要。

#### 为什么 OpenAI 先在 ChatGPT 用 PPO 而不是 DPO？

> "OpenAI 做 ChatGPT 的团队里很多是 RL 专家——包括 PPO 论文的原创作者。对他们来说 RL 是最直观的方案。"

### PPO 和 DPO 的代价对比效果

```
Pre-trained → SFT → PPO/DPO
                 ↓
        性能稳步提升（PPO ≈ DPO）
        在某些基准上甚至超过人类
```

## 2.3 人类反馈数据的挑战

### 标注的实际困难

> **现场感受**：即使用户亲自读两个回答，也很难立即判断哪个更好。
>
> 示例 prompt："告诉我关于自动驾驶汽车"
> - 回答 A："自动驾驶汽车是能感知周围环境的车辆..."
> - 回答 B："自动驾驶汽车是配备了传感器的汽车..."
>
> "两者看起来都还行。哪个更好？真的很难一眼看出来。"

### 五大挑战

**① 慢且贵**
- Mechanical Turk：1,000 条标注 ≈ $300
- 每条标注需要仔细阅读两个回答并做出判断（开放性问题，不是简单的分类）

**② 难以聚焦关键维度**
- 人类倾向于选**更长的回答**（即使内容更差）
- **随着 RLHF 进行，模型输出越来越长**——这被称作 length bias
- "如果你曾经被 ChatGPT 的超长回答烦到过，这就是 RLHF 的副作用"

**③ 标注员之间方差极大**
- 人类之间的二选一一致性仅约 **66%**
- 论文作者五人在讨论 3 小时标注指南后，自我一致性也仅 **67-68%**
- "在二选一任务上，两个人有 1/3 的概率意见不一致——这不是人类水平差，是这个任务太难了"

**④ 标注员分布漂移**
- 谁来做标注员？他们代表谁的观点？
- 不同文化背景、不同教育水平的人偏好不同

**⑤ 标注伦理问题**
- 标注员工资普遍低
- 需要大量接触有害/有毒内容

### 用 LLM 替代人类标注（LLM-as-Judge）

**做法**：用另一个 LLM（如 GPT-4）替代人类做偏好评判。

**结果**：
- **成本低约 50 倍**
- **与人类众数的一致性比人类之间的一致性更高**
- 原因：人类方差大（随机性高），模型几乎无方差（虽然可能有系统性偏差，但极稳定）

**现状**：开源社区和工业界已广泛采用 LM + 人类混合标注模式。

## 2.4 后训练模型的评测

### 评测的核心困境

> "如何评估一个回答可以是无限多样的模型？"

**传统方法全部失效**：

| 方法 | 失效原因 |
|------|----------|
| ❌ Validation loss | PPO 和 DPO 模型之间不可比 |
| ❌ Perplexity | 模型不再是最大似然训练，似然值无意义 |
| ❌ 传统 ML 基准 | 回答高度开放，无法自动化判定 |

> **补充**：PPO 训练后，最优 policy 本质上是对每个 prompt 给出一个 delta 分布（接近单一回答）。语义上略有不同的输入，模型可能直接给出零概率。

### Chatbot Arena（聊天机器人竞技场）

- 上线两个匿名模型
- 用户与两者对话后盲选哪个更好
- 积累数十万名用户的投票 → 生成 Elo 排名

**缺点**：
- 用户群体偏向技术爱好者（提问内容多与技术相关）
- 成本高、速度慢（不适合开发过程中的快速迭代）

### LM-as-Judge 用于评测

与标注类似，用 LLM 替代人类做盲审：

1. 对每条指令，用待评测模型和基线模型各生成一个回答
2. 用 GPT-4 等评判模型判断哪个更好
3. 在所有测试用例上计算胜率

**优势**：
- 与 Chatbot Arena 的**相关性高达 98%**
- 3 分钟内即可完成，成本 <$10
- 几近免费替代人类众包

**⚠️ 长度偏差问题**：
- GPT-4 也偏好更长的输出（= 更啰嗦的答案）
- 实验：在 prompt 中加入"请用简洁语言回答" → GPT-4 胜率从 50% 降至 20%
- 在 prompt 中加入"请用详细语言回答" → 胜率升至 64.4%
- **解决方案**：使用回归/因果推断方法控制长度变量

# 第三部分：Systems（系统优化）

> **（由于时间关系，讲者只讲了约 8 分钟）**

## 3.1 为什么系统优化重要？

> "为什么不多买些 GPU？——GPU 贵，但更关键的是，即使你有 $1000 万，现在也买不到最好的 GPU。"

**物理限制**：
- 多 GPU 之间需要通信，通信成本随 GPU 数量增长而增加
- IO 瓶颈日益严重：**计算性能提升速度 > 内存/通信性能提升速度**
- 大多数 GPU 在跑未优化的代码时，大部分时间处于空闲状态

所以**系统优化的目标**：尽可能让 GPU 处于工作状态。

## 3.2 GPU 基础

| 特性 | CPU | GPU |
|------|-----|-----|
| 优化目标 | 低延迟（快速响应单任务） | 高吞吐（并行处理大量同类计算） |
| 核心数量 | 数个到数十个 | 数千个 |
| 黄金操作 | 任意指令 | **矩阵乘法**（比其它操作快 10× 以上） |
| 瓶颈 | 通用计算 | 内存带宽+通信 |

## 3.3 混合精度训练（Mixed Precision）

**直觉**：
- 如果我用低精度浮点数 → 更少比特需要传输给 GPU → 传输更快 → 内存占用更少 → 处理更快
- 深度学习对精度不敏感（SGD 本身已经有大量噪声）

**标准做法**：
- **权重存储**：32 位浮点数（FP32）
- **矩阵运算**：前向/反向传播 → 转为 16 位（FP16/BF16） → 运算完成后转回 32 位更新权重
- 原因：如果学习率很小（如 $1e^{-7}$），FP16 无法表示这样小的增量，必须用 FP32 做累积

**MFU（Model FLOP Utilization）**：
- 等于"实际吞吐量 ÷ GPU 理论最大吞吐量"
- 达到 50% 已经很不错了
- Meta 训练 LLaMA 时 MFU ≈ 45%
- 意味着 GPU 有一半时间在空等

## 3.4 算子融合（Operator Fusion）

**问题**：PyTorch 每行代码都会将变量从 GPU 全局内存搬到计算单元 → 计算完再搬回内存。

```python
x = torch.cos(x)  # 移动到 GPU → 算 cos → 搬回 DRAM
x = torch.sin(x)  # 再移动到 GPU → 算 sin → 搬回 DRAM
```

**每次搬移都是巨大的开销**。

**解决方案**：算子融合（Fused Kernel）

```python
# 一次性搬到 GPU → 连续算完所有操作 → 一次性搬回
x = fused_cos_sin(x)  # 只搬移一次
```

**实际操作**：`torch.compile(model)` → 自动将 PyTorch 代码编译为优化的 CUDA C++ kernel → **约 2× 加速**。

# 第四部分：现场 Q&A 精选

### Q：后训练微调是在改所有参数还是部分参数？

> **A**：工业界做法——**改所有参数**。开源社区常用 LoRA（只在各层输出添加低秩差值），但工业界全量微调。

### Q：为什么少量 SFT 数据（2K 条）能有这么大影响？预训练见过 15T token 啊！

> **A**：你要意识到**预训练只是模型的初始化**。如果用足够大的学习率在一条数据上反复训，最终模型只会生成那句话。
>
> 关键在于：后训练数据不是和预训练数据混在一起的。你**先做预训练（作为权重初始化）**，然后**只在后训练数据上继续训练**。一旦这样理解，那 15T tokens 的预训练只是提供了好的初始状态。

### Q：PPO 微调时实际会跑几个 epoch？

> **A**：在我们的实验（Packard Farm）中，大约做了 **3 个 epoch**。但重复次数不重要——重要的是**有效学习率（effective learning rate）**。

### Q：后训练数据的规模有多大？

> **A**：
> - SFT：约 5,000 ~ 50,000 条
> - RLHF：约 **100 万条**量级
> - 预训练：15 万亿 tokens
>
> 预训练的量级是 RLHF 的**数亿倍**，但后训练不同的学习率和训练方式让它能对模型行为产生巨大的影响。
