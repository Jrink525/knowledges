## 训练的"两步法"

训练一个能做对话的 LLM，分**两个大阶段**：

### 第一阶段：预训练（Pre-training）
- **目标**：让模型学会"人是怎么说话的"
- **方法**：把整个互联网的文字（数万亿个词）喂给模型，让它学会预测下一个词
- **结果**：一个"语言模型"——它知道语法、知道很多知识，但**不会和人类对话**

### 第二阶段：后训练（Post-training）
- **目标**：让语言模型变成"AI 助手"
- **方法**：用人类对话数据进一步训练，用人类偏好优化回答
- **结果**：你用的 ChatGPT

> GPT-3 是纯预训练模型，ChatGPT 是 GPT-3 做了后训练之后的产品。**后训练是 ChatGPT 如此受欢迎的关键创新。**

# 第一部分：预训练（Pre-training）

预训练是构建 LLM 的第一步，也是最费钱的一步。我们来一点一点拆解。

---

## 1.1 什么是"语言模型"？

### 核心概念

语言模型，简单说，就是能**计算一句话出现在互联网上的可能性有多大**的模型。

比如给你三个句子：

| 句子 | 互联网上出现的可能性 |
|------|---------------------|
| "The mouse ate the cheese."（老鼠吃了奶酪） | ✅ 很高 |
| "The the mouse ate cheese."（语法错误） | ❌ 很低 |
| "The cheese ate the mouse."（奶酪吃了老鼠——荒谬） | ❌ 极低 |

一个有水平的语言模型应该能区分这三个句子：**第一个可能，第二、三个不可能**。

要做到这一点，模型需要：
- **语法知识**：知道 "the the" 是错误的
- **语义知识**：知道老鼠吃奶酪是正常的，奶酪不会吃老鼠

### 自回归语言模型——今天所有 LLM 都在用的方法

目前所有主流 LLM（ChatGPT、Claude、Gemini 等）都是一种叫做"自回归语言模型"的东西。名字很拗口，但思想很简单：

**把一个句子的概率，分解成一个个词的概率的乘积。**

公式长这样：

```
P(老鼠吃了奶酪) = P(老鼠) × P(吃了 | 老鼠) × P(奶酪 | 老鼠, 吃了)
```

翻译成人话就是：
1. 先问：第一个词是"老鼠"的概率？
2. 再问：已知第一个词是"老鼠"，第二个词是"吃了"的概率？
3. 再问：已知前两个词是"老鼠吃了"，第三个词是"奶酪"的概率？
4. 所有概率乘起来，就是整句话的概率。

这就是**链式法则**——高中概率课上应该学过。

### 训练的时候做什么？

1. 给模型一段文字："老鼠吃了"
2. 让模型预测下一个词："奶酪"？
3. 看正确答案："奶酪"✅
4. 调整模型参数，让模型下次更倾向于说"奶酪"而不是别的词

### 生成（推理）的时候做什么？

1. 给模型："解释一下相对论"
2. 模型预测第一个词："相对"
3. 把"相对"加回去，再预测下一个词："论"
4. 继续循环，直到生成完整的回答

> **一个缺点**：因为要一个个词地生成，**长文本很慢**。这也解释了为什么 ChatGPT 打字是一字一字"蹦"出来的。

## 1.2 分词器（Tokenizer）——模型怎么"读"文字

### 为什么需要分词器？

你可能会想：为什么不直接把每个单词当作一个"基本单位"？

| 方案 | 问题 |
|------|------|
| 按单词分（word-level） | "猫"是一个词，"猫猫"（打字错误）又是另一个词？拼写错了模型就不认识。而且像泰语这种没有空格的文字，根本没法按单词分。 |
| 按字母分（character-level） | "老鼠吃了奶酪"会变成 6 个字母，序列太长。Transformer 模型处理长序列的计算量是**平方增长**的，吃不消。 |

**解决方案**：取中间值——分词器（Tokenizer），把文字切分成"子词"。一个 token 大约 3-4 个字母。

### BPE 算法——最常用的分词方法

字节对编码（Byte Pair Encoding）的原理很简单：

**第一步**：把所有文字拆成一个个字母
```
老鼠吃了奶酪 → 老 鼠 吃 了 奶 酪
```

**第二步**：统计最常出现的字母对
```
"鼠"+"吃"一起出现了 5 次，是最多的
```

**第三步**：把这对合并成一个新 token
```
老 鼠吃 了 奶 酪
```

**第四步**：重复——继续找最常出现的对，继续合并
```
老鼠吃 了 奶酪
```

最终词汇表里会有：
- 最常用的词：一个完整词作为一个 token
- 不常用的词：分成 2-3 个子词 token
- 生僻词/错别字：回退到字母级别

**实际例子**：GPT 的分词器会把 "tokenization" 分成 "token" + "ization" 两个 token。

### 分词器的一个"大问题"：数学不好

假设 "327" 在分词器里是一个独立的 token，"328" 是另一个 token。

模型看到 327 时，它看到的不是三个数字 3、2、7，而是一个符号 🏷️"327"。

这就意味着：**模型不知道 327 + 1 = 328**。它没有数字的概念！

> 这就是为什么大语言模型在做简单数学题时经常出错。GPT-4 花了很大功夫改进代码的分词方式，比如把 Python 中的 4 空格缩进编成独立 token，这样模型写代码就更好一些。

### 未来方向

很多研究者希望在 5-10 年内抛弃分词器，直接基于字符或字节。但这需要新的架构来解决"序列太长"的问题。在那之前，分词器还会一直存在。

## 1.3 数据——模型"吃"什么长大

> "人们常说在互联网上训练模型，但互联网到底是什么？"

你可能以为训练数据就是维基百科那种干净整洁的文章。错。**真实的互联网很脏。**

### 数据从哪里来？

最常用的来源是 **Common Crawl**——一个每月的互联网快照。

- 页面数量：约 **2,500 亿个网页**
- 数据量：约 **1 PB**（1,000 TB，或 100 万 GB）
- 里面什么都有：博客、论坛、垃圾邮件、代码、色情内容、机器生成的废话

### 从"脏数据"到"干净数据"的七步流程

```
第一步：下载原始网页（2,500 亿页 ≈ 1PB 的纯文本）
        ↓
第二步：提取正文
   - 把 HTML 标签去掉，只保留文字
   - 数学公式很难提取（但很重要）
   - 去掉重复的页眉页脚
        ↓
第三步：过滤不良内容
   - 色情、暴力、仇恨言论
   - 个人隐私信息（身份证号、电话号码）
   - 每个公司都有一份很长的黑名单（但不公开）
        ↓
第四步：去重
   - 同一个 URL 出现多次 → 只留一次
   - 相同段落出现多次 → 只留一次
   - 比如《三体》的盗版在网上出现了一万次 → 只留一次
        ↓
第五步：规则过滤
   - 页面只有 3 个词 → 没什么用，扔掉
   - 页面有 1000 万个词 → 可能有问题，扔掉
   - 非目标语言占主导 → 扔掉
        ↓
第六步：模型过滤（聪明的一招）
   - 训练一个分类器：用"维基百科引用的网站"作为好数据的标准
   - 给所有网页打分，只留高分页面
        ↓
第七步：领域分类与加权
   - 把数据分成：代码、学术论文、书籍、新闻、娱乐、社交媒体
   - 人为调整比例：
     💻 代码多放一点 → 提升推理能力
     📚 书籍多放一点 → 提升语言质量和长文理解
     🎮 娱乐少放一点
        ↓
第八步：高质收尾（冲刺阶段）
   - 训练快结束时，降低学习率
   - 只用维基百科、高质量图书这些"精品"数据再训一小段
   - 相当于考前"刷重点"
```

### 数据量有多大？——从 150B 到 15T

| 时间 | 模型 | 训练数据量 | 增长倍数 |
|------|------|-----------|---------|
| 2018-2020 | 早期学术模型 | 1,500 亿 tokens | 基准 |
| 2020 | GPT-3 | 3,000 亿 tokens | ×2 |
| 2023 | LLaMA | 1.4 万亿（T）tokens | ×9 |
| 2023 | LLaMA 2 | 2T tokens | |
| 2024 | LLaMA 3 | **15.6T tokens** | ×100 |

**1 个 token ≈ 3/4 个英文单词**，所以 15.6T tokens ≈ 12 万亿个英文单词。

> 一个 70 人的团队，大约有 **15 个人专门做数据工作**。数据是 AI 公司的"商业机密"，所以很少有公司公开细节。

## 1.4 评测——怎么知道模型在变好？

### 方法一：困惑度（Perplexity）

这是预训练阶段**最常用的评测指标**。

**公式**：`困惑度 = 2的(平均损失)次方`

**怎么理解**：
- 范围：**1 ~ 词汇表大小**（比如 1 ~ 32,000）
- 1 分：模型完美——永远知道下一个词是什么
- 32,000 分：模型完全瞎猜——等于在 3.2 万个词里随机猜
- GPT-3 级别：约 **10 分**——模型在 10 个候选项之间犹豫
- 2017 年模型：约 **70 分**

> 过去 6 年，困惑度从 70 降到了 10 以下，说明模型进步巨大。

**局限**：不同模型用不同的分词器，困惑度不能跨模型比较。比如 ChatGPT 用的分词器有 10 万个 token，Gemini 用了 3 万个——这本身就是不一样的尺子。

### 方法二：学术基准（Benchmark）

因为困惑度没法跨模型比较，学术界设计了各种标准化测试。

**MMLU（大规模多任务语言理解）** 是最常用的：
- 覆盖 57 个学科：大学医学、大学物理、天文学、法律……
- 每个问题是四选一选择题
- 评估方法：看模型选 A/B/C/D 哪个 token 概率最高

**两个类似的综合测评集**：
- **HELM**：斯坦福出品
- **Open LLM Leaderboard**：Hugging Face 出品

### 评测的两个大坑

**坑一：评测方式不统一**

同一个模型，用了不同的评测方法，结果可以差很多。

> LLaMA 65B 在 HELM 上准确率 **63.7%**，但在另一个基准上只有 **48.8%**——差了 15 个点！

**坑二：数据污染**

如果模型的训练数据里包含了测试题，那它等于"考试前看过了答案"。

**检测方法**：如果把测试题顺序打乱后，模型的表现大幅下降 → 说明它可能是"背"过答案，而不是"理解"了题目。

## 1.5 规模定律（Scaling Laws）——为什么"大"就是好？

### 核心发现

2017 年以来，研究者发现了一个非常简单的规律：**模型越大，数据越多，效果越好。**

这不是随便说的，而是有具体公式的：

$$损失（Loss） \propto \frac{1}{参数量的\alpha次方}$$

**在对数坐标纸上**，这条线是直的——说明规律非常稳定。

**关键结论**：
- ✅ 增加计算量 → 可预测的损失下降
- ✅ 增大模型（更多参数）→ 损失下降
- ✅ 增加训练数据 → 损失下降
- 📍 **迄今为止，没有任何证据表明这个规律会停止**——模型还可以继续变大

### 这个规律的实际用途

**场景**：假如给你 10,000 块顶级 GPU，训练一个月，你怎么选模型架构？

**❌ 以前的笨方法**：
- 训练 30 个不同配置的大模型，每个训 1 天
- 挑最好的那个
- **问题**：最终模型只训了 1 天！

**✅ 现在的聪明方法**：
1. 先训一批不同大小的小模型（每个训几小时）
2. 用结果拟合"规模定律"曲线
3. **预测**更大模型的表现
4. 按预测选择最佳配置
5. 训最终大模型 **27 天**

### Chinchilla 法则

**问题**：如果我有固定的算力预算（比如 1000 万美元），我该怎么分配？

- 把钱花在**更大的模型**上？
- 还是花在**更多的数据**上？

**Chinchilla 论文（DeepMind, 2022）的答案**：

> 每增加 1 个参数，就要增加 **20 个训练 token**。

比例是 **1 : 20**
- 如果你的模型有 1000 亿参数，就需要 2 万亿 token 的数据
- 这是**训练最优**的比例

**但是**：如果还要考虑推理成本（模型上线后每次回答都要花钱）：
- 你偏向于更小的模型（省钱）
- 推理最优比例约 **1 : 150**
- LLaMA 3 405B 的比例是 1 : 40（介于两者之间）

### 苦味教训（Bitter Lesson）

> 这是 AI 研究者 Richard Sutton 在 2019 年写的一篇著名博客。他说：

"过去 70 年的 AI 研究告诉我们一个残酷的教训：**唯一确定在增长的就是算力**。所以我们应该做的是设计**能充分利用算力的架构**，而不是花时间研究那些看似聪明的小技巧。"

**对 LLM 的启示**：
- 别花太多时间在一个新的"激活函数"或"注意力机制变种"上
- 真正重要的是：**系统怎么优化 + 数据多好多干净 + 规模怎么扩大**
- OpenAI 的成功不是因为架构创新，而是**把简单的事情做到了极致**（大规模、高质量数据、工程优化）

## 1.6 到底要花多少钱？——以 LLaMA 3 405B 为例

### 算力公式

训练一个模型需要多少计算量？有一个简单的估算公式：

$$\text{FLOPs} \approx 6 \times \text{参数量} \times \text{训练 token 数}$$

- FLOPs = 浮点运算次数（算力的基本单位）
- 乘以 6 是因为：前向传播（3 次运算）+ 反向传播（3 次运算）

**代入 LLaMA 3 405B**：
$$6 \times 4050 \text{ 亿} \times 15.6 \text{ 万亿} = 3.8 \times 10^{25} \text{ FLOPs}$$

### 完整成本账

| 项目 | 数据 | 说明 |
|------|------|------|
| 参数量 | **4050 亿** | 405 billion parameters |
| 训练数据 | **15.6 万亿 tokens** | 约 12 万亿个英文单词 |
| 总计算量 | $3.8 \times 10^{25}$ FLOPs | 写出来是 38,000,000,000,000,000,000,000,000 |
| GPU 型号 | **NVIDIA H100** | 当时最好的 AI 训练卡，每块约 3 万美元 |
| GPU 数量 | **16,000 块** | 摆满好几个大机房 |
| 训练时间 | **约 70 天** | 2 个多月不停机 |
| GPU 小时数 | 约 **3,000 万**小时 | 16,000 × 24 × 70 |
| GPU 租金成本 | **约 $5,200 万** | 按 $2/块·小时 计算 |
| 人力成本 | **约 $2,500 万** | 50 人的团队，每人年薪 50 万美元（硅谷行情） |
| **总成本** | **约 $7,500 万** | ≈ 5.4 亿人民币 |
| 碳排放 | 约 **4,400 吨 CO₂** | ≈ 2,000 次纽约↔伦敦的往返机票 |

> **讲者点评**：目前碳排放在全球范围内还不是大问题。但到 GPT-6/GPT-7 时代（规模再扩大 100 倍），就会成为真正的环境问题。

# 第二部分：后训练（Post-training）

预训练结束后，你有了一个"语言模型"。它能续写文字，但**不会和人对话**。

举个例子：

> **问 GPT-3**："给一个六岁小孩解释登月"
> **GPT-3 回答**："给一个六岁小孩解释万有引力"
>
> 为什么？因为在互联网数据中，这种问题是"列表"形式的：
> - 给一个六岁小孩解释登月
> - 给一个六岁小孩解释光合作用
> - 给一个六岁小孩解释万有引力
>
> 模型以为你要的是"解释万有引力"——它只是在"续写列表"！

后训练的目标就是：**让语言模型变成 AI 助手**。

---

## 2.1 SFT（监督微调）——教模型"好好说话"

### 怎么做？

做法很简单：收集一批高质量的"问题→优质回答"对，然后用**和预训练完全一样的损失函数**，继续训练模型。

- 预训练时模型学的是"所有可能的人类风格"（问答、列表、博客、闲聊……）
- SFT 时告诉模型："你现在要做一个**问答助手**，选这种风格"

本质上是**行为克隆**——让模型模仿人类给出的"标准答案"。

### 数据从哪里来？

**方法一：人类手动写**
- 贵！要人们写完整的高质量答案
- 示例：OpenAssistant 数据集

**方法二：用 AI 生成数据（Alpaca 方法——里程碑式创新）**
- 2023 年，斯坦福的 Alpaca 团队只写了 175 条"种子数据"
- 然后让 GPT-3.5 **自动生成了 52,000 条问答对**
- 用这 52K 条数据微调 LLaMA 7B → 效果惊人！
- 从此开创了"**用 LLM 训练 LLM**"的时代

**方法三：人机协作（未来方向）**
- 让 AI 先生成回答
- 人类只做针对性修改（比从头写快得多）
- 关键：只在模型最不确定的地方让人类介入（主动学习）

### 一个惊人的发现：不需要太多数据

**LIMA 论文**做了一个实验：
- 用 2,000 条 SFT 数据微调 → 效果很好
- 把数据量加到 32,000 条 → 性能几乎**没有提升**

**原因**：SFT 不教模型新知识。预训练时模型已经见过所有人类说话方式了。SFT 只是告诉它"选哪种风格"。2,000 条足够让它理解这个选择。

### SFT 的三大问题

**问题 1：受限于人类能力**
> 我能判断一本书好不好看，但我写不出那本书。
>
> 同样，人类评判答案好坏的能力 ≥ 写出最好答案的能力。
>
> SFT 让模型模仿"人类能写出来的答案"——而不是"人类能想象出来的最好答案"。

**问题 2：可能加剧幻觉**

假设 SFT 数据里，人类给了这样一个回答：
> 问："《xxx》这本书怎么样？"
> 答："这是一本关于……的经典之作，参考文献是 [某论文]。"

这个回答本身没有问题——但它提到的论文可能**模型在预训练中从未见过**。模型会想："哦，人类让我编造看起来合理的参考文献。"于是在推理时，它也会**编造**参考文献。

这就是幻觉（Hallucination）的一个重要来源。

**问题 3：贵**
- 让人写出完整的、最优的答案
- 比让人做个"二选一"标注贵很多
- 这就引出了下一个方法：RLHF

---

## 2.2 RLHF——用人类偏好优化模型

> RLHF = Reinforcement Learning from Human Feedback（基于人类反馈的强化学习）

### 核心思想

不克隆人类的行为（SFT 的做法），而是**最大化人类的偏好**。

人是这样判断的：
- "我写不出一篇满分作文，但我能看出哪篇更好。"
- "我不会写代码，但我能看出哪个 AI 写的代码更好用。"

所以 RLHF 的做法是：
1. 让模型生成**两个回答**
2. 人类选**哪个更好**
3. 让模型多学"好的"、少学"差的"

### 2.2.1 第一步：训练一个"裁判"——奖励模型

不能直接用"好/坏"来打分（信息太少）——回答 A 比 B 好一点点，和好非常多，都是"好"。

**做法**：训练一个奖励模型（Reward Model），输出连续值：
- 比如回答 A 得分 0.8，回答 B 得分 0.3
- A 比 B 好不少

**Bradley-Terry 模型**（数学部分，不想看可以跳过）：

```
P(回答A比B好) = e^(奖励A) / (e^(奖励A) + e^(奖励B))
```

简单理解：把"回答比另一个好多少"变成"0%～100% 的概率"。

### 2.2.2 第二步：用 PPO 算法优化

> PPO = Proximal Policy Optimization（近端策略优化）

1. 把 LLM 看作"决策者"（policy）
2. 让 LLM 生成回答
3. 用奖励模型给回答打分
4. 用 PPO 算法更新 LLM 的权重
5. 加"正则化"防止走极端（因为奖励模型也不完美）

**PPO 的问题**：
> "听过 RL 的人都知道，RL 理论上很漂亮，但用过的人都头疼。"

- 涉及到 rollout（采样）、clipping（裁剪）、out-of-distribution（分布外问题）
- 实现极其复杂
- 超参数特别多
- 文档还不全

### ⚠️ PPO 之后的一个重要变化

PPO 优化后的模型不再追求"预测最准"——它追求"给出最好的回答"。

后果：
- 模型输出的概率值不再有意义
- 模型更倾向于给出**唯一的最佳回答**（而不是各种回答的概率分布）
- **Perplexity 完全不可靠**

---

## 2.3 DPO——PPO 的简化版

> DPO = Direct Preference Optimization（直接偏好优化）
>
> 也是斯坦福提出的（2023 年）

### 核心思想

不用搞复杂的强化学习了。直接做两件事：

1. **偏好回答** → 增大它的概率
2. **不偏好回答** → 减小它的概率

简单公式：

```
损失 = -log(偏好回答的概率) + log(不偏好回答的概率)
```

**不需要奖励模型！不需要 rollout！不需要 clipping！**

### DPO vs PPO

| 维度 | PPO | DPO |
|------|-----|-----|
| 复杂程度 | 很高 | 很低 |
| 需要额外模型 | 需要奖励模型 | 不需要 |
| 能否用无标注数据 | ✅ 能（奖励模型打分） | ❌ 只能用已标注的 |
| 效果 | ✅ 好 | ✅ 至少一样好 |

> **为什么 OpenAI 先用 PPO 而不是 DPO？**
>
> 因为 ChatGPT 团队里很多 RL 专家——**包括 PPO 论文的原创作者**。对他们来说，RL 是最直观的方法。

### 效果对比

```
效果排名（从差到好）：
预训练模型 < SFT < PPO ≈ DPO
在某些任务上，后两个甚至超过了人类写的参考答案
```

## 2.4 人类反馈数据的现实困难

### 标注一个"好回答"有多难？

想象你在标注 ChatGPT 的两个回答，哪个更好：

> 问："介绍一下自动驾驶汽车"
>
> 回答 A："自动驾驶汽车是能够感知周围环境的车辆……"
>
> 回答 B："自动驾驶汽车是配备了传感器来实现自主导航的车辆……"

**两个看起来都很好。哪个更好？真的很难说。**

### 标注的五大难题

**1. 慢且贵**
- 在 Mechanical Turk（众包平台）上：1,000 条标注 ≈ $300
- 每条标注都需要仔细阅读两个回答

**2. 人类有偏见——喜欢更长的回答**
- 标注员下意识地选**更长的回答**（即使内容更差）
- 结果：**ChatGPT 的回答越来越长**——这就是 RLHF 的副作用
- 如果你被 ChatGPT 的啰嗦回答烦到过，这就是原因

**3. 标注员之间意见不一致**
- 两个人标注同一对回答，一致性只有约 **66%**
- 论文的五个作者花了 3 小时讨论标注标准后，自我一致性也才 **67-68%**
- 这意味着：三个人里有一个人会选不同答案——不是因为笨，是因为这个任务真的难

**4. 标注员的背景会影响结果**
- 不同文化、不同教育水平的人偏好不同
- 那你让哪些人来标注？代表了谁的观点？

**5. 伦理问题**
- 标注员工资低
- 还要大量接触有害内容（色情、暴力）来教会模型避免这些

### 解决办法：用另一个 AI 来标注

**LLM-as-Judge（LLM 当裁判）**：
- 用 GPT-4 代替人类来做偏好评判
- **优势**：
  - 便宜约 50 倍
  - 一致性比人类还高（虽然可能有偏差，但几乎无随机波动）
  - 开源社区和工业界已广泛采用

> 人类之间的一致性只有 66%，但 LLM 与人类的一致性反而更高——因为人类每次判断都不一样，LLM 每次都一样。

## 2.5 后训练模型的评测——比想象中更难

### 传统方法全失效

| 方法 | 为什么不能用 |
|------|-------------|
| ❌ Validation loss | PPO 和 DPO 模型之间不可比 |
| ❌ Perplexity | 模型不再是"预测下一个词"训练出来的，概率值无意义 |
| ❌ 传统选择题基准 | 后训练模型的回答是开放的，不能用 ABCD 选答案 |

### Chatbot Arena（聊天机器人竞技场）——目前最可信的方法

**做法**：
1. 在网站上并排放两个匿名聊天机器人
2. 用户随意提问，看哪个回答更好
3. 收集**数十万用户**的盲审投票
4. 生成 Elo 排名（和象棋比赛一样的计分方法）

**优点**：接近真实使用场景
**缺点**：
- 参与用户偏向"技术爱好者"（提问内容多与技术相关）
- 慢且贵——不适合开发过程中快速迭代

### LM-as-Judge 用于评测

用 GPT-4 代替人类做评测：
1. 对 N 个测试问题，让模型 A 和模型 B 各生成回答
2. 让 GPT-4 判断哪个更好
3. 计算"胜率"

**效果**：
- 与 Chatbot Arena 的相关性高达 **98%**
- 3 分钟内完成，成本 < $10
- 几乎免费替代了数万人的投票

**⚠️ 但有个坑：长度偏差**

实验发现：
- 如果让 GPT-4 用"简洁语言"回答 → 胜率 20%
- 如果让 GPT-4 用"详细语言"回答 → 胜率 64.4%
- 同一个模型，只是改了下说话长短……

**解决方案**：用统计方法把"回答长度"这个因素剔除掉。

# 第三部分：系统优化（Systems）

## 3.1 为什么系统优化如此重要？

> "为什么不多买些 GPU？——贵是一方面，关键是就算你有 1000 万美元，现在也买不到最好的 GPU。"

GPU 多了以后，还有一个物理限制：**GPU 之间要互相通信**。GPU 越多，通信时间越长。大部分 GPU 在跑没优化过的代码时，**大部分时间在空等**。

## 3.2 GPU 和 CPU 有什么区别？

| | CPU | GPU |
|--|-----|-----|
| 类比 | 一个诸葛亮（极聪明但只有一个） | 一千个臭皮匠（每个一般但一起干活） |
| 擅长 | 做复杂的、不同的任务 | 做简单的、重复的任务 |
| 黄金操作 | 各种指令 | **矩阵乘法**（比其它操作快 10 倍以上） |

GPU 的结构是：大量的小计算单元（叫 SM / streaming multiprocessor）同时工作。

## 3.3 两个关键优化技巧

### 技巧一：混合精度训练

**直觉**：用更少的位数表示数字 → 传输更快 → 内存占用更少 → 更快

- 通常用 **32 位浮点数**（FP32）存权重
- 计算时转为 **16 位浮点数**（FP16/BF16）
- 计算完转回 32 位去更新权重

**为什么？** 如果学习率很小（比如 0.0000001），16 位浮点数根本表示不了这么小的变化。

### 技巧二：算子融合（Operator Fusion）

**问题**：PyTorch 每执行一行代码，都要把数据从 GPU 内存搬到计算单元，算完再搬回去。

```python
x = torch.cos(x)  # 搬数据到 GPU → 算 cos → 搬回来
x = torch.sin(x)  # 再搬过去 → 算 sin → 再搬回来
```

每次搬移都很慢！

**解决**：把多次计算合并成一次：`torch.compile(model)`
- 一次性把所有计算搬到 GPU
- 全部算完再一次性搬回来
- **速度提升约 2 倍**

### MFU（模型 FLOP 利用率）

衡量 GPU 被利用了多少的指标：

$$MFU = \frac{\text{实际计算吞吐量}}{\text{GPU 理论最大吞吐量}}$$

- 达到 **50%** 就很不错了
- Meta 训练 LLaMA 时 MFU 约 **45%**
- 也就是说 GPU 有一半时间在空等——还有很大优化空间

# 第四部分：现场问答精选

### Q：后训练微调是改全部参数，还是只改一部分？

> **A**：工业界——**改全部参数**。开源社区常用 LoRA（只加少量新参数来近似），但公司在生产环境全量微调。

### Q：为什么 2,000 条 SFT 数据就能撼动 15 万亿 tokens 的预训练？

> **A**：关键是要换一个思路——**把预训练看作模型的"初始化"**。预训练给了模型一个好的起点（好的权重初始值），后训练才真正定义了模型的"行为"。
>
> 如果我用足够大的学习率，只用 1 个句子反复训，最终模型也只会生成那 1 个句子。所以**数据量不重要，重要的是怎么训**。

### Q：PPO 微调时实际会跑几个循环（epoch）？

> **A**：在实验中，大约做了 **3 个 epoch**。但重复次数不重要——**有效学习率**才是关键。

### Q：后训练的数据量有多大？

> **A**：和后训练对比一下就知道多夸张了：
> - SFT 数据：约 **5,000 ~ 50,000 条**
> - RLHF 数据：约 **100 万条**
> - 预训练数据：**15 万亿 tokens**
>
> 15 万亿 ÷ 100 万 = **1,500 万倍**。但后训练用不同的学习率和训练方式，对模型行为的影响巨大。

### Q：为什么会有人选择 PPO 而不是更简单的 DPO？

> **A**：PPO 有一个独特优势——可以用奖励模型给**无标注的数据**打分。如果你有 100 万条未标注的数据 + 1 万条标注数据，PPO 可以先训好奖励模型，再用它对那 100 万条未标注数据打分。DPO 只能直接用已标注的 1 万条数据。
>
> 但 DPO 胜在简单。现在开源社区基本都在用 DPO。
