# 第一部分：Pre-training（预训练）

## 1.1 什么是语言模型（LM）

语言模型本质上是**词元序列的概率分布模型**：
- 给定句子 "the mouse ate the cheese" → 模型输出这句话出现在互联网上的概率
- 语法错误的句子（"the the mouse ate cheese"）→ 概率更低
- 语义荒谬的句子（"the cheese ate the mouse"）→ 概率更低

### 自回归语言模型（Autoregressive LM）

利用**链式法则**分解联合概率：

$$P(x_1, x_2, ..., x_L) = P(x_1) \cdot P(x_2|x_1) \cdot P(x_3|x_1, x_2) \cdot ... \cdot P(x_L|x_1, ..., x_{L-1})$$

**训练**：预测下一个 token → 用交叉熵损失（Cross Entropy Loss）与真实 token 比较 → 更新权重
**推理**：逐个生成 token，每次将新生成的 token 加入上下文，再预测下一个

### ⚠️ 局限
- 生成长文本时，必须逐 token 循环生成，速度慢
- 当前范式虽有缺点，但仍是主流

## 1.2 Tokenizer（分词器）

### 为什么需要 Tokenizer？

| 方案 | 问题 |
|------|------|
| 按单词分 | 拼写错误 → 找不到对应 token；非拉丁语言（如泰语）无空格 |
| 按字符分 | 序列过长 → Transformer 复杂度随长度大幅增长 |

### BPE（Byte Pair Encoding）算法

1. 从字符级别 token 开始
2. 不断合并最频繁出现的 token 对
3. 最终得到常用子词（subword）的 token 集合

**示例**："token" 在 GPT tokenizer 中被拆成 "token" + "izer"

### 局限与争议
- **数学能力差**：数字"327"可能是一个独立 token，模型无法进行分解式推理
- GPT-4 改进了代码的 token 化方式（如 Python 的 4 空格缩进）
- 学术界希望未来抛弃 tokenizer，直接用字符级别，但需要新架构解决序列长度问题

## 1.3 评测方法

### Perplexity（困惑度）

$$\text{perplexity} = 2^{\text{avg_loss}}$$

- 取值范围：1（完美）~ 词汇表大小（完全随机）
- GPT-3 级别模型：~10（在 10 个词左右犹豫）
- 2017 年模型：~70
- **不适用于学术基准**，因为不同模型使用不同 tokenizer → 不具可比性

### 学术基准

- **MMLU**：覆盖各学科的多选题，评测方式有歧义（只看 A/B/C/D 哪个 token 概率最高）
- **HELM**（Stanford）和 **Open LLM Leaderboard**（Hugging Face）：聚合多个可自动评测的任务

### 评测的两大挑战

**① 评测方法不一致**
- LLaMA 65B 在 HELM 上准确率 63.7%，但在另一个基准上仅 48.8%
- 错误使用其实就来自不同的评测方式

**② 数据污染（Train-Test Contamination）**
- 训练数据可能包含了测试集
- 检测技巧：如果按原序生成的 perplexity 低于随机打乱后的顺序 → 可能数据泄露

## 1.4 数据流水线

> "人们常说在互联网上训练，但互联网到底是什么？"

### 从 Common Crawl 到可用训练数据

```
原始网页 (250B 页 ≈ 1PB)
    ↓
① 文本提取（处理 HTML、数学公式提取很难）
    ↓
② 过滤不良内容（NSFW、有害内容、PII — 每个公司有很长的黑名单）
    ↓
③ 去重（相同 URL、相同段落、重复文本）
    ↓
④ 基于规则的过滤（token 分布异常、词长异常、页面太短/太长）
    ↓
⑤ 基于模型的过滤（训练分类器：参照维基百科引用的网站 → 区分高质量/低质量）
    ↓
⑥ 领域分类与加权（代码↑→推理提升、书籍↑→质量提升、娱乐↓）
    ↓
⑦ 高质收尾（降低学习率，在维基百科等高质数据上过拟合）
```

### 数据规模演化

| 数据集 | 规模 |
|--------|------|
| 早期学术基准 | 150B tokens (~800 GB) |
| LLaMA 2 | 2T tokens |
| LLaMA 3 / 当前顶级模型 | 15T tokens |

### 仍未解决的问题
- 如何高效处理海量数据？
- 如何平衡各领域比例？
- 是否可以合成数据生成？
- **数据是大多数公司的竞争壁垒，所以很少公开细节**

## 1.5 Scaling Laws（规模定律）

### 核心发现

训练损失与计算量 / 数据量 / 参数量之间呈**幂律关系**（对数坐标下为一条直线）：

$$loss \propto \text{compute}^{-\alpha}$$

- 增加计算量 → 可预测的 loss 下降
- 增大模型 → loss 下降
- 增加数据 → loss 下降
- **至今未见平台期！**

### 应用场景

> 假如给你 10,000 块 GPU 训练一个月，你该怎么设计实验？

**老方法**：在大模型上调超参（30 个模型各训 1 天 → 选最好的，最终模型只训了 1 天）
**新方法**：先在不同规模的小模型上调超参 → 拟合 scaling law → 外推最优配置 → 训最终大模型 27 天

### Chinchilla 法则

- **最优训练资源分配**：每 1 个参数需要 20 个 token
- 如果加一个参数，就要多训 20 个 token
- **但这是训练最优**，考虑推理成本时，倾向更小的模型 → 实际比例约 1:150

### 苦味教训（Bitter Lesson）

> Richard Sutton（2019）：计算量持续增长是确定的，唯一重要的是能利用计算量的架构。
> 小而精致的架构改进意义不大，真正重要的是 Systems、Data 和 Scale。

# 第二部分：Post-training（后训练 / 对齐）

## 为什么需要后训练？

> GPT-3（纯预训练模型）回答 "explain the moon landing to a six year old"
> → 它会生成 "explain the theory of gravity to a six year old"
> → 因为互联网上问题是列表形式，不是 QA 对话形式。

后训练的目标：让 LLM **从语言模型变成 AI 助手**

## 2.1 SFT（Supervised Fine-Tuning）

### 做法
- 就是用**相同的语言建模损失**（交叉熵），在高质量的人类问答数据上继续微调
- 本质上是 behavioral cloning（行为克隆）

### 数据来源

**① 人工收集**（昂贵、慢）
- 示例：OpenAssistant 数据集

**② 用 LLM 生成（Synthetic Data）**
- **Alpaca（Stanford）**：用 175 条人工种子数据 → 调用 GPT-3.5 生成了 52,000 条问答 → 微调 LLaMA 7B
- 从此开创了 "用 LLM 训练 LLM"（合成数据）的研究方向

### 关键发现：数据不需要多

**LIMA 论文**：2,000 条 SFT 数据 → 32,000 条 → 性能几乎没有提升

> 原因是：预训练模型已经见过所有类型的"用户"（列表党、问答党、博客党），SFT 只是告诉模型"你该选这个特定用户风格"，不教新知识。

### SFT 的问题

| 问题 | 说明 |
|------|------|
| 受限于人类能力 | 评判好坏的能力 ⩾ 生成最优答案的能力 |
| 可能加剧幻觉 | 人类给了模型一个"合理但没在预训练中出现"的回答 → 模型学会编造 |
| 昂贵 | 让人类写出完整的最优答案，比简单的二选一标注贵得多 |

## 2.2 RLHF（Reinforcement Learning from Human Feedback）

### 核心思想

把人类偏好作为优化目标，而不是克隆人类行为。

**流程**：
1. 对每条指令，由模型生成两个回答
2. 人类标注员选择哪一个更好
3. 用算法训练模型多产生"好的"，少产生"坏的"

### 2.2.1 PPO（Proximal Policy Optimization）

#### 第 1 步：训练奖励模型

- 不是用二进制奖励（好/坏），因为信息太少
- 训练一个**奖励模型** $R(x, y)$，输出连续值，表达"这个回答比那个好多少"
- 使用 **Bradley-Terry 模型**：将两个回答的奖励对数转化为 softmax 概率

#### 第 2 步：强化学习

- 将 LLM 视为 **RL 中的 policy**
- 采样子回答 → 奖励模型评分 → PPO 更新 → 添加正则化避免过优化（reward hacking）
- 正则化原因：奖励模型本身不完美，不能让它无限最大化

#### PPO 的痛点

> "RL 理论上很漂亮，实践中用过的人都知道有多乱。"

- rollout、clipping、out-of-distribution 等大量复杂机制
- 实现极难，超参数多，文档不全

#### ⚠️ 重要后果

PPO 优化后的 LLM 不再以**最大似然**为目标，而是以**生成最优回答**为目标。
- 输出的**似然值（likelihood）不再有概率意义**
- **Perplexity 完全不可靠**（模型没有熵正则化，倾向于生成单一回答）

### 2.2.2 DPO（Direct Preference Optimization）

> 也是 Stanford 提出的（一年前），本质上是 PPO 的简化版

**核心思想**：
- 不用 RL，直接用**最大化偏好回答的概率，最小化不偏好回答的概率**
- 损失函数 ≈ $-\log P(\text{preferred}|\text{input}) + \log P(\text{dispreferred}|\text{input})$

**关键结论**：
- 在某些假设下，DPO 和 PPO 的全局最优解等价
- DPO 只需要做**最大似然**，不需要 rollout、clipping 等复杂 RL 机制
- 性能与 PPO 相当（见实验数据）

**PPO 的独特优势**：
- 可以用奖励模型给**无标记数据**打分（半监督带正则化）
- DPO 只能用已标注的偏好数据

**为什么 OpenAI 先做了 PPO 而不是 DPO？**
- 因为 ChatGPT 团队的成员很多是 RL 专家（包括 PPO 的原创作者）
- 对他们来说，RL 更直观

### RLHF 效果对比

```
Pre-trained → SFT → PPO/DPO
性能稳步提升，PPO 和 DPO 效果相当
在摘要任务上甚至超过人类参照
```

## 2.3 人类反馈数据的问题

### 挑战

**① 慢且贵**
- 人工标注 1,000 条 ≈ $300（Mechanical Turk）
- 标注员面对的是开放性问题，评判很难

**② 难以聚焦关键维度**
- 人类倾向于选**更长的回答**（即使内容更差）
- ChatGPT 回答越来越长 → RLHF 的副作用（length bias）

**③ 标注工具方差大**
- 人类之间的二选一一致性仅约 66%
- 作者五人讨论了 3 小时标注指南后，自我一致性也只有 67-68%

**④ 标注员分布漂移**
- 谁来做标注？代表谁的观点？

**⑤ 标注伦理**
- 标注员工资低，还要面对大量有害数据

### 用 LLM 替代人类标注

- **LLM-as-Judge**：让模型替代人类做偏好评判
- 结果：比人类更便宜（约 50×）、与人类众数的一致性比人类自身还高
- 原因：人类方差大，模型无方差（虽有偏，但方差低）
- 现在开源社区和工业界都已广泛采用 LM + 人类混合标注

## 2.4 后训练模型的评测

### 评测的核心困境

> "如何评估一个回答可以是无限多样的模型？"

**传统方法失效**：
- ❌ **validation loss**：PPO 和 DPO 模型之间不可比
- ❌ **perplexity**：模型不再是最大似然训练的，似然值无意义
- ❌ **传统 ML**：问题太开放，无法自动化

### Chatbot Arena 方案

- 上线两个匿名模型，让随机网友盲测
- 覆盖超数十万用户 → 生成模型排名
- 缺点：用户群可能有偏见（技术爱好者为主）
