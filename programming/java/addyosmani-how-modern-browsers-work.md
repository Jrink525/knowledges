# 现代浏览器是如何工作的

> **来源：** [How modern browsers work](https://x.com/addyosmani/status/2068394292796871019) by Addy Osmani
>
> Addy Osmani 是 Google Chrome 团队的高级工程师，本文全面解析了以 Chromium 为代表的现代浏览器内部机制。

---

## 概述

Web 开发者经常把浏览器视为一个黑盒——它神奇地将 HTML、CSS 和 JavaScript 转化为交互式 Web 应用。但事实上，现代浏览器（Chrome/Chromium、Firefox/Gecko、Safari/WebKit）是极其复杂的软件。它协调网络通信、解析和执行代码、通过 GPU 加速渲染图形，并在沙盒进程中隔离内容以保证安全。

本文将深入探讨现代浏览器的工作原理——重点聚焦 Chromium 的架构和内部机制，同时指出其他引擎的差异。我们将涵盖：网络栈、解析流水线、Blink 渲染过程、V8 JavaScript 引擎、模块加载、多进程架构、安全沙盒和开发者工具。

## 一、网络与资源加载

每次页面加载都始于浏览器的网络栈从 Web 获取资源。输入 URL 或点击链接时，浏览器进程中的 UI 线程发起导航请求。

> **浏览器进程**是主要的控制进程，管理所有其他进程和浏览器用户界面。特定网页标签之外的一切都由浏览器进程控制。

### 导航步骤

1. **URL 解析与安全检查** — 解析 URL 确定协议（http/https 等）和目标域名。决定输入是搜索查询还是 URL。检查安全黑名单以避免钓鱼网站。

2. **DNS 查询** — 将域名解析为 IP 地址（除非已缓存）。现代浏览器可能使用系统 DNS 或配置的 DNS over HTTPS（DoH）。

3. **建立连接** — 如果没有打开的连接，浏览器建立一个新的。对于 HTTPS URL，包括 TLS 握手以安全交换密钥和验证证书。

4. **发送 HTTP 请求** — 发送 HTTP GET 请求。现代浏览器默认使用 HTTP/2 或 HTTP/3（如果服务器支持），允许在一个连接上多路复用多个资源请求，避免了 HTTP/1.1 的每个主机约 6 个并行连接的限制。

5. **接收响应** — 服务器返回 HTTP 状态码与报头，后跟响应体。浏览器可能需要进行 MIME 嗅探（如果 Content-Type 缺失或错误）。同时进行安全检查，比如 CORB（跨源读取阻塞）和安全浏览（Safe Browsing）。

6. **重定向与后续** — 如果响应是 HTTP 重定向（如 301/302），网络代码会跟随重定向并重复请求。只有获得最终的真实内容响应后，浏览器才会处理该内容。

所有这些步骤发生在网络栈中，Chromium 的网络栈运行在专用的网络服务（Network Service）中，通常是独立的进程。**渲染器（renderer）不直接访问网络**——它请求浏览器进程代为获取，这是一个安全设计。

### 推测加载与资源优化

- **预加载扫描器（Preload Scanner）** — Chromium 在主解析器之前对 HTML 标记进行分词，即使主解析器被 CSS 或同步 JavaScript 阻塞，预加载扫描器仍会继续扫描原始标记以发现图片、脚本和样式表等资源，并发地启动加载。这是现代浏览器性能的基础机制，自动运行，无需开发者干预。需要注意：预加载扫描器**无法发现通过 JavaScript 动态注入的资源**。

- **Early Hints（HTTP 103）** — 允许服务器在生成主响应时发送资源提示（使用 HTTP 103 状态码），在服务器思考时间内发送 preconnect 和 preload 提示，可将 Largest Contentful Paint 提升数百毫秒。

- **Speculation Rules API** — 新的 Web 标准，允许基于用户交互模式动态预取和预渲染 URL。与传统 link prefetch 不同，此 API 可以**预渲染完整页面**（包括 JavaScript 执行），实现近乎即时的加载。

- **HTTP/2 和 HTTP/3** — 大多数基于 Chromium 的浏览器和 Firefox 完全支持 HTTP/2，HTTP/3（基于 QUIC）也得到广泛支持。从开发者角度看，这可能意味着不再需要雪碧图或域名分片（domain sharding）技巧——浏览器可以在一个连接上高效并行获取许多小文件。

- **资源优先级** — 浏览器对某些资源赋予更高优先级。HTML 和 CSS 通常是高优先级（因为它们阻塞渲染），脚本可能是中优先级，图片可能更低。开发者可以使用 `rel=preload` 和 Fetch Priority API 影响资源优先级。

网络阶段结束后，浏览器已获得页面的初始 HTML。此时浏览器进程选择一个渲染器进程来处理内容，提交导航（commit navigation），地址栏更新，安全指示器显示。然后渲染器进程接管：解析 HTML、加载子资源、执行脚本和绘制页面。

## 二、解析 HTML、CSS 和 JavaScript

### HTML 解析与 DOM 构建

当渲染器进程收到 HTML 内容时，其主线程开始按照 HTML 规范进行解析。解析的结果是 **DOM（文档对象模型）**——代表页面结构的对象树。解析是**增量式**的，可以与网络读取交错进行（浏览器流式解析 HTML，DOM 甚至在完整 HTML 文件下载完成前就能开始构建）。

HTML 标准定义的解析是**容错**的：即使标记格式错误，解析器也会隐式修复或调整 DOM 树，使其有效。

一个重要方面：HTML 解析器在解析过程中会遇到需要获取的资源——遇到 `<link rel="stylesheet">` 时会请求 CSS，遇到 `<img>` 会触发图片请求。这些与解析并行发生，但有一个**重大例外**：脚本。

### \<script\> 标签处理

默认情况下，HTML 解析器遇到 `<script>` 标签时会暂停解析，必须执行脚本后才能继续。这是因为脚本可以使用 `document.write()` 或其他 DOM 操作改变页面结构。解析器将脚本交给 JavaScript 引擎执行，只有脚本完成（以及任何 DOM 变更被应用）后，HTML 解析才能恢复。

开发者可以使用属性改变这一行为：
- **`async`** — 脚本文件并行获取，准备就绪后立即执行，不暂停 HTML 解析
- **`defer`** — 脚本文件并行获取，但推迟到 HTML 解析完成后执行（按原始顺序）
- **ES6 模块（`<script type="module">`）** — 自动 defer，还可使用 `import` 语句

### CSS 解析与 CSSOM

CSS 文本必须解析为浏览器可以使用的结构——**CSSOM（CSS 对象模型）**。CSSOM 本质上代表了应用于文档的所有样式（规则、选择器、属性）。

浏览器将 DOM 和 CSSOM 结合起来，为每个 DOM 节点确定哪些 CSS 规则生效以及最终的**计算样式**（computed styles）。即使没有任何作者 CSS，每个元素也有默认的浏览器样式（用户代理样式表）。

**渲染阻塞行为**：虽然 HTML 解析可以在没有完全加载 CSS 的情况下进行，但存在渲染阻塞关系——**浏览器通常等待 CSS 加载后才进行首次渲染**，以避免未样式内容闪烁（FOUC）。

## 三、样式与布局

此时浏览器已知晓 DOM 结构和每个元素的计算样式。**布局（layout，也称为 reflow）**计算每个元素的几何信息——大小和位置。

### 布局树构建

浏览器遍历 DOM 树并生成**布局树**（layout tree，也称为渲染树或帧树）。布局树在结构上类似 DOM 树，但它省略了非可视化元素（如 `<script>` 或 `<meta>` 标签不产生盒子），并根据需要拆分元素（例如跨多行的同一 HTML 元素可能对应多个布局盒子）。

布局树的关键属性：
- `display: none` 的元素完全从布局树中移除
- `visibility: hidden` 的元素会获得布局盒子（占据空间），只是不绘制
- `::before` 和 `::after` 等伪元素会包含在布局树中

### 布局计算

布局通常是递归过程。从根元素（`<html>`）开始，浏览器计算视口大小，然后在其中布局子元素，以此类推。许多元素的大小取决于子元素或父元素。

布局是一个计算昂贵的操作。JavaScript 改变元素大小或添加内容会强制重新布局。开发者常听到避免**布局抖动（layout thrashing）**的建议——紧跟在 DOM 修改后读取布局信息，可能强制同步重新计算。

### 阶段总结

从 HTML 和 CSS 构建出来的结构：
1. **DOM 树** — 结构和内容
2. **CSSOM** — 解析后的 CSS 规则
3. **计算样式** — 将 CSS 规则匹配到每个 DOM 节点的结果
4. **布局树** — 过滤到可视化元素并包含几何信息的 DOM 树

## 四、绘制、合成与 GPU 渲染

绘制是将结构化布局信息转化为屏幕像素的过程。

### 绘制/光栅化

在渲染器主线程上，布局后 Chrome 生成**绘制记录（paint records 或 display list）**——按正确 z-index 顺序列出的绘制操作列表："在 (x,y) 以 W×H 绘制蓝色矩形，在 (x2,y2) 以字体 XYZ 绘制文本'Hello'，然后绘制图片..."等。

### 分层与合成

**合成（compositing）**是优化手段：页面被拆分为多个可以独立处理的层。带有 CSS transform 或动画的定位元素可能会获得自己的层。层就像独立的"草稿画布"——浏览器可以分别光栅化每层，然后在 GPU 上合成在一起。

在 Chromium 的流水线中，主线程将构建层树后的工作交给**合成线程（compositor thread）**。合成线程与主线程分离，即使主 JS 线程忙碌也能继续工作——这对于流畅滚动和动画非常重要。

合成线程将层划分为更小的**图块（tiles）**，分发给光栅化工作线程，使用 Skia 图形库（CPU 或 GPU）生成位图，然后上传到 GPU 内存。最后由 GPU 进程将这些纹理合成到屏幕。

**Chrome 的渲染流水线**：
```
DOM → 样式 → 布局 → 绘制（记录显示项）→ 分层 → 光栅化（图块）→ 合成（GPU）
```

**合成优化的关键**：只改变 transform 或 opacity 的动画不需要重新布局或绘制，可以在合成线程上以 60 FPS 顺畅运行——即使主线程繁忙。

### GPU 进程

Chromium 中有一个独立的 GPU 进程，负责与图形硬件交互。它接收来自所有渲染器合成器和浏览器 UI 的绘制命令，转换为实际的 GPU API 调用。隔离在独立进程中的好处：崩溃的 GPU 驱动不会拖垮整个浏览器——只有 GPU 进程崩溃，可以重新启动。

## 五、深入了解 JavaScript 引擎（V8）

V8 是 Chromium 的 JavaScript 引擎，负责执行 JavaScript（和 WebAssembly）。了解 V8 的工作原理有助于编写高性能 JS。

### 现代 V8 解析与编译流水线

- **后台编译** — 从 Chrome 66 开始，V8 在后台线程编译 JavaScript 源代码，将主线程上的编译时间减少了 5% 到 20%
- **流式解析** — 从 Chrome 41 开始，通过网络流式传输文件时可以并行解析
- 目前，顶层脚本代码和立即调用函数表达式（IIFE）在后台线程编译，内层函数仍按需惰性编译

### 解析与字节码

1. V8 首先解析 JavaScript 源代码，生成抽象语法树（AST）
2. 使用名为 **Ignition** 的字节码解释器将 JS 编译为紧凑的字节码格式
3. 解释器执行字节码的同时收集描述信息（类型反馈）

### JIT 编译层级

V8 目前有**四个执行层级**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 1 | **Ignition** | 字节码解释器，快速启动 |
| 2 | **Sparkplug** | 基线 JIT（2021 年引入），快速生成机器码，无深度优化 |
| 3 | **Maglev** | 中端优化编译器（2023 年引入），比 Sparkplug 慢约 20 倍，但比 TurboFan 快 10-100 倍 |
| 4 | **TurboFan** | 最强的优化编译器，应用内联函数、边界检查消除等高级优化 |

这类似于 Java HotSpot VM 的多级 JIT（C1 和 C2）。引擎根据执行配置文件动态决定优化哪些函数以及何时优化。

### 内存管理：垃圾回收

V8 使用名为 **Orinoco** 的分代、增量、并发垃圾收集器：

- **分代收集** — 新对象在新生代（nursery）分配，频繁但快速回收；存活足够多次的对象升级到老生代
- **标记-清除/压缩** — 老生代使用标记-清除收集器，但 Orinoco 将大部分标记工作放在后台线程，最小化暂停时间
- **增量 GC** — 将 GC 工作分散到小片段中，利用空闲时间

效果：现代 V8 已大幅减少了 GC 暂停时间，在大型应用中 GC 也几乎不可察觉。

### V8 与 Web API

V8 只覆盖核心 JavaScript 语言和运行时。浏览器 API（如 DOM 方法、`fetch()`、`alert()`）不由 V8 提供，而是通过绑定暴露给 JS。例如调用 `document.querySelector`，底层进入 C++ DOM 实现的绑定层。

## 六、模块加载与 Import Maps

ES6 模块引入了与经典 `<script>` 标签不同的加载和执行模型。

### 静态模块导入

当浏览器遇到 `<script type="module" src="main.js">` 时：
1. 获取 main.js，解析为 ES 模块
2. 发现 `import` 语句，构建模块依赖图
3. 递归获取所有导入模块，解析其导入
4. 整个模块图全部获取和解析后，按依赖顺序执行（深度优先、后序遍历）

模块脚本本质上是 **defer** 的——浏览器在所有依赖就绪前不执行模块代码。

### 动态 import()

`import(moduleSpecifier)` 作为表达式允许在运行时按需加载模块。例如 `const module = await import('./analytics.js')`。这在非模块脚本中也可以使用。

### Import Maps

Import Map 是一个 JSON 配置，告诉浏览器如何将模块标识符解析为真实 URL。通过 `<script type="importmap">` 标签提供。例如将 `"react"` 映射到 CDN 的完整 URL。这使得裸标识符在 Web 上工作的方式，对于无构建步骤的开发非常有用。

截至 2023 年，所有主流浏览器都已支持 Import Maps。

## 七、浏览器多进程架构

现代浏览器（Chrome、Firefox、Safari）都采用多进程架构，以实现**稳定性、安全性和性能隔离**。

### Chromium 的进程模型

- **浏览器进程（Browser Process）** — 中心进程，负责 UI（地址栏、书签、菜单）和协调资源加载与导航。是生成其他进程的父进程。
- **渲染器进程（Renderer Process）** — 每个标签（有时每个站点）一个。运行 Blink 渲染引擎和 V8 JS 引擎。沙盒化，不能直接访问文件系统或网络。
- **GPU 进程** — 与 GPU 通信。独立进程确保 GPU 崩溃不会拖垮渲染器。
- **网络进程（Network Process）** — 处理网络请求、DNS 等，可以单独沙盒化。
- **工具进程（Utility Processes）** — 音频播放、图片解码等服务。
- **扩展进程（Extension Processes）** — Chrome 扩展运行在独立进程中。

### 多进程的优势

1. **稳定性** — 一个标签页崩溃不会拖垮整个浏览器
2. **安全（沙盒）** — 网页内容在受限进程中运行，即使渲染引擎存在漏洞，攻击者也被困在沙盒中
3. **性能隔离** — 一个标签页的重负荷不会阻塞其他标签页
4. **内存分段** — 关闭标签页时，OS 可以高效回收该进程的所有内存

### 站点隔离（Site Isolation）

最初 Chrome 是一个标签一个进程。Spectre 漏洞（2018）后，Chrome 默认启用**站点隔离**——不同站点运行在不同渲染器进程中。跨源 iframe 也被强制放入不同进程（称为 OOPIF——Out-of-Process iframe）。

这意味着一个标签页可能运行 N 个进程（主文档一个，每个跨源子文档各一个）。站点隔离使内存使用增加约 10-13%，但提供了关键的安全保障。

### Firefox 与 Safari

- **Firefox** — 默认 8 个内容进程共享所有标签（不一定是每标签一进程）。通过 Project Fission 实现站点隔离。
- **Safari** — WebKit2 架构，每个标签页的内容在独立的 WebContent 进程中，中央 UI 进程控制。

### 进程间通信（IPC）

浏览器使用 IPC 机制（Chrome 有自己的 Mojo 库）实现进程间通信。例如网络响应到达网络进程后，通过浏览器进程协调交付给正确的渲染器进程。

## 八、站点隔离与沙盒

### 站点隔离

我们已经涵盖了基本概念。站点隔离构建在多进程基础之上，确保即使恶意代码在浏览器中运行，也无法轻易从其他站点窃取数据或访问你的系统。

从开发者角度看，站点隔离基本是透明的。一个影响是：嵌入的 iframe 与父页面之间的通信可能跨进程边界，`postMessage` 等操作底层通过 IPC 实现，但浏览器使其无缝。

### 沙盒

每个渲染器进程（和其他辅助进程）在**受限权限的沙盒**中运行。例如：
- **Windows** — 使用 job 对象降低权限，渲染器无法调用大多数访问系统的 Win32 API
- **Linux** — 使用命名空间和 seccomp 过滤器限制系统调用

渲染器基本上只能计算和渲染内容。如果试图打开文件、摄像头或麦克风，会被阻止——必须通过浏览器进程的正当渠道请求用户权限。

这种分层方法（站点隔离 + 沙盒）是当前浏览器安全的最先进水平。攻击者通常需要多个链式漏洞（一个突破渲染器，一个逃离沙盒）才能造成严重损害。

### 跨站点预取限制

出于隐私原因，跨站点预取目前仅在用户未为目标站点设置 cookie 时生效，防止站点通过预取页面跟踪用户活动。

## 九、Chromium、Gecko 与 WebKit 对比

### CSS/样式系统差异

| 引擎 | 特性 |
|------|------|
| **Blink（Chromium）** | 单线程 C++ 样式引擎，有增量样式失效优化 |
| **Gecko（Firefox）** | **Stylo**：用 Rust 编写的新 CSS 引擎，**多线程**，可并行计算不同 DOM 子树的样式 |
| **WebKit（Safari）** | 单线程（与 Blink 同源），但使用 CSS 选择器的字节码 JIT 编译器加速匹配 |

### 布局与图形

- **Firefox（WebRender）** — 用 Rust 开发，**将显示列表直接在 GPU 上渲染**，不先生成位图。类似游戏引擎每帧重绘场景，对于复杂页面可能更快
- **Chrome（Skia）** — 光栅化主要在 CPU 上进行（大多数内容），然后作为位图发送到 GPU
- **Safari（CoreAnimation）** — 使用 CALayer 技术，早期就实现了硬件加速合成

### JavaScript 引擎

| 引擎 | 浏览器 | 层级 |
|------|--------|------|
| **V8** | Chrome | Ignition → Sparkplug → Maglev → TurboFan |
| **SpiderMonkey** | Firefox | Baseline Interpreter → Baseline JIT → WarpMonkey |
| **JavaScriptCore** | Safari | LLInt → Baseline → DFG → FTL（使用 LLVM） |

### 多进程模型差异

- **Chrome** — 每标签通常独立，站点隔离按源级别隔离，进程可能达数十个
- **Firefox** — 默认 8 个内容进程处理所有标签，内存使用更保守，但一个内容进程崩溃可能影响多个标签
- **Safari** — 每标签独立进程，iOS 上 WKWebView 确保每个 webview 隔离

## 十、总结与进一步建议

### 给开发者的关键提示

1. **优化网络使用** — 更少的往返次数和更小的文件 = 更快的首次渲染。使用资源提示和高效缓存

2. **结构化 HTML/CSS** — 结构良好的 DOM 和简洁的 CSS（避免过深的树或过于复杂的选择器）有助于解析和样式系统

3. **批处理 DOM 更新** — 避免重复的样式/布局抖动。使用 DevTools Performance 面板发现脚本导致多次布局或绘制的情况

4. **使用合成友好的 CSS 做动画** — 对 transform 或 opacity 的动画保持在主线程之外，由合成器处理，获得流畅动画

5. **关注 JS 执行** — JS 引擎虽然超快，但长任务会阻塞主线程。分解长操作，考虑使用 Web Workers

6. **拥抱安全特性** — 使用 iframe sandbox 或 `rel=noopener`，因为浏览器已经在底层做隔离

7. **DevTools 是你的朋友** — Performance 和 Network 面板是了解浏览器行为的金矿
